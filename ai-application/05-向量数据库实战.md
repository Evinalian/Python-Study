## 1. Embedding 模型

### 1.1 什么是 Embedding —— 让机器"理解"语义

在计算机的世界里，一切最终都是数字。文本也不例外——但问题是，如何用数字来表示"意思"？如果只是简单地把每个汉字映射为一个整数（比如"猫"=1，"汽车"=2），那么"猫"和"小猫"之间的关系与"猫"和"汽车"之间的关系在数字上没有任何区别——都是差1。这显然无法反映语义上的亲疏远近。

Embedding 技术的核心思想是：**用高维空间中的向量来表示语义**。一条文本被转换为一组浮点数（比如1536个float32），这组数字不是随机的——它分布在这样一个空间里：语义相近的文本，向量也"指向"相近的方向；语义无关的文本，向量则分散在不同区域。

举个例子，如果我们把"猫"、"小猫"、"汽车"分别编码为向量：
- "猫"的向量和"小猫"的向量在空间中距离很近 —— 因为它们在语义上都指向猫科动物
- "猫"的向量和"汽车"的向量距离很远 —— 因为在语义上它们毫无关联

这种将非结构化的文本映射到结构化的向量空间的能力，是整个现代语义搜索、RAG、推荐系统的基础。有了它，"搜索包含'猫'的文档"就升级为"搜索与'猫'语义相关的内容"——即使文档里根本没有出现"猫"这个字，只要内容谈到"宠物"、"喵星人"、"撸猫"等，也能被检索到。这就是语义搜索比关键词搜索强大的根本原因。

Embedding 模型本质上是一个训练好的神经网络：输入一段文本，输出一个固定维度的向量。训练过程通常使用对比学习（contrastive learning）：模型被训练成让相似文本的向量靠近，不相干文本的向量远离。经过海量数据训练后，模型就学会了把文本"翻译"到语义空间中。

```
"猫"      to [0.12, -0.34, 0.56, ...]  (1536维)
"小猫"    to [0.13, -0.32, 0.54, ...]  (与"猫"的向量很接近)
"汽车"    to [-0.45, 0.78, -0.23, ...]  (与"猫"的向量很远)
```

理解了 Embedding 的本质之后，我们来看具体怎么使用它。主流选择分两类：调用 OpenAI 的 API（简单但收费），或者使用开源的本地模型（免费但需要计算资源）。

### 1.2 OpenAI Embedding 模型

OpenAI 提供了三个 Embedding 模型。其中 `text-embedding-3-small` 和 `text-embedding-3-large` 是2024年发布的新一代模型，它们不仅效果比旧的 `text-embedding-ada-002` 更好，价格也更便宜。特别是 `text-embedding-3-small`，每100万token仅需 $0.02，在处理大批量文档时成本极低。

一个容易被忽略但很实用的特性是 `dimensions` 参数：这两个新模型支持在调用时指定输出维度。比如 `text-embedding-3-large` 默认输出3072维的向量，但你可以通过 `dimensions=512` 将其压缩到512维。虽然精度会略有下降（通常 < 5%），但存储空间和检索速度会大幅改善。当你需要存储数千万条向量时，从3072维降到512维意味着存储空间减少83%。

```python
"""
OpenAI Embedding API 使用指南。

模型对比:
- text-embedding-3-small: 1536维, $0.02/1M tokens, 性能好, 性价比最高
- text-embedding-3-large: 3072维, $0.13/1M tokens, 效果最好
- text-embedding-ada-002: 1536维, $0.10/1M tokens, 旧版(不推荐新项目)
"""
import os
import numpy as np
import time
from openai import OpenAI

client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)


class OpenAIEmbeddings:
    """
    封装 OpenAI Embedding API。
    """

    def __init__(self, model: str = "text-embedding-3-small"):
        self.model = model
        self.dimensions = {
            "text-embedding-3-small": 1536,
            "text-embedding-3-large": 3072,
            "text-embedding-ada-002": 1536,
        }

    def embed(self, texts: list[str]) -> list[list[float]]:
        """
        将文本列表转换为向量列表。

        参数:
            texts: 文本列表，每批最多 2048 条

        返回:
            [[0.1, -0.2, ...], [0.3, 0.4, ...]] 维度 = len(texts) x dim
        """
        # OpenAI 建议每批不超过 2048 条
        batch_size = 100
        all_embeddings = []

        for i in range(0, len(texts), batch_size):
            batch = texts[i : i + batch_size]

            response = client.embeddings.create(
                model=self.model,
                input=batch,
                # text-embedding-3-small/large 支持指定输出维度
                # dimensions=512,  # 可选: 降低维度以减少存储和加速检索
            )

            embeddings = [item.embedding for item in response.data]
            all_embeddings.extend(embeddings)

            print(f"  批次 {i // batch_size + 1}: {len(batch)} 条文本完成")

        return all_embeddings

    def embed_single(self, text: str) -> list[float]:
        """将单条文本转换为向量"""
        return self.embed([text])[0]

    def similarity(self, a: list[float], b: list[float]) -> float:
        """计算两个向量的余弦相似度"""
        a_arr = np.array(a)
        b_arr = np.array(b)
        return np.dot(a_arr, b_arr) / (np.linalg.norm(a_arr) * np.linalg.norm(b_arr))


def demo_openai_embeddings():
    """演示 OpenAI Embedding 的基本用法"""

    model = OpenAIEmbeddings("text-embedding-3-small")

    # 对比语义相近和不相近的文本
    texts = [
        "Python是一种编程语言",
        "Java也是一种编程语言",
        "今天天气很好适合出去散步",
        "编程语言包括Python、Java、C++等",
    ]

    print("=== OpenAI Embedding 演示 ===\n")
    print(f"模型: {model.model}, 维度: {model.dimensions[model.model]}")

    t0 = time.time()
    embeddings = model.embed(texts)
    elapsed = time.time() - t0

    print(f"耗时: {elapsed:.2f}s")

    # 计算两两相似度
    print("\n--- 相似度矩阵 ---")
    for i, text_i in enumerate(texts):
        for j, text_j in enumerate(texts):
            if i < j:
                sim = model.similarity(embeddings[i], embeddings[j])
                print(f"  \"{text_i[:20]}...\" vs \"{text_j[:20]}...\": {sim:.3f}")

    # 预期结果:
    # "Python是..." vs "Java也是..." : 高相似度（都是编程语言）
    # "Python是..." vs "今天天气..." : 低相似度（话题无关）


if __name__ == "__main__":
    # demo_openai_embeddings()
    print("(取消注释以运行 OpenAI Embedding 演示)")
```

### 1.3 开源 Embedding 模型 —— 免费且可控的替代方案

使用 OpenAI API 虽然方便，但存在三个痛点：第一，每次调用都要花钱，当你有上千万条文档需要向量化时成本不菲；第二，数据必须离开你的服务器，这在金融、医疗等合规场景下是不可接受的；第三，你无法控制模型的行为——如果 OpenAI 升级了模型，你索引的所有向量都需要重新计算。

开源 Embedding 模型解决了这些问题。通过 `sentence-transformers` 库，你可以在本地运行这些模型。当前社区公认的标杆是 BGE-M3（由北京智源研究院 BAAI 开发），它在多语言场景下（特别是中英混合）表现优异，在 MTEB 评测榜单上名列前茅。

选择开源模型时，有几个关键考量：

**模型尺寸与资源的关系**：更大的模型通常效果更好，但对显存的要求也更高。`bge-small`（512维）在 CPU 上也能跑，`bge-m3`（1024维）则需要 GPU 才能有可接受的速度。如果你只有一台普通的开发服务器，`bge-base`（768维）往往是最佳平衡点。

**BGE 模型的特殊用法**：这是一个很重要的细节。BGE 系列模型对"查询"和"文档"使用不同的编码策略——查询文本在编码前需要加上一个特定的前缀（instruction prefix）。这个设计让模型能够区分"我正在找一个问题的答案"和"我是一段文档内容"两种不同的语义角色。如果你忘记加这个前缀，检索精度会明显下降。这就是为什么下面的代码中 `embed_query` 和 `embed` 是分开的。

**维度与性能的权衡**：向量的维度越高，表达能力越强，但存储和计算成本也越大。384维的向量占约1.5KB，1536维则占约6KB。如果索引1亿条向量，仅向量数据就从150GB飙升到600GB。在实际项目中，这是一个必须认真考虑的工程决策。

```python
"""
开源 Embedding 模型对比和使用。

模型选择指南:
- BGE-M3 (BAAI): 多语言(中英), 1024维, MTEB排名靠前, 多语言场景首选
- M3E (moka-ai): 中文优化, 768维, 轻量
- text2vec-large-chinese: 中文, 1024维, 效果好
- all-MiniLM-L6-v2: 英文, 384维, 非常轻量

安装: pip install sentence-transformers
"""
import time
import numpy as np


class SentenceTransformerEmbeddings:
    """
    基于 sentence-transformers 的开源 Embedding 模型封装。

    优势:
    - 本地运行，无需 API 调用
    - 无调用次数限制
    - 数据不出本地（隐私合规）
    - 可以 fine-tune

    劣势:
    - 需要 GPU 才能高效运行
    - 效果可能不如商业 API
    - 占用本地内存
    """

    def __init__(self, model_name: str = "BAAI/bge-small-zh-v1.5"):
        """
        初始化模型。

        推荐模型:
        - "BAAI/bge-small-zh-v1.5": 512维, 中文, 轻量
        - "BAAI/bge-base-zh-v1.5": 768维, 中文, 均衡
        - "BAAI/bge-large-zh-v1.5": 1024维, 中文, 效果好
        - "BAAI/bge-m3": 1024维, 多语言
        - "moka-ai/m3e-base": 768维, 中文
        - "shibing624/text2vec-base-chinese": 768维, 中文
        """
        try:
            from sentence_transformers import SentenceTransformer
        except ImportError:
            raise ImportError("请安装: pip install sentence-transformers")

        print(f"加载模型: {model_name} ...")
        t0 = time.time()
        self.model = SentenceTransformer(model_name)
        self.model_name = model_name
        print(f"加载完成 ({time.time() - t0:.1f}s)")

    def embed(self, texts: list[str], batch_size: int = 32) -> list[list[float]]:
        """
        将文本列表转换为向量列表。

        参数:
            texts: 文本列表
            batch_size: 批处理大小（GPU 内存允许下越大越快）

        返回:
            向量列表
        """
        # BGE 模型需要在 query 前加前缀
        # 注意: 这里是将所有文本作为文档编码，所以不需要 query prefix
        embeddings = self.model.encode(
            texts,
            batch_size=batch_size,
            show_progress_bar=True,
            normalize_embeddings=True,  # L2 归一化，方便直接用点积
        )
        return embeddings.tolist()

    def embed_query(self, query: str) -> list[float]:
        """
        对查询文本进行编码（BGE 模型需要加 query prefix）。

        BGE 模型的特殊要求:
        - 文档: 直接编码
        - 查询: 编码前加 "为这个句子生成表示以用于检索相关文章：" 前缀
        """
        try:
            # BGE 模型的 query 指令
            query_with_instruction = f"为这个句子生成表示以用于检索相关文章：{query}"
            embedding = self.model.encode(
                [query_with_instruction],
                normalize_embeddings=True,
            )
            return embedding[0].tolist()
        except Exception:
            # 非 BGE 模型可能不需要 prefix
            return self.embed([query])[0]

    @property
    def dimension(self) -> int:
        """返回向量维度"""
        return self.model.get_sentence_embedding_dimension()


# ============================================================
# 模型选择决策树
# ============================================================
def recommend_embedding_model(
    language: str = "chinese",
    budget: str = "free",
    scale: str = "small",
) -> str:
    """
    根据需求推荐 Embedding 模型。

    参数:
        language: 主要语言 "chinese" / "english" / "multilingual"
        budget: 预算 "free" / "api"
        scale: 规模 "small" / "medium" / "large"
    """
    recommendations = {
        ("chinese", "free", "small"): "BAAI/bge-small-zh-v1.5",
        ("chinese", "free", "medium"): "BAAI/bge-base-zh-v1.5",
        ("chinese", "free", "large"): "BAAI/bge-large-zh-v1.5",
        ("english", "free", "small"): "sentence-transformers/all-MiniLM-L6-v2",
        ("english", "free", "medium"): "BAAI/bge-base-en-v1.5",
        ("multilingual", "free", "medium"): "BAAI/bge-m3",
        ("chinese", "api", "small"): "OpenAI text-embedding-3-small",
        ("chinese", "api", "large"): "OpenAI text-embedding-3-large",
        ("multilingual", "api", "medium"): "OpenAI text-embedding-3-small",
    }

    return recommendations.get(
        (language, budget, scale),
        "BAAI/bge-small-zh-v1.5",  # 默认推荐
    )


if __name__ == "__main__":
    print("=== 开源 Embedding 模型推荐 ===\n")
    print(f"中文/免费/小规模 to {recommend_embedding_model('chinese', 'free', 'small')}")
    print(f"多语言/免费/中等 to {recommend_embedding_model('multilingual', 'free', 'medium')}")
    print(f"中文/API/大规模 to {recommend_embedding_model('chinese', 'api', 'large')}")

    print("\n=== MTEB 排行榜参考 (2024) ===")
    print("  https://huggingface.co/spaces/mteb/leaderboard")
    print("  中文推荐: BGE-M3 > text2vec-large > M3E")
    print("  英文推荐: BGE-en > E5-mistral > GTE")
```

### 1.4 Embedding 价格和维度对比

理解 Embedding 模型的参数不仅仅是"看数字"，更重要的是理解这些数字背后的工程含义。下面这张对比表看似简单，但每个列都对应一个实际决策场景：

- **维度**直接决定了存储成本。1536维的向量使用float32存储，每个向量约占6KB。如果索引100万条文档（这在企业知识库场景中很常见），仅向量数据就需要约6GB存储。如果每个文档被切为3-5个chunk，存储量还要乘以3-5倍。
- **Max Tokens**限制了单次编码的文本长度。8191 tokens大约对应中文4000字左右。如果一篇长文档有20000字，你必须先切分再分别编码，而不是一股脑塞进模型。
- **多语言支持**不是"有"或"没有"的简单二元属性。同一模型对英文的编码质量通常优于中文，因为训练数据中英文占比更高。如果你主要处理中文文档，优先选择明确针对中文优化的模型（如BGE系列和M3E）。
- **价格**方面，开源模型的一次性成本是GPU服务器（如果已经有的话边际成本为零），API模型则是按用量付费。粗略估算：100万条文本用text-embedding-3-small编码约花费$2-5，同样的量用开源模型在GPU上跑则是电费（几乎可以忽略）。

## 2. 向量相似度 —— 如何衡量"接近"

### 2.1 三种相似度计算方法及其直觉

有了向量之后，下一个问题是：怎么判断两个向量是否"接近"？这就引出了三种最常用的相似度（或距离）计算方法。它们各有优劣，适用场景不同。理解它们之间的区别，是做向量检索的基本功。

**余弦相似度（Cosine Similarity）**：测量的是方向的一致性。想象从原点出发的两支箭，余弦相似度告诉你它们指向同一个方向的程度，而不关心箭的长度。这就是为什么余弦相似度特别适合文本语义比较——"人工智能改变世界"和"人工智能正在改变世界"这两句话长度不同，但它们"说的事情"是相同的。公式为 `cos(theta) = (A dot B) / (||A|| x ||B||)`。值域是[-1, 1]，越接近1越相似。

**欧氏距离（Euclidean Distance）**：测量的是空间中两点之间的直线距离。`d = sqrt(Sigma(Ai - Bi)^2)`。它既关心方向，也关心向量的模长（即"箭有多长"）。因此它更适合那些向量模长本身有物理含义的场景，比如用户画像中"消费金额"这个维度的大小直接反映了消费能力，不能忽略。值域是[0, +infinity)，越小越相似。通常我们会通过 `1 / (1 + distance)` 将它转为 (0, 1] 的相似度。

**内积（Dot Product）**：`A dot B = Sigma(Ai x Bi)`。它可以视为余弦相似度的未归一化版本。当向量已经做了L2归一化（即所有向量的模长都是1），内积就等于余弦相似度。在神经网络训练中，内积常用于倒数第二层的输出（称为logits），因为它天然适合矩阵运算加速。

**如何选择？** 一个简单的决策法则：如果你的向量已经做了归一化（大多数Embedding模型输出时可以选择 `normalize_embeddings=True`），那么余弦相似度和内积是等价的，你只需根据向量数据库支持哪种度量方式来选择。如果向量未归一化，一般来说：文本语义比较用余弦相似度（消除长度偏差）；图像特征比较用欧氏距离（像素值的大小有意义）；推荐系统中用户和物品的匹配用内积（方便大规模矩阵运算）。

```python
"""
三种向量相似度计算方法的详解和代码实现。

1. 余弦相似度 (Cosine Similarity): 测量方向相似度
2. 欧氏距离 (Euclidean Distance): 测量绝对距离
3. 内积 (Dot Product): 测量方向 + 长度

选择指南:
- 余弦相似度: 最常用，适合文本语义相似（忽略文本长度差异）
- 欧氏距离: 适合数值特征（如用户画像、商品属性）
- 内积: 当向量做了 L2 归一化后，等价于余弦相似度
"""
import numpy as np
import math


class VectorSimilarity:
    """向量相似度计算工具集"""

    @staticmethod
    def cosine(a: list[float], b: list[float]) -> float:
        """
        余弦相似度。

        公式: cos(theta) = (A dot B) / (||A|| x ||B||)

        范围: [-1, 1]，越接近 1 越相似

        使用 Python 循环（便于理解）:
        dot = sum(ai * bi for ai, bi in zip(a, b))
        norm_a = math.sqrt(sum(ai * ai for ai in a))
        norm_b = math.sqrt(sum(bi * bi for bi in b))
        return dot / (norm_a * norm_b) if norm_a and norm_b else 0.0
        """
        a_arr = np.array(a)
        b_arr = np.array(b)
        dot = np.dot(a_arr, b_arr)
        norm_a = np.linalg.norm(a_arr)
        norm_b = np.linalg.norm(b_arr)

        if norm_a == 0 or norm_b == 0:
            return 0.0
        return float(dot / (norm_a * norm_b))

    @staticmethod
    def euclidean(a: list[float], b: list[float]) -> float:
        """
        欧氏距离。

        公式: d = sqrt(Sigma(Ai - Bi)^2)

        范围: [0, +infinity)，越接近 0 越相似

        注意: 这是距离指标，值越小说明越相似。
        如果要转成相似度: similarity = 1 / (1 + distance)
        """
        a_arr = np.array(a)
        b_arr = np.array(b)
        distance = np.linalg.norm(a_arr - b_arr)
        return float(distance)

    @staticmethod
    def euclidean_similarity(a: list[float], b: list[float]) -> float:
        """将欧氏距离转为相似度 (0~1 之间)"""
        distance = VectorSimilarity.euclidean(a, b)
        return 1.0 / (1.0 + distance)

    @staticmethod
    def dot_product(a: list[float], b: list[float]) -> float:
        """
        内积。

        公式: A dot B = Sigma(Ai x Bi)

        范围: (-infinity, +infinity)，值越大越相似

        当向量已 L2 归一化后（||A|| = ||B|| = 1），内积 = 余弦相似度
        """
        return float(np.dot(np.array(a), np.array(b)))

    @staticmethod
    def compare_all(a: list[float], b: list[float]) -> dict:
        """一次性计算三种相似度"""
        return {
            "cosine": VectorSimilarity.cosine(a, b),
            "euclidean_distance": VectorSimilarity.euclidean(a, b),
            "euclidean_similarity": VectorSimilarity.euclidean_similarity(a, b),
            "dot_product": VectorSimilarity.dot_product(a, b),
        }


if __name__ == "__main__":
    # 演示: 归一化向量 vs 未归一化向量
    print("=== 向量相似度对比 ===\n")

    vec_a = [0.1, 0.2, 0.3, 0.4]
    vec_b = [0.15, 0.25, 0.35, 0.45]  # 与 a 方向相同，模长略大
    vec_c = [-0.1, -0.2, -0.3, -0.4]  # 与 a 方向相反

    for label, vec in [("相似向量", vec_b), ("相反向量", vec_c)]:
        result = VectorSimilarity.compare_all(vec_a, vec)
        print(f"vec_a vs {label}:")
        for key, value in result.items():
            print(f"  {key}: {value:.4f}")
        print()

    print("观察:")
    print("  - 余弦相似度: 关注方向，b与a方向相同to接近1")
    print("  - 欧氏距离: 关注绝对位置，c与a方向相反to距离大")
    print("  - 内积: 同时关注方向和长度")
```

## 3. ChromaDB 实战

### 3.1 为什么选择 ChromaDB —— 从零到语义搜索的最短路径

在众多向量数据库中，ChromaDB 有一个鲜明的定位：**它是最容易上手的向量数据库**。`pip install chromadb` 一行命令完成安装，无需 Docker、无需配置网络、无需学习复杂的 API。它就像向量数据库世界里的 SQLite——轻量、嵌入式的、开箱即用的。

但这并不意味着 ChromaDB 只适合玩具项目。它的底层使用了 HNSW（Hierarchical Navigable Small World）图索引算法，这是一种在学术和工业界都得到充分验证的高性能近似最近邻搜索算法。对于百万级别的向量检索，ChromaDB 的表现完全够用。

三种运行模式覆盖了不同阶段的需求：内存模式（数据不持久化，适合单元测试）、持久化模式（数据存磁盘，适合原型开发）、和客户端-服务器模式（适合多人协作）。你可以在原型阶段用持久化模式快速迭代，验证想法后再决定是否需要迁移到更重型的向量数据库。

ChromaDB 的另一个重要设计是**自动 Embedding**：如果你在添加文档时没有提供预计算好的向量，ChromaDB 会使用内置的 Embedding 函数自动将文本转为向量。这在本章的学习阶段简化了很多步骤，但在生产环境中，你通常应该自己管理 Embedding 过程（为了一致性和可控性）。

### 3.2 ChromaDB 的高级功能 —— 元数据过滤与自定义 Embedding

当你的知识库规模增长时，"搜索所有文档"已经不够了——你需要精确控制搜索范围。ChromaDB 的元数据过滤机制正是为此设计。

**元数据过滤的实际价值**：假设你有一个包含公司所有制度文档的知识库，用户问"我的年假有多少天？"如果你不做过滤，搜索结果可能混入"往年假数据统计"、"年假制度的历史沿革"等信息，这会让LLM生成的回答不够精准。但如果你提前在文档的元数据中标记了 `{"category": "福利", "year": 2024}`，就可以在查询时加上 `where={"category": "福利"}` 的过滤条件，只搜索最新版本的福利政策文档。这种**向量语义匹配 + 元数据精确过滤**的组合，是生产级RAG系统的标准做法。

ChromaDB 的过滤语法类似于 MongoDB 的查询语法，支持比较运算符（`$eq`、`$ne`、`$gt`、`$gte`、`$lt`、`$lte`）、集合成员（`$in`、`$nin`）、以及逻辑组合（`$and`、`$or`）。这意味着你可以构建相当复杂的过滤条件，比如"2024年发布的中级难度的Python相关文档"。

**自定义 Embedding Function 的意义**：默认情况下，ChromaDB 使用 `all-MiniLM-L6-v2`（一个384维的英文轻量模型）作为自动 Embedding 工具。如果你处理的是中文文档或者对精度有更高要求，就需要替换为自定义的 Embedding 函数。ChromaDB 内置了 OpenAI、Cohere、HuggingFace 等多种 Embedding Function 的适配器，你只需在创建 Collection 时传入即可。

```python
"""
ChromaDB 高级特性:
- 自定义 Embedding Function
- 复杂元数据过滤
- 批量操作
- Collection 管理
"""
import chromadb
from chromadb.utils import embedding_functions
import numpy as np


class ChromaDBAdvanced:
    """
    ChromaDB 高级用法封装。
    """

    def __init__(
        self,
        collection_name: str,
        persist_path: str = "./chroma_advanced",
        use_openai: bool = False,
        openai_api_key: str = None,
    ):
        self.client = chromadb.PersistentClient(path=persist_path)

        # 选择 Embedding 函数
        if use_openai and openai_api_key:
            # OpenAI Embedding
            self.embedding_fn = embedding_functions.OpenAIEmbeddingFunction(
                api_key=openai_api_key,
                model_name="text-embedding-3-small",
            )
            print("使用 OpenAI Embedding")
        else:
            # 默认: sentence-transformers (本地)
            self.embedding_fn = embedding_functions.SentenceTransformerEmbeddingFunction(
                model_name="all-MiniLM-L6-v2",
            )
            print("使用本地 Sentence-Transformer Embedding")

        self.collection = self.client.get_or_create_collection(
            name=collection_name,
            embedding_function=self.embedding_fn,
            metadata={"hnsw:space": "cosine"},
        )

    def add_documents(self, documents: list[dict]):
        """
        批量添加文档。

        参数:
            documents: [{"text": "...", "metadata": {...}, "id": "..."}, ...]
        """
        ids = [doc.get("id", f"doc_{i}") for i, doc in enumerate(documents)]
        texts = [doc["text"] for doc in documents]
        metadatas = [doc.get("metadata", {}) for doc in documents]

        self.collection.add(
            documents=texts,
            metadatas=metadatas,
            ids=ids,
        )

        print(f"添加了 {len(documents)} 条文档，总计 {self.collection.count()} 条")

    def search(
        self,
        query: str,
        top_k: int = 5,
        where: dict = None,
        where_document: dict = None,
    ) -> list[dict]:
        """
        语义搜索。

        参数:
            query: 查询文本
            top_k: 返回数量
            where: 元数据过滤条件，如 {"category": "技术"}
            where_document: 文档内容过滤条件，如 {"$contains": "Python"}

        返回:
            [{"id": "...", "text": "...", "metadata": {...}, "score": 0.95}, ...]

        元数据过滤语法（where）:
        - 等于: {"field": "value"}
        - 不等于: {"field": {"$ne": "value"}}
        - 大于: {"field": {"$gt": 10}}
        - 范围: {"field": {"$gte": 5, "$lte": 100}}
        - 包含: {"field": {"$in": ["a", "b"]}}
        - AND组合: {"$and": [{"field1": "a"}, {"field2": "b"}]}
        - OR组合: {"$or": [{"field1": "a"}, {"field1": "b"}]}
        """
        results = self.collection.query(
            query_texts=[query],
            n_results=top_k,
            where=where,
            where_document=where_document,
        )

        # 整理输出格式
        formatted = []
        for i in range(len(results["ids"][0])):
            formatted.append(
                {
                    "id": results["ids"][0][i],
                    "text": results["documents"][0][i],
                    "metadata": results["metadatas"][0][i] if results["metadatas"][0] else {},
                    "score": 1.0 - results["distances"][0][i] / 2.0,  # 距离转相似度(近似)
                }
            )

        return formatted

    def get_all(self) -> list[dict]:
        """获取所有文档"""
        results = self.collection.get()
        return [
            {
                "id": results["ids"][i],
                "text": results["documents"][i],
                "metadata": results["metadatas"][i],
            }
            for i in range(len(results["ids"]))
        ]

    def delete_collection(self):
        """删除整个集合"""
        self.client.delete_collection(self.collection.name)
```

## 4. Milvus —— 当你的数据量超过百万时

### 4.1 ChromaDB vs Milvus：不只是"简单 vs 复杂"

很多教程把 ChromaDB 和 Milvus 的关系描述为"轻量 vs 重型"，但这样的二分法掩盖了更重要的选型逻辑。更准确的说法是：**它们解决的是同一个问题（向量检索）的不同规模版本**。

ChromaDB 的设计哲学是"开发者体验优先"：一行 `pip install`、Python 原生 API、自动 Embedding。它假设你的数据量在百万级别以下，服务器只有一台。在这个假设下，ChromaDB 做得非常好——你不需要关心分布式一致性、索引调优、资源隔离这些概念。

Milvus 的设计哲学是"性能和规模优先"：它采用了计算与存储分离的架构。这意味着你可以独立扩缩容——当查询量激增时，只增加计算节点（query node）而不需要动存储层；当数据量膨胀时，只扩容存储节点。这种架构在十亿级别的向量检索中至关重要，但也带来了更高的运维复杂度。

具体到技术细节：Milvus 支持多种索引类型，每种都针对不同的"精度 vs 速度 vs 内存"权衡场景。IVF_FLAT 先把向量空间划分为N个区域（聚类），查询时只搜索最近的几个区域，大幅减少了需要精确比较的向量数量。HNSW 则构建一个多层图结构，查询时在图中"走"到最接近的区域。选择哪种索引取决于你的数据规模和对查询精度的要求。

**选型的实操建议**：如果你正在学习或做原型验证，选 ChromaDB——它让你专注于"语义搜索能做什么"，而不是"向量数据库怎么配置"。当你面临以下任一情况时，考虑迁移到 Milvus：（1）数据量超过500万条向量；（2）需要多台服务器分担查询负载；（3）对查询延迟有严格的 P99 要求（比如 < 50ms）；（4）需要数据隔离和多租户支持。

### 4.2 Milvus 基础操作示例

```python
        FieldSchema(name="id", dtype=DataType.VARCHAR, is_primary=True, max_length=100),

def create_index(collection: Collection):
    """
    创建索引。

    索引类型选择:

    - nlist: 聚类数 (IVF_FLAT)，建议 4*sqrt(N)
```

### 4.3 Weaviate / Qdrant 简介 —— 向量数据库的完整图景

除了 ChromaDB 和 Milvus，向量数据库领域还有两个重要的玩家值得了解。它们各自在特定方面优于前面两者，了解它们能帮助你在不同场景下做出更好的技术选型。

**Weaviate** 的独特之处在于它内建了大量 NLP 模块——文本向量化、命名实体识别、摘要生成、问答等。如果你需要的不仅是一个向量存储，而是一个能自动处理文本的智能数据平台，Weaviate 可能比"向量数据库 + 单独的 NLP 管道"更简洁。它使用 GraphQL 作为查询语言（而不是 SQL 或自定义 DSL），这对熟悉 GraphQL 的团队是个加分项。但代价是它的学习曲线比 ChromaDB 陡峭得多，官方文档也比较"学院派"。

## 5. 完整语义搜索实现 —— 从文本到答案的全链路

### 5.1 端到端流程

前面各节分别介绍了 Embedding、相似度计算、向量数据库。现在把它们串在一起，构建一个完整的语义搜索引擎。这个引擎做的事情是：用户输入一个问题；系统从知识库中找到最相关的文档片段；将这些片段作为"参考资料"附加到 Prompt 中；由 LLM 基于这些资料生成答案。

这条流水线本质上是第04章 RAG 架构的具体实现。关键在于每个环节的衔接要顺畅：文档被切分后通过 Embedding 模型转为向量存入 ChromaDB；用户问题也通过同一个 Embedding 模型转为向量；在 ChromaDB 中找到与问题向量最相似的文档片段；将片段拼成上下文；LLM 阅读上下文后生成答案。

在实际工程中，有几个细节值得注意。第一，Embedding 模型必须保持一致——入库时和查询时使用同一个模型，否则向量的"空间坐标系"不同，相似度计算没有意义。第二，检索到的文档片段需要标注来源，这不仅是为了透明度，更是因为 LLM 可能在"参考资料不足"时自行编造内容——标注来源能让你检查回答是否真的有据可依。第三，Prompt 中应该明确告诉 LLM："如果资料中没有答案，就说不知道"——这就形成了"基于参考资料的受控生成"，而不是任由模型自由发挥。

```python
"""
完整的语义搜索流程:
  文档 to 切块 to Embedding to 向量存储 to 查询向量化 to 检索 to 生成回答

这是第04章的 RAG 管道加上本章的向量数据库形成完整闭环。
"""
import os
import json
from openai import OpenAI
import chromadb

client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)


class SemanticSearchEngine:
    """
    完整的语义搜索引擎。

    集成: 文档加载 to Embedding to ChromaDB 存储 to 语义检索 to LLM 生成
    """

    def __init__(
        self,
        collection_name: str = "semantic_search",
        embedding_model: str = "text-embedding-3-small",
        generation_model: str = "gpt-4o",
        persist_path: str = "./semantic_search_db",
    ):
        self.embedding_model = embedding_model
        self.generation_model = generation_model

        # ChromaDB 客户端（通过 OpenAI embedding function）
        self.chroma_client = chromadb.PersistentClient(path=persist_path)

        # 使用 OpenAI Embedding
        from chromadb.utils import embedding_functions

        self.embedding_fn = embedding_functions.OpenAIEmbeddingFunction(
            api_key=os.getenv("OPENAI_API_KEY"),
            model_name=embedding_model,
        )

        self.collection = self.chroma_client.get_or_create_collection(
            name=collection_name,
            embedding_function=self.embedding_fn,
            metadata={"hnsw:space": "cosine"},
        )

    def index_texts(
        self,
        texts: list[str],
        metadatas: list[dict] = None,
        ids: list[str] = None,
    ):
        """
        索引文本数据。

        参数:
            texts: 文本列表
            metadatas: 元数据列表（可选）
            ids: ID 列表（可选）
        """
        if ids is None:
            ids = [f"doc_{i}" for i in range(len(texts))]

        self.collection.add(
            documents=texts,
            metadatas=metadatas if metadatas else [{}] * len(texts),
            ids=ids,
        )

        print(f"已索引 {len(texts)} 条文本，总计 {self.collection.count()} 条")

    def search(
        self,
        query: str,
        top_k: int = 5,
        where: dict = None,
    ) -> list[dict]:
        """
        语义搜索。

        返回: [{"id": "...", "text": "...", "metadata": {...}, "score": 0.95}, ...]
        """
        results = self.collection.query(
            query_texts=[query],
            n_results=top_k,
            where=where,
        )

        formatted = []
        for i in range(len(results["ids"][0])):
            score = 1.0 - results["distances"][0][i] / 2.0  # 距离转相似度
            formatted.append(
                {
                    "id": results["ids"][0][i],
                    "text": results["documents"][0][i],
                    "metadata": results["metadatas"][0][i] if results["metadatas"][0] else {},
                    "score": round(score, 4),
                }
            )

        return formatted

    def ask(
        self,
        question: str,
        top_k: int = 5,
        where: dict = None,
    ) -> dict:
        """
        检索 + LLM 生成 = RAG 问答。

        返回: {"answer": "...", "sources": [...]}
        """
        # 1. 检索相关文档
        chunks = self.search(question, top_k=top_k, where=where)

        if not chunks:
            return {
                "answer": "抱歉，没有找到相关信息。",
                "sources": [],
            }

        # 2. 构建上下文
        context_parts = []
        for i, chunk in enumerate(chunks):
            source = chunk["metadata"].get("source", f"文档{i + 1}")
            context_parts.append(f"[来源: {source}]\n{chunk['text']}")

        context = "\n\n---\n\n".join(context_parts)

        # 3. 生成回答
        response = client.chat.completions.create(
            model=self.generation_model,
            messages=[
                {
                    "role": "system",
                    "content": "你是一个基于知识库的问答助手。只根据提供的参考资料回答问题。如果资料中没有答案，明确告知用户。标注引用来源。",
                },
                {
                    "role": "user",
                    "content": f"参考资料:\n{context}\n\n问题: {question}\n\n请根据以上资料回答。",
                },
            ],
            temperature=0.3,
        )

        answer = response.choices[0].message.content

        # 4. 提取来源
        sources = list(
            set(
                chunk["metadata"].get("source", chunk["id"])
                for chunk in chunks
            )
        )

        return {
            "answer": answer,
            "sources": sources,
            "retrieved_chunks": chunks,
        }

    def get_stats(self) -> dict:
        """获取搜索引擎的统计信息"""
        return {
            "total_documents": self.collection.count(),
            "collection_name": self.collection.name,
            "embedding_model": self.embedding_model,
            "generation_model": self.generation_model,
        }
```

## 6. 高级话题

### 6.1 增量更新与数据一致性 —— 保持向量库与源数据同步

一个容易被忽视但非常重要的问题：当源文档更新后，向量数据库中的旧向量怎么处理？如果不管它，用户查询时可能得到已经过时的信息——比如公司政策变更了，但搜索结果还在引用旧的版本。

处理这个问题的常见策略有三种。**全量重建**每次从头索引所有文档，逻辑最简单但代价最大——100万条向量的全量重建可能需要数小时。**增量更新**只处理发生变化的文档：通过内容哈希判断是否变化，对变化的文档执行 upsert 操作。**版本标记**在元数据中记录版本号和时间戳，查询时过滤掉过期版本。在生产环境中，增量更新配合定期全量重建（比如每周日夜间执行一次）是常见的组合策略。

Chromium 的 `upsert` 操作是对此的最佳支持——如果 ID 已存在则更新，否则插入。配合内容哈希，可以实现高效的增量同步。

```python
"""
向量数据库的增量更新策略。

问题: 当源文档改变时，如何保证向量数据库中的内容是最新的？

策略:
1. 全量重建: 清空旧数据，重新索引（适合数据量小）
2. 增量更新: 只更新变化的文档（适合数据量大）
3. 版本标记: 每个文档带版本号，过期版本标记删除

ChromaDB 支持:
- collection.upsert(): 插入或更新（基于 ID 匹配）
- collection.update(): 更新指定 ID
- collection.delete(): 删除指定 ID
"""
import chromadb
from datetime import datetime


class IncrementalIndexer:
    """
    增量索引器 —— 只更新变化的文档。
    """

    def __init__(self, collection: chromadb.Collection):
        self.collection = collection
        self._doc_versions: dict[str, str] = {}  # {doc_id: content_hash}

    def _hash_content(self, content: str) -> str:
        """对内容做简单哈希"""
        import hashlib

        return hashlib.md5(content.encode()).hexdigest()

    def sync(self, documents: list[dict]) -> dict:
        """
        同步文档到向量数据库（增量更新）。

        参数:
            documents: [{"id": "...", "text": "...", "metadata": {...}}, ...]

        返回:
            {"added": n, "updated": n, "deleted": n, "unchanged": n}
        """
        stats = {"added": 0, "updated": 0, "deleted": 0, "unchanged": 0}

        # 当前文档 ID 集合
        current_ids = set()

        for doc in documents:
            doc_id = doc["id"]
            current_ids.add(doc_id)
            content_hash = self._hash_content(doc["text"])

            # 检查是否已存在且未变化
            if doc_id in self._doc_versions:
                if self._doc_versions[doc_id] == content_hash:
                    stats["unchanged"] += 1
                    continue
                else:
                    # 内容变了，更新
                    self.collection.update(
                        ids=[doc_id],
                        documents=[doc["text"]],
                        metadatas=[doc.get("metadata", {})],
                    )
                    stats["updated"] += 1
            else:
                # 新文档，添加
                self.collection.add(
                    ids=[doc_id],
                    documents=[doc["text"]],
                    metadatas=[doc.get("metadata", {})],
                )
                stats["added"] += 1

            # 记录版本
            self._doc_versions[doc_id] = content_hash

        # 删除已不存在的文档
        removed_ids = set(self._doc_versions.keys()) - current_ids
        if removed_ids:
            self.collection.delete(ids=list(removed_ids))
            stats["deleted"] = len(removed_ids)
            for rid in removed_ids:
                del self._doc_versions[rid]

        print(f"同步完成: 新增{stats['added']} 更新{stats['updated']} "
              f"删除{stats['deleted']} 不变{stats['unchanged']}")

        return stats


if __name__ == "__main__":
    print("=== 增量更新策略 ===\n")
    print("1. upsert: 根据 ID 自动判断插入或更新")
    print("2. 内容哈希: 对比内容是否变化，避免无效更新")
    print("3. 删除检测: 找出源中不存在的旧文档并清理")
```

### 6.2 性能优化 —— 从"能用"到"好用"

向量数据库的性能调优主要围绕三个维度：**插入吞吐、查询延迟、内存占用**。它们之间往往存在权衡关系。

**批量插入**是最直接的优化手段。1000条向量分100批插入和一次性插入的差异，主要是网络往返开销（Round-Trip Time）——每次HTTP请求都有固定的连接建立和序列化开销。批量越大，这些固定开销被分摊得越薄。但批量也不能无限大：太大的批量可能导致单次请求超时，或者占满服务器内存导致 OOM。通常 500-1000 条一批是比较稳妥的选择。

**索引选择**的决策必须基于你的数据规模：[N<100万]时 IVF_FLAT 简单够用；[N在100万-1000万之间] 时考虑 IVF_SQ8（减少50%内存）或 HNSW（提升查询QPS）；[N>1000万] 时 IVF_PQ 或 DiskANN 几乎是必须的。索引参数中 `nprobe`（IVF）和 `ef`（HNSW）控制着精度-速度的权衡——更大的值意味着更精细的搜索和更慢的响应。生产环境通常分两阶段：在线查询用较快的参数（如 nprobe=16），离线评估用较精确的参数（如 nprobe=128）。

```python
"""
向量数据库性能优化指南。

优化维度:
1. 批量插入 to 降低网络往返开销
2. 索引类型选择 to 根据数据量和精度需求
3. 向量压缩 to 减少内存和存储
4. 分区策略 to 限制搜索范围
"""
import time
import numpy as np


def benchmark_batch_insert():
    """
    对比逐条插入 vs 批量插入的性能差异。

    ChromaDB 的 add() 方法已经是批量操作，内置优化。
    这个示例演示的是通用原则。
    """

    dim = 128
    batch_sizes = [10, 100, 500, 1000]
    total = 5000

    print("=== 批量插入性能对比 (模拟) ===\n")
    print(f"总数据量: {total} 条, 维度: {dim}")
    print()

    for batch_size in batch_sizes:
        # 模拟: 按批处理
        batches = total // batch_size
        t0 = time.time()

        for _ in range(batches):
            vectors = np.random.rand(batch_size, dim).tolist()
            # 模拟插入操作 (实际代码中这里调用 collection.add)
            np.array(vectors)  # 模拟处理时间

        elapsed = time.time() - t0
        print(f"batch_size={batch_size:>4}: {batches} 批, 耗时 {elapsed:.3f}s")
        print(f"    相当于 {total / elapsed:.0f} 条/秒")

    print("\n结论: batch_size 越大，吞吐量越高（但受内存限制）")


# ============================================================
# 索引类型选择指南
# ============================================================
INDEX_SELECTION_GUIDE = """
索引类型选择 (以 Milvus 为例):

数据量 < 100万:
  to HNSW (查询快) 或 IVF_FLAT (精度高)

数据量 100万 ~ 1000万:
  to IVF_SQ8 (内存友好) 或 HNSW (如果内存足够)

数据量 > 1000万:
  to IVF_PQ (内存最优) 或 DiskANN (磁盘索引)

索引参数调优:
  - nlist (IVF): 建议 4*sqrt(N), 例如 N=100万 to nlist=4000
  - nprobe (IVF 搜索): 建议 nlist/10, nprobe 越大精度越高但越慢
  - M (HNSW): 建议 16-64, 越大精度越高但内存越多
  - ef (HNSW 搜索): 建议 64-512, 越大精度越高但越慢
"""

print(INDEX_SELECTION_GUIDE)


if __name__ == "__main__":
    benchmark_batch_insert()
```

## 常见错误

### 错误 1: 向量维度不匹配

```python
# 错误: Embedding 模型输出 1536 维，但数据库设置了 768 维
collection = client.create_collection(name="docs", dimension=768)
embeddings = embed_fn.embed(texts)  # 返回 1536 维向量
collection.add(embeddings=embeddings, ...)  # 报错: dimension mismatch

# 修正: 确保维度一致
dim = 1536  # 与 embedding 模型匹配
collection = client.create_collection(name="docs", dimension=dim)
```

### 错误 2: 忘记 normalize_embeddings 导致相似度不准

```python
# 错误: 使用内积作为相似度，但向量没有归一化
# 结果: 长文本的向量模长大，内积天然更大，不公平

# 修正: 编码时设置 normalize_embeddings=True
embeddings = model.encode(texts, normalize_embeddings=True)
# 此时内积 = 余弦相似度
```

### 错误 3: 批量操作时忽略 API 速率限制

```python
# 错误: 一次发送 10000 条文本到 OpenAI Embedding API
embeddings = client.embeddings.create(input=texts)  # 可能触发速率限制

# 修正: 分批发送，每批 100-500 条
batch_size = 100
for i in range(0, len(texts), batch_size):
    batch = texts[i : i + batch_size]
    embeddings = client.embeddings.create(input=batch)
    time.sleep(0.1)  # 避让速率限制
```

### 错误 4: ChromaDB 使用默认的 l2 距离而非 cosine

```python
# 错误: 使用默认的 l2 距离（欧氏距离）
collection = client.create_collection(name="docs")
# 对于归一化向量，欧氏距离和余弦相似度的排序相同，
# 但对未归一化的向量，欧氏距离可能偏好短文本

# 修正: 显式指定余弦相似度
collection = client.create_collection(
    name="docs",
    metadata={"hnsw:space": "cosine"},
)
```

### 错误 5: 查询向量和索引向量使用了不同的 embedding 模型

```python
# 错误: 索引时用 text-embedding-3-small, 查询时用 text-embedding-3-large
# 结果: 虽然都是语义向量，但分布在不同的向量空间中，相似度计算无意义

# 修正: 确保使用完全相同的 embedding 模型
# 在系统中统一管理 embedding model 实例
```

### 错误 6: 忘记关闭 Milvus 连接导致资源泄露

```python
# 错误: 脚本结束时没有断开连接
connections.connect(host="localhost", port="19530")
# ... 操作 ...
# 脚本结束，连接未正确释放

# 修正: 使用上下文管理器或显式断开

    connections.connect(host="localhost", port="19530")
    # ... 操作 ...

    connections.disconnect("default")
```
