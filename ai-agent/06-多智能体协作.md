## 1. 为什么需要多 Agent

### 1.1 单 Agent 的能力边界

在第01-05章中，我们构建的 Agent 都是"全才型"的——一个 Agent 承担所有角色：理解任务、制定计划、调用工具、处理错误。这种单体 Agent 架构在任务简单时工作良好，但随着任务复杂度的增长，会遇到以下瓶颈：

**认知负载过重**

当一个 Agent 需要同时扮演"数据分析师"、"报告撰写者"、"代码审查员"时，它的 System Prompt 会变得极其冗长。每个角色需要不同的知识背景、输出风格和行为约束。把所有这些塞进一个 System Prompt 中，会导致：

- 指令越来越多，模型越来越难全部遵循
- 不同角色的指令可能互相冲突（"数据要详细" vs "报告要简洁"）
- Token 消耗随 prompt 膨胀而线性增长，但有效注意力随着增长而非线性衰减

**技能深度不足**

一个好的数据分析师和一个好的文案撰写者需要的是不同的能力。在单 Agent 架构中，Agent 在每类任务上都是"半吊子"——它知道怎么做数据分析，但不如专门的"数据分析 Agent"精深；它也能写报告，但不如"写作 Agent"专业。

**单点故障风险**

单体 Agent 的一次 LLM 调用出错或一个工具调用失败，可能影响整个任务链。在多 Agent 架构中，一个 Agent 的失败可以被其他 Agent 检测和补偿。

**无法利用真正的并行性**

虽然第05章讲了并行执行子任务，但那些子任务由同一个 Agent 决策。多 Agent 架构中，不同 Agent 可以真正独立地思考和行动，不仅执行并行，决策也是并行的。

### 1.2 多 Agent 的核心价值：专业化分工

类比人类组织：你不会让一个人同时做 CEO、CTO、CFO、COO 的工作。每个角色有专注的领域和决策边界。多 Agent 系统遵循同样的原则——**将大模型的能力通过角色分工进行"聚焦"，每个 Agent 在有限的认知边界内做到最好**。

```
单 Agent 架构:
┌─────────────────────────────────────────┐
│            全能型 Agent                  │
│  System Prompt: 5000 tokens            │
│  包含：数据分析规则 + 写作风格 +         │
│        代码规范 + 安全边界 + ...         │
│                                         │
│  问题：规则太多，每条规则的"有效信号"    │
│        被稀释在噪音中                    │
└─────────────────────────────────────────┘

多 Agent 架构:
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 数据分析  │  │ 报告写作  │  │ 质量审查  │
│ Agent    │  │ Agent    │  │ Agent    │
│          │  │          │  │          │
│ Prompt:  │  │ Prompt:  │  │ Prompt:  │
│ 300词     │  │ 300词    │  │ 300词    │
│ 专注分析  │  │ 专注写作  │  │ 专注审查  │
└──────────┘  └──────────┘  └──────────┘
      │             │             │
      └─────────────┼─────────────┘
                    │
            ┌───────┴───────┐
            │  协调者 Agent  │
            │  Prompt: 200词 │
            │  专注分配任务  │
            └───────────────┘

三个专业 Agent + 一个轻量协调者，总 prompt token 反而更少，
每个 Agent 的指令更清晰、更易遵循。
```

### 1.3 什么时候需要多 Agent —— 决策树

不是所有任务都需要多 Agent。以下决策树帮助判断：

```
任务是否涉及多个不同的专业领域？
  ├── 否 → 使用单 Agent（第01-05章的模式）
  └── 是 → 这些领域是否可以清晰分离？
            ├── 否 → 使用单 Agent + 详细的角色 Prompt
            └── 是 → 任务是否有并行机会？
                      ├── 否 → 流水线模式（串行的专业 Agent）
                      └── 是 → 主从或协商模式（并行协作）
```

**不需要多 Agent 的场景**：
- "查询天气并回复用户"——单一领域，简单任务
- "翻译一段文字"——单一工具，一步完成
- "解释一个编程概念"——知识问答

**需要多 Agent 的场景**：
- 软件开发：需求分析 Agent + 架构设计 Agent + 编码 Agent + 测试 Agent + 代码审查 Agent
- 内容创作：选题分析 Agent + 素材搜集 Agent + 文章撰写 Agent + 编辑审校 Agent
- 金融分析：数据采集 Agent + 技术分析 Agent + 基本面分析 Agent + 风险评估 Agent + 报告生成 Agent

```python
"""
多 Agent 需求判断器 —— 帮助决定是否应该使用多 Agent 架构。

这是一个"元认知"工具：在构建 Agent 系统之前，先判断架构选型。
"""
from dataclasses import dataclass, field
from enum import Enum


class ArchitectureAdvice(Enum):
    SINGLE_AGENT = "single_agent"           # 单 Agent 足够了
    SINGLE_AGENT_ENHANCED = "single_enhanced"  # 单 Agent + 增强 Prompt
    PIPELINE = "pipeline"                   # 流水线多 Agent
    ORCHESTRATED = "orchestrated"           # 主从多 Agent
    COLLABORATIVE = "collaborative"         # 平等协商多 Agent


@dataclass
class TaskProfile:
    """
    任务画像 —— 描述一个任务的多个维度特征，
    用于判断最适合的 Agent 架构。
    """
    description: str
    
    # 维度评分（1-5）
    domain_diversity: int = 1      # 涉及多少个不同专业领域？
    step_count: int = 1             # 大约需要多少步骤？
    parallelism_potential: int = 1  # 步骤之间能并行执行的程度？
    expertise_depth: int = 1        # 每个领域需要多深的专业知识？
    uncertainty: int = 1            # 执行过程中的不确定性程度？
    quality_requirement: int = 1    # 对输出质量的要求有多高？


def recommend_architecture(profile: TaskProfile) -> tuple[ArchitectureAdvice, str]:
    """
    根据任务画像推荐 Agent 架构。
    
    决策逻辑（规则可根据实际经验调整）：
    """
    score = (
        profile.domain_diversity * 2 +
        profile.expertise_depth * 2 +
        profile.parallelism_potential +
        profile.uncertainty
    )

    if profile.domain_diversity <= 1 and profile.step_count <= 3:
        return (
            ArchitectureAdvice.SINGLE_AGENT,
            "任务简单、领域单一，直接用单 Agent 即可。不需要多 Agent 的额外复杂度和通信开销。"
        )
    
    elif profile.domain_diversity <= 2 and profile.parallelism_potential <= 1:
        return (
            ArchitectureAdvice.PIPELINE,
            "多步骤但串行、领域有区分。用流水线模式：每个 Agent 处理一个阶段，传递中间结果。"
        )
    
    elif profile.parallelism_potential >= 3:
        return (
            ArchitectureAdvice.ORCHESTRATED,
            "有明确并行机会。用主从模式：一个协调者 Agent 分配任务给多个专业 Agent 并行执行。"
        )
    
    elif profile.uncertainty >= 3 and profile.quality_requirement >= 4:
        return (
            ArchitectureAdvice.COLLABORATIVE,
            "不确定性高、质量要求高。用协商模式：多个 Agent 互相审查和补充，提高结果可靠性。"
        )
    
    else:
        return (
            ArchitectureAdvice.SINGLE_AGENT_ENHANCED,
            "任务有一定复杂度但领域不够分离。加强单 Agent 的 System Prompt 比引入多 Agent 更经济。"
        )


if __name__ == "__main__":
    # 测试不同任务的架构建议
    tasks = [
        TaskProfile("查询天气并回复", 1, 2, 0, 1, 0, 1),
        TaskProfile("翻译一篇技术文档", 1, 3, 0, 1, 1, 3),
        TaskProfile("开发一个完整的Web应用（前端+后端+数据库+测试）",
                   4, 10, 4, 4, 3, 5),
        TaskProfile("对上市公司的财报进行深度分析并生成投资建议报告",
                   3, 6, 3, 4, 3, 5),
    ]

    for task in tasks:
        advice, reason = recommend_architecture(task)
        print(f"\n任务: {task.description}")
        print(f"  推荐架构: {advice.value}")
        print(f"  理由: {reason}")
```

## 2. 多 Agent 协作模式

### 2.1 三种基本模式概览

多 Agent 系统的协作模式决定了 Agent 之间如何交换信息、如何分配任务、如何汇总结果。三种基本模式构成了所有复杂多 Agent 系统的构建块：

```
模式1: 主从模式 (Orchestrator-Worker)
┌──────────────┐
│ Orchestrator │  ← 一个"大脑"负责任务分配和结果汇总
│ (协调者)      │
└──┬───┬───┬──┘
   │   │   │
   ▼   ▼   ▼
┌──┐ ┌──┐ ┌──┐
│A │ │B │ │C │  ← Worker Agent，只负责执行分配的任务
└──┘ └──┘ └──┘
特点: 控制集中、易于管理、单点瓶颈
适用: 任务可被清晰分解、并行机会明确


模式2: 平等协商模式 (Peer-to-Peer)
┌──┐ ←→ ┌──┐
│A │    │B │  ← Agent 之间自由通信、互相请求
└──┘ ←→ └──┘
   ↕       ↕
┌──┐ ←→ ┌──┐
│C │    │D │
└──┘    └──┘
特点: 去中心化、灵活、难以预测和调试
适用: 需要多角度讨论、开放性问题、头脑风暴


模式3: 流水线模式 (Pipeline)
┌──┐    ┌──┐    ┌──┐
│A │ → │B │ → │C │  ← 串行处理，每个 Agent 负责一个阶段
└──┘    └──┘    └──┘
特点: 简单清晰、延迟较高、缺乏反馈
适用: 有明确工序、标准化的处理流程
```

### 2.2 主从模式详解

主从模式是最常用也最容易实现的多 Agent 模式。一个 Orchestrator（协调者）Agent 负责全局决策，多个 Worker Agent 负责执行具体任务。

**协调者的职责**：
- 分析用户任务，理解整体目标
- 将任务分解为子任务，分配给最合适的 Worker
- 制定 Worker 之间的执行顺序
- 汇总 Worker 的结果，形成最终输出
- 处理 Worker 的失败（重试或重新分配）

**Worker 的职责**：
- 接收来自协调者的具体任务
- 在自己的专业领域内完成工作
- 将结果返回给协调者
- 如果自己的领域内也需要帮助，可以请求协调者安排

```python
"""
主从模式 (Orchestrator-Worker) 的框架实现。

这是多 Agent 协作的"入门模式"——结构清晰，易于理解和调试。
"""
import os
import json
import asyncio
from dataclasses import dataclass, field
from typing import Any, Optional
from enum import Enum
from openai import OpenAI


# ============================================================
# Agent 基类
# ============================================================
@dataclass
class AgentConfig:
    """Agent 配置"""
    name: str                          # Agent 名称
    role: str                          # 角色描述（如"数据分析专家"）
    expertise: list[str]               # 专业领域
    system_prompt: str                 # 系统提示词
    tools: list[str] = field(default_factory=list)  # 可用工具列表
    temperature: float = 0.3


class Agent:
    """
    Agent 基类 —— 所有 Worker Agent 和 Orchestrator 的父类。
    
    每个 Agent 有自己的：
    - 角色定义（system prompt）
    - 专业领域（影响任务分配）
    - 工具集（可调用的外部能力）
    """

    def __init__(self, config: AgentConfig, llm_client: OpenAI):
        self.config = config
        self.client = llm_client

    async def think(self, task: str, context: str = "") -> str:
        """
        Agent 的"思考"方法 —— 调用 LLM 进行推理。
        
        这是 Agent 最核心的能力：接收任务、结合上下文、产生输出。
        """
        messages = [
            {"role": "system", "content": self.config.system_prompt},
        ]
        if context:
            messages.append({"role": "user", "content": f"上下文信息:\n{context}"})
        messages.append({"role": "user", "content": f"请完成以下任务:\n{task}"})

        # 注意：实际中使用 await，这里为了兼容性使用同步调用
        response = self.client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            temperature=self.config.temperature,
        )
        return response.choices[0].message.content

    def get_profile(self) -> dict:
        """返回 Agent 的能力画像（用于协调者做任务分配决策）"""
        return {
            "name": self.config.name,
            "role": self.config.role,
            "expertise": self.config.expertise,
            "tools": self.config.tools,
        }


# ============================================================
# 协调者 Agent
# ============================================================
@dataclass
class TaskAssignment:
    """协调者分配的一个任务"""
    worker_name: str      # 分配给哪个 Worker
    task_description: str  # 任务描述
    expected_output: str   # 期望输出格式
    priority: int = 5      # 优先级
    depends_on: list[str] = field(default_factory=list)  # 依赖的前置任务 ID


class Orchestrator(Agent):
    """
    协调者 Agent —— 主从模式的大脑。
    
    核心能力：
    1. 分析用户任务 → 确定需要哪些专业能力
    2. 将任务分解 → 分配给合适的 Worker
    3. 汇总结果 → 形成最终输出
    4. 处理异常 → Worker 失败时重新分配
    """

    def __init__(self, config: AgentConfig, llm_client: OpenAI):
        super().__init__(config, llm_client)
        self.workers: list[Agent] = []
        self.worker_profiles: list[dict] = []

    def register_worker(self, worker: Agent):
        """注册一个 Worker Agent"""
        self.workers.append(worker)
        self.worker_profiles.append(worker.get_profile())

    def plan(self, user_task: str) -> list[TaskAssignment]:
        """
        分析用户任务，生成任务分配计划。
        
        协调者需要回答：
        - 这个任务需要哪些专业能力？
        - 应该分配给哪个 Worker？
        - 任务之间的依赖关系是什么？
        - 哪些任务可以并行？
        """
        # 构建分配计划的 prompt
        workers_desc = json.dumps(self.worker_profiles, ensure_ascii=False, indent=2)

        prompt = f"""你是一个任务协调专家。根据以下 Worker 的能力画像，将用户任务分解并分配。

## 可用 Worker
{workers_desc}

## 用户任务
{user_task}

## 分配原则
1. 每个子任务分配给最擅长该领域的 Worker
2. 标注子任务之间的依赖关系
3. 识别可以并行的子任务
4. 为每个子任务定义明确的期望输出格式

## 输出格式
返回 JSON 数组:
[
  {{
    "worker_name": "分配给哪个 Worker 的名称",
    "task_description": "具体的任务描述",
    "expected_output": "期望的输出格式",
    "priority": 1-10,
    "depends_on": ["依赖的前置分配项的索引（从0开始）"],
    "reason": "为什么分配给这个 Worker"
  }}
]"""

        response = self.client.chat.completions.create(
            model="gpt-4o",
            messages=[
                {"role": "system", "content": prompt},
                {"role": "user", "content": "请为以上任务生成 Worker 分配计划"},
            ],
            temperature=0.0,
            response_format={"type": "json_object"},
        )

        assignments_raw = json.loads(response.choices[0].message.content)

        # 解析为 TaskAssignment 对象
        assignments = []
        if isinstance(assignments_raw, dict):
            # LLM 可能返回 {"assignments": [...]} 或直接返回 [...]
            items = assignments_raw.get("assignments", [assignments_raw])
        else:
            items = assignments_raw

        for item in items:
            assignments.append(TaskAssignment(
                worker_name=item["worker_name"],
                task_description=item["task_description"],
                expected_output=item.get("expected_output", ""),
                priority=item.get("priority", 5),
                depends_on=item.get("depends_on", []),
            ))

        return assignments

    async def execute(
        self,
        assignments: list[TaskAssignment],
        max_parallel: int = 3,
    ) -> dict:
        """
        按依赖关系执行任务分配计划。
        
        与第05章的 Plan-Execute 类似：按依赖关系决定执行顺序，
        无依赖的任务（或依赖已满足的任务）可以并行执行。
        """
        worker_map = {w.config.name: w for w in self.workers}
        results = {}  # assignment_index → result

        while len(results) < len(assignments):
            # 找出所有就绪的任务（依赖都已满足）
            ready = []
            for i, assignment in enumerate(assignments):
                if i in results:
                    continue
                deps_satisfied = all(
                    dep_idx in results
                    for dep_idx in assignment.depends_on
                )
                if deps_satisfied:
                    ready.append((i, assignment))

            if not ready:
                print("⚠ 没有就绪任务但还有未完成的任务 — 检查依赖关系")
                break

            # 并行执行就绪任务
            tasks_coroutines = []
            for idx, assignment in ready[:max_parallel]:
                worker = worker_map.get(assignment.worker_name)
                if worker is None:
                    results[idx] = f"错误: 未找到 Worker '{assignment.worker_name}'"
                    continue

                # 构建上下文（包含依赖任务的输出）
                context = ""
                for dep_idx in assignment.depends_on:
                    if dep_idx in results:
                        context += f"\n[{assignments[dep_idx].worker_name} 的输出]:\n{results[dep_idx]}\n"

                tasks_coroutines.append(
                    self._execute_one(idx, worker, assignment, context, results)
                )

            if tasks_coroutines:
                await asyncio.gather(*tasks_coroutines)

        return results

    async def _execute_one(
        self,
        idx: int,
        worker: Agent,
        assignment: TaskAssignment,
        context: str,
        results: dict,
    ):
        """执行单个 Worker 任务"""
        print(f"  ▶ {worker.config.name} 正在处理: {assignment.task_description[:50]}...")
        try:
            result = await worker.think(
                task=assignment.task_description,
                context=context,
            )
            results[idx] = result
            print(f"  ✅ {worker.config.name} 完成")
        except Exception as e:
            results[idx] = f"执行失败: {e}"
            print(f"  ❌ {worker.config.name} 失败: {e}")

    def synthesize(self, assignments: list[TaskAssignment], results: dict) -> str:
        """
        汇总所有 Worker 的结果，形成最终回复。

        协调者不是简单拼接结果，而是理解每个结果、提取关键信息、
        去除冗余、组织成连贯的最终输出。
        """
        # 构建汇总 prompt
        results_text = ""
        for i, assignment in enumerate(assignments):
            result = results.get(i, "未完成")
            results_text += f"\n## {assignment.worker_name} 的输出\n{result}\n"

        prompt = f"""你是一个报告汇总专家。以下是你团队的成员针对用户任务各自完成的工作。

## 原始任务分配
{json.dumps([{
    "worker": a.worker_name,
    "task": a.task_description,
    "expected": a.expected_output,
} for a in assignments], ensure_ascii=False, indent=2)}

## 各成员的输出
{results_text}

## 汇总要求
1. 整合各成员的输出，形成连贯的最终回复
2. 去除重复信息
3. 如果某些输出相互矛盾，标注出来
4. 按逻辑顺序组织内容（引言 → 主体分析 → 结论建议）
5. 保持专业、清晰的表达风格"""

        response = self.client.chat.completions.create(
            model="gpt-4o",
            messages=[
                {"role": "system", "content": prompt},
                {"role": "user", "content": "请汇总以上结果。"},
            ],
            temperature=0.3,
        )
        return response.choices[0].message.content
```

### 2.3 平等协商模式

在主从模式中，Worker 只和协调者通信，Worker 之间不直接交流。平等协商模式打破了这个限制——Agent 之间可以直接通信、互相提问、辩论、修正。

协商模式的特点：
- **没有中心权威**：没有协调者，Agent 自主决定和谁通信
- **消息自由路由**：Agent A 可以直接请求 Agent B 帮助
- **共识机制**：多个 Agent 可能得出不同结论，需要通过共识机制解决分歧
- **涌现行为**：系统的行为可能超出单个 Agent 的设计（有利有弊）

```python
"""
平等协商模式 —— Agent 之间直接通信和协商。

核心组件：
1. MessageBus: 消息总线，Agent 通过它发送和接收消息
2. Agent 自主决策：每个 Agent 自己决定何时发送消息、发给谁
3. 共识协议：当 Agent 意见不一致时如何达成一致
"""
import asyncio
from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Optional
from datetime import datetime


class MessageType(Enum):
    """消息类型"""
    REQUEST = "request"        # 请求帮助
    RESPONSE = "response"      # 回复请求
    PROPOSAL = "proposal"      # 提出方案
    OBJECTION = "objection"    # 反对某个方案
    AGREEMENT = "agreement"    # 同意某个方案
    QUESTION = "question"      # 提问
    INFORMATION = "information"  # 共享信息


@dataclass
class AgentMessage:
    """Agent 间通信的消息格式"""
    id: str
    sender: str           # 发送者 Agent 名称
    receiver: str         # 接收者 Agent 名称（"broadcast" 表示广播）
    msg_type: MessageType
    content: str          # 消息正文
    timestamp: str = field(default_factory=lambda: datetime.now().isoformat())
    reply_to: Optional[str] = None  # 回复哪条消息
    metadata: dict = field(default_factory=dict)  # 附加元数据


class MessageBus:
    """
    消息总线 —— Agent 之间通信的基础设施。
    
    不是让 Agent 直接调用彼此的方法，而是通过消息总线异步传递消息。
    这带来了几个好处：
    - 解耦：Agent 不需要知道其他 Agent 的内部实现
    - 异步：发送者不需要等待接收者处理完
    - 可审计：所有消息都有记录
    - 可拦截：可以在消息传递过程中添加过滤、转换、日志
    """

    def __init__(self):
        self._queues: dict[str, asyncio.Queue] = {}  # agent_name → message queue
        self._history: list[AgentMessage] = []       # 完整的消息历史
        self._subscriptions: dict[str, set[str]] = {}  # message_type → {agent_names}

    def register(self, agent_name: str):
        """注册一个 Agent 到消息总线"""
        self._queues[agent_name] = asyncio.Queue()

    def subscribe(self, agent_name: str, msg_type: MessageType):
        """订阅特定类型的消息"""
        key = msg_type.value
        if key not in self._subscriptions:
            self._subscriptions[key] = set()
        self._subscriptions[key].add(agent_name)

    async def send(self, message: AgentMessage) -> None:
        """发送消息到指定接收者"""
        self._history.append(message)

        if message.receiver == "broadcast":
            # 广播：发送给所有 Agent（除了发送者自己）
            for name, queue in self._queues.items():
                if name != message.sender:
                    await queue.put(message)
        else:
            # 单播：发送给指定 Agent
            target_queue = self._queues.get(message.receiver)
            if target_queue is not None:
                await target_queue.put(message)
            else:
                print(f"⚠ 消息发送失败: 接收者 '{message.receiver}' 未注册")

    async def receive(self, agent_name: str, timeout: float = None) -> Optional[AgentMessage]:
        """接收发给指定 Agent 的消息"""
        queue = self._queues.get(agent_name)
        if queue is None:
            return None
        try:
            if timeout is not None:
                return await asyncio.wait_for(queue.get(), timeout=timeout)
            else:
                return await queue.get()
        except asyncio.TimeoutError:
            return None

    def get_history(self, between: tuple[str, str] = None) -> list[AgentMessage]:
        """查询消息历史，可选过滤"""
        if between is None:
            return list(self._history)
        a, b = between
        return [
            m for m in self._history
            if (m.sender == a and m.receiver == b) or
               (m.sender == b and m.receiver == a)
        ]


@dataclass
class PeerAgent:
    """
    协商模式下的 Agent —— 具有更大的自主性。
    
    每个 PeerAgent 有自己的"个性"：
    - domain: 专业领域，影响它参与哪些讨论
    - assertiveness: 自信程度（0-1），影响它是否坚持己见
    - cooperativeness: 合作程度（0-1），影响它是否愿意妥协
    """

    name: str
    domain: str
    system_prompt: str
    llm_client: Any  # OpenAI client
    bus: MessageBus
    
    # 行为特征
    assertiveness: float = 0.5    # 自信程度
    cooperativeness: float = 0.7  # 合作程度
    
    def __post_init__(self):
        self.inbox: list[AgentMessage] = []

    async def think_and_respond(self, message: AgentMessage) -> Optional[AgentMessage]:
        """
        接收到消息后，Agent 自主决定如何处理。
        
        这模拟了真实的协商过程：
        1. 理解消息内容
        2. 评估与自己专业领域的相关性
        3. 决定是否回复、如何回复
        """
        # 构建推理 prompt
        prompt = f"""你是 {self.name}，专业领域是 {self.domain}。

根据你的专业领域和个性，你应该：
1. 如果消息与你的领域无关 → 回复 "IGNORE"
2. 如果你同意消息内容 → 回复 "AGREE: <你的补充>"

4. 如果你需要更多信息 → 回复 "QUESTION: <你的问题>"

- 只回应你专业领域内的问题
- 如果你不自信（低 assertiveness），即使不同意也不要太强硬
- 如果你高度合作（高 cooperativeness），可以接受不完美的方案"""

        response = self.llm_client.chat.completions.create(
            model="gpt-4o",
            messages=[
                {"role": "system", "content": self.system_prompt},
                {"role": "user", "content": prompt},
            ],
            temperature=0.5,  # 协商需要一定随机性
        )

        reply_text = response.choices[0].message.content

        if "IGNORE" in reply_text.upper():
            return None

        # 判断回复类型
        if reply_text.upper().startswith("AGREE:"):
            msg_type = MessageType.AGREEMENT
        elif reply_text.upper().startswith("OBJECT:"):
            msg_type = MessageType.OBJECTION
        elif reply_text.upper().startswith("QUESTION:"):
            msg_type = MessageType.QUESTION
        else:
            msg_type = MessageType.INFORMATION

        return AgentMessage(
            id=f"msg-{datetime.now().timestamp()}",
            sender=self.name,
            receiver=message.sender,  # 默认回复给发送者
            msg_type=msg_type,
            content=reply_text,
            reply_to=message.id,
        )
```

### 2.4 流水线模式

流水线模式是最简单直接的多 Agent 模式：Agent 按顺序排成一列，每个 Agent 处理上一个 Agent 的输出。这是"工厂装配线"在 Agent 系统中的对应物。

```python
"""
流水线模式 —— Agent 串行处理，每个 Agent 负责一个阶段。

典型的流水线：
输入 → Agent A (处理) → Agent B (处理) → Agent C (处理) → 输出

适用场景：
- 有清晰阶段划分的任务（如 审稿 → 翻译 → 排版 → 发布）
- 需要质量把关的流程（每个阶段都是一道质检）
- 数据有明确处理顺序的 ETL 流程
"""
import asyncio
from typing import Any, Callable


class PipelineStage:
    """
    流水线的一个阶段。
    
    每个阶段包含：
    - Agent (处理逻辑)
    - 验证规则 (输入/输出的约束)
    - 错误处理策略
    """

    def __init__(
        self,
        name: str,
        agent: Any,  # Agent 实例
        input_validator: Callable = None,
        output_validator: Callable = None,
        on_error: str = "stop",  # "stop" | "skip" | "retry"
    ):
        self.name = name
        self.agent = agent
        self.input_validator = input_validator
        self.output_validator = output_validator
        self.on_error = on_error

    async def process(self, input_data: Any) -> dict:
        """处理输入，返回包含结果和元信息的字典"""
        result = {
            "stage": self.name,
            "input": input_data,
            "output": None,
            "success": False,
            "error": None,
        }

        # 1. 输入验证
        if self.input_validator:
            try:
                self.input_validator(input_data)
            except Exception as e:
                result["error"] = f"输入验证失败: {e}"
                return result

        # 2. Agent 处理
        try:
            output = await self.agent.think(str(input_data))
            result["output"] = output
        except Exception as e:
            result["error"] = f"Agent 执行失败: {e}"
            return result

        # 3. 输出验证
        if self.output_validator:
            try:
                self.output_validator(output)
                result["success"] = True
            except Exception as e:
                result["error"] = f"输出验证失败: {e}"
                result["success"] = False
        else:
            result["success"] = True

        return result


class Pipeline:
    """
    多 Agent 流水线 —— 串联多个处理阶段。
    """

    def __init__(self, name: str):
        self.name = name
        self.stages: list[PipelineStage] = []
        self.artifacts: dict[str, Any] = {}  # 阶段名称 → 中间结果

    def add_stage(self, stage: PipelineStage):
        """添加一个处理阶段"""
        self.stages.append(stage)

    async def run(self, initial_input: Any, verbose: bool = True) -> dict:
        """运行完整流水线"""
        current_input = initial_input

        for i, stage in enumerate(self.stages):
            if verbose:
                print(f"  🔄 阶段 {i + 1}/{len(self.stages)}: {stage.name}")

            result = await stage.process(current_input)
            self.artifacts[stage.name] = result

            if result["success"]:
                current_input = result["output"]
                if verbose:
                    output_preview = str(result["output"])[:100].replace("\n", " ")
                    print(f"  ✅ {stage.name} 完成 → {output_preview}...")
            else:
                if verbose:
                    print(f"  ❌ {stage.name} 失败: {result['error']}")
                if stage.on_error == "stop":
                    return {
                        "success": False,
                        "failed_at": stage.name,
                        "error": result["error"],
                        "artifacts": self.artifacts,
                    }
                elif stage.on_error == "skip":
                    continue  # 保持当前输入不变，跳到下一阶段
                elif stage.on_error == "retry":
                    # 简单重试一次
                    result = await stage.process(current_input)
                    if not result["success"]:
                        return {
                            "success": False,
                            "failed_at": stage.name,
                            "error": result["error"],
                            "artifacts": self.artifacts,
                        }
                    current_input = result["output"]

        return {
            "success": True,
            "final_output": current_input,
            "stages_completed": len(self.stages),
            "artifacts": self.artifacts,
        }
```

## 3. Agent 间通信协议

### 3.1 通信协议设计原则

多 Agent 系统中，Agent 之间的通信协议决定了整个系统的效率和可靠性。好的协议设计遵循以下原则：

**原则1：消息结构化，而非自由文本**

Agent A 对 Agent B 说"帮我看一下这个问题"——这句话有太多歧义。看什么？怎么反馈？在多 Agent 系统中，自由文本通信会导致大量 LLM 调用被浪费在"澄清需求"上。

结构化消息明确了"谁要对谁做什么，期望什么输出"。

**原则2：异步优先**

同步调用（Agent A 等待 Agent B 完成才继续）会浪费并行机会。异步消息 + 回调/轮询允许 Agent A 在等待 Agent B 时继续做其他工作。

**原则3：可审计和可追溯**

每条消息必须有唯一 ID、时间戳、发送者和接收者。这不仅是调试需要，也是安全和合规要求（谁说了什么、谁影响了谁的决策）。

**原则4：版本化**

消息格式会演进。在消息中标注协议版本，确保不同版本的 Agent 仍能通信。

```python
"""
Agent 通信协议的标准消息格式。

这个设计借鉴了 HTTP 和 JSON-RPC 的设计思想：
- 有请求和响应的配对（通过 reply_to）
- 有标准的错误格式
- 有元数据字段用于扩展
"""
import uuid
from datetime import datetime
from enum import Enum
from typing import Any, Optional
from dataclasses import dataclass, field
from pydantic import BaseModel, Field


class TaskPriority(str, Enum):
    LOW = "low"
    MEDIUM = "medium"
    HIGH = "high"
    CRITICAL = "critical"


class TaskStatus(str, Enum):
    ACCEPTED = "accepted"    # 已接受，等待开始
    IN_PROGRESS = "in_progress"
    COMPLETED = "completed"
    FAILED = "failed"
    REJECTED = "rejected"    # 被拒绝（能力不匹配或其他原因）


class TaskRequest(BaseModel):
    """
    任务委派消息 —— Agent A 请求 Agent B 执行一个任务。
    
    这是多 Agent 系统中最重要的消息类型。
    """
    # 消息元信息
    message_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    message_type: str = "task_request"
    protocol_version: str = "1.0"
    timestamp: str = Field(default_factory=lambda: datetime.now().isoformat())
    
    # 通信双方
    sender: str           # 委派者的 Agent 名称
    receiver: str         # 被委派者的 Agent 名称
    
    # 任务内容
    task_id: str          # 任务的唯一标识
    description: str      # 自然语言任务描述
    context: dict = Field(default_factory=dict)  # 上下文数据（前面任务的输出等）
    
    # 约束
    priority: TaskPriority = TaskPriority.MEDIUM
    deadline: Optional[str] = None  # ISO 时间戳
    max_tokens: int = 2000
    expected_format: str = "text"   # "text" | "json" | "markdown" | "code"
    
    # 协作信息
    parent_task_id: Optional[str] = None  # 在这个更大的任务框架下
    depends_on: list[str] = Field(default_factory=list)  # 依赖的任务 ID


class TaskResponse(BaseModel):
    """
    任务结果汇报消息 —— Agent B 完成（或拒绝）任务后回复 Agent A。
    """
    message_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    message_type: str = "task_response"
    protocol_version: str = "1.0"
    timestamp: str = Field(default_factory=lambda: datetime.now().isoformat())
    
    sender: str           # 执行者的 Agent 名称
    receiver: str         # 原委派者的 Agent 名称
    
    # 关联信息
    reply_to: str         # 对应 TaskRequest 的 message_id
    task_id: str          # 对应 TaskRequest 的 task_id
    status: TaskStatus
    
    # 结果内容
    result: Optional[Any] = None     # 成功时的输出
    error: Optional[str] = None      # 失败时的错误信息
    
    # 执行元信息
    tokens_used: int = 0
    execution_time_seconds: float = 0.0
    retry_count: int = 0


class InformationMessage(BaseModel):
    """
    信息共享消息 —— Agent 主动向其他 Agent 广播信息。
    
    不是"请求"，而是"通知"。例如："我刚发现了这个重要的数据，大家注意"
    """
    message_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    message_type: str = "information"
    protocol_version: str = "1.0"
    timestamp: str = Field(default_factory=lambda: datetime.now().isoformat())
    
    sender: str
    receiver: str = "broadcast"  # 默认广播给所有人
    
    topic: str                    # 信息主题
    content: Any                  # 信息内容
    urgency: TaskPriority = TaskPriority.LOW


class ClarificationRequest(BaseModel):
    """
    澄清请求 —— Agent B 不理解 Agent A 的任务时需要更多信息。
    """
    message_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    message_type: str = "clarification"
    protocol_version: str = "1.0"
    timestamp: str = Field(default_factory=lambda: datetime.now().isoformat())
    
    sender: str
    receiver: str
    reply_to: str         # 对应原始 TaskRequest 的 message_id
    
    questions: list[str]  # 需要澄清的具体问题
    suggested_refinement: Optional[str] = None  # 建议的任务修改


# ============================================================
# 通信协议管理器
# ============================================================
class CommunicationManager:
    """
    通信管理器 —— 在 MessageBus 之上提供更高层的通信语义。
    
    它在基础的消息收发之上增加了：
    - 任务分配与结果收集的关联
    - 超时处理
    - 重试逻辑
    - 协议版本兼容性检查
    """

    def __init__(self, bus: MessageBus):
        self.bus = bus
        self._pending_tasks: dict[str, TaskRequest] = {}  # task_id → request
        self._task_results: dict[str, TaskResponse] = {}  # task_id → response

    async def delegate_task(
        self,
        sender_agent: Any,
        receiver_name: str,
        task_id: str,
        description: str,
        context: dict = None,
        timeout: float = 60.0,
    ) -> TaskResponse:
        """
        委派任务并等待结果。
        
        这个方法封装了完整的"发送请求 → 等待响应 → 处理超时"流程。
        """
        request = TaskRequest(
            sender=sender_agent.name,
            receiver=receiver_name,
            task_id=task_id,
            description=description,
            context=context or {},
        )
        self._pending_tasks[task_id] = request

        # 发送请求
        message = AgentMessage(
            id=request.message_id,
            sender=request.sender,
            receiver=request.receiver,
            msg_type=MessageType.REQUEST,
            content=request.model_dump_json(),
        )
        await self.bus.send(message)

        # 等待响应（带超时）
        start_time = datetime.now()
        while (datetime.now() - start_time).total_seconds() < timeout:
            response_msg = await self.bus.receive(sender_agent.name, timeout=1.0)
            if response_msg and response_msg.msg_type == MessageType.RESPONSE:
                try:
                    response = TaskResponse.model_validate_json(response_msg.content)
                    if response.task_id == task_id:
                        self._task_results[task_id] = response
                        return response
                except Exception:
                    continue

        # 超时
        timeout_response = TaskResponse(
            sender=receiver_name,
            receiver=sender_agent.name,
            reply_to=request.message_id,
            task_id=task_id,
            status=TaskStatus.FAILED,
            error=f"任务超时 (>{timeout}s)",
        )
        self._task_results[task_id] = timeout_response
        return timeout_response

    def get_all_results(self) -> dict[str, TaskResponse]:
        """获取所有已完成（或失败）的任务结果"""
        return dict(self._task_results)


if __name__ == "__main__":
    print("=== Agent 通信协议演示 ===\n")
    
    # 创建一条任务委派消息
    request = TaskRequest(
        sender="Orchestrator",
        receiver="DataAnalyst",
        task_id="task-001",
        description="分析 Q1 2024 销售数据中增长最快的三个品类",
        context={"start_date": "2024-01-01", "end_date": "2024-03-31"},
        priority=TaskPriority.HIGH,
        expected_format="json",
    )
    print("任务委派消息:")
    print(request.model_dump_json(indent=2))

    # 创建对应的响应消息
    response = TaskResponse(
        sender="DataAnalyst",
        receiver="Orchestrator",
        reply_to=request.message_id,
        task_id="task-001",
        status=TaskStatus.COMPLETED,
        result={
            "top_categories": [
                {"name": "电子产品", "growth": "35%", "revenue": "$1.2M"},
                {"name": "家居用品", "growth": "28%", "revenue": "$800K"},
                {"name": "运动装备", "growth": "22%", "revenue": "$650K"},
            ],
            "analysis_period": "2024-Q1",
            "methodology": "同比 (YoY) 增长率",
        },
        execution_time_seconds=3.5,
        tokens_used=1200,
    )
    print("\n任务响应消息:")
    print(response.model_dump_json(indent=2))
```

## 4. 角色定义与分工

### 4.1 从 System Prompt 到 Agent Profile

在第01章和 ai-application 中，我们学习了 System Prompt 的设计。在多 Agent 系统中，System Prompt 不仅定义"如何做"，更定义了"你是谁"和"你的边界是什么"。

```python
"""
Agent 角色定义的最佳实践。

一个完整的 Agent Profile 包含：
1. 身份 (Identity): 我是谁？我的专业背景是什么？
2. 职责 (Responsibilities): 我应该做什么？
3. 能力 (Capabilities): 我能使用哪些工具？
4. 边界 (Boundaries): 什么不是我的职责？遇到超出能力的事怎么办？
5. 协作偏好 (Collaboration): 我如何与其他 Agent 交互？
"""
from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Optional


class CommunicationStyle(str, Enum):
    """Agent 的沟通风格"""
    CONCISE = "concise"       # 简洁：直击要点，不废话
    DETAILED = "detailed"    # 详细：提供完整上下文
    SKEPTICAL = "skeptical"  # 审慎：总是质疑和验证
    SUPPORTIVE = "supportive"  # 支持性：倾向于帮助和补充


@dataclass
class AgentProfile:
    """
    Agent 的完整角色定义。
    
    这个 Profile 不仅在创建 Agent 时使用，也被协调者用来做任务分配决策。
    """
    # 1. 身份定义
    name: str                          # Agent 的唯一名称
    title: str                         # 职位/角色（如"资深数据分析师"）
    background: str                    # 背景描述（教育、经验、专长）
    
    # 2. 职责定义
    primary_duties: list[str]          # 主要职责
    secondary_duties: list[str] = field(default_factory=list)  # 次要职责
    
    # 3. 能力定义
    tools: list[str] = field(default_factory=list)         # 可用的工具
    knowledge_domains: list[str] = field(default_factory=list)  # 知识领域
    skill_level: dict[str, int] = field(default_factory=dict)   # 各项技能等级 (1-10)
    
    # 4. 边界定义
    out_of_scope: list[str] = field(default_factory=list)  # 明确不处理的事情
    escalation_policy: str = ""        # 遇到超出能力的事时的升级策略
    # 例如："如果遇到法律问题，标记为需要人工律师审查"
    
    # 5. 协作偏好
    communication_style: CommunicationStyle = CommunicationStyle.CONCISE
    preferred_collaborators: list[str] = field(default_factory=list)
    # 偏好与哪些 Agent 合作
    
    # 6. 输出偏好
    output_format: str = "text"        # 默认输出格式
    output_detail_level: str = "standard"  # "brief" | "standard" | "comprehensive"

    def to_system_prompt(self) -> str:
        """将 Profile 转化为 System Prompt"""
        prompt_parts = [
            f"# 身份",
            f"你是 {self.name}，一位 {self.title}。",
            f"背景: {self.background}",
            "",
            f"# 主要职责",
        ]
        for duty in self.primary_duties:
            prompt_parts.append(f"- {duty}")
        
        prompt_parts.append("")
        prompt_parts.append("# 能力")
        prompt_parts.append(f"知识领域: {', '.join(self.knowledge_domains)}")
        prompt_parts.append(f"可用工具: {', '.join(self.tools)}")
        
        prompt_parts.append("")
        prompt_parts.append("# 边界")
        prompt_parts.append(f"不处理: {', '.join(self.out_of_scope)}")
        prompt_parts.append(f"超纲处理: {self.escalation_policy}")
        
        prompt_parts.append("")
        prompt_parts.append(f"# 协作风格")
        prompt_parts.append(f"沟通方式: {self.communication_style.value}")
        prompt_parts.append(f"输出格式: {self.output_format}")
        prompt_parts.append(f"详细程度: {self.output_detail_level}")
        
        return "\n".join(prompt_parts)

    def match_score(self, task_description: str) -> float:
        """
        计算此 Agent 与任务的匹配度。
        
        简单的关键词匹配（实际中可用 embedding 相似度或 LLM 评估）。
        协调者使用此分数决定将任务分配给哪个 Agent。
        """
        task_lower = task_description.lower()
        score = 0.0
        
        # 知识领域匹配
        for domain in self.knowledge_domains:
            if domain.lower() in task_lower:
                score += 2.0
        
        # 职责匹配
        for duty in self.primary_duties:
            duty_keywords = duty.lower().split()
            matches = sum(1 for kw in duty_keywords if kw in task_lower)
            score += matches * 0.5
        
        # 如果任务明确在 out_of_scope 中，大幅降低分数
        for scope in self.out_of_scope:
            if scope.lower() in task_lower:
                score -= 5.0
        
        return max(0.0, score)
```

## 5. CrewAI 实战

### 5.1 CrewAI 简介

CrewAI 是一个专门用于构建多 Agent 协作系统的 Python 框架。它将我们前面讨论的概念（角色定义、任务委派、顺序/并行执行）封装成了简洁的 API。

CrewAI 的核心概念：

```
Crew (团队)
├── Agent 1 (角色 + 背景 + 目标)
│   └── Task 1, Task 2, ...
├── Agent 2 (角色 + 背景 + 目标)
│   └── Task 3, ...
└── Process (执行流程)
    ├── sequential (顺序执行)
    └── hierarchical (层级执行，类似主从模式)
```

### 5.2 安装和基础使用

```python
"""
CrewAI 实战 —— 从零构建多 Agent 写作团队。

CrewAI 官方文档: https://docs.crewai.com/
```

## 6. 多 Agent 系统的挑战

### 6.1 通信开销

多 Agent 系统中，Agent 之间的每次通信都是一次 LLM 调用。如果有 5 个 Agent，每个 Agent 在完成任务的过程中平均发送 3 条消息，那就是 15 次 LLM 调用——仅通信开销。相比之下，单 Agent 可能只需要 3-5 次调用。

**量化通信开销**：

```python
"""
多 Agent 通信开销分析器。

帮助决定多 Agent 架构是否值得：
如果通信开销 > 专业化带来的收益，说明不应该用多 Agent。
"""
from dataclasses import dataclass


@dataclass
class CommunicationCost:
    """通信开销分析"""
    num_agents: int
    avg_messages_per_agent: int
    avg_tokens_per_message: int
    cost_per_1k_tokens: float = 0.01  # GPT-4o 约 $5/M input tokens

    @property
    def total_tokens(self) -> int:
        """总通信 token 消耗"""
        return self.num_agents * self.avg_messages_per_agent * self.avg_tokens_per_message

    @property
    def total_cost(self) -> float:
        """总通信费用"""
        return self.total_tokens / 1000 * self.cost_per_1k_tokens

    def compare_with_single_agent(
        self,
        single_agent_tokens: int,
    ) -> dict:
        """与单 Agent 方案做对比"""
        return {
            "single_agent_tokens": single_agent_tokens,
            "multi_agent_communication_only": self.total_tokens,
            "overhead_ratio": f"{self.total_tokens / single_agent_tokens * 100:.1f}%",
            "conclusion": (
                "多 Agent 通信开销过高，建议单 Agent"
                if self.total_tokens > single_agent_tokens * 0.5
                else "通信开销在可接受范围内"
            ),
        }


if __name__ == "__main__":
    # 示例：5 个 Agent，平均每人发 3 条消息，每条 500 token
    cost = CommunicationCost(
        num_agents=5,
        avg_messages_per_agent=3,
        avg_tokens_per_message=500,
    )
    print(f"多 Agent 通信开销: {cost.total_tokens} tokens, ${cost.total_cost:.3f}")
    
    comparison = cost.compare_with_single_agent(single_agent_tokens=8000)
    for k, v in comparison.items():
        print(f"  {k}: {v}")
```

### 6.2 任务冲突

当多个 Agent 对同一问题给出不同结论时，需要解决冲突。这不是 bug，而是一个设计问题——你必须决定如何解决分歧。

```python
"""
任务冲突解决机制。

常见的冲突场景：
- Agent A 说"做 X"，Agent B 说"做 Y"，X 和 Y 互斥
- Agent A 和 Agent B 都声称某个任务是自己的职责（重叠）
- Agent A 的输出和 Agent B 的输出逻辑矛盾
"""
from enum import Enum


class ConflictResolutionStrategy(str, Enum):
    """冲突解决策略"""
    COORDINATOR_DECIDES = "coordinator_decides"  # 协调者仲裁
    MAJORITY_VOTE = "majority_vote"               # 多数投票
    WEIGHTED_VOTE = "weighted_vote"               # 加权投票（考虑专业度）
    ESCALATE_TO_HUMAN = "escalate_to_human"       # 升级到人类决策
    REVISE_AND_RETRY = "revise_and_retry"         # 让双方重新审视并修改


def resolve_conflict(
    conflict_type: str,
    agents_involved: list[str],
    their_positions: dict[str, str],
    context: str,
    strategy: ConflictResolutionStrategy = ConflictResolutionStrategy.WEIGHTED_VOTE,
) -> str:
    """
    解决多 Agent 冲突。
    
    参数:
        conflict_type: 冲突类型描述
        agents_involved: 涉及冲突的 Agent 名称列表
        their_positions: {agent_name: 其立场和论证}
        context: 冲突发生的上下文
        strategy: 解决策略
    
    返回:
        解决结果描述
    """
    # 这里应根据策略调用不同的解决方法
    # 以下为逻辑框架（实际中需要 LLM 参与）
    
    if strategy == ConflictResolutionStrategy.COORDINATOR_DECIDES:
        # 协调者理解各方意见后做最终决定
        return "协调者已裁决: ..."
    
    elif strategy == ConflictResolutionStrategy.WEIGHTED_VOTE:
        # 根据每个 Agent 与当前问题的专业相关度加权
        return "加权投票结果: ..."
    
    elif strategy == ConflictResolutionStrategy.ESCALATE_TO_HUMAN:
        # 将冲突详细信息呈现给人类决策者
        return "已升级给人类决策者: ..."
    
    return "未解决"


# 冲突解决日志
print("=== 多 Agent 冲突场景与解决 ===")
print("""
场景1: 范围重叠
  DataAnalyst: "这个分析应该我来做"
  BusinessAnalyst: "不，这属于商业分析范畴"
  解决: 协调者根据具体任务内容判定 → 如果是纯数据分析 → DataAnalyst
       → 如果涉及商业解读 → BusinessAnalyst

场景2: 结论矛盾
  Analyst A: "数据显示增长趋势"
  Analyst B: "数据显示下滑趋势"
  解决: 先检查双方使用的数据源是否一致 → 可能用了不同时间范围的数据
       → 要求双方标注数据来源和方法论 → 协调者比对后裁定

场景3: 责任推诿
  Task: "找出数据异常的原因"
  DataEngineer: "这是分析问题"
  Analyst: "这是数据质量问题"
  解决: 协调者介入 → 先让 DataEngineer 确认数据管线没问题
       → 再让 Analyst 确认分析逻辑没问题 → 通常问题在中间地带
""")
```

### 6.3 无限循环

多 Agent 系统的一个典型陷阱是"回复循环"——Agent A 要求 Agent B 修改，B 修改后发回 A，A 又要求修改...直到 token 耗尽。

```python
"""
多 Agent 无限循环的检测和预防。

常见循环模式：
1. 修改-审查循环: A 修改 → B 审查 → B 不满意 → A 再修改 → B 再审查 → ...
2. 澄清循环: A 要求 B 澄清 → B 澄清 → A 还是不清楚 → 继续要求 → ...
3. 协商循环: A 和 B 各持己见 → 来回争论 → ...
"""
from dataclasses import dataclass, field
from collections import defaultdict


@dataclass
class LoopDetector:
    """
    循环检测器 —— 监控 Agent 间的消息模式，识别潜在的循环。
    
    工作原理：
    1. 记录每个 Agent 对之间的最近 N 条消息
    2. 检测消息内容的相似度（高相似度 = 可能在循环）
    3. 检测回合数（同一主题的来回次数）
    """

    max_rounds_per_topic: int = 5   # 同一主题最多来回回合
    similarity_threshold: float = 0.8  # 内容相似度阈值
    message_history: list = field(default_factory=list)
    topic_rounds: dict = field(default_factory=lambda: defaultdict(int))

    def check(self, sender: str, receiver: str, content: str, topic: str) -> dict:
        """检查是否进入了循环"""
        self.message_history.append({
            "sender": sender,
            "receiver": receiver,
            "content": content[:200],  # 只记录前 200 字符的摘要
            "topic": topic,
        })

        # 统计同一主题的来回次数
        pair = tuple(sorted([sender, receiver]))
        key = f"{pair[0]}-{pair[1]}-{topic}"
        self.topic_rounds[key] += 1

        result = {
            "is_looping": False,
            "rounds": self.topic_rounds[key],
            "warning": None,
        }

        if self.topic_rounds[key] > self.max_rounds_per_topic:
            result["is_looping"] = True
            result["warning"] = (
                f"检测到 {sender} 和 {receiver} 在主题 '{topic}' 上"
                f"已来回 {self.topic_rounds[key]} 次，可能陷入循环。"
                f"建议：引入协调者仲裁或升级给人类决策者。"
            )

        return result

    def break_loop(self, sender: str, receiver: str, topic: str) -> str:
        """
        打破循环的策略：
        1. 引入第三方 Agent (协调者) 强制仲裁
        2. 对当前主题设置"最终决定"标志，禁止继续讨论
        3. 修改为不同的 prompt（改变思维角度）
        """
        return (
            f"循环打破方案: 协调者已介入 {sender} 和 {receiver} 关于 '{topic}' 的讨论。"
            f"协调者做出最终决定，双方必须接受。如需申诉，请升级给人类。"
        )


if __name__ == "__main__":
    detector = LoopDetector(max_rounds_per_topic=3)

    # 模拟一个循环场景
    for i in range(5):
        result = detector.check(
            sender="分析师",
            receiver="写作者",
            content="请再修改一下这部分的分析结论...",
            topic="销售报告结论",
        )
        if result["is_looping"]:
            print(f"⚠ {result['warning']}")
            print(f"  {detector.break_loop('分析师', '写作者', '销售报告结论')}")
            break
```

## 常见错误

### 错误 1: 多 Agent 过度使用 —— "能用单 Agent 解决的问题偏要用多 Agent"

```python
# 错误：用户问"今天天气怎么样"
# → 启动 3 个 Agent: 气象数据 Agent + 自然语言 Agent + 格式化 Agent
# → 开销是单 Agent 的 5-10 倍，结果没有更好

# 正确：先用决策树判断是否需要多 Agent
# 简单任务 → 单 Agent
```

### 错误 2: Agent 角色定义过于宽泛

```python
# 错误 Profile:
# "你是一个全栈工程师，负责前端、后端、数据库、DevOps、测试、文档..."
# → 这等于没有分工，失去了多 Agent 的意义

# 正确 Profile:
# "你是一个 React 前端开发者，专注于 UI 组件开发、状态管理、性能优化"
```

### 错误 3: Worker 之间的隐式耦合

```python
# 错误：Worker A 的输出直接给 Worker B，但输出格式没约定
# Worker A 返回 {"analysis": "..."} 
# Worker B 期望 {"data": [...]} 
# → B 解析失败

# 正确：每个 Worker 之间通过协调者通信，协调者负责格式转换和适配
```

### 错误 4: 协调者成为单点瓶颈

```python
# 错误：所有通信都要经过协调者
# 10 个 Worker 等待协调者一个一个处理请求
# → 协调者的处理速度成为整个系统的瓶颈

# 正确：协调者只做关键决策（任务分配、冲突仲裁）
# Worker 之间可以直接交换信息（通过消息总线）
```

### 错误 5: 无限制的消息传递导致 Token 爆炸

```python
# 错误：Agent A 把完整 5000 字的上下文全部发给 Agent B
# Agent B 又附加了 3000 字注释发给 Agent C
# → 消息越来越长，token 消耗指数增长

# 正确：Agent 发送消息时只传递必要信息
# 长文档通过共享存储引用（发 URL 或文档 ID，不发全文）
```

### 错误 6: 忘记设置最大回合数

```python
# 错误：Agent 可以无限次来回讨论
# → 当两个 Agent 意见不一致且都坚持己见时 → 无限对话

# 正确：为每个讨论主题设置 `max_rounds`
# 超过上限 → 协调者强制仲裁或升级给人类
```

### 错误 7: 并行任务之间存在隐式写冲突

```python
# 错误：Analyst 和 Writer 同时运行
# 但 Writer 的任务描述中引用了 "Analyst 的结论"
# 而 Analyst 还没完成 → Writer 基于"幻影"数据写作

# 正确：明确任务依赖：Writer 的 depends_on=["Analyst"]
# CrewAI 中确保 Context 正确传递
```

## 本章小结

本章系统学习了多智能体协作的知识体系：

| 知识点 | 核心要点 |
|--------|----------|
| 为什么多 Agent | 单 Agent 的认知负载瓶颈、技能深度不足、无法真正并行决策 |
| 架构决策树 | 根据领域多样性、并行潜力、质量要求判断是否需要多 Agent |
| 主从模式 | 协调者集中管理，Worker 专注执行，最常用也最容易调试 |
| 平等协商模式 | Agent 自由通信，通过消息总线和共识协议自主协调 |
| 流水线模式 | 串行处理，每阶段有明确的输入输出契约和验证规则 |
| 通信协议 | 结构化消息（TaskRequest/TaskResponse）、异步优先、可审计 |
| 角色定义 | AgentProfile 六要素：身份、职责、能力、边界、协作偏好、输出格式 |
| CrewAI | 多 Agent 框架，支持 sequential 和 hierarchical 两种执行模式 |
| 通信开销 | 多 Agent 通信本身就是 LLM 调用消耗，需量化评估 |
| 任务冲突 | 通过加权投票、协调者仲裁、人类升级等策略解决 |
| 无限循环 | 设置 max_rounds、引入循环检测器、强制仲裁 |

下一章将学习 LangGraph 工作流引擎——当 Agent 逻辑变得复杂到无法用简单的"调一个工具 → 返回"模式描述时，如何用状态图来建模和执行复杂的 Agent 工作流。
