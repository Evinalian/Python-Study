## 1. 为什么需要任务规划

### 1.1 复杂任务的本质：不可原子化

在 ReAct 模式中，Agent 遵循"观察 → 思考 → 行动 → 观察 → ..."的循环：每一步只决定下一步做什么。这种"走一步看一步"的策略在简单任务中表现良好——用户问"北京今天天气怎么样"，Agent 调用天气 API，返回结果，结束。

但现实中的复杂任务往往具有以下特征：

- **多步骤**：不是调用一次工具就能完成。例如"分析过去一周的销售数据，找出增长最快的三个品类，给每个品类写一段分析报告，最后汇总成一封邮件发给老板"
- **有依赖关系**：步骤 B 需要步骤 A 的结果才能开始。例如你需要先拿到数据，才能分析数据；你需要先分析完，才能写报告
- **有不确定性**：某些步骤可能失败，后续步骤需要根据失败情况调整
- **有并行机会**：某些步骤之间没有依赖，可以同时执行

如果让 ReAct Agent 处理这种复杂任务，会遇到什么问题？

```
用户: 分析销售数据并写报告发邮件
Agent (ReAct):
  Thought: 需要分析销售数据，我先获取数据
  Action: get_sales_data()
  Observation: 返回了100MB的原始数据...
  Thought: 数据量太大了，我需要先过滤
  Action: filter_data(criteria="最近一周")
  Observation: 过滤后还有10MB...
  Thought: 接下来要分析了...但我应该做什么分析？用户到底想看什么？
  ...(每一步都重新思考，没有全局视角)
```

问题很明显：**ReAct 会陷入局部最优——每一步看起来合理，但缺少对全局的把握**。它可能在中途"迷路"，忘记最初的目标，或者重复执行不必要的步骤。

### 1.2 ReAct vs Plan-Execute：两种思维范式

从认知科学的角度，这两种模式对应人类解决问题的两种思维方式：

| 维度 | ReAct（边想边做） | Plan-Execute（先想后做） |
|------|-------------------|--------------------------|
| 决策粒度 | 每步一决策 | 先出全局计划，再按计划执行 |
| 全局观 | 无，只看当前步 | 有，先理解整体 |
| 适应性 | 强，每一步都可调整 | 通过重规划机制实现 |
| Token 消耗 | 每一步都调用 LLM 做决策 | 计划阶段一次大调用，执行阶段轻量 |
| 适合任务 | 简单、步骤少的任务 | 复杂、多步骤、有依赖的任务 |
| 失败模式 | 中途"迷路"，目标漂移 | 计划不切实际，需重规划 |
| 可审计性 | 难审计，决策分散 | 可审计，计划是显式产物 |

用更直观的方式理解：

```
ReAct 模式:
  执行步骤1 → 观察 → 思考 → 执行步骤2 → 观察 → 思考 → 执行步骤3 → ...
  ┌─┐   ┌─┐   ┌─┐
  │?│   │?│   │?│   每步都不知道下一步做什么
  └─┘   └─┘   └─┘

Plan-Execute 模式:
  先制定计划: [步骤1, 步骤2, 步骤3, ..., 步骤N]  ← 一次性看清全局
  然后逐步执行:
  步骤1 ✅ → 步骤2 ✅ → 步骤3 ❌ → 重规划 → 步骤3' ✅ → 步骤4 ✅ → ...
  ┌───────────┐
  │ 清晰的全局 │   知道自己在什么位置，还剩多少
  └───────────┘
```

这并不意味着 Plan-Execute 一定优于 ReAct。事实上，**最强大的 Agent 往往结合两者**——先有一个粗略的计划（Plan-Execute 的全局视图），执行过程中再使用 ReAct 应对每一步的细节（ReAct 的灵活性）。本章重点讲解 Plan-Execute 模式的机制，但理解 ReAct（第03章）是前提——许多 Plan-Execute 系统的执行单元本身就使用 ReAct。

### 1.3 规划在 Agent 架构中的位置

回顾第01章的 Agent 基础架构：

```
┌─────────────────────────────────────────────────┐
│                  Agent 架构                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │  感知     │→│  规划     │→│  执行     │       │
│  │ Perceive  │  │  Plan    │  │ Execute   │       │
│  └──────────┘  └──────────┘  └──────────┘       │
│                     ↑              │              │
│                     │    ┌──────────┐            │
│                     └────│  观察    │←────────────┘
│                          │ Observe   │
│                          └──────────┘
└─────────────────────────────────────────────────┘
```

本章聚焦于"规划"模块。它接收感知层的输出（对任务和环境的理解），产生一个结构化的执行计划，传递给执行层。规划模块的核心职责是回答三个问题：

1. **做什么**（What）：任务需要完成哪些子目标
2. **什么顺序**（When）：这些子目标的执行顺序是什么
3. **用什么**（How）：每个子目标需要调用哪些工具

## 2. Plan-and-Execute 模式详解

### 2.1 核心流程

Plan-and-Execute 模式的核心思想可以用一句话概括：**规划器负责"做什么"，执行器负责"怎么做"**。

完整流程分为三个阶段：

```
阶段一：规划 (Planning)
  输入: 用户任务描述 + 可用工具列表 + 环境上下文
  处理: LLM 分析任务 → 分解为子任务 → 确定执行顺序 → 识别依赖
  输出: 结构化的任务计划 (Plan)

阶段二：执行 (Execution)
  输入: 任务计划 + 当前环境状态
  处理: 按计划顺序执行每个子任务 → 收集结果 → 更新计划状态
  输出: 每个子任务的执行结果

阶段三：重规划 (Re-planning)
  触发: 某个子任务执行失败 OR 结果与预期不符 OR 发现新的信息
  处理: LLM 分析失败原因 / 新信息 → 修改后续计划
  输出: 更新后的任务计划
```

这三个阶段不是线性的——执行阶段可能触发重规划，重规划后回到执行阶段，形成动态循环。

```
流程图:

     ┌──────────┐
     │  用户任务  │
     └────┬─────┘
          │
          ▼
     ┌──────────┐
     │ 阶段1:   │  调用 LLM 分析任务
     │ 规划     │  输出结构化计划
     └────┬─────┘
          │ 计划 = [子任务A, 子任务B, 子任务C]
          ▼
     ┌──────────┐
     │ 阶段2:   │  按顺序执行子任务
     │ 执行     │←─────────┐
     └────┬─────┘          │
          │               │
          ├── 全部成功 ────┘
          │               │
          ├── 部分失败 ──→ 阶段3: 重规划
          │               │
          │               │ 分析失败原因
          │               │ 修改部分计划
          │               │ 标记已完成的子任务
          │               └──→ 回到阶段2
          │
          ▼
     ┌──────────┐
     │ 交付结果  │
     └──────────┘
```

### 2.2 计划的表示：结构化设计

计划不是一个自然语言的"待办列表"，而是一个结构化的数据结构。一个良好设计的计划包含以下字段：

```python
"""
计划 (Plan) 的数据结构设计。

一个好的计划表示需要满足：
1. 机器可解析（程序能读懂并自动执行）
2. 人类可读（方便调试和审计）
3. 可跟踪状态（知道执行到哪一步了）
4. 可修改（重规划时能局部更新）
"""
from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Optional
import uuid


class TaskStatus(Enum):
    """子任务执行状态"""
    PENDING = "pending"       # 待执行
    IN_PROGRESS = "in_progress"  # 执行中
    COMPLETED = "completed"   # 已完成
    FAILED = "failed"         # 执行失败
    SKIPPED = "skipped"       # 因依赖失败而跳过
    WAITING = "waiting"       # 等待依赖完成


@dataclass
class SubTask:
    """
    子任务 —— 计划的最小执行单元。
    
    每个子任务包含：
    - 要做什么（description, action）
    - 需要什么输入（depends_on, inputs）
    - 会产生什么输出（output_schema, expected_output_key）
    - 执行状态（status, result, error）
    """
    # 基本信息
    id: str                        # 唯一标识符
    description: str               # 人类可读的描述（"从数据库查询销售数据"）
    
    # 执行信息
    action: str                    # 要执行的工具/函数名（"get_sales_data"）
    action_args: dict = field(default_factory=dict)  # 工具参数
    
    # 依赖信息
    depends_on: list[str] = field(default_factory=list)  # 依赖的子任务 ID 列表
    # 例子: subtask_3.depends_on = ["subtask_1", "subtask_2"]
    # 表示 subtask_3 需要 subtask_1 和 subtask_2 都完成后才能开始
    
    # 输入输出
    inputs: dict[str, str] = field(default_factory=dict)
    # 例子: {"data": "$subtask_1.result.data", "date_range": "$context.start_date"}
    # 用 $ 前缀引用其他子任务的输出或上下文变量
    output_key: str = ""           # 该子任务结果的存储键名
    
    # 状态追踪
    status: TaskStatus = TaskStatus.PENDING
    result: Any = None             # 执行成功的结果
    error: Optional[str] = None    # 执行失败的错误信息
    
    # 元信息
    priority: int = 0              # 优先级（数字越大优先级越高）
    max_retries: int = 1           # 最大重试次数
    retry_count: int = 0           # 已重试次数
    timeout_seconds: int = 60      # 超时时间
    created_at: str = ""           # 创建时间
    started_at: str = ""           # 开始执行时间
    completed_at: str = ""         # 完成时间


@dataclass
class Plan:
    """
    执行计划 —— 包含一组有序的子任务及其依赖关系。
    
    一个计划在创建后不是静止的 —— 执行过程中子任务的状态在不断变化，
    重规划时可能会添加、删除或修改子任务。
    """
    # 标识
    id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    name: str = ""                 # 计划名称（来自用户任务）
    
    # 核心内容
    tasks: list[SubTask] = field(default_factory=list)  # 所有子任务
    
    # 元信息
    original_task: str = ""        # 原始用户任务描述
    created_at: str = ""           # 计划创建时间
    updated_at: str = ""           # 最后更新（含重规划）时间
    version: int = 1               # 计划版本（每次重规划 +1）
    revision_history: list[str] = field(default_factory=list)
    # 重规划历史记录，便于审计
    
    # 全局上下文
    context: dict = field(default_factory=dict)  # 计划级别的共享上下文
    # 例子: {"start_date": "2024-01-01", "end_date": "2024-01-07"}
    
    def get_ready_tasks(self) -> list[SubTask]:
        """获取所有可以立即开始执行的任务（依赖全部满足的 PENDING 任务）"""
        completed_ids = {
            t.id for t in self.tasks if t.status == TaskStatus.COMPLETED
        }
        ready = []
        for task in self.tasks:
            if task.status != TaskStatus.PENDING:
                continue
            # 检查所有依赖是否都已完成
            if all(dep in completed_ids for dep in task.depends_on):
                ready.append(task)
        return ready
    
    def get_blocked_tasks(self) -> list[SubTask]:
        """获取所有被依赖阻塞的任务"""
        completed_ids = {
            t.id for t in self.tasks if t.status == TaskStatus.COMPLETED
        }
        blocked = []
        for task in self.tasks:
            if task.status != TaskStatus.PENDING:
                continue
            if not all(dep in completed_ids for dep in task.depends_on):
                blocked.append(task)
        return blocked
    
    def is_complete(self) -> bool:
        """判断计划是否执行完毕（所有任务最终状态确定）"""
        terminal_statuses = {TaskStatus.COMPLETED, TaskStatus.FAILED, TaskStatus.SKIPPED}
        return all(t.status in terminal_statuses for t in self.tasks)
    
    def progress(self) -> dict:
        """获取计划执行进度"""
        total = len(self.tasks)
        completed = sum(1 for t in self.tasks if t.status == TaskStatus.COMPLETED)
        failed = sum(1 for t in self.tasks if t.status == TaskStatus.FAILED)
        in_progress = sum(1 for t in self.tasks if t.status == TaskStatus.IN_PROGRESS)
        return {
            "total": total,
            "completed": completed,
            "failed": failed,
            "in_progress": in_progress,
            "pending": total - completed - failed - in_progress,
            "percentage": round(completed / total * 100, 1) if total > 0 else 0,
        }


if __name__ == "__main__":

### 2.3 规划器的核心：LLM 如何生成计划

规划器是整个系统的大脑。它的输入是"用户想做什么"的自然语言描述，输出是结构化的计划。这个转化过程由 LLM 完成，核心 Prompt 设计如下：

```python

规划器 —— 用 LLM 将用户任务转化为结构化执行计划。

规划器 Prompt 的关键要素：
1. 角色设定：你是一个任务规划专家
2. 可用工具：列出所有可调用的工具及其参数
3. 输出格式：严格定义 JSON Schema
4. 规划原则：分解粒度、依赖判断、并行识别的规则
5. 示例：至少一个完整的计划生成示例
"""

def build_planner_prompt(user_task: str) -> str:

    # 生成工具列表描述
    tools_desc = "可用工具列表：\n"
    for name, info in AVAILABLE_TOOLS.items():
        tools_desc += f"\n- **{name}**: {info['description']}\n"
        tools_desc += f"  参数: {json.dumps(info['parameters'], ensure_ascii=False)}\n"

1. **粒度控制**: 每个子任务应该是一个单一的、可执行的动作（一次工具调用）。不要合并多个工具调用到一个子任务中。
2. **依赖分析**: 如果子任务 B 需要子任务 A 的输出作为输入，则 B 依赖 A。在 depends_on 字段中正确标注。
3. **并行识别**: 如果多个子任务都只依赖同一个已完成的任务，且它们之间没有数据依赖，则它们可以并行执行。
4. **先搜索后分析**: 如果任务需要外部信息（市场趋势等），先安排搜索任务获取信息。
5. **验证步骤**: 对于可能失败的任务（如数据查询、网络请求），考虑安排验证/重试逻辑。
6. **输出复用**: 如果一个子任务的输出被多个后续任务使用，给它一个有意义的 output_key。

## 3. 任务分解的策略

### 3.1 层次分解：从粗到细

任务分解不是一次性把任务拆成最细粒度的子任务。更好的做法是**层次分解**——先拆成大块，再细化每个大块。

```
层次1（目标层）: "分析销售数据并生成报告"
  │
  ├── 层次2（阶段层）:
  │   ├── 数据获取阶段
  │   ├── 数据处理阶段
  │   ├── 分析阶段
  │   └── 报告生成阶段
  │
  └── 层次3（任务层）:
      ├── 数据获取:
      │   ├── 查询销售数据库
      │   └── 查询市场趋势（外部数据）
      ├── 数据处理:
      │   ├── 数据清洗
      │   ├── 按类别汇总
      │   └── 计算增长率
      ├── 分析:
      │   ├── 分析品类A
      │   ├── 分析品类B
      │   └── 分析品类C
      └── 报告生成:
          ├── 撰写邮件
          └── 发送邮件
```

层次分解的好处：

1. **降低 LLM 的认知负担**：一次规划 3 个阶段，比一次规划 15 个子任务容易得多
2. **支持增量规划**：执行完"数据获取阶段"后，可以根据实际获取的数据调整"分析阶段"的规划
3. **便于人工审查**：用户可以 review 阶段层的计划，而不需要看每一个原子操作

    Phase 1 实际产出: 获取了 5000 行销售数据和 10 篇市场趋势文章

    ├── Task 2.1: 数据清洗（知道有 5000 行，可能需去重）

    """)

### 3.2 依赖分析：谁必须在谁之前

依赖关系是任务分解中最关键的分析。错误的依赖标注会导致两种后果：

- **漏标依赖**：B 依赖 A 但没有标注 → B 在 A 完成前执行 → B 缺少输入数据 → 执行失败
- **过度依赖**：B 不依赖 A 但标注了依赖 → B 被不必要地阻塞 → 浪费并行执行机会

依赖的类型：

| 依赖类型 | 描述 | 示例 |
|----------|------|------|
| 数据依赖 | B 需要 A 的输出作为输入 | 分析数据 依赖 获取数据 |
| 资源依赖 | B 需要 A 释放的资源 | 写入数据库 依赖 锁定数据库 |
| 因果依赖 | B 的合理性基于 A 的结果 | 发邮件 依赖 写邮件 |
| 条件依赖 | B 是否执行取决于 A 的结果 | 重试 依赖 首次尝试失败 |

```python
"""
依赖分析器 —— 自动检测计划中的依赖问题。

一个好的计划应该在执行前通过以下检查：
1. 不存在循环依赖（A→B→A）
2. 不存在对不存在任务的依赖
3. 所有非初始任务至少有一个依赖（除非确实是独立的）
4. 依赖链的终点可达（没有"孤岛"任务）
"""
from collections import defaultdict, deque


class DependencyAnalyzer:
    """
    依赖分析器 —— 验证和优化计划中的依赖关系。
    """

    @staticmethod
    def detect_cycles(plan) -> list[list[str]]:
        """
        检测依赖图中的环。使用 DFS 染色法。
        
        环的存在会导致部分任务永远无法执行（互相等待）。
        
        返回: 检测到的所有环，每个环是一个 task_id 列表
        """
        # 构建邻接表：A → [B, C] 表示 B 和 C 依赖 A
        # （注意方向：如果 B 依赖 A，则边是 A→B）
        graph = defaultdict(list)
        for task in plan.tasks:
            for dep_id in task.depends_on:
                graph[dep_id].append(task.id)

        WHITE, GRAY, BLACK = 0, 1, 2
        color = {t.id: WHITE for t in plan.tasks}
        parent = {}
        cycles = []

        def dfs(node_id):
            color[node_id] = GRAY
            for neighbor_id in graph.get(node_id, []):
                if color.get(neighbor_id) == GRAY:
                    # 发现环：回溯路径
                    cycle = [neighbor_id]
                    curr = node_id
                    while curr != neighbor_id:
                        cycle.append(curr)
                        curr = parent.get(curr, "")
                    cycle.append(neighbor_id)
                    cycle.reverse()
                    cycles.append(cycle)
                elif color.get(neighbor_id) == WHITE:
                    parent[neighbor_id] = node_id
                    dfs(neighbor_id)
            color[node_id] = BLACK

        for task in plan.tasks:
            if color[task.id] == WHITE:
                dfs(task.id)

        return cycles

    @staticmethod
    def detect_orphan_tasks(plan) -> list[str]:
        """
        检测"孤儿"任务：没有被任何其他任务依赖，也不依赖任何任务。
        
        孤儿任务可能意味着：
        - 该任务是独立的顶层工作（合理）
        - 该任务应该依赖某个任务但忘了标注（问题）
        """
        all_ids = {t.id for t in plan.tasks}
        referenced_as_dependency = set()
        for t in plan.tasks:
            for dep_id in t.depends_on:
                referenced_as_dependency.add(dep_id)

        orphans = []
        for t in plan.tasks:
            has_deps = len(t.depends_on) > 0
            is_depended_on = t.id in referenced_as_dependency
            if not has_deps and not is_depended_on and len(plan.tasks) > 1:
                orphans.append(t.id)

        return orphans

    @staticmethod
    def find_parallel_groups(plan) -> list[list[str]]:
        """
        识别可以并行执行的任务组。
        
        规则：如果多个任务具有完全相同的依赖集，且它们之间没有相互依赖，
        则它们可以并行执行。
        """
        # 按依赖集分组
        groups = defaultdict(list)
        for task in plan.tasks:
            dep_key = tuple(sorted(task.depends_on))
            groups[dep_key].append(task.id)

        # 筛选出包含多个任务的组
        parallel_groups = [
            task_ids for task_ids in groups.values() if len(task_ids) > 1
        ]
        return parallel_groups

    @staticmethod
    def validate(plan) -> dict:
        """
        对计划执行所有检查，返回检查报告。
        """
        report = {
            "total_tasks": len(plan.tasks),
            "has_cycles": False,
            "cycles": [],
            "orphan_tasks": [],
            "parallel_groups": [],
            "unresolved_dependencies": [],
            "warnings": [],
        }

        # 1. 循环检查
        cycles = DependencyAnalyzer.detect_cycles(plan)
        if cycles:
            report["has_cycles"] = True
            report["cycles"] = cycles
            report["warnings"].append(
                f"发现 {len(cycles)} 个循环依赖！"
                f"循环中的任务将永远等待。请检查并打破循环。"
            )

        # 2. 孤儿任务检查
        orphans = DependencyAnalyzer.detect_orphan_tasks(plan)
        report["orphan_tasks"] = orphans
        if orphans:
            report["warnings"].append(
                f"发现 {len(orphans)} 个孤立任务: {orphans}。"
                f"请确认它们确实是独立任务还是缺少依赖标注。"
            )

        # 3. 依赖解析检查
        all_ids = {t.id for t in plan.tasks}
        for t in plan.tasks:
            for dep_id in t.depends_on:
                if dep_id not in all_ids:
                    report["unresolved_dependencies"].append(
                        {"task": t.id, "missing_dep": dep_id}
                    )
                    report["warnings"].append(
                        f"{t.id} 依赖了不存在的任务 {dep_id}！"
                    )

        # 4. 并行机会识别
        report["parallel_groups"] = DependencyAnalyzer.find_parallel_groups(plan)
        if report["parallel_groups"]:
            report["warnings"].append(
                f"发现 {len(report['parallel_groups'])} 组可并行执行的任务，"
                f"可提升执行效率。"
            )

        return report


if __name__ == "__main__":
    # 构建一个包含问题的计划来测试
    from dataclasses import dataclass

    # 模拟一个简单的计划结构用于测试
    @dataclass
    class SimplePlan:
        tasks: list

    @dataclass
    class SimpleTask:
        id: str
        depends_on: list

    good_plan = SimplePlan(tasks=[
        SimpleTask("A", []),
        SimpleTask("B", ["A"]),
        SimpleTask("C", ["A"]),    # B 和 C 可并行（依赖相同）

    # 有循环依赖的计划
    cycle_plan = SimplePlan(tasks=[
        SimpleTask("A", ["C"]),    # A 依赖 C

        SimpleTask("C", ["B"]),    # C 依赖 B → A→B→C→A 的环
    ])

    print("=== 正常计划检查 ===")

    print("\n=== 有循环的计划检查 ===")
    report2 = DependencyAnalyzer.validate(cycle_plan)

### 3.3 并行子任务识别

并行执行是提升 Agent 效率的关键。在任务分解时，规划器应该有意识地识别哪些子任务可以并行。

```python
"""
并行子任务识别的规则和策略。

识别并行的关键问题："任务 A 和任务 B 之间有没有数据依赖？"

如果没有——即 B 不需要 A 的输出作为输入——它们就可以并行。
但有例外：
- 共享资源的写冲突（两个任务写同一个文件）
- 逻辑上的先后顺序要求（即使没有数据依赖）
"""
import asyncio
import time
from typing import Callable


class ParallelExecutor:
    """
    并行执行器 —— 在依赖允许的情况下并行执行子任务。
    
    核心逻辑：
    1. 持续检查哪些任务的依赖已全部满足
    2. 将所有"就绪"任务同时启动
    3. 每完成一个任务，重新检查是否有新任务就绪
    4. 重复直到所有任务完成或无法继续
    """

    def __init__(self, max_parallel: int = 5):
        self.max_parallel = max_parallel  # 最大并行数（避免资源耗尽）

    async def execute_plan(
        self,
        plan,
        tool_executor: Callable,
        on_progress: Callable = None,
    ) -> dict:
        """
        并行执行一个计划。
        
        参数:
            plan: Plan 对象
            tool_executor: 执行单个子任务的函数，签名为 async def(subtask) -> result
            on_progress: 进度回调，签名为 def(plan, completed_task)
        
        返回:
            {task_id: result} 字典
        """
        results = {}  # task_id → result
        running_tasks = {}  # task_id → asyncio.Task
        completed_ids = set()

        def get_ready_tasks():
            """获取当前所有就绪的任务（未完成且依赖都满足）"""
            ready = []
            for task in plan.tasks:
                if task.id in completed_ids:
                    continue
                if task.id in running_tasks:
                    continue
                if task.status == TaskStatus.FAILED:
                    continue
                if all(dep in completed_ids for dep in task.depends_on):
                    ready.append(task)
            return ready

        async def execute_single(task):
            """执行单个子任务"""
            task.status = TaskStatus.IN_PROGRESS
            try:
                result = await tool_executor(task)
                task.status = TaskStatus.COMPLETED
                task.result = result
                results[task.id] = result
                completed_ids.add(task.id)
                if on_progress:
                    on_progress(plan, task)
                return task.id, True, result
            except Exception as e:
                task.status = TaskStatus.FAILED
                task.error = str(e)
                completed_ids.add(task.id)  # 失败也算"完成"（状态终态）
                if on_progress:
                    on_progress(plan, task)
                return task.id, False, str(e)

        # 主循环：持续执行直到所有任务到达终态
        while not plan.is_complete():
            ready = get_ready_tasks()

            if not ready and not running_tasks:
                # 没有就绪任务也没有运行中的任务=死锁
                # 可能是因为失败任务阻塞了依赖链
                print("⚠ 检测到可能的死锁：无就绪任务且无运行中任务")
                self._handle_deadlock(plan, results)
                break

            # 启动就绪任务（受 max_parallel 限制）
            available_slots = self.max_parallel - len(running_tasks)
            for task in ready[:available_slots]:
                running_tasks[task.id] = asyncio.create_task(execute_single(task))

            if not running_tasks:
                # 没有就绪任务，等待也无效
                break

            # 等待任意一个任务完成
            done, _ = await asyncio.wait(
                running_tasks.values(),
                return_when=asyncio.FIRST_COMPLETED,
            )

            # 处理完成的任务
            for done_task in done:
                task_id, success, _ = done_task.result()
                del running_tasks[task_id]

        return results

    def _handle_deadlock(self, plan, results: dict):
        """
        死锁处理：当没有任务可执行时，分析原因。
        
        常见原因：
        1. 某个依赖任务失败了 → 后续依赖它的任务都无法执行
        2. 循环依赖 → 互相等待
        
        策略：将失败任务的"必须"依赖降级为"可选"，
        或者标记依赖失败任务的所有下游任务为 SKIPPED。
        """
        for task in plan.tasks:
            if task.status == TaskStatus.PENDING:
                # 检查是否有依赖任务失败了
                failed_deps = [
                    dep_id for dep_id in task.depends_on
                    if dep_id in results and results.get(dep_id) is False
                ]
                if failed_deps:
                    task.status = TaskStatus.SKIPPED
                    task.error = f"跳过了：依赖任务 {failed_deps} 执行失败"


# 演示并行执行
async def demo_parallel_execution():

        # 模拟不同的执行时间

## 4. 子任务依赖图

### 4.1 依赖图的可视化表示

- 哪些任务在关键路径上（延迟它们会延迟整个计划）

- 某个任务失败会影响多少下游任务

```
任务依赖图（Mermaid 格式 — 可在支持 Mermaid 的工具中渲染）:

    T3 --> T4[task_4: 分析品类A]

    并行区间: T4, T5, T6 可同时执行

### 4.2 关键路径分析

关键路径（Critical Path）是依赖图中从起点到终点最长的路径。关键路径决定了整个计划的最短完成时间——如果你不缩短关键路径上的任务，就无法缩短总时间。

```python
"""
关键路径分析 —— 计算计划的最短完成时间和瓶颈任务。

算法：拓扑排序 + 动态规划
1. 按拓扑序计算每个任务的最早完成时间
2. 反向传播计算每个任务的最晚完成时间
3. 总松弛时间为0的任务 = 关键路径上的任务
"""
from collections import deque


class CriticalPathAnalyzer:
    """
    关键路径分析器。
    使用 CPM（Critical Path Method）算法。
    """

    @staticmethod
    def analyze(plan) -> dict:
        """
        分析计划的关键路径。
        
        返回:
        {
            "critical_path": ["task_1", "task_2", ...],  # 关键路径上的任务ID
            "total_duration": 4.4,  # 计划最短完成时间
            "task_details": {
                "task_1": {
                    "earliest_start": 0.0,
                    "earliest_finish": 1.0,
                    "latest_start": 0.0,
                    "latest_finish": 1.0,
                    "slack": 0.0,  # 松弛时间，0=在关键路径上
                },
                ...
            }
        }
        """
        # 简化：假设每个任务的估算耗时
        # 实际中这个耗时来自历史数据或 LLM 估算
        task_map = {t.id: t for t in plan.tasks}
        durations = {}
        for t in plan.tasks:
            # 这里简化处理，实际中应有估算机制
            durations[t.id] = 1.0  # 默认 1 单位时间

        # 1. 拓扑排序
        in_degree = {t.id: len(t.depends_on) for t in plan.tasks}
        graph = {t.id: [] for t in plan.tasks}
        for t in plan.tasks:
            for dep_id in t.depends_on:
                if dep_id in graph:
                    graph[dep_id].append(t.id)

        queue = deque([tid for tid, deg in in_degree.items() if deg == 0])
        topo_order = []

        while queue:
            node = queue.popleft()
            topo_order.append(node)
            for neighbor in graph.get(node, []):
                in_degree[neighbor] -= 1
                if in_degree[neighbor] == 0:
                    queue.append(neighbor)

        if len(topo_order) != len(plan.tasks):
            return {"error": "依赖图中存在循环，无法计算关键路径"}

        # 2. 正向传播：计算最早开始/完成时间
        earliest_start = {tid: 0.0 for tid in task_map}
        earliest_finish = {}

        for tid in topo_order:
            ef = earliest_start[tid] + durations[tid]
            earliest_finish[tid] = ef
            # 更新后继任务的最早开始时间
            for neighbor in graph.get(tid, []):
                earliest_start[neighbor] = max(
                    earliest_start[neighbor], ef
                )

        project_duration = max(earliest_finish.values())

        # 3. 反向传播：计算最晚开始/完成时间
        latest_finish = {tid: project_duration for tid in task_map}
        latest_start = {}

        for tid in reversed(topo_order):
            lf = latest_finish[tid]
            # 该任务所有后继的最晚开始时间的最小值
            for neighbor in graph.get(tid, []):
                lf = min(lf, latest_start.get(neighbor, project_duration))
            latest_finish[tid] = lf
            ls = lf - durations[tid]
            latest_start[tid] = ls
            # 更新前驱任务的最晚完成时间
            for dep_id in task_map[tid].depends_on:
                if dep_id in latest_finish:
                    latest_finish[dep_id] = min(
                        latest_finish[dep_id], ls
                    )

        # 4. 计算松弛时间和关键路径
        task_details = {}
        critical_path = []

        for tid in task_map:
            slack = latest_start[tid] - earliest_start[tid]
            detail = {
                "earliest_start": earliest_start[tid],
                "earliest_finish": earliest_finish[tid],
                "latest_start": latest_start[tid],
                "latest_finish": latest_finish[tid],

## 5. 动态重规划

### 5.1 什么时候需要重规划

计划不是一成不变的。执行过程中可能遇到各种意外：

**触发重规划的常见场景**：

1. **任务执行失败**：某个子任务调用的工具返回错误（数据库连接超时、API 返回空数据等）
2. **结果不符合预期**：任务执行成功了，但输出格式/内容与预期不同（获取的销售数据居然是空的，表示上周没有销售记录）
3. **发现新信息**：执行过程中获得了计划时未知的信息（原来数据需要从三个不同的数据库获取，而不是一个）
4. **用户中断并修改需求**：用户说"等一下，不需要分析了，只要原始数据就行"
5. **资源变化**：某个工具突然不可用，需要找替代方案

### 5.2 重规划的策略

重规划不是"放弃已有成果重新来"。好的重规划策略应该在**保留已完成工作的基础上，只修改受影响的部分**。

```python
"""
动态重规划器 —— 响应执行过程中的变化，智能调整后续计划。

三种重规划策略（按影响范围从小到大）：
1. 局部重试：仅重试失败的任务（最小改动）
2. 路径替换：找到一条新的执行路径绕过障碍（中等改动）
3. 全局重规划：放弃现有计划，重新生成（最大改动）
"""
import json
import copy
from datetime import datetime


class Replanner:
    """
    重规划器 —— 管理计划调整的完整生命周期。
    
    关键设计决策：
    - 已成功完成的任务**永远不重做**（除非用户明确要求）
    - 重规划只影响 PENDING 和 FAILED 状态的任务
    - 每次重规划都记录在 revision_history 中
    """

    def __init__(self, llm_client):
        self.client = llm_client

    def replan(
        self,
        plan,
        failed_task: SubTask,
        strategy: str = "auto",
    ) -> Plan:
        """
        根据失败信息重新规划。
        
        参数:
            plan: 当前计划（包含执行状态）
            failed_task: 触发重规划的任务
            strategy: "retry" / "replace_path" / "full_replan" / "auto"
        
        返回:
            修改后的计划（原地修改 + 返回引用）
        """
        # 记录重规划前的状态
        snapshot = self._snapshot(plan)

        if strategy == "auto":
            strategy = self._choose_strategy(failed_task)

        if strategy == "retry":
            # 策略1：仅重试失败任务
            self._retry_strategy(plan, failed_task)

        elif strategy == "replace_path":
            # 策略2：替换执行路径
            self._replace_path_strategy(plan, failed_task)

        elif strategy == "full_replan":
            # 策略3：全局重规划
            self._full_replan_strategy(plan, failed_task)

        # 记录重规划历史
        plan.version += 1
        plan.updated_at = datetime.now().isoformat()
        plan.revision_history.append(
            f"v{plan.version}: 任务 {failed_task.id} ({failed_task.description}) "
            f"失败，原因: {failed_task.error}，采用策略: {strategy}"
        )

        return plan

    def _choose_strategy(self, failed_task: SubTask) -> str:
        """
        自动选择重规划策略。
        
        决策逻辑：
        - 如果失败任务的重试次数 < max_retries → retry
        - 如果失败任务有替代工具 → replace_path
        - 如果失败影响大量下游任务 → full_replan
        - 其他 → replace_path（保守策略）
        """
        if failed_task.retry_count < failed_task.max_retries:
            return "retry"
        return "replace_path"

    def _retry_strategy(self, plan, failed_task: SubTask):
        """简单重试：将失败任务重置为 PENDING"""
        failed_task.status = TaskStatus.PENDING
        failed_task.retry_count += 1
        failed_task.error = None

    def _replace_path_strategy(self, plan, failed_task: SubTask):
        """
        路径替换：用 LLM 生成替代的执行路径。
        
        思路：
        - 保留已成功的任务不变
        - 告诉 LLM 哪个任务失败了、失败原因是什么
        - LLM 生成 1-3 个新的替代子任务来替换失败的任务
        - 新任务的依赖关系从失败任务的依赖继承
        - 原来依赖失败任务的下游任务改为依赖新的替代任务
        """
        # 构建重规划 prompt
        completed_tasks = [
            {"id": t.id, "description": t.description, "result": str(t.result)[:200]}
            for t in plan.tasks if t.status == TaskStatus.COMPLETED
        ]
        pending_tasks = [
            {"id": t.id, "description": t.description, "depends_on": t.depends_on}
            for t in plan.tasks if t.status == TaskStatus.PENDING
        ]

        prompt = f"""你是一个任务重规划助手。当前计划中的一个任务执行失败了，请生成替代方案。

## 已完成的任务
{json.dumps(completed_tasks, ensure_ascii=False, indent=2)}

## 失败的任务
- ID: {failed_task.id}
- 描述: {failed_task.description}
- 动作: {failed_task.action}
- 失败原因: {failed_task.error}
- 依赖: {failed_task.depends_on}

## 待执行的任务
{json.dumps(pending_tasks, ensure_ascii=False, indent=2)}

## 可用工具
{json.dumps(AVAILABLE_TOOLS, ensure_ascii=False, indent=2) if 'AVAILABLE_TOOLS' in dir() else "参考原计划中的工具"}

请生成 1-3 个替代子任务来替代失败的任务。输出 JSON 数组，每个子任务格式与原始计划相同。

重要：
- 替代任务的依赖应继承失败任务的依赖
- 如果失败的原因是某个工具不可用，请使用不同的工具
- 如果有多个替代任务，它们之间可以是并行关系"""
        
        # 调用 LLM 生成替代任务（这里省略实际调用代码）
        # new_tasks_from_llm = self._call_llm(prompt)
        
        # 简化演示：手动构造替代任务
        new_task = SubTask(
            id=f"{failed_task.id}_retry",
            description=f"(重规划) {failed_task.description} - 使用替代方案",
            action=failed_task.action + "_v2",  # 使用变体工具
            action_args=failed_task.action_args,
            depends_on=list(failed_task.depends_on),
            output_key=failed_task.output_key,
            priority=failed_task.priority + 1,  # 优先级稍低
        )
        
        # 将失败任务标记为已跳过
        failed_task.status = TaskStatus.SKIPPED
        
        # 插入新任务
        plan.tasks.append(new_task)
        
        # 更新下游任务的依赖：从失败任务改为新任务
        for task in plan.tasks:
            if task.status == TaskStatus.PENDING and failed_task.id in task.depends_on:
                task.depends_on = [
                    new_task.id if dep == failed_task.id else dep
                    for dep in task.depends_on
                ]

    def _full_replan_strategy(self, plan, failed_task: SubTask):
        """
        全局重规划：放弃 PENDING 任务，重新生成后续计划。
        
        适用场景：
        - 失败任务的失败揭示了计划阶段的根本性误判
        - 已有结果与预期大相径庭，需要完全不同的后续路径
        
        做法：
        1. 保留 COMPLETED 任务及其结果
        2. 将所有 PENDING 任务标记为 SKIPPED
        3. 将已完成的结果和失败信息传给 LLM，生成全新的后续计划
        4. 新计划的第一个任务依赖最后完成的那个任务
        """
        # 找到最后完成的任务作为新计划的起点
        completed = [t for t in plan.tasks if t.status == TaskStatus.COMPLETED]
        last_completed = completed[-1] if completed else None
        
        # 标记所有 PENDING 任务
        for task in plan.tasks:
            if task.status == TaskStatus.PENDING:
                task.status = TaskStatus.SKIPPED
        
        # 这里调用 LLM 重新规划...
        # skip（原理与 _replace_path_strategy 类似）

    def _snapshot(self, plan) -> dict:
        """保存计划快照（用于回滚）"""
        return {
            "version": plan.version,
            "tasks": [
                {
                    "id": t.id,
                    "status": t.status.value,
                    "result": t.result,
                    "error": t.error,
                }
                for t in plan.tasks
            ],
            "revision_history": list(plan.revision_history),
        }

## 6. 任务执行追踪

### 6.1 可观察的执行过程

一个可维护的 Agent 系统必须让执行过程可观察。执行追踪系统记录每个子任务的状态变化、耗时、输入输出，为调试、审计和优化提供数据基础。

```python
"""
任务执行追踪器 —— 记录和可视化每个子任务的完整生命周期。

追踪粒度：
- 状态变化（PENDING → IN_PROGRESS → COMPLETED / FAILED）
- 时间戳（创建、开始、结束）
- 输入输出快照
- 错误详情
- 重规划事件
"""
import time
import json
from datetime import datetime
from enum import Enum
from typing import Any, Optional, Callable
from dataclasses import dataclass, field


@dataclass
class TaskEvent:
    """子任务生命周期中的一个事件"""
    timestamp: str
    task_id: str
    event_type: str  # "created", "started", "completed", "failed", "retried", "skipped"
    details: dict = field(default_factory=dict)


class ExecutionTracker:
    """
    执行追踪器 —— 负责记录计划执行的完整日志。
    
    功能：
    1. 记录每个子任务的状态变化事件
    2. 记录每次重规划事件
    3. 生成执行报告
    4. 支持事件回调（如实时推送到前端）
    """

    def __init__(self, plan_id: str):
        self.plan_id = plan_id
        self.events: list[TaskEvent] = []
        self.start_time: Optional[float] = None
        self.end_time: Optional[float] = None
        self._callbacks: list[Callable] = []

    def on_event(self, callback: Callable[[TaskEvent], None]):
        """注册事件回调函数"""
        self._callbacks.append(callback)

    def record(self, event_type: str, task_id: str, **details):
        """记录一个事件"""
        event = TaskEvent(
            timestamp=datetime.now().isoformat(),
            task_id=task_id,
            event_type=event_type,
            details=details,
        )
        self.events.append(event)
        # 通知所有回调
        for cb in self._callbacks:
            try:
                cb(event)
            except Exception as e:
                print(f"回调执行失败: {e}")

    def start_plan(self):
        """标记计划开始执行"""
        self.start_time = time.time()
        self.record("plan_started", "plan", plan_id=self.plan_id)

    def end_plan(self, success: bool, summary: str = ""):
        """标记计划执行结束"""
        self.end_time = time.time()
        elapsed = self.end_time - self.start_time if self.start_time else 0
        self.record("plan_ended", "plan",
                    success=success, elapsed_seconds=round(elapsed, 2),
                    summary=summary)

    def task_created(self, task: SubTask):
        self.record("created", task.id,
                    description=task.description,
                    action=task.action,
                    depends_on=task.depends_on)

    def task_started(self, task: SubTask):
        self.record("started", task.id)

    def task_completed(self, task: SubTask, result: Any):
        self.record("completed", task.id,
                    result_preview=str(result)[:200])

    def task_failed(self, task: SubTask, error: str):
        self.record("failed", task.id, error=error)

    def task_retried(self, task: SubTask):
        self.record("retried", task.id, retry_count=task.retry_count)

    def replan_occurred(self, plan_version: int, reason: str):
        self.record("replan", "plan",
                    new_version=plan_version, reason=reason)

    def generate_report(self) -> dict:
        """生成执行追踪报告"""
        if not self.events:
            return {"error": "No events recorded"}

        # 按任务聚合事件
        task_events = {}
        for event in self.events:
            if event.task_id not in task_events:
                task_events[event.task_id] = []
            task_events[event.task_id].append(event)

        # 计算每个任务的耗时
        task_timings = {}
        for task_id, events in task_events.items():
            created_at = None
            started_at = None
            completed_at = None
            failed_at = None

            for e in events:
                ts = datetime.fromisoformat(e.timestamp)
                if e.event_type == "created":
                    created_at = ts
                elif e.event_type == "started":
                    started_at = ts
                elif e.event_type == "completed":
                    completed_at = ts
                elif e.event_type == "failed":
                    failed_at = ts

            timing = {}
            if started_at and created_at:
                timing["wait_time_seconds"] = (started_at - created_at).total_seconds()
            end_time = completed_at or failed_at
            if started_at and end_time:
                timing["execution_time_seconds"] = (end_time - started_at).total_seconds()

            task_timings[task_id] = timing

        # 统计
        completed_count = sum(
            1 for events in task_events.values()
            if any(e.event_type == "completed" for e in events)
        )
        failed_count = sum(
            1 for events in task_events.values()
            if any(e.event_type == "failed" for e in events)
        )

        total_elapsed = 0
        if self.start_time and self.end_time:
            total_elapsed = round(self.end_time - self.start_time, 2)

        return {
            "plan_id": self.plan_id,
            "total_elapsed_seconds": total_elapsed,
            "total_events": len(self.events),
            "task_count": len(task_events) - (1 if "plan" in task_events else 0),
            "completed_tasks": completed_count,
            "failed_tasks": failed_count,
            "success_rate": round(
                completed_count / max(completed_count + failed_count, 1) * 100, 1
            ),
            "task_timings": task_timings,
            "events": [
                {
                    "timestamp": e.timestamp,
                    "task_id": e.task_id,
                    "event_type": e.event_type,
                    "details": e.details,
                }
                for e in self.events
            ],
        }

    def print_timeline(self):
        """打印执行时间线（文本版甘特图）"""
        if not self.events:
            print("无事件记录")
            return

        task_events = {}
        for event in self.events:
            if event.task_id not in task_events:
                task_events[event.task_id] = []
            task_events[event.task_id].append(event)

        print("\n" + "=" * 70)
        print("执行时间线")
        print("=" * 70)

        for task_id, events in task_events.items():
            if task_id == "plan":
                continue

            # 确定最终状态
            final_state = "UNKNOWN"
            for e in events:
                if e.event_type == "completed":
                    final_state = "✅ COMPLETED"
                elif e.event_type == "failed":
                    final_state = "❌ FAILED"
                elif e.event_type == "skipped":
                    final_state = "⏭ SKIPPED"

            # 找到时间范围
            timestamps = [datetime.fromisoformat(e.timestamp) for e in events]
            if len(timestamps) >= 2:
                duration = (timestamps[-1] - timestamps[0]).total_seconds()
                print(f"  {task_id}: {final_state} ({duration:.1f}s)")
            else:
                print(f"  {task_id}: {final_state}")

            for e in events:
                print(f"    [{e.event_type}] {e.timestamp}")
                if e.details:
                    for k, v in e.details.items():
                        if k != "plan_id":
                            print(f"      {k}: {str(v)[:80]}")

    async def execute(self, task: SubTask, context: dict) -> Any:
        """
        执行一个子任务。
        
        参数:
            task: 要执行的子任务
            context: 包含前面任务结果的上下文
        
        返回:
            工具执行的返回值
        """
        tool_fn = self.tools.get(task.action)
        if tool_fn is None:
            raise ValueError(f"未找到工具: {task.action}")

        # 解析参数中的引用（$task_X.result 等）
        resolved_args = self._resolve_args(task.action_args, context)

        # 执行工具
        if asyncio.iscoroutinefunction(tool_fn):
            result = await tool_fn(**resolved_args)
        else:
            result = tool_fn(**resolved_args)

        return result

    def _resolve_args(self, args: dict, context: dict) -> dict:
        """
        解析参数中的变量引用。
        例如: "$task_1.raw_data" → context["task_1"]["raw_data"]
              "$context.start_date" → context["start_date"]
        """
        resolved = {}
        for key, value in args.items():
            if isinstance(value, str) and value.startswith("$"):
                # 解析引用路径
                resolved[key] = self._resolve_ref(value, context)
            else:
                resolved[key] = value
        return resolved

    def _resolve_ref(self, ref: str, context: dict) -> Any:
        """解析单个 $ 引用"""
        # 去掉 $ 前缀
        path = ref[1:]  # 例如 "task_1.raw_data" 或 "context.start_date"

        # 处理 $context.xxx 引用
        if path.startswith("context."):
            key = path.split(".", 1)[1]
            return context.get("_global_context", {}).get(key, ref)

        # 处理 $task_X.result.xxx 引用
        parts = path.split(".")
        task_id = parts[0]
        task_result = context.get("_results", {}).get(task_id)

        if task_result is None:
            return ref  # 无法解析，返回原引用

        # 遍历嵌套路径
        current = task_result
        for part in parts[1:]:
            if isinstance(current, dict):
                current = current.get(part)
            elif isinstance(current, list) and part.isdigit():
                current = current[int(part)]
            else:
                return ref

        return current if current is not None else ref

class PlanExecuteAgent:
    """
    Plan-and-Execute Agent —— 本章的集大成实现。
    
    工作流程：
    1. 理解用户任务
    2. 调用 LLM 生成执行计划
    3. 并行执行就绪的子任务
    4. 遇到失败时触发重规划
    5. 记录完整的执行轨迹
    6. 返回最终结果
    """

    def __init__(
        self,
        llm_client: OpenAI,
        tool_executor: ToolExecutor = None,
        max_parallel: int = 3,
        verbose: bool = True,
    ):
        self.client = llm_client
        self.executor = tool_executor or ToolExecutor()
        self.max_parallel = max_parallel
        self.verbose = verbose
        self.tracker: Optional[ExecutionTracker] = None
        self.replanner = Replanner(llm_client)

    async def run(
        self,
        user_task: str,
        context: dict = None,
        max_replans: int = 3,
    ) -> dict:
        """
        执行用户任务的主入口。
        
        参数:
            user_task: 用户任务的自然语言描述
            context: 额外的上下文（日期、用户信息等）
            max_replans: 允许的最大重规划次数
        
        返回:
            {
                "success": bool,
                "plan": Plan 对象,
                "results": {task_id: result},
                "report": 执行报告,
                "total_time": 总耗时秒数,
            }
        """
        start_time = time.time()

        # 初始化追踪器
        self.tracker = ExecutionTracker(plan_id=f"run-{int(start_time)}")
        self.tracker.start_plan()

        if self.verbose:
            print(f"\n{'=' * 60}")
            print(f"Plan-Execute Agent 开始执行")
            print(f"{'=' * 60}")
            print(f"任务: {user_task}")

        # ===== 阶段1: 规划 =====
        if self.verbose:
            print(f"\n📋 阶段1: 生成执行计划...")

        plan = await self._plan(user_task, context or {})
        self.tracker.record("plan_generated", "plan",
                           task_count=len(plan.tasks),
                           plan_name=plan.name)

        if self.verbose:
            print(f"计划: {plan.name} ({len(plan.tasks)} 个子任务)")
            for t in plan.tasks:
                deps = f" ← 等待: {', '.join(t.depends_on)}" if t.depends_on else ""
                print(f"  {t.id}: {t.description}{deps}")

        # ===== 阶段2: 执行 =====
        if self.verbose:
            print(f"\n⚡ 阶段2: 执行计划...")

        results = {}
        replan_count = 0

        while not plan.is_complete() and replan_count <= max_replans:
            # 获取所有就绪任务
            ready = plan.get_ready_tasks()

            if not ready:
                # 检查是否死锁
                pending = [t for t in plan.tasks if t.status == TaskStatus.PENDING]
                if pending:
                    if self.verbose:
                        print(f"  ⚠ 检测到 {len(pending)} 个 PENDING 任务但无就绪任务")
                        print(f"  可能是依赖任务失败导致阻塞，尝试重规划...")

                    # 找到第一个失败的任务作为重规划触发
                    failed_tasks = [t for t in plan.tasks if t.status == TaskStatus.FAILED]
                    if failed_tasks:
                        plan = self.replanner.replan(plan, failed_tasks[0])
                        self.tracker.replan_occurred(plan.version,
                                                     f"死锁恢复: {failed_tasks[0].id}")
                        replan_count += 1
                        continue
                    else:
                        # 没有失败任务但无法继续 —— 可能是循环依赖
                        if self.verbose:
                            print(f"  ❌ 无法继续执行，可能是循环依赖")
                        break
                else:
                    # 所有任务已完成
                    break

            # 并行执行就绪任务
            if self.verbose:
                print(f"  → 就绪任务 ({len(ready)}): {[t.id for t in ready]}")

            tasks_coroutines = []
            for task in ready[:self.max_parallel]:
                tasks_coroutines.append(self._execute_single_task(task, plan, results))

            # 等待就绪任务完成
            if tasks_coroutines:
                await asyncio.gather(*tasks_coroutines, return_exceptions=True)

            # 检查是否有任务失败需要重规划
            failed_tasks = [
                t for t in plan.tasks
                if t.status == TaskStatus.FAILED and t.retry_count >= t.max_retries
            ]
            if failed_tasks and replan_count < max_replans:
                if self.verbose:
                    print(f"  🔄 触发重规划 (失败任务: {failed_tasks[0].id})")
                plan = self.replanner.replan(plan, failed_tasks[0])
                self.tracker.replan_occurred(plan.version,
                                             f"任务失败: {failed_tasks[0].id}")
                replan_count += 1

        # ===== 阶段3: 收集结果 =====
        if self.verbose:
            print(f"\n📊 阶段3: 执行完成")

        total_time = time.time() - start_time
        success = all(
            t.status == TaskStatus.COMPLETED
            for t in plan.tasks
            if t.status != TaskStatus.SKIPPED  # 跳过的任务不算失败
        )

        self.tracker.end_plan(success=success,
                             summary=f"完成 {plan.progress()['completed']}/{len(plan.tasks)} 个任务")

        return {
            "success": success,
            "plan": plan,
            "results": results,
            "report": self.tracker.generate_report(),
            "total_time": round(total_time, 2),
        }

    async def _plan(self, user_task: str, context: dict) -> Plan:
        """调用 LLM 生成计划（使用第2.3节的规划器逻辑）"""
        # 注意：这里可以使用第2.3节中的 generate_plan 函数
        # 但由于依赖外部 API，演示中使用模拟数据
        plan = Plan(
            name=f"计划: {user_task[:30]}...",
            original_task=user_task,
            created_at=datetime.now().isoformat(),
            context=context,
        )

        # 模拟 LLM 生成的计划
        task_templates = [
            ("task_1", "获取销售数据", "get_sales_data",
             {"start_date": "$context.start_date", "end_date": "$context.end_date"},
             [], "raw_data"),
            ("task_2", "按品类汇总", "aggregate_by_category",
             {"data": "$task_1.raw_data", "metrics": ["revenue", "growth"]},
             ["task_1"], "agg_data"),
            ("task_3", "搜索市场趋势", "search_web",
             {"query": "2024年消费电子市场趋势", "max_results": 5},
             [], "market_trends"),  # 注意：task_3 和 task_2 可以并行
            ("task_4", "找增长最快品类", "find_top_growth",
             {"categories": "$task_2.agg_data", "top_n": 3, "sort_by": "growth"},
             ["task_2"], "top_categories"),
            ("task_5", "分析品类A", "analyze_category",
             {"category_name": "$task_4.top_categories.top_categories[0].name",
              "data": "$task_2.agg_data"},
             ["task_4"], "analysis_a"),
            ("task_6", "分析品类B", "analyze_category",

             ["task_4"], "analysis_b"),
            ("task_7", "分析品类C", "analyze_category",

             ["task_4"], "analysis_c"),
            ("task_8", "撰写分析邮件", "compose_email",

              "body": "$task_8.email_draft.body"},
             ["task_8"], "send_result"),
        ]

        for tid, desc, action, args, deps, output_key in task_templates:
            task = SubTask(
                id=tid, description=desc, action=action,
                action_args=args, depends_on=deps, output_key=output_key,
            )
            plan.tasks.append(task)
            self.tracker.task_created(task)

        return plan

    async def _execute_single_task(
        self, task: SubTask, plan: Plan, results: dict
    ) -> None:
        """执行单个子任务并更新状态"""
        task.status = TaskStatus.IN_PROGRESS
        task.started_at = datetime.now().isoformat()
        self.tracker.task_started(task)

        if self.verbose:
            print(f"    ▶ 执行 {task.id}: {task.description}")

        try:
            # 构建执行上下文
            context = {
                "_results": results,
                "_global_context": plan.context,
            }

            # 执行工具
            result = await self.executor.execute(task, context)

            # 成功
            task.status = TaskStatus.COMPLETED
            task.result = result
            task.completed_at = datetime.now().isoformat()
            results[task.id] = result
            self.tracker.task_completed(task, result)

            if self.verbose:
                result_preview = str(result)[:100].replace("\n", " ")
                print(f"    ✅ {task.id} 完成: {result_preview}...")

        except Exception as e:
            # 失败
            task.status = TaskStatus.FAILED
            task.error = str(e)
            task.completed_at = datetime.now().isoformat()
            self.tracker.task_failed(task, str(e))

            if self.verbose:
                print(f"    ❌ {task.id} 失败: {e}")

    def print_final_report(self, result: dict) -> None:
        """打印最终执行报告"""
        print(f"\n{'=' * 60}")
        print(f"执行完成")
        print(f"{'=' * 60}")
        print(f"状态: {'✅ 成功' if result['success'] else '❌ 失败'}")
        print(f"总耗时: {result['total_time']}s")

        plan = result['plan']
        progress = plan.progress()
        print(f"\n任务统计:")
        print(f"  总计: {progress['total']}")
        print(f"  完成: {progress['completed']}")
        print(f"  失败: {progress['failed']}")
        print(f"  重规划次数: {plan.version - 1}")

        if self.tracker:

## 常见错误

### 错误 1: 混淆 ReAct 和 Plan-Execute 的适用场景

```python
# 错误：对简单任务使用 Plan-Execute（过度设计）
# 用户问"今天天气怎么样"，生成一个 8 步计划
# → 浪费 token 和时间，直接用 ReAct 一步完成

# 错误：对复杂任务使用纯 ReAct（规划不足）
# 用户要求"分析数据→写报告→发邮件"，让 Agent 一步步想
# → 容易中途迷失目标，无法利用并行执行
```

### 错误 2: 子任务粒度过细或过粗

```python
# 错误：粒度过细
# 把"获取数据"拆成: 连接数据库→发送查询→接收结果→关闭连接
# → 这些是工具内部的实现细节，不应该暴露为计划层子任务

# 错误：粒度过粗
# 一个子任务 = "完成整个数据分析和报告"
# → 失去规划意义，等于没有分解

# 正确粒度：每个子任务 = 一次工具调用 = 一个有清晰输入输出的原子操作
```

### 错误 3: 依赖标注错误——漏标

```python
# 规划器输出了以下依赖：
# task_2.depends_on = []        ← 漏标！task_2 需要 task_1 的数据
# task_3.depends_on = ["task_1"] ← 漏标！task_3 还需要 task_2 的输出

# 后果: task_2 在 task_1 完成前执行 → task_2 缺少输入数据 → 失败
```

### 错误 4: 依赖标注错误——过度标注

```python
# task_4.depends_on = ["task_1", "task_2", "task_3"]
# 实际上 task_4 只需要 task_3 的输出，和 task_1、task_2 无关

# 后果: task_4 被不必要地延迟，浪费执行效率
# 正确: task_4.depends_on = ["task_3"]
```

### 错误 5: 忽略重规划中的"已完成任务保护"

```python
# 错误：重规划时重新生成全部计划，包括已完成的任务
# → 已完成的任务被标记为 PENDING，可能被重新执行
# → 数据库写入被重复执行，邮件被重复发送

# 正确：重规划只影响 PENDING 和 FAILED 状态的任务
# → 已完成的任务保持 COMPLETED 不变
```

### 错误 6: 循环依赖检测缺失

```python
# task_A.depends_on = ["task_C"]
# task_B.depends_on = ["task_A"]
# task_C.depends_on = ["task_B"]
# → A 等 C, C 等 B, B 等 A → 形成环 → 所有三个任务永远无法执行

# 解决：在执行前调用 DependencyAnalyzer.detect_cycles() 检查
```

### 错误 7: 并行执行时的并发安全问题

```python
# 错误：两个并行任务写入同一个文件
# task_4 和 task_5 同时执行 → 都写入 /tmp/report.txt → 数据竞争

# 正确：为每个任务分配独立的输出文件或使用文件锁
# task_4 写入 /tmp/report_a.txt, task_5 写入 /tmp/report_b.txt
```

### 错误 8: 工具参数中引用解析失败

```python
# 规划器生成的参数引用: "$task_3.result.top_categories[0].name"
# 但 task_3 的实际结果是: {"data": {"items": [...]}}  ← 键名是 "data" 不是 "top_categories"

# 解决：
# - 在参数解析时做好 error handling（解析失败时返回有意义的信息）
# - 为每个工具定义明确的输出 schema
# - 让 LLM 在生成计划时了解工具的输出格式
```

## 本章小结

本章深入学习了 Agent 任务规划与分解的完整知识体系：

| 知识点 | 核心要点 |
|--------|----------|
| 规划的动机 | 复杂任务不能一步完成、需要全局视图、支持并行和审计 |
| ReAct vs Plan-Execute | ReAct 边想边做（灵活但易迷失），Plan-Execute 先谋后动（有序但需重规划） |
| 计划的数据结构 | Plan 包含 SubTask 列表，每个 SubTask 有 ID/描述/动作/依赖/输出/状态 |
| 规划器设计 | LLM + 结构化 Prompt → JSON 格式计划，关键是工具描述和规则定义 |
| 层次分解 | 先拆阶段（粗），再细化子任务（细），支持增量规划和不确定性适应 |
| 依赖分析 | 数据依赖 / 资源依赖 / 因果依赖 / 条件依赖，循环检测用 DFS 染色法 |
| 并行识别 | 相同依赖集 + 无相互依赖 = 可并行，用 `get_ready_tasks()` 动态发现 |
| 关键路径 | CPM 算法：拓扑排序 + 正反向传播 → 最短完成时间 + 瓶颈任务 |
| 动态重规划 | 三种策略：重试（最小改动）、路径替换（中等）、全局重规划（最大改动） |
| 执行追踪 | 记录状态流转 + 时间戳 + 输入输出 → 审计、调试、优化 |
| 完整 Agent | 规划(LLM) → 并行执行(ToolExecutor) → 失败时重规划(Replanner) → 全程追踪(Tracker) |

下一章将学习多智能体协作——当单个 Agent 的能力不足以覆盖所有任务时，如何让多个专业 Agent 分工合作，共同完成更复杂的任务。
