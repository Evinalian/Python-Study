## 第一部分：Agent 为什么需要记忆

### 1.1 对比人类对话与无状态 API 调用

想象你和一位朋友进行以下对话：

> 你："我今天去了那家新开的咖啡店。"
> 朋友："哦？味道怎么样？"
> 你："很不错，他们的拿铁特别香。"
> 朋友："下次带我一起去吧。"
> 你："好啊，就在公司楼下，很方便。"
> 朋友："对了，你上次说的那个项目进展如何？"

这段对话看起来非常自然，但如果你仔细思考，它包含了大量的"记忆依赖"：

- 当朋友说"味道怎么样"时，他知道你指的是什么——你上一句提到的"咖啡店"。这是**最近对话的记忆**。
- 当你说"就在公司楼下"时，朋友不需要问"哪家公司"，因为他知道你在哪里工作。这是**长期共享知识的记忆**。
- 当朋友提到"上次说的那个项目"时，他引用的是更早的一次对话——可能发生在几天甚至几周前。这是**跨会话的长期记忆**。

现在，对比一下典型的 LLM API 调用：

```python
# 每次调用都是完全独立的
response1 = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "我今天去了那家新开的咖啡店。"}
    ]
)

# 第二次调用：LLM 对第一次调用的内容一无所知
# 除非你手动把第一次的内容也塞进 messages 数组
response2 = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "味道怎么样？"}  # LLM 无法知道"什么"的味道
    ]
)
```

这就是无状态 API 调用的本质：**每一次请求都是独立的，LLM 不记得上一轮对话、上一个会话、上一个用户说了什么。** 它拥有的唯一"记忆"就是你传入的 messages 数组——你必须手动维护这个数组，把历史消息一条一条地塞进去。

对于简单的"一问一答"场景（如翻译一句话、回答一个独立的常识问题），无状态调用完全够用。但对于下面这些场景，无状态就无法胜任了：

- **多轮对话**：用户说"后来呢？"——你需要知道"后来"指什么
- **长任务**：Agent 在 10 轮循环中搜集信息，每一轮需要知道前几轮的结果
- **跨会话延续**：用户第二天回到对话，"上次我们聊到哪了？"
- **个性化**：Agent 需要记住用户的偏好、习惯、背景信息

### 1.2 Agent 记忆的核心挑战

为 Agent 添加记忆不是简单地把所有历史消息塞进 messages 数组——如果你这样做，很快就会遇到以下问题：

**Token 限制问题**：每个 LLM 模型都有上下文窗口大小限制（GPT-4o 是 128K tokens，GPT-3.5 是 16K tokens，某些开源模型只有 4K 或 8K）。当一个 Agent 在循环中累积了大量消息后（用户消息、LLM 的回复、工具调用的请求和返回），很快就会超出限制。

**注意力衰减问题**：即使上下文窗口够大（比如 128K），LLM 对"长文本中间部分"的关注度也远低于开头和结尾。这是一个被多次验证的现象——LLM 倾向于更关注 prompt 的开头（primacy effect）和结尾（recency effect），中间的内容容易被忽略。所以把 100 轮对话全塞进去，第 30-70 轮的信息可能被 LLM 忽略。

**成本问题**：每次调用 LLM 都传入完整的对话历史，token 消耗随着对话增长呈线性增长。如果一次对话 100 轮，第 100 轮的输入 token 数可能是第 1 轮的 100 倍。这不仅是钱的浪费，也是延迟的灾难。

**信息质量衰减**：并不是所有的历史信息都同等重要。用户可能说了很多闲话（"嗯"、"好的"、"等一下我看看"），这些填充语不传达有效信息，但占用了宝贵的 token。你需要一种机制来过滤噪音、保留信号。

## 第二部分：记忆的三个层次

### 2.1 工作记忆（Working Memory）

**定义**：当前对话中，LLM 正在"思考"的即时上下文。它是 LLM 当前决策的直接依据。

**类比**：人类的"工作记忆"——你正在想的事情。比如你在算 27 x 34 时，脑子里同时 hold 住的中间结果。容量有限，遗忘迅速。

**在 Agent 中的体现**：当前 messages 数组中的内容——最近的几轮对话、最近的几次工具调用和返回结果。工作记忆是"此刻可用"的信息。

**容量建议**：保持在工作记忆中的内容应该尽量精简。一般来说，保留最近 3-5 轮完整对话，加上当前轮的内容，是比较合理的范围。更早的内容应该被"压缩"到短期记忆中。

### 2.2 短期记忆（Short-term Memory）

**定义**：当前会话中，已经被"压缩"但保留的关键信息。不是完整的对话记录，而是从对话中提取的摘要和要点。

**类比**：人类在听完一场讲座后，第二天能回忆起的"主要内容"——不是完整的录音，而是几个关键点和整体脉络。

**在 Agent 中的体现**：从对话历史中提取的结构化摘要。例如："用户询问了 Python 和 JavaScript 在 Web 开发中的对比，Agent 已经搜索了两者的性能、生态系统、学习曲线三个维度的信息，目前正在等待用户确认下一步。"

**存储形式**：可以是自然语言摘要文本，也可以是结构化的 JSON 对象（如 `{"topic": "...", "gathered_info": [...], "pending_questions": [...]}`）。

**更新时机**：当工作记忆"满了"（即对话轮数超过阈值），或者当 Agent 完成了当前对话的一个"阶段"时，将当前工作记忆中的关键信息提取并合并到短期记忆中。

### 2.3 长期记忆（Long-term Memory）

**定义**：跨会话的、持久化的信息存储。存储的是用户偏好、历史交互记录、学到的知识等。

**类比**：你记得"我的同事小王喜欢喝美式咖啡，不加糖"——这是跨时间的、稳定的记忆。你不会每次见到小王都重新问他喜欢喝什么。

**在 Agent 中的体现**：存储在外部数据库（向量数据库、关系型数据库、键值存储等）中的信息。通过检索（而非直接全部加载）来按需获取相关信息。

**存储的内容类型**：
- **用户画像**：用户的偏好、习惯、背景、常用语言
- **事实知识**：在历史对话中确认过的事实信息（如"用户所在城市是北京"）
- **经验教训**：Agent 在历史任务中积累的经验（如"搜索学术论文用 Google Scholar 比普通搜索效果好"）
- **任务模板**：用户常用的任务流程（如"用户做市场调研时通常需要：竞品分析→市场规模→趋势预测"）

**检索方式**：因为长期记忆的数据量可能非常庞大，不能全部加载到工作记忆中。典型的做法是：根据当前查询，从长期记忆中检索最相关的几条信息，作为上下文注入到当前 Agent 的决策中。

## 第三部分：上下文窗口管理

### 3.1 Token 限制问题的本质

每个 LLM 模型的上下文窗口都有上限。当 messages 数组的 token 数超过这个上限时，API 调用会失败（抛出 context_length_exceeded 错误）。

更微妙的问题是：即使没有超过硬限制，接近上限时 LLM 的性能也会下降。这体现在：
- 对中间部分的信息"遗忘"
- 对指令的遵循度下降
- 回答质量下降

解决策略主要有三种：滑动窗口、摘要压缩、关键信息提取。这三种策略不是互斥的，通常结合起来使用。

### 3.2 滑动窗口策略

**原理**：只保留最近的 N 条消息，超出 N 条的最老消息被丢弃。

**优点**：
- 实现简单，不需要额外 LLM 调用
- 保证了最新的对话上下文始终在窗口中

**缺点**：
- 直接丢弃早期信息，可能丢失关键上下文
- 如果早期有重要信息（如用户说"我的预算上限是 5000 元"），这个信息会在窗口滑过后丢失

**适用场景**：信息重要性随时间是均匀衰减的场景，或者"早期信息不太重要"的闲聊场景。

**实现**：

```python
def sliding_window(messages: list, max_messages: int = 20) -> list:
    """
    滑动窗口策略：保留最近的 max_messages 条消息。

    注意：system message 始终保留在最前面。
    """
    # 保留 system message
    system_messages = [m for m in messages if m["role"] == "system"]
    other_messages = [m for m in messages if m["role"] != "system"]

    # 截取最近的 max_messages 条
    recent = other_messages[-max_messages:]

    return system_messages + recent
```

### 3.3 摘要压缩策略

**原理**：当对话历史超过阈值时，调用 LLM 将早期对话压缩为一个摘要。摘要保留关键信息，但体积远小于原始对话。后续对话中，用摘要代替原始对话历史。

**优点**：
- 保留了早期对话的关键信息
- 压缩比例高（通常 10:1 到 100:1）
- 摘要可以分层（可以再压缩已有摘要）

**缺点**：
- 需要额外的 LLM 调用（产生延迟和费用）
- 摘要质量取决于 LLM 的能力
- 可能会遗漏细微但重要的信息

**适用场景**：信息密度高、早期信息不能被丢弃的场景。

**实现**：

```python
def compress_to_summary(messages: list, client, max_summary_tokens: int = 500) -> str:
    """
    将对话历史压缩为摘要。

    参数:
        messages: 需要压缩的消息列表
        client: OpenAI 客户端
        max_summary_tokens: 摘要的最大 token 数（大约值）

    返回:
        对话摘要字符串
    """
    # 将消息列表格式化为可读文本
    formatted = []
    for msg in messages:
        role = msg.get("role", "unknown")
        content = msg.get("content", "")
        if isinstance(content, list):
            content = str(content)
        # 跳过工具调用的技术细节，只保留关键信息
        if role == "tool":
            formatted.append(f"[工具返回]: {content[:200]}")
        elif role == "assistant" and msg.get("tool_calls"):
            # 助理消息中包含了工具调用，简化表示
            tool_names = [tc.get("function", {}).get("name", "?") for tc in msg.get("tool_calls", [])]
            formatted.append(f"[Agent 调用工具]: {', '.join(tool_names)}")
        else:
            formatted.append(f"[{role}]: {content[:300]}")

    conversation_text = "\n".join(formatted)

    # 调用 LLM 生成摘要
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "system",
            "content": (
                "请将以下对话历史压缩为一个简洁的摘要。要求：\n"
                "1. 保留所有关键信息（用户的问题、Agent 的发现、重要数据、用户偏好等）\n"
                "2. 省略闲聊、问候语、无信息量的内容\n"
                "3. 保留尚未解决的问题\n"
                f"4. 摘要控制在 {max_summary_tokens} 个 token 以内"
            )
        }, {
            "role": "user",
            "content": f"对话历史:\n{conversation_text}\n\n请生成摘要:"
        }],
        temperature=0.1,
    )
    return response.choices[0].message.content
```

### 3.4 关键信息提取保留

**原理**：从对话中提取结构化的事实和状态，存为独立的"知识条目"。这些条目在后续对话中按需注入（而不是一直占据上下文）。

**与摘要压缩的区别**：摘要是一段"叙述性文字"，关键信息提取是"结构化数据"。两者的区别类似于：把一本小说的内容总结成一段书评（摘要）vs 把小说中的关键信息整理成一个表格（结构化提取）。

**优点**：
- 信息结构化，易于检索
- 不会丢失细节（每条信息独立存储）
- 可以按需加载相关条目，不占用无关上下文

**缺点**：
- 需要额外的提取和检索逻辑
- 结构化提取的粒度难以把控

**适合提取的关键信息类型**：
- 用户事实：姓名、位置、偏好、限制条件
- 任务状态：已完成的步骤、待完成的事项、中间结果
- 重要数据：搜索到的关键数据点、计算得到的数值
- 待办事项：用户提出的后续需要跟进的事项

### 3.5 综合策略：分层上下文管理

在实际项目中，通常将这三种策略组合使用：

```
[短期记忆(摘要)] + [最近 N 轮原始对话] + [从长期记忆中检索的相关信息]
```

具体的分层方案：

1. **第 1 层（工作区）**：最近 5 轮完整对话 → 直接放在 messages 中
2. **第 2 层（短记忆）**：5 轮之前的对话 → 压缩为摘要，放在 messages 中
3. **第 3 层（长记忆）**：跨会话的关键信息 → 按需检索，注入到 messages 中

每次对话轮次超过 5 时，触发压缩：将最早的那轮对话从"工作区"移到"短记忆"，更新摘要。

## 第四部分：向量记忆

### 4.1 什么是向量记忆

向量记忆是利用 embedding（向量嵌入）技术实现的长期记忆系统。其核心思想是：

1. 将一段文字（如一次对话的摘要、一个用户偏好、一个历史事实）通过 embedding 模型转换为一个向量（一组浮点数）
2. 将向量存储在向量数据库中
3. 当需要检索记忆时，将当前查询也转换为向量，在向量数据库中搜索最相似的向量
4. 返回与相似向量对应的原始文字，作为上下文注入到当前对话中

向量记忆的优势在于**语义检索**——你不需要精确记住关键词，LLM 会根据语义相似度找到相关的记忆。例如用户说"我上次跟你聊过喝的"，向量搜索可以找到"用户喜欢喝美式咖啡，不加糖"这条记忆，尽管"喝的"和"咖啡"在字面上并不匹配。

### 4.2 向量记忆的实现

下面我们实现一个完整的向量记忆系统，包括记忆的创建、存储、检索和更新。

```python
"""
向量记忆系统 —— 基于 Embedding + 向量数据库的长期记忆实现

本实现使用一个简单的内存向量存储（可替换为 ChromaDB、Pinecone、Milvus 等）
并展示记忆的创建、检索、更新和遗忘机制。
"""

import json
import os
import time
import math
from typing import List, Dict, Any, Optional, Tuple
from dataclasses import dataclass, field
from openai import OpenAI


@dataclass
class Memory:
    """一条记忆的数据结构"""
    id: str                          # 唯一标识
    content: str                     # 记忆内容（自然语言文本）
    embedding: List[float] = None    # 向量嵌入（检索用）
    memory_type: str = "fact"        # 记忆类型: fact / preference / event / lesson
    importance: float = 0.5          # 重要程度 (0.0-1.0)
    created_at: float = field(default_factory=time.time)
    last_accessed_at: float = field(default_factory=time.time)
    access_count: int = 0            # 被检索次数
    source: str = ""                 # 来源（哪次对话产生的）
    tags: List[str] = field(default_factory=list)  # 标签，辅助分类检索


class VectorMemory:
    """
    向量记忆系统。

    核心功能:
    1. add() —— 添加新记忆（自动生成 embedding）
    2. search() —— 语义检索相关记忆
    3. update() —— 更新已有记忆
    4. forget() —— 删除记忆或让低重要度的记忆衰减
    5. consolidate() —— 合并相似的记忆，去重

    这是一个内存实现（数据不持久化）。生产环境中应替换为
    ChromaDB、Pinecone、Weaviate 等持久化向量数据库。
    """

    def __init__(self, client: OpenAI, embedding_model: str = "text-embedding-3-small"):
        """
        初始化向量记忆系统。

        参数:
            client: OpenAI 客户端（用于生成 embedding）
            embedding_model: embedding 模型名称
        """
        self.client = client
        self.embedding_model = embedding_model
        self.memories: Dict[str, Memory] = {}  # 内存存储

    def _generate_embedding(self, text: str) -> List[float]:
        """
        使用 OpenAI Embedding API 生成文本的向量嵌入。

        向量嵌入是一个高维空间中的点（通常 1536 维或 3072 维）。
        语义相似的文本在这个空间中距离更近。
        """
        response = self.client.embeddings.create(
            model=self.embedding_model,
            input=text,
        )
        return response.data[0].embedding

    def add(
        self,
        content: str,
        memory_type: str = "fact",
        importance: float = 0.5,
        source: str = "",
        tags: List[str] = None,
    ) -> str:
        """
        添加一条新记忆。

        参数:
            content: 记忆内容（自然语言描述）
            memory_type: 记忆类型 —— "fact"(事实), "preference"(偏好),
                        "event"(事件), "lesson"(经验教训)
            importance: 重要程度 0.0-1.0
            source: 来源标识
            tags: 分类标签列表

        返回:
            新记忆的 ID
        """
        # 生成向量嵌入
        embedding = self._generate_embedding(content)

        # 创建记忆对象
        memory_id = f"mem_{int(time.time() * 1000)}_{len(self.memories)}"
        memory = Memory(
            id=memory_id,
            content=content,
            embedding=embedding,
            memory_type=memory_type,
            importance=importance,
            source=source,
            tags=tags or [],
        )

        self.memories[memory_id] = memory
        return memory_id

    def search(
        self,
        query: str,
        top_k: int = 5,
        memory_type: str = None,
        min_importance: float = 0.0,
    ) -> List[Tuple[Memory, float]]:
        """
        语义检索最相关的记忆。

        参数:
            query: 检索查询（用于匹配相关记忆的文本）
            top_k: 返回最相关的前 K 条记忆
            memory_type: 按类型筛选（None 表示不筛选）
            min_importance: 最低重要度阈值

        返回:
            [(Memory, similarity_score), ...] 列表，按相似度降序排列
        """
        if not self.memories:
            return []

        # 生成查询的向量嵌入
        query_embedding = self._generate_embedding(query)

        # 计算余弦相似度
        results = []
        for memory in self.memories.values():
            # 类型筛选
            if memory_type and memory.memory_type != memory_type:
                continue
            # 重要度筛选
            if memory.importance < min_importance:
                continue
            # 计算相似度
            similarity = self._cosine_similarity(query_embedding, memory.embedding)
            results.append((memory, similarity))

        # 按相似度降序排列，取 top_k
        results.sort(key=lambda x: x[1], reverse=True)

        # 更新访问记录
        for memory, _ in results[:top_k]:
            memory.last_accessed_at = time.time()
            memory.access_count += 1

        return results[:top_k]

    def _cosine_similarity(self, vec1: List[float], vec2: List[float]) -> float:
        """
        计算两个向量的余弦相似度。

        余弦相似度 = (A·B) / (|A| * |B|)
        值域: [-1, 1]，1 表示完全相同，0 表示正交（无关），-1 表示完全相反。

        对于文本 embedding，值通常在 0 到 1 之间，越接近 1 越相似。
        """
        dot_product = sum(a * b for a, b in zip(vec1, vec2))
        norm1 = math.sqrt(sum(a * a for a in vec1))
        norm2 = math.sqrt(sum(b * b for b in vec2))
        if norm1 == 0 or norm2 == 0:
            return 0.0
        return dot_product / (norm1 * norm2)

    def update(self, memory_id: str, new_content: str = None, importance_delta: float = 0.0) -> bool:
        """
        更新一条记忆。

        参数:
            memory_id: 要更新的记忆 ID
            new_content: 新内容（None 表示不修改内容）
            importance_delta: 重要度增量（正数提升，负数降低）

        返回:
            True 表示成功，False 表示记忆不存在
        """
        memory = self.memories.get(memory_id)
        if memory is None:
            return False

        if new_content:
            memory.content = new_content
            memory.embedding = self._generate_embedding(new_content)

        memory.importance = max(0.0, min(1.0, memory.importance + importance_delta))

        return True

    def forget(self, memory_id: str) -> bool:
        """
        删除一条记忆。

        参数:
            memory_id: 要删除的记忆 ID

        返回:
            True 表示成功，False 表示记忆不存在
        """
        if memory_id in self.memories:
            del self.memories[memory_id]
            return True
        return False

    def decay(self, min_importance: float = 0.1, decay_rate: float = 0.05):
        """
        记忆衰减: 降低所有记忆的重要度。
        长时间未被访问的记忆会逐渐衰减，模拟人类的"遗忘曲线"。

        参数:
            min_importance: 低于此重要度的记忆将被删除
            decay_rate: 每次衰减的比例
        """
        to_remove = []
        for mem_id, memory in self.memories.items():
            memory.importance -= decay_rate
            if memory.importance < min_importance:
                to_remove.append(mem_id)

        for mem_id in to_remove:
            del self.memories[mem_id]

        print(f"记忆衰减: 删除了 {len(to_remove)} 条低重要度记忆, "
              f"剩余 {len(self.memories)} 条")

    def consolidate(self, similarity_threshold: float = 0.85):
        """
        记忆合并: 找到内容高度相似的记忆，合并为一条。

        这可以防止同一信息以稍微不同的措辞被重复存储。

        参数:
            similarity_threshold: 相似度阈值，超过此值视为重复
        """
        mem_list = list(self.memories.values())
        merged = set()

        for i in range(len(mem_list)):
            if mem_list[i].id in merged:
                continue
            for j in range(i + 1, len(mem_list)):
                if mem_list[j].id in merged:
                    continue
                sim = self._cosine_similarity(
                    mem_list[i].embedding, mem_list[j].embedding
                )
                if sim > similarity_threshold:
                    # 合并: 保留重要度更高的那条，删除另一条
                    if mem_list[i].importance >= mem_list[j].importance:
                        merged.add(mem_list[j].id)
                    else:
                        merged.add(mem_list[i].id)

        for mem_id in merged:
            del self.memories[mem_id]

        if merged:
            print(f"记忆合并: 合并了 {len(merged)} 条相似记忆")

    def get_context_for_query(
        self,
        query: str,
        max_tokens: int = 2000,
        top_k: int = 10,
    ) -> str:
        """
        获取与查询相关的记忆上下文，格式化为可直接注入 prompt 的文本。

        这是向量记忆系统与 Agent 集成的核心接口。

        参数:
            query: 当前查询或对话上下文
            max_tokens: 返回上下文的最大 token 数（估计）
            top_k: 最多检索多少条相关记忆

        返回:
            格式化的上下文字符串，可直接拼接到 system prompt 中
        """
        results = self.search(query, top_k=top_k, min_importance=0.2)

        if not results:
            return ""

        # 按类型分组格式化
        context_parts = ["## 相关历史记忆\n"]

        # 估算 token 数 (粗略: 中文 1 字 ≈ 1.5 token, 英文 1 词 ≈ 1.3 token)
        total_estimate = 0
        for memory, score in results:
            entry = (
                f"- [{memory.memory_type}] {memory.content} "
                f"(相关度: {score:.2f}, 重要度: {memory.importance:.2f})\n"
            )
            # 粗略估算 entry 的 token 数
            entry_tokens = len(entry) * 1.5
            if total_estimate + entry_tokens > max_tokens:
                break
            context_parts.append(entry)
            total_estimate += entry_tokens

        return "".join(context_parts)

    def stats(self) -> Dict[str, Any]:
        """获取记忆系统的统计信息"""
        if not self.memories:
            return {"total": 0}

        types = {}
        total_importance = 0
        for m in self.memories.values():
            types[m.memory_type] = types.get(m.memory_type, 0) + 1
            total_importance += m.importance

        return {
            "total": len(self.memories),
            "by_type": types,
            "avg_importance": total_importance / len(self.memories),
            "total_accesses": sum(m.access_count for m in self.memories.values()),
        }
```

### 4.3 向量记忆与 RAG 的关系

你可能会发现，向量记忆系统的实现和 RAG（第 ai-application 模块中学过的检索增强生成）非常相似。两者都使用 embedding + 向量数据库 + 相似度检索的模式。区别在于：

**RAG** 存储的是**外部知识库的文档**。知识是静态的，由人（或系统）预先准备好的。检索的目的是为 LLM 提供回答问题所需的"参考材料"。

**向量记忆**存储的是**Agent 自身的历史交互记录**。记忆是动态的，随着 Agent 的运行不断积累。检索的目的是让 Agent"记住"之前和用户的交互、学到的教训、用户的偏好。

两者的技术路径一致，但使用场景和更新机制不同。如果你已经会构建 RAG，实现向量记忆就是水到渠成的。

## 第五部分：记忆的检索与更新策略

### 5.1 什么时候检索记忆

不是每次 LLM 调用都需要检索记忆。如果用户问"今天天气怎么样"，你不需要去回顾用户三个月前的偏好。检索记忆的时机需要根据上下文来判断。

**触发检索的时机**：

1. **会话开始时**：用户开始新对话时，检索与用户相关的关键信息（用户画像、未完成的任务、最近的交互记录），作为 system prompt 的一部分注入。

2. **用户提到模糊引用时**：用户说"上次那个方案"、"之前聊过的那个问题"、"继续上次的讨论"——这些引用表明用户期望 Agent 记得之前的内容。此时应检索相关记忆。

3. **Agent 需要做个性化决策时**：当 Agent 需要"猜测"用户的偏好时（比如推荐咖啡、推荐技术方案），应该检索用户偏好记忆。

4. **Agent 遇到与历史相似的困难时**：如果 Agent 在当前任务中遇到困难（如搜索不到信息、工具调用失败），可以检索相似的历史经验记忆，寻找替代方案。

**不应该检索记忆的时机**：

1. 用户问的是完全独立的、事实性的问题
2. 用户明确表示"开始新话题"或"重来"
3. 当前对话历史已经包含了足够的上下文

### 5.2 什么时候写入新记忆

不是每一条信息都值得被记住。如果 Agent 把用户的每一句话都存成记忆，很快就会面临信息过载和检索质量下降。

**值得写入记忆的信息**：

1. **用户明确分享的个人信息**：姓名、职业、兴趣、偏好、限制条件
2. **用户确认过的事实**：经用户确认的数据、决定、结论
3. **Agent 自己学到的重要经验**：哪种搜索策略对某种类型的问题更有效、哪些网站的信息质量更高
4. **未完成的任务**：用户提出了但尚未完成的任务或问题

**不值得写入记忆的信息**：

1. 闲聊、寒暄、问候语
2. 已经在当前对话中解决的问题（不需要跨会话记忆）
3. 用户一次性的、临时的需求（"帮我查一下今天天气"——今天的天气不需要记忆到明天）
4. 不确定的信息（Agent 推测的但用户没有确认的信息）

### 5.3 记忆的自动管理策略

手动决定"什么时候存、什么时候查"是不现实的。一个实用的方法是实现自动记忆管理：

**自动提取**：在每次对话结束时（或会话达到一定长度时），调用 LLM 自动分析对话历史，提取值得记忆的信息。LLM 根据预设的"记忆提取规则"来决定哪些信息值得存储。

```python
def auto_extract_memories(
    conversation_history: list,
    client,
    memory_system: VectorMemory,
    source: str = "auto",
):
    """
    自动从对话历史中提取值得记忆的信息。

    这个函数在对话结束或达到一定长度时调用。
    它使用 LLM 来分析对话历史，自动识别和提取关键信息。
    """
    # 将对话历史格式化为文本
    formatted = []
    for msg in conversation_history[-50:]:  # 只分析最近 50 条
        role = msg.get("role", "unknown")
        content = msg.get("content", "")
        if isinstance(content, list):
            content = str(content)
        formatted.append(f"[{role}]: {content[:500]}")

    conversation_text = "\n".join(formatted)

    # 调用 LLM 提取记忆
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "system",
            "content": """你是一个记忆提取助手。请从对话历史中提取值得长期记忆的信息。

提取规则:
1. 用户明确分享的个人信息（姓名、职业、偏好、习惯、限制条件）
2. 用户确认过的事实、决定或结论
3. 未完成的任务或用户提出的后续需求
4. 对后续交互可能有用的经验教训

不要提取:
- 闲聊和问候
- 一次性的临时查询
- 不确定的推测

请以 JSON 数组格式输出，每条记忆包含:
- content: 记忆内容（自然语言）
- type: 类型 (fact/preference/event/lesson)
- importance: 重要度 0.0-1.0
- tags: 标签列表

如果对话中没有值得记忆的内容，返回空数组 []。"""
        }, {
            "role": "user",
            "content": f"对话历史:\n{conversation_text}\n\n请提取值得记忆的信息:"
        }],
        temperature=0.1,
    )

    # 解析提取结果
    try:
        extracted = json.loads(response.choices[0].message.content)
    except json.JSONDecodeError:
        print("记忆提取失败: LLM 返回的不是有效 JSON")
        return []

    # 将提取的记忆写入记忆系统
    for item in extracted:
        memory_system.add(
            content=item["content"],
            memory_type=item.get("type", "fact"),
            importance=item.get("importance", 0.5),
            source=source,
            tags=item.get("tags", []),
        )

    print(f"自动提取了 {len(extracted)} 条记忆")
    return extracted

# 如果高度相似，更新已有记忆而不是创建新记忆，避免信息冗余。
```

## 第六部分：带记忆的 Agent 完整实现

现在我们把所有知识点整合起来——实现一个带有分层记忆（工作记忆 + 短期记忆 + 长期向量记忆）的 Agent。

```python
"""
带记忆的 Agent 完整实现

集成了三个记忆层次:
1. 工作记忆: 最近 N 轮完整对话（在 messages 中）
2. 短期记忆: 早期对话的压缩摘要（在 messages 中）
3. 长期记忆: 向量记忆系统（按需检索注入）

Agent 流程:
1. 用户发起对话时，从长期记忆中检索相关记忆，注入到 system prompt
2. 在对话过程中，维护工作记忆（最近 N 轮完整消息）
3. 当对话超过一定轮数时，压缩早期对话为摘要（短期记忆）
4. 对话结束后，自动从对话历史中提取新的长期记忆
"""

import json
import os
import time
from typing import List, Dict, Any, Optional
from openai import OpenAI

# ---- 初始化 ----
client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY", "your-api-key"),
    base_url=os.environ.get("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)

# ---- 全局记忆系统 ----
# 在实际生产环境中，VectorMemory 应该使用持久化的向量数据库
# 这里为了演示，使用上一节实现的内存 VectorMemory
# (假设已从上一节的代码中导入)
from __main__ import VectorMemory  # noqa
memory_system = VectorMemory(client)


class MemoryAugmentedAgent:
    """
    带分层记忆的 Agent。

    记忆架构:
    - 工作记忆: self.working_memory (最近 MAX_WORKING_TURNS 轮的 messages)
    - 短期记忆: self.short_term_summary (早期对话的压缩摘要文本)
    - 长期记忆: memory_system (全局向量记忆，跨会话)

    工作流程:
    1. 用户输入到达
    2. 检索长期记忆中与当前查询相关的内容
    3. 构建完整的 messages (system prompt + 长期记忆上下文 + 短期摘要 + 工作记忆 + 用户输入)
    4. 调用 LLM 获取回复
    5. 更新工作记忆，必要时压缩到短期记忆
    """

    MAX_WORKING_TURNS = 6  # 工作记忆中保留的最近对话轮数
    COMPRESSION_THRESHOLD = 8  # 对话超过此轮数时触发压缩

    def __init__(self, system_prompt: str = None):
        """
        初始化 Agent。

        参数:
            system_prompt: 基础 system prompt（不包含记忆上下文的部分）
        """
        self.base_system_prompt = system_prompt or "你是一个乐于助人的智能助手。"
        self.working_memory: List[Dict] = []  # 最近 N 轮的完整消息
        self.short_term_summary: str = ""     # 压缩后的早期对话摘要
        self.total_turns: int = 0             # 对话总轮数
        self.user_id: str = "default_user"    # 用户标识

    def _build_system_prompt(self, user_input: str) -> str:
        """
        构建完整的 system prompt，注入各层记忆。

        组装顺序:
        1. 基础 system prompt
        2. 长期记忆上下文（从向量记忆中检索的）
        3. 短期记忆（压缩摘要）
        4. 用户画像提示
        """
        prompt_parts = [self.base_system_prompt]

        # 注入长期记忆（向量检索）
        long_term_context = memory_system.get_context_for_query(
            user_input,
            max_tokens=800,
            top_k=5,
        )
        if long_term_context:
            prompt_parts.append("\n" + long_term_context)

        # 注入短期记忆（压缩摘要）
        if self.short_term_summary:
            prompt_parts.append(
                "\n## 之前的对话摘要\n"
                f"{self.short_term_summary}\n"
                "请基于以上摘要理解当前对话的上下文。"
            )

        return "\n".join(prompt_parts)

    def chat(self, user_input: str) -> str:
        """
        处理用户的输入，返回 Agent 的回复。

        参数:
            user_input: 用户输入文本
        返回:
            Agent 的回复文本
        """
        self.total_turns += 1

        # 步骤 1: 构建完整的 messages 列表
        # 顺序: system prompt → 工作记忆(历史消息) → 当前用户输入
        full_system_prompt = self._build_system_prompt(user_input)

        messages = [
            {"role": "system", "content": full_system_prompt},
        ]

        # 添加工作记忆中的历史消息（已经是 assistant/tool 格式）
        messages.extend(self.working_memory)

        # 添加当前用户消息
        messages.append({"role": "user", "content": user_input})

        # 步骤 2: 调用 LLM
        print(f"\n[对话轮次 {self.total_turns}] messages 数量: {len(messages)}")
        print(f"[短期记忆长度]: {len(self.short_term_summary)} 字符")
        print(f"[工作记忆轮数]: {len(self.working_memory) // 2}")

        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            temperature=0.3,
        )

        assistant_reply = response.choices[0].message.content

        # 步骤 3: 更新工作记忆
        self.working_memory.append({"role": "user", "content": user_input})
        self.working_memory.append({"role": "assistant", "content": assistant_reply})

        # 步骤 4: 如果工作记忆达到压缩阈值，触发压缩
        if len(self.working_memory) > self.COMPRESSION_THRESHOLD * 2:
            self._compress_working_memory()

        # 步骤 5: 如果对话较长 (超过 3 轮)，自动提取长期记忆
        if self.total_turns >= 3 and self.total_turns % 5 == 0:
            self._auto_memorize()

        return assistant_reply

    def _compress_working_memory(self):
        """
        将工作记忆中最老的消息压缩到短期记忆中。

        策略:
        1. 取出工作记忆中最老的 N 条消息（保留最近的 MAX_WORKING_TURNS * 2 条）
        2. 调用 LLM 将这些消息压缩为摘要
        3. 将新摘要合并到已有的 short_term_summary 中
        """
        # 计算需要压缩的消息数量
        keep_count = self.MAX_WORKING_TURNS * 2  # *2 是因为每轮有 user + assistant
        if len(self.working_memory) <= keep_count:
            return

        # 取出最老的消息
        to_compress = self.working_memory[:-keep_count]
        self.working_memory = self.working_memory[-keep_count:]

        # 调用 LLM 压缩
        formatted = []
        for msg in to_compress:
            role = msg.get("role", "?")
            content = msg.get("content", "")[:400]
            formatted.append(f"[{role}]: {content}")

        conversation_snippet = "\n".join(formatted)

        try:
            response = client.chat.completions.create(
                model="gpt-4o",
                messages=[{
                    "role": "system",
                    "content": (
                        "请将以下对话片段压缩为一个简洁的摘要。保留: "
                        "1) 讨论的主题 2) 关键信息和数据 3) 用户的偏好和决策 "
                        "4) 未解决的问题。摘要控制在 300 字以内。"
                    )
                }, {
                    "role": "user",
                    "content": f"对话片段:\n{conversation_snippet}\n\n请生成摘要:"
                }],
                temperature=0.1,
            )
            new_summary = response.choices[0].message.content

            # 合并到现有短期记忆中
            if self.short_term_summary:
                # 递归压缩：将旧摘要和新摘要合并再压缩
                combined = (
                    f"旧摘要: {self.short_term_summary}\n\n"
                    f"新内容: {new_summary}"
                )
                response2 = client.chat.completions.create(
                    model="gpt-4o",
                    messages=[{
                        "role": "system",
                        "content": "将以下两段摘要合并为一段连贯的摘要，避免重复信息。控制在 400 字以内。"
                    }, {
                        "role": "user",
                        "content": combined,
                    }],
                    temperature=0.1,
                )
                self.short_term_summary = response2.choices[0].message.content
            else:
                self.short_term_summary = new_summary

            print(f"\n[记忆压缩] 压缩了 {len(to_compress)} 条消息 → "
                  f"摘要 {len(self.short_term_summary)} 字符")

        except Exception as e:
            print(f"[记忆压缩失败]: {e}")

    def _auto_memorize(self):
        """
        自动从最近的工作记忆中提取长期记忆。
        在对话进行到一定程度时调用。
        """
        # 取最近 10 条消息用于提取
        recent = self.working_memory[-10:] if len(self.working_memory) >= 10 else self.working_memory
        if len(recent) < 4:
            return

        formatted = []
        for msg in recent:
            role = msg.get("role", "?")
            content = msg.get("content", "")[:400]
            formatted.append(f"[{role}]: {content}")

        conversation_snippet = "\n".join(formatted)

        try:
            response = client.chat.completions.create(
                model="gpt-4o",
                messages=[{
                    "role": "system",
                    "content": (
                        "从以下对话片段中提取值得长期记忆的信息。"
                        "提取: 用户个人信息、偏好、重要决策、未完成任务。"
                        "忽略: 闲聊、临时查询。"
                        "以 JSON 数组返回，每项含 content/type/importance/tags。"
                        "如无值得记忆的内容，返回空数组 []。"
                    )
                }, {
                    "role": "user",
                    "content": conversation_snippet,
                }],
                temperature=0.1,
            )

            extracted = json.loads(response.choices[0].message.content)

            for item in extracted:
                memory_system.add(
                    content=item["content"],
                    memory_type=item.get("type", "fact"),
                    importance=item.get("importance", 0.5),
                    source=f"session_{int(time.time())}",
                    tags=item.get("tags", []),
                )

            if extracted:
                print(f"  [长期记忆] 自动提取了 {len(extracted)} 条新记忆")

        except Exception as e:
            print(f"  [记忆提取失败]: {e}")

    def end_session(self):
        """
        结束当前会话时调用。
        执行最终的长期记忆提取（分析完整的会话历史）。
        """
        print(f"\n[会话结束] 执行最终记忆提取...")
        self._auto_memorize()
        print(f"[记忆统计] {memory_system.stats()}")

    def reset_session(self):
        """
        重置会话状态（开始新的对话），但保留长期记忆。
        """
        self.working_memory = []
        self.short_term_summary = ""
        self.total_turns = 0
        print("[会话已重置] 工作记忆和短期记忆已清空，长期记忆保留。")
```

### 代码详解

**分层记忆的组装顺序**：在 `_build_system_prompt` 方法中，我们按照"基础 prompt → 长期记忆 → 短期记忆（摘要）"的顺序组装 system prompt。这个顺序不是随意的——我们通常希望长期记忆（来自向量检索的上下文）出现在 prompt 中部偏前的显著位置，而短期摘要信息则放在中部偏后。

**压缩触发时机**：`COMPRESSION_THRESHOLD` 设为 8，即当工作记忆中超过 8 轮对话（16 条消息）时触发压缩。这是一个经验值，实际项目中应该根据模型的上下文窗口大小和每轮消息的平均长度来调整。

**自动记忆提取**：`_auto_memorize` 在对话进行到一定轮数时自动执行。我们设置了"每 5 轮提取一次"的策略，这样可以渐进地积累长期记忆，而不是等到会话完全结束才一次性提取。

**会话结束时的记忆提取**：`end_session` 是会话级别的收尾操作。它在对话完全结束时执行最后一次全面的记忆提取，确保不会遗漏任何重要信息。

## 常见错误

**错误 1：把所有历史消息都塞进 messages**
这是最直观但最糟糕的做法。对话越长，每次调用的 token 消耗越大，最终一定会超过上下文限制。必须对历史消息进行管理——要么丢弃最老的，要么压缩。

**错误 2：压缩摘要时丢失关键信息**
摘要压缩的本质是"用 LLM 做信息提取"。如果压缩 prompt 写得不好（比如只说"请总结对话"而没有给出"保留哪些信息"的具体要求），LLM 可能会丢失关键细节。压缩 prompt 中必须明确列举需要保留的信息类别。

**错误 3：长期记忆只存不查**
有些开发者实现了向量记忆的存储功能，但在每次对话中没有自动检索相关记忆并注入 prompt。这等于建了一个图书馆但从不去查阅——记忆存了等于白存。必须实现"每次对话开始时自动检索相关记忆"的机制。

**错误 4：检索时的查询词太短或太长**
向量检索的质量高度依赖于查询 embedding 的质量。如果查询词只是用户输入的前 5 个字，可能无法充分表达查询意图。如果查询词是整个对话历史，可能包含了太多噪声。建议将"当前用户输入 + 前一轮对话的摘要"作为检索查询，平衡信息量和精准度。

**错误 5：记忆数量膨胀后检索性能下降**
当记忆中存储了成千上万条信息时，全量检索的延迟会变得不可接受。需要使用近似最近邻（ANN）索引来加速检索（如 FAISS、HNSW），或者使用支持索引的向量数据库（如 Pinecone、Milvus）。

**错误 6：在 system prompt 中同时注入过多记忆**
如果从向量记忆中检索了 20 条相关信息但全部注入 system prompt，可能会导致 prompt 冗长无比，反而干扰 LLM 的判断。建议限制注入的记忆条数（通常 3-5 条最相关的最理想），并设置注入的总 token 上限。

**错误 7：忘记处理记忆的时效性**
有些记忆是"有时效性的"——比如"用户正在出差"三天后可能就不成立了。如果不在存储时记录时间戳并在检索时考虑时效性，Agent 可能会使用过时信息做出错误判断。

**错误 8：跨会话用户识别错误**
在多用户系统中，如果 A 用户的记忆被错误地注入到 B 用户的对话中，这是严重的隐私问题。确保用户识别和记忆隔离的机制健壮——使用不可伪造的 user_id 而非用户自称的姓名。

## 本章小结

记忆与状态管理是 Agent 从"工具"进化为"伙伴"的关键。一个没有记忆的 Agent 永远是"初次见面"，而一个有记忆的 Agent 能够建立持续的关系。

本章的核心内容回顾：

1. **记忆分层**：工作记忆（即时上下文）、短期记忆（会话摘要）、长期记忆（向量数据库）——三者共同构成了 Agent 的记忆体系。

2. **上下文窗口管理**：滑动窗口、摘要压缩、关键信息提取——三种策略不是互斥的，实践中通常是它们的组合。

3. **向量记忆**：利用 embedding + 向量数据库实现语义化的长期记忆存储和检索。与 RAG 技术路径一致，但存储的内容和更新方式不同。

4. **检索与更新策略**：何时检索记忆（会话开始、模糊引用、个性化决策、遇到困难）和何时写入记忆（明确的个人信息、确认的事实、学到的经验、未完成任务）——这些规则可以部分自动化。

5. **实践要点**：分层组装 system prompt、渐进式压缩、自动提取、重要性衰减——这些工程细节决定了一个记忆系统是"能用"还是"好用"。

记忆系统的设计和调优是一个持续迭代的过程。不同的应用场景对记忆的需求差异很大——一个客服机器人需要的记忆（客户信息、工单历史）和一个个人生活助理需要的记忆（用户偏好、日常习惯、社交关系）完全不同。理解记忆的基本原理后，更重要的是根据你的具体场景来设计恰当的记忆架构。
