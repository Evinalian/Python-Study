## 1. 没有类型注解的痛点

假设你接手了学长留下的代码，里面有一个函数：

```python
def process(data, config, mode):
    """处理数据"""
    result = []
    for item in data:
        if mode == 1:
            result.append(item * config["factor"])
        else:
            result.append(item + config["offset"])
    return result
```

你拿着这段代码，脑子里会冒出三个问题：

1. **`data` 是什么？** 是列表还是数组？里面的元素是数字还是字符串？IDE 无法告诉你 `item` 有哪些方法可用，你只能靠猜或者去翻调用处的代码。
2. **`config` 是字典吗？** 里面应该有哪些 key？`factor` 和 `offset` 是必填还是可选？传错了 key 不会报错，直到运行到那一行才炸。
3. **返回值是什么？** 是列表还是别的什么？调用方拿到结果后同样一脸茫然。

这三个问题正好对应动态类型语言在**中型以上项目**中的三大痛点：

| 痛点 | 具体表现 |
|------|---------|
| IDE 无法补全 | 编辑器不知道变量类型，无法提示可用方法/属性 |
| 重构高风险 | 改了函数签名，调用方不报错，运行时才炸 |
| 文档与代码脱节 | 类型信息写在注释里，很容易过时 |

**类比**：学过 C/Java 的同学习惯了编译器帮你检查类型。Python 选择把类型检查推迟到运行时——这在小脚本里很爽快，但当代码量超过几千行、多人协作时，"自由"就成了"混乱"。

类型注解从 Python 3.5（PEP 484）开始引入，核心原则是：**可选的、渐进式的**。你可以只给关键函数加注解，不用一下子全改。

## 2. 从简单变量开始

### 2.1 变量注解：在冒号后面写类型

```python
# 最基础的变量注解
name: str = "小明"            # name 是字符串
age: int = 20                 # age 是整数
height: float = 1.75          # height 是浮点数
is_student: bool = True       # is_student 是布尔值

# 容器类型
scores: list[int] = [85, 92, 78]           # 整数组成的列表
info: dict[str, str] = {"city": "北京"}     # 字符串→字符串的字典
point: tuple[int, int] = (10, 20)          # 两个整数的元组
```

**关键理解**：这些注解**运行时不被强制检查**。下面这段代码运行不会报错：

```python
name: str = 123  # 运行不会报错！但 IDE 和 mypy 会提醒你
print(name)      # 输出: 123
```

注解存储在 `__annotations__` 字典里，供**工具读取**，Python 解释器完全忽略它们。你可以把它理解成"写给 IDE 看的正式文档"。

### 2.2 类比 Java/C

| 语言 | 写法 | 检查时机 |
|------|------|---------|
| Java | `String name = "小明";` | 编译期强制 |
| C | `int age = 20;` | 编译期强制 |
| Python | `name: str = "小明"` | 不强制，工具可选检查 |

Java/C 的编译器和 Python 的 mypy 做的是同一件事（静态类型检查），只不过一个内建在语言里，一个是外部工具。

## 3. 函数注解：参数和返回值

### 3.1 基本语法

```python
def greet(name: str) -> str:
    """冒号后写参数类型，箭头后写返回值类型"""
    return f"你好，{name}！"

# 调用时 IDE 会提示你 name 应该是 str，返回值也是 str
print(greet("小明"))  # 你好，小明！

def divide(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("除数不能为零")
    return a / b

# 带默认值的写法：默认值在类型注解后面
def connect(host: str, port: int = 8080, timeout: float | None = None) -> bool:
    """连接服务器，timeout 可以是 None（表示不设超时）"""
    print(f"连接 {host}:{port}，超时设置: {timeout}")
    return True
```

### 3.2 选哪个函数？——分三步理解

回到开头的 `process` 函数，我们分三步给它加上类型注解：

```python
# 第一步：最简单的注解——参数和返回值都标上类型
def process(data: list[float], config: dict[str, float], mode: int) -> list[float]:
    result = []
    for item in data:
        if mode == 1:
            result.append(item * config["factor"])
        else:
            result.append(item + config["offset"])
    return result
```

现在 IDE 至少知道 `data` 是 `list[float]`，`item` 也就能推断出是 `float`——补全 `item.` 会提示 float 的方法。

```python
# 第二步：mode 只有 1 和 2 两种取值，用 Literal 限定
from typing import Literal

def process(
    data: list[float],
    config: dict[str, float],
    mode: Literal[1, 2]       # 只能是 1 或 2
) -> list[float]:
    ...
# 如果有人传 mode=3，IDE 和 mypy 都会警告
```

```python
# 第三步：config 的字典结构太模糊——用 TypedDict 精确描述
from typing import TypedDict

class ProcessConfig(TypedDict):
    factor: float    # 模式 1 用的乘数
    offset: float    # 模式 2 用的偏移量

def process(
    data: list[float],
    config: ProcessConfig,   # 现在 IDE 知道 config 里有哪些 key
    mode: Literal[1, 2]
) -> list[float]:
    ...
```

## 4. 泛型容器：`list[int]` vs `List[int]`

### 4.1 现代写法（Python 3.9+）

从 Python 3.9 开始，内置容器直接支持泛型写法，不需要从 `typing` 导入：

```python
# Python 3.9+ 推荐写法（小写，简洁）
users: list[str] = ["alice", "bob"]
scores: dict[str, int] = {"math": 95, "english": 88}
point: tuple[int, int] = (10, 20)
unique_ids: set[int] = {1, 2, 3}

# 嵌套也很直观
matrix: list[list[float]] = [[1.0, 2.0], [3.0, 4.0]]
config: dict[str, list[str]] = {"hosts": ["srv1", "srv2"]}
```

### 4.2 旧版写法（了解即可）

Python 3.8 及更早版本需要从 `typing` 导入大写版本：

```python
from typing import List, Dict, Tuple, Set

users: List[str] = ["alice"]        # 旧写法
scores: Dict[str, int] = {}
```

如果你的项目最低支持 Python 3.9+，优先使用小写内置泛型。本章后续均使用 3.9+ 写法。

### 4.3 对比表

| 功能 | 旧版写法 | 现代写法 |
|------|---------|---------|
| 字符串列表 | `List[str]` | `list[str]` |
| 字典 | `Dict[str, int]` | `dict[str, int]` |
| 元组 | `Tuple[int, str]` | `tuple[int, str]` |
| 嵌套 | `Dict[str, List[int]]` | `dict[str, list[int]]` |

## 5. `X | None`：这个值可能是 None

### 5.1 为什么需要 "或 None"

在实际项目中，很多情况返回值可能是 `None`：

```python
# 查询用户——如果没找到，返回什么？
def find_user(name: str) -> dict | None:   # 找到了返回字典，没找到返回 None
    users = {"alice": {"age": 25}}
    return users.get(name)  # dict.get() 找不到就返回 None

# 调用方必须先判断 None
user = find_user("bob")
if user is not None:          # 先判空！
    print(user["age"])        # 安全访问
else:
    print("用户不存在")
```

### 5.2 `X | None` vs `Optional[X]`

两种写法**完全等价**：

```python
# Python 3.10+ 推荐：| 语法（更简洁）
def get_age(user: dict | None) -> int:
    if user is None:
        return 0
    return user.get("age", 0)

# 旧写法（能读懂即可）
from typing import Optional
def get_age(user: Optional[dict]) -> int:
    ...
```

**建议**：写新代码用 `X | None`，读旧代码时要知道 `Optional[X]` 就是 `X | None`。

### 5.3 `|` 的更多用法（Python 3.10+ 的 Union 语法）

```python
# 参数可以是 int 或 str
def process(value: int | str) -> str:
    return str(value)

# 返回值可能是 float 或 str（错误信息）
def safe_divide(a: float, b: float) -> float | str:
    if b == 0:
        return "错误：除数不能为零"
    return a / b
```

在 C/Java 中，函数只能返回一种类型。Python 中 `int | str` 不是说"同时返回两种"，而是说"**可能是 int，也可能是 str**"——调用方需要对不同情况分别处理。

## 6. Pydantic：让类型注解在运行时生效

### 6.1 从实际需求出发

类型注解只在**编写代码时**被 IDE 和 mypy 检查。但程序运行时会收到外部数据——API 返回的 JSON、用户上传的文件、配置文件……这些数据长什么样，只有运行时才知道。

**场景**：你调用一个天气 API，文档说返回格式是：

```json
{
    "city": "北京",
    "temperature": 26.5,
    "humidity": 60,
    "condition": "晴"
}
```

你怎么确保收到的数据真的符合这个格式？如果 API 出 bug 返回了字符串 `"26.5"` 而不是数字 `26.5`？如果 `humidity` 字段直接消失？

**过去你可能会写很多 `if isinstance(...)` 来逐个字段检查**。Pydantic 让你**声明一次模型，自动校验所有数据**。

### 6.2 安装 Pydantic

```bash
pip install pydantic
```

### 6.3 第一个 Pydantic 模型

```python
from pydantic import BaseModel

class WeatherData(BaseModel):
    """天气数据的模型——声明字段和类型"""
    city: str
    temperature: float
    humidity: int
    condition: str

# 从字典创建——Pydantic 自动校验每个字段的类型
data = WeatherData(
    city="北京",
    temperature=26.5,
    humidity=60,
    condition="晴"
)
print(data.city)          # 北京
print(data.temperature)   # 26.5

# 类型不匹配时，能转就自动转，不能转就报错
data2 = WeatherData(city="上海", temperature="28", humidity="55", condition="阴")
print(type(data2.temperature))  # <class 'float'> —— "28" 自动转成了 28.0

# 无法转换时抛出 ValidationError，给出清晰的错误信息
try:
    data3 = WeatherData(city="广州", temperature="abc", humidity=60, condition="雨")
except Exception as e:
    print(e)
    # temperature: Input should be a valid number, unable to parse string as a number
```

**核心理解**：Pydantic = 类型注解 + 运行时校验。你写的 `temperature: float` 不仅被 IDE 看见，Pydantic 在运行时也会真的检查它是不是 float。

### 6.4 Field()：添加约束条件

```python
from pydantic import BaseModel, Field

class Product(BaseModel):
    name: str = Field(min_length=2, max_length=100, description="商品名称")
    price: float = Field(gt=0, le=999999, description="价格（元）")
    quantity: int = Field(ge=0, default=0, description="库存数量")
    tags: list[str] = Field(default_factory=list, max_items=10, description="标签")

# 尝试创建非法数据
try:
    p = Product(name="X", price=-5.0, quantity=-1)
except Exception as e:
    print(e)
    # name: 长度至少为 2；price: 必须大于 0；quantity: 必须 >= 0
```

Field 常用参数速查：

| 参数 | 作用 | 示例 |
|------|------|------|
| `gt`, `ge`, `lt`, `le` | 数值范围 | `Field(gt=0, le=100)` |
| `min_length`, `max_length` | 字符串长度 | `Field(min_length=2)` |
| `pattern` | 正则匹配 | `Field(pattern=r"^\d{11}$")` |
| `default` | 默认值 | `Field(default="未知")` |
| `default_factory` | 默认值工厂（可变类型） | `Field(default_factory=list)` |
| `description` | 字段说明（给文档生成用） | `Field(description="用户名")` |

### 6.5 序列化：模型转字典/JSON

```python
# 模型 → 字典
print(data.model_dump())        # {'city': '北京', 'temperature': 26.5, ...}

# 模型 → JSON 字符串
print(data.model_dump_json())   # '{"city":"北京","temperature":26.5,...}'

# JSON 字符串 → 模型（运行时校验）
raw = '{"city":"深圳","temperature":30.0,"humidity":70,"condition":"多云"}'
parsed = WeatherData.model_validate_json(raw)
print(parsed.city)  # 深圳
```

**对比 Java/C**：在 Java 中你需要用 Jackson/Gson 库做 JSON 到对象的映射，还要自己加 `@NotNull` 等注解做校验。Pydantic 把类型声明和校验合二为一了。

## 7. mypy：在提交代码前发现类型错误

### 7.1 安装和第一次运行

```bash
pip install mypy
```

写一段有类型错误的代码 `test_types.py`：

```python
# test_types.py
def add(a: int, b: int) -> int:
    return a + b

# 故意传错类型
result = add("hello", 5)   # 传了字符串和整数
print(result)

# 函数声明返回 int，但实际可能返回 None
def get_user(name: str) -> dict:
    users = {"alice": {"age": 25}}
    return users.get(name)   # .get() 可能返回 None！
```

运行 mypy：

```bash
mypy test_types.py
```

mypy 会报告：

```
test_types.py:5: error: Argument 1 to "add" has incompatible type "str"; expected "int"
test_types.py:8: error: Incompatible return value type (got "dict[Any, Any] | None", expected "dict[Any, Any]")
```

**注意**：Python 解释器运行这个文件不会报错（它不检查类型），但 mypy 会提前发现这两处类型不匹配。

### 7.2 修复错误

```python
# 修复后
def add(a: int, b: int) -> int:
    return a + b

result = add(10, 5)   # 传正确的类型
print(result)          # 15

def get_user(name: str) -> dict | None:   # 返回值加上 | None
    users = {"alice": {"age": 25}}
    return users.get(name)
```

### 7.3 常用命令

```bash
# 检查单个文件
mypy app.py

# 检查整个目录
mypy src/

# 严格模式（推荐新项目）
mypy --strict src/
```

## 8. 常见错误

### 错误 1：以为类型注解会改变运行时行为

```python
# 错误认知：以为 Python 会检查类型
def add(a: int, b: int) -> int:
    return a + b

add("a", "b")  # 返回 "ab"，不会报错！类型注解只给工具看

# 正确理解：类型注解是写给工具（IDE、mypy）看的文档
```

### 错误 2：Python 3.8 环境下用小写泛型

```python
# Python 3.8 中直接写会报错
x: list[int]   # TypeError: 'type' object is not subscriptable

# 解决办法 1：文件开头加这一行
from __future__ import annotations
x: list[int]   # 现在可以了

# 解决办法 2：用 typing 的大写版本
from typing import List
x: List[int]
```

### 错误 3：Pydantic v1 vs v2 API 混用

```python
# Pydantic v2（当前主流，推荐）:
user.model_dump()        # 模型转字典
user.model_dump_json()   # 模型转 JSON 字符串

# Pydantic v1（遗留代码中可能出现）:
user.dict()              # v2 中已废弃！改用 model_dump()
user.json()              # v2 中已废弃！改用 model_dump_json()
```

### 错误 4：`search` 后不判空直接调 `.group()`——虽与类型无关但有关联

```python
# mypy 会报告：可能为 None 的值未处理
def get_username(email: str) -> str | None:
    m = re.search(r"(\w+)@", email)
    return m.group(1)   # mypy 警告：m 可能是 None！

# 正确写法：先判空
def get_username(email: str) -> str | None:
    m = re.search(r"(\w+)@", email)
    if m is not None:
        return m.group(1)
    return None
```

### 错误 5：过度使用 Any——让它传染了

```python
# 不好：Any 有传染性
from typing import Any
def load_config() -> Any:             # 返回 Any
    return {"host": "localhost"}
config = load_config()                 # config 类型是 Any
print(config["host"].upper())         # IDE 不会提示，mypy 不会检查

# 好：用具体类型
def load_config() -> dict[str, str]:   # 返回具体类型
    return {"host": "localhost"}
config = load_config()                 # config 类型是 dict[str, str]
print(config["host"].upper())         # IDE 能补全，mypy 能检查
```

**练习 3：用 Pydantic 定义一个数据模型**

用 Pydantic 定义一个 `Student` 模型：
- `name`: str，最少 2 个字符
- `student_id`: str，必须匹配模式 `"S\d{6}"`（S 后面 6 位数字）
- `score`: float，在 0-100 之间
- `grade`: Literal["A", "B", "C", "D", "F"]，默认 "C"

创建几个实例测试校验是否生效。

<details>
<summary>参考答案（点击展开）</summary>

```python
from pydantic import BaseModel, Field
from typing import Literal

class Student(BaseModel):
    name: str = Field(min_length=2, description="姓名")
    student_id: str = Field(pattern=r"^S\d{6}$", description="学号")
    score: float = Field(ge=0, le=100, description="成绩")
    grade: Literal["A", "B", "C", "D", "F"] = Field(default="C", description="等级")

# 测试合法数据
s1 = Student(name="小明", student_id="S001234", score=88.5)
print(s1.model_dump())  # {'name': '小明', 'student_id': 'S001234', 'score': 88.5, 'grade': 'C'}

# 测试非法数据
try:
    s2 = Student(name="X", student_id="A001234", score=150)
except Exception as e:
    print(e)
    # name: 长度至少为 2
    # student_id: 不匹配模式
    # score: 必须 <= 100
```

**练习 4：用 Pydantic 设计 LLM API 的请求和响应模型**

设计一套 Pydantic 模型来描述一个"情感分析 API"的交互：

**请求模型** (`AnalysisRequest`)：
- `text`: str，最少 1 个字符
- `model`: str，默认 "default"
- `language`: str，默认 "zh"
- `max_tokens`: int，默认 512，必须大于 0

- `sentiment`: Literal["positive", "negative", "neutral"]
- `confidence`: float，0-1 之间
- `tokens_used`: int，非负
- `details`: 嵌套模型 `SentimentDetails`，包含三个 0-1 范围的 float 字段：`positive_score`、`negative_score`、`neutral_score`

from pydantic import BaseModel, Field
from typing import Literal

class SentimentDetails(BaseModel):
    """情感分析详情——三个维度的得分"""
    positive_score: float = Field(ge=0, le=1)
    negative_score: float = Field(ge=0, le=1)
    neutral_score: float = Field(ge=0, le=1)

class AnalysisRequest(BaseModel):
    """情感分析请求"""
    text: str = Field(min_length=1)
    model: str = Field(default="default")
    language: str = Field(default="zh")
    max_tokens: int = Field(default=512, gt=0)

class AnalysisResponse(BaseModel):
    """情感分析响应"""
    sentiment: Literal["positive", "negative", "neutral"]
    confidence: float = Field(ge=0, le=1)
    tokens_used: int = Field(ge=0)
    details: SentimentDetails   # 嵌套模型

# ---- 测试 ----
# 模拟 API 返回的 JSON
mock_response_json = """
{

    "details": {

        "negative_score": 0.03,
        "neutral_score": 0.08
    }
}
"""

result = AnalysisResponse.model_validate_json(mock_response_json)
print(f"情感: {result.sentiment}, 置信度: {result.confidence}")
print(f"负面分数: {result.details.negative_score}")  # 可以链式访问嵌套字段

# 测试错误数据（sentiment 不合法 + confidence 超出范围）

except Exception as e:
    print(e)

    # confidence: 必须 <= 1
    # tokens_used: 必须 >= 0

## 10. 本章小结

| 工具 | 时机 | 作用 |
|------|------|------|
| 类型注解语法 (`x: int`) | 编写代码时 | 让 IDE 能补全、让代码自文档化 |
| mypy | 提交前 / CI 流程 | 静态类型检查，提前发现类型不匹配 |
| Pydantic BaseModel | 运行时 | 自动校验外部数据，类型转换 + 序列化 |

核心原则：

1. **渐进式采用**：先给接口函数加注解，不要试图一次性覆盖整个项目。
2. **`list[int]` 优于 `List[int]`**（Python 3.9+），**`X | None` 优于 `Optional[X]`**（Python 3.10+）。
3. **类型注解不改变运行时行为**——它写给工具看，Python 解释器不管它。
4. **Pydantic 是 AI 开发的标配**：FastAPI、LangChain、LlamaIndex 等框架都用 Pydantic 做数据建模。
5. **警惕 Any 的传染性**：一旦变量类型变成 Any，后续所有操作 mypy 都不会检查。
