## 22.1 先体验一下：同步的"痛"和异步的"快"

### 22.1.1 同步版本：逐一下载，很慢

想象你要下载 10 个网页。最自然的写法就是一个 for 循环：

```python
import requests
import time

URLS = [
    "https://httpbin.org/delay/1",   # 这个地址会故意延迟 1 秒
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
]


def sync_download():
    """同步下载：一个接一个，像排队买饭。"""
    start = time.perf_counter()
    for i, url in enumerate(URLS):
        resp = requests.get(url, timeout=10)
        print(f"  第 {i + 1} 个完成", flush=True)
    elapsed = time.perf_counter() - start
    print(f"同步版本总耗时: {elapsed:.2f} 秒")
    return elapsed


sync_download()
# 典型输出：
#   第 1 个完成
#   第 2 个完成
#   ...
#   第 10 个完成
#   同步版本总耗时: 15.23 秒
```

10 个请求，每个大约 1.5 秒（网络传输也有开销），串行执行就是大约 15 秒。但仔细想：每个请求的 1.5 秒里，CPU 真正干活的时间可能只有几毫秒——其余时间都在**等网络**。

这就像你去食堂打饭，排了 10 个窗口的队，但你只能一次排一个——排完第一个才能排第二个。

### 22.1.2 异步版本：同时发出，很快

```python
import asyncio
import aiohttp
import time


async def async_download():
    """异步下载：10 个请求同时发出，像 10 个窗口同时排队。"""
    start = time.perf_counter()

    async with aiohttp.ClientSession() as session:
        tasks = []
        for i, url in enumerate(URLS):
            task = asyncio.create_task(fetch_one(session, url, i))
            tasks.append(task)
        await asyncio.gather(*tasks)

    elapsed = time.perf_counter() - start
    print(f"异步版本总耗时: {elapsed:.2f} 秒")
    return elapsed


async def fetch_one(session, url, index):
    async with session.get(url, timeout=aiohttp.ClientTimeout(total=10)) as resp:
        await resp.text()
        print(f"  第 {index + 1} 个完成", flush=True)


asyncio.run(async_download())
# 典型输出（顺序不确定，因为同时完成）：
#   第 3 个完成
#   第 7 个完成
#   第 1 个完成
#   ...
#   异步版本总耗时: 1.62 秒
```

10 个请求的完成顺序是乱的（谁先回来先打印谁），但总耗时从 **15 秒变成了 1.6 秒**——接近单个请求的耗时。这就是异步并发的价值。

**关键洞察**：异步不是让单个请求变快了，而是让多个请求的等待时间**重叠**了。在等第 1 个请求的网络回复时，CPU 去处理第 2 个请求，再去处理第 3 个…… 就像你在等水烧开的同时去切菜，而不是傻站着等。

## 22.2 使用层面：先会写，再理解

不要一开始就去想"事件循环底层怎么实现的"。先学会四个核心操作，能写出能跑的异步代码：

### 22.2.1 `async def` —— 定义一个协程

```python
# 普通函数用 def
def normal_func():
    return "我是一个普通函数"

# 协程函数用 async def
async def coroutine_func():
    return "我是一个协程"
```

调用它试试：

```python
result = coroutine_func()
print(type(result))   # <class 'coroutine'>
print(result)         # <coroutine object coroutine_func at 0x...>
# 注意：函数体没有执行！只是创建了一个"协程对象"
```

这和普通函数完全不同：调用 `async def` 函数**不会执行函数体**，而是返回一个协程对象——你可以把它理解为"一个还没开始执行的任务计划书"。

> **对比 Java**：这类似于创建一个 `Runnable` 对象但还没有 `Thread.start()`。或者更接近 Java 19+ 的虚拟线程——你创建了它，但它还没开始跑。

### 22.2.2 `await` —— 等待一个协程完成

```python
import asyncio


async def fetch_data():
    print("开始获取数据...")
    await asyncio.sleep(1)     # 模拟 1 秒的网络等待
    print("数据获取完成！")
    return "data123"


async def main():
    print("调用 fetch_data() 之前")
    result = await fetch_data()   # await = "等这个协程完成，拿到返回值"
    print(f"拿到结果: {result}")
    print("main 结束")


asyncio.run(main())
```

输出：

```text
调用 fetch_data() 之前
开始获取数据...
（等 1 秒）
数据获取完成！
拿到结果: data123
main 结束
```

`await` 的含义是：**暂停当前协程，转去执行其他可以执行的任务，直到 await 后面的东西完成为止**。

### 22.2.3 `asyncio.run()` —— 启动整个异步程序

```python
# asyncio.run() 是异步世界的"入口"
# 它创建一个事件循环，运行你的主协程，然后关闭事件循环
asyncio.run(main())
```

一条重要的规则：**`asyncio.run()` 在一个进程中只能调用一次**。因为这个函数会创建事件循环并在 main 完成后关闭它。

> 在 Jupyter Notebook 里不需要 `asyncio.run()`——Jupyter 自身已经运行在事件循环中，直接 `await` 即可。

### 22.2.4 `asyncio.gather()` —— 并发执行多个协程

这是最重要的并发工具。它同时启动多个协程，等所有都完成后返回结果列表。

```python
import asyncio
import time


async def task(name: str, delay: float) -> str:
    """模拟一个耗时任务。"""
    print(f"[{name}] 开始，需要 {delay} 秒")
    await asyncio.sleep(delay)
    print(f"[{name}] 完成")
    return f"{name} 的结果"


async def main():
    start = time.perf_counter()

    # gather 同时启动 3 个 task，总耗时 = 最长的那个（2 秒）
    results = await asyncio.gather(
        task("任务A", 2.0),
        task("任务B", 1.0),
        task("任务C", 0.5),
    )

    elapsed = time.perf_counter() - start
    print(f"结果: {results}")
    print(f"总耗时: {elapsed:.2f}s")      # ~2.0s，不是 3.5s


asyncio.run(main())
```

输出示例：

```text
[任务A] 开始，需要 2.0 秒
[任务B] 开始，需要 1.0 秒
[任务C] 开始，需要 0.5 秒
[任务C] 完成
[任务B] 完成
[任务A] 完成
结果: ['任务A 的结果', '任务B 的结果', '任务C 的结果']
总耗时: 2.00s
```

注意：`gather()` 返回的结果顺序**和传入参数的顺序一致**，而不是按完成时间排序。任务 C 虽然最先完成，但它在结果的第 3 个位置。

## 22.3 回头理解事件循环

现在你已经能写出基本的异步代码了，来理解背后的机制。

**事件循环**相当于一个"待办事项管理器"：

```text
事件循环（一直在跑）：
  1. 看看有没有任务可以执行 → 有就执行一段
  2. 当前任务遇到 await → 把它挂起，切换到另一个任务
  3. 之前挂起的任务等到了结果 → 把它重新加入待办队列
  4. 重复 1-3，直到所有任务都完成
```

关键概念：

- **协程**（coroutine）：`async def` 定义的函数，可以暂停和恢复
- **Task**：被事件循环"托管"的协程。用 `asyncio.create_task()` 创建
- **事件循环**：调度所有 Task 的调度器

```python
import asyncio


async def say_hello():
    print("Hello")
    await asyncio.sleep(0.1)    # 暂停，让出控制权
    print("World")


async def main():
    # create_task 把协程包装成 Task 并注册到事件循环
    task1 = asyncio.create_task(say_hello())
    task2 = asyncio.create_task(say_hello())

    print("Tasks 已创建，等待完成...")
    await task1
    await task2
    print("全部完成")


asyncio.run(main())
```

> **与 Java 对比**：Python 的协程类似于 Java 19+ 的虚拟线程（Virtual Threads），但有一个关键区别：Java 的虚拟线程是**抢占式调度**的（JVM 自动切换），而 Python 的协程是**协作式调度**的——只有显式 `await` 时才切换。这意味着 Python 协程里如果写了一个 `while True: pass`，整个事件循环都会被卡死。

## 22.4 aiohttp —— 异步 HTTP 请求

requests 很好用，但它是同步的——在 `await` 的世界里会阻塞事件循环。需要用 aiohttp 替代。

```bash
pip install aiohttp
```

```python
import asyncio
import aiohttp
import time


async def fetch(session: aiohttp.ClientSession, url: str) -> str:
    """异步获取一个 URL 的内容。"""
    async with session.get(url, timeout=aiohttp.ClientTimeout(total=10)) as resp:
        return await resp.text()


async def main():
    urls = [f"https://httpbin.org/delay/1?n={i}" for i in range(5)]

    start = time.perf_counter()
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url) for url in urls]
        results = await asyncio.gather(*tasks)
    elapsed = time.perf_counter() - start

    print(f"完成 {len(results)} 个请求，总耗时: {elapsed:.2f}s")
    # 5 个请求并发，总耗时约 1.5s，而不是 7.5s


asyncio.run(main())
```

**控制并发数量**：如果同时发 100 个请求会撑爆服务器，用 Semaphore 限流：

```python
async def fetch_limited(session, sem, url):
    async with sem:   # 等待一个"槽位"空出来
        async with session.get(url, timeout=aiohttp.ClientTimeout(total=10)) as resp:
            return await resp.text()


async def main():
    sem = asyncio.Semaphore(3)   # 最多 3 个并发

    async with aiohttp.ClientSession() as session:
        urls = [f"https://httpbin.org/delay/1?n={i}" for i in range(15)]
        tasks = [fetch_limited(session, sem, url) for url in urls]
        results = await asyncio.gather(*tasks)
    # 15 个请求，最多 3 个并发 → 约 5 批 * 1.5s ≈ 7.5s
```

## 22.5 GIL —— 全局解释器锁

### 22.5.1 用一个比喻理解 GIL

想象 Python 解释器是一个办公室，里面有一个白板（Python 对象的共享内存）。GIL 就是这个办公室唯一的一支白板笔。

- **任何时候只有一个人（线程）能拿笔写字（执行 Python 字节码）**
- 有人写完一段就会放下笔让别人拿（默认每 5 毫秒切换一次）
- 如果有人在打电话等回复（I/O 操作），他会主动放下笔让别人用

所以：
- **I/O 密集**（打电话、等外卖）：放下笔的时间很多，大家都能干活 → 多线程有效
- **CPU 密集**（疯狂演算数学题）：一直拿着笔不放手 → 多线程没有用，甚至因为抢笔更慢

### 22.5.2 对实际开发的影响

| 场景 | GIL 影响 | 正确方案 |
|------|----------|---------|
| 网络请求、文件读写、数据库查询 | 几乎无（I/O 时释放 GIL） | asyncio 或 ThreadPoolExecutor |
| 纯 Python 数学计算、图像处理 | 严重（多线程比单线程慢） | ProcessPoolExecutor |
| 使用 NumPy/pandas | 无（底层是 C，已释放 GIL） | 随便用，C 扩展自己管锁 |

## 22.6 ThreadPoolExecutor —— I/O 密集型任务的备选方案

如果你的任务是 I/O 密集型但你不想全面改成 async，可以用线程池：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed
import requests
import time

URLS = [f"https://httpbin.org/delay/1?n={i}" for i in range(10)]


def download(url: str) -> int:
    """同步下载函数（跟平时写的一模一样，不需要 async）。"""
    resp = requests.get(url, timeout=10)
    return resp.status_code


start = time.perf_counter()
with ThreadPoolExecutor(max_workers=5) as executor:
    # executor.map 保持输入顺序
    results = list(executor.map(download, URLS))

elapsed = time.perf_counter() - start
print(f"10 个请求，5 个线程并发: {elapsed:.2f}s")
# 约 2 批 * 1.5s ≈ 3s，而不是串行的 15s
```

**ThreadPoolExecutor vs asyncio 怎么选？**

- 项目中已经在用 `async/await`：用 aiohttp
- 调用不支持异步的阻塞库（如某些数据库驱动）：用 ThreadPoolExecutor
- 简单脚本，不想改代码结构：用 ThreadPoolExecutor

## 22.7 ProcessPoolExecutor —— CPU 密集型的正确打开方式

对于纯 Python 的 CPU 计算，要用多进程绕过 GIL：

```python
from concurrent.futures import ProcessPoolExecutor
import time


def heavy_calculation(n: int) -> int:
    """一个 CPU 密集型计算：求 0 到 n 的平方和。"""
    total = 0
    for i in range(n):
        total += i ** 2
    return total


numbers = [10_000_000] * 8     # 8 个相同的计算任务
start = time.perf_counter()

with ProcessPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(heavy_calculation, numbers))

elapsed = time.perf_counter() - start
print(f"4 进程并行: {elapsed:.2f}s")
# 4 核机器上约是单进程的 1/4 时间
```

**为什么 ProcessPoolExecutor 能绕过 GIL？** 因为每个进程有自己独立的 Python 解释器实例，也就有自己独立的 GIL。多个进程之间不抢同一把锁。

值得注意的代价是：进程间数据要序列化传递（pickle），而且创建进程比创建线程重很多。

## 22.8 选择指南：一图胜过千言

```text
你的任务瓶颈在哪？
│
├── 网络/磁盘/数据库等待 (I/O 密集)
│   ├── 代码已是 async？ → 用 asyncio + aiohttp
│   ├── 需要超高并发（>100）？ → 用 asyncio
│   └── 不想重构成 async？ → 用 ThreadPoolExecutor
│
├── 数学计算/图像处理 (CPU 密集)
│   ├── 纯 Python 计算？ → 用 ProcessPoolExecutor
│   └── 能用 numpy/pandas？ → 直接用（底层是 C，不受 GIL 限制）
│
└── 既有 I/O 又有 CPU？
    └── asyncio + loop.run_in_executor() 混合使用
```

**AI 开发的经验法则**：90% 的并发需求是 I/O 密集型（调模型 API、查向量数据库、读写文件），所以 asyncio 是首选。数据预处理尽量用 NumPy/Pandas——它们底层是 C 代码，已经释放了 GIL，多线程也能加速。

## 22.9 本章小结

1. **异步解决什么问题**：I/O 等待期间不浪费 CPU，让多个 I/O 操作的等待时间重叠。
2. **四个核心操作**：`async def` 定义协程，`await` 等待结果，`asyncio.run()` 启动，`asyncio.gather()` 并发。
3. **事件循环**：一个"调度器"，管理所有协程的暂停和恢复。先会用再理解。
4. **aiohttp**：异步的 HTTP 客户端，配合 Semaphore 控制并发数量。
5. **GIL**：CPython 的全局锁，同一时刻只有一个线程执行 Python 字节码。I/O 密集无影响，CPU 密集是瓶颈。
6. **ThreadPoolExecutor**：I/O 密集但不想改异步时的备选。**ProcessPoolExecutor**：CPU 密集时绕过 GIL 的方案。

### 练习 1：用 asyncio.sleep 模拟并发等待

写一个程序，用 `asyncio.gather()` 同时运行 3 个协程，分别等待 3 秒、2 秒、1 秒后打印完成信息。验证总耗时约为 3 秒而非 6 秒。

**参考答案：**

```python
import asyncio
import time


async def wait_and_print(name: str, delay: float):
    print(f"[{name}] 开始等待 {delay} 秒")
    await asyncio.sleep(delay)
    print(f"[{name}] 完成！")
    return f"{name}-done"


async def main():
    start = time.perf_counter()

    results = await asyncio.gather(
        wait_and_print("协程A", 3.0),
        wait_and_print("协程B", 2.0),
        wait_and_print("协程C", 1.0),
    )

    elapsed = time.perf_counter() - start
    print(f"结果: {results}")
    print(f"总耗时: {elapsed:.2f}s (预期约 3.0s)")


asyncio.run(main())
# 输出：
# [协程A] 开始等待 3.0 秒
# [协程B] 开始等待 2.0 秒
# [协程C] 开始等待 1.0 秒
# [协程C] 完成！
# [协程B] 完成！
# [协程A] 完成！
# 结果: ['协程A-done', '协程B-done', '协程C-done']
# 总耗时: 3.00s (预期约 3.0s)
```

### 练习 2：用 asyncio.gather 并发执行 3 个不同逻辑的协程

三个协程分别模拟：（1）查询用户信息（等 1 秒），（2）查询订单列表（等 2 秒），（3）写日志（等 0.5 秒）。用 gather 并发执行，收集所有结果。

import asyncio


async def fetch_user():

### 练习 3：用 ThreadPoolExecutor 并发下载

用 ThreadPoolExecutor 并发获取 5 个不同的 GitHub 用户信息（URL 列表），打印每个用户的 login 和 public_repos。

**参考答案：**

```python
from concurrent.futures import ThreadPoolExecutor, as_completed
import requests

URLS = [
    "https://api.github.com/users/torvalds",
    "https://api.github.com/users/gvanrossum",
    "https://api.github.com/users/kennethreitz",
    "https://api.github.com/users/mitsuhiko",
    "https://api.github.com/users/tiangolo",
]


def get_user(url: str) -> dict:
    """获取单个用户信息。"""
    resp = requests.get(url, timeout=10)
    resp.raise_for_status()
    return resp.json()


with ThreadPoolExecutor(max_workers=3) as executor:
    # 用 as_completed 按完成顺序处理
    future_to_url = {executor.submit(get_user, url): url for url in URLS}
    for future in as_completed(future_to_url):
        url = future_to_url[future]
        try:
            user = future.result()
            print(f"{user['login']}: {user['public_repos']} repos")
        except Exception as e:
            print(f"{url} 失败: {e}")
# 输出顺序不确定，谁先返回先打印谁
```

### 练习 4：用 aiohttp 并发请求并对比同步版本耗时

用 aiohttp 并发请求 10 个 URL（相同的或不同的），统计总耗时。再写一个同步版本，对比两者的耗时差异，计算加速比。

```python
import asyncio
import aiohttp
import requests
import time

# 测试用 URL 列表（httpbin.org/delay/N 会延迟 N 秒后返回）
URLS = [f"https://httpbin.org/delay/1?n={i}" for i in range(10)]


# ============ 同步版本 ============
def sync_fetch_all():
    start = time.perf_counter()
    results = []
    for url in URLS:
        resp = requests.get(url, timeout=15)
        results.append(resp.status_code)
    elapsed = time.perf_counter() - start
    return elapsed, results


# ============ 异步版本 ============
async def async_fetch_one(session, url):
    async with session.get(url, timeout=aiohttp.ClientTimeout(total=15)) as resp:
        return resp.status


async def async_fetch_all():
    start = time.perf_counter()
    async with aiohttp.ClientSession() as session:
        tasks = [async_fetch_one(session, url) for url in URLS]
        results = await asyncio.gather(*tasks)
    elapsed = time.perf_counter() - start
    return elapsed, results


# ============ 运行对比 ============
print("运行同步版本...")
sync_time, sync_results = sync_fetch_all()
print(f"同步: {sync_time:.2f}s, 结果数: {len(sync_results)}")

print("\n运行异步版本...")
async_time, async_results = asyncio.run(async_fetch_all())
print(f"异步: {async_time:.2f}s, 结果数: {len(async_results)}")

print(f"\n加速比: {sync_time / async_time:.1f}x")
# 典型输出：
# 同步: 15.2s
# 异步: 1.6s
# 加速比: 9.5x
```

## 常见错误

### 错误 1：调了 async def 函数但没 await

```python
# ❌ 错误：函数体不执行，只得到一个 coroutine 对象
async def get_data():
    return "data"

result = get_data()              # 不会执行函数体！
print(result)                    # <coroutine object get_data at 0x...>
# 还会看到警告: RuntimeWarning: coroutine 'get_data' was never awaited
```

```python
# ✓ 正确：用 asyncio.run() 运行
result = asyncio.run(get_data())
print(result)                    # "data"
```

### 错误 2：在协程里用了 time.sleep() 而不是 asyncio.sleep()

```python
# ❌ 错误：time.sleep() 会阻塞整个线程，包括事件循环
async def bad_sleep():
    time.sleep(5)     # 整个程序卡死 5 秒，其他协程也动不了
    return "done"
```

```python
# ✓ 正确：用 asyncio.sleep()，让出控制权
async def good_sleep():
    await asyncio.sleep(5)   # 暂停当前协程，其他协程可以运行
    return "done"
```

### 错误 3：create_task 创建的任务没有 await 就退出了

```python
# ❌ 错误：task 还没完成，main() 就退出了，任务被静默取消
async def main():
    asyncio.create_task(asyncio.sleep(10))
    print("main 退出")  # 10 秒的等待被取消了！
```

```python
# ✓ 正确：保持对 task 的引用并 await
async def main():
    task = asyncio.create_task(asyncio.sleep(10))
    await task            # 等任务完成再退出
    print("main 退出")
```

### 错误 4：多线程跑 CPU 密集任务，反而更慢

```python
# ❌ 错误：CPU 密集 + 多线程 = 更慢（GIL + 上下文切换开销）
from concurrent.futures import ThreadPoolExecutor
def calc(n):
    return sum(i**2 for i in range(n))

with ThreadPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(calc, [10_000_000]*4))
# 比单线程按顺序跑还慢！
```

```python
# ✓ 正确：CPU 密集用进程池
from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(calc, [10_000_000]*4))
# 4 核机器上约是单进程的 1/4
```

### 错误 5：在已运行的事件循环中又调 asyncio.run()

```python
# ❌ 错误：Jupyter 中这样写会报错
asyncio.run(some_coroutine())
# RuntimeError: asyncio.run() cannot be called from a running event loop
```

```python
# ✓ 正确：Jupyter 中直接 await
await some_coroutine()
# Jupyter 自身已经在事件循环里运行了
```

### 错误 6：async 代码中忘了 await，导致串行执行

```python
# ❌ 错误：虽然写了 async def，但没有并发
async def main():
    r1 = await fetch("url1")     # 等 r1 完成
    r2 = await fetch("url2")     # 再等 r2 完成
    r3 = await fetch("url3")     # 再等 r3 完成
    # 三个请求串行，总耗时 = 三个之和，异步白写了！
```

```python
# ✓ 正确：用 gather 并发
async def main():
    r1, r2, r3 = await asyncio.gather(
        fetch("url1"),
        fetch("url2"),
        fetch("url3"),
    )
    # 三个请求同时发出，总耗时 ≈ 最慢的那个
```
