## 1. 没有异常处理会怎样？

先看一个"没有异常处理"的程序——用户输入了非数字，程序直接崩溃：

```python
# 一个简单的除法程序
a = input("请输入被除数: ")      # 用户输入 "10"
b = input("请输入除数: ")        # 用户输入 "abc"（不小心）
a = int(a)
b = int(b)                        # 报错！程序在这里崩溃
print(f"{a} / {b} = {a / b}")    # 这行永远不会执行
```

运行结果：
```
请输入被除数: 10
请输入除数: abc
Traceback (most recent call last):
  File "test.py", line 5, in <module>
    b = int(b)
ValueError: invalid literal for int() with base 10: 'abc'
```

**发生了什么？** 程序在第 5 行 `int(b)` 处抛出 `ValueError`，因为没有 try/except 捕获它，Python 直接终止程序并打印调用栈（traceback）。用户看到这个非常困惑，而且之前输入的数据也白费了。

**Java 对比**：Java 中类似的情况会抛出 `NumberFormatException`，如果你没写 try-catch，在线程的 `uncaughtExceptionHandler` 处理或直接导致线程终止。但 Java 有 checked exception 强制你处理，Python 则完全依赖自觉。

现在引入 try/except 来优雅地处理：

```python
try:
    a = int(input("请输入被除数: "))
    b = int(input("请输入除数: "))
    print(f"{a} / {b} = {a / b}")
except ValueError:
    # 当 int() 转换失败时执行这里
    print("输入错误：请输入整数！")
except ZeroDivisionError:
    # 当除数为 0 时执行这里
    print("数学错误：除数不能为 0！")
```

现在用户输入 "abc" 时，程序不会崩溃，而是友好地提示"输入错误：请输入整数！"。**这就是异常处理的价值：让程序在出错时依然可控。**

## 2. 常见异常类型逐个演示

下面每种异常都故意用错代码触发，并详细解读报错信息。**建议你在本地 Python 中亲自试一遍**，看报错看得多了，以后一眼就能定位问题。

### 2.1 NameError —— 用了未定义的变量

```python
print(my_variable)  # 没定义过 my_variable
```

报错解读：
```
NameError: name 'my_variable' is not defined
```
"name ... is not defined" 就是 Python 在说"我不认识这个东西"。通常是因为变量名打错了，或者忘了先赋值就使用。

### 2.2 TypeError —— 操作类型不匹配

```python
result = "hello" + 5     # 字符串 + 整数，类型不兼容
```

报错解读：
```
TypeError: can only concatenate str (not "int") to str
```
把整数 5 和字符串拼在一起没意义。解决方案：`"hello" + str(5)` 把整数转为字符串后再拼接。

```python
len(42)    # len() 需要容器类型，不接收整数
# TypeError: object of type 'int' has no len()
```

### 2.3 ValueError —— 类型对但值不合法

```python
int("abc")    # "abc" 确实是字符串，但不能表示一个整数
```

报错解读：
```
ValueError: invalid literal for int() with base 10: 'abc'
```
"invalid literal" 意思是"字符串的内容不是合法的整数"。

```python
float("hello")    # 同样报错
# ValueError: could not convert string to float: 'hello'
```

### 2.4 KeyError —— 字典中不存在的键

```python
student = {"name": "小明", "age": 20}
print(student["score"])   # "score" 这个键不存在
```

报错解读：
```
KeyError: 'score'
```
字典没有 "score" 这个键。解决方案：用 `student.get("score", 0)` 带默认值的安全取值方式。

### 2.5 IndexError —— 列表索引越界

```python
nums = [10, 20, 30]
print(nums[5])   # 列表只有 3 个元素，索引 5 越界了
```

报错解读：
```
IndexError: list index out of range
```
"out of range" 就是说索引超过了列表的有效范围（0 到 长度-1）。

### 2.6 FileNotFoundError —— 文件不存在

```python
open("不存在的文件.txt")    # 文件根本不存在
```

报错解读：
```
FileNotFoundError: [Errno 2] No such file or directory: '不存在的文件.txt'
```
文件路径写错了，或者文件还没有创建。开文件前要先确认文件是否存在。

**汇总表格**（对照 Java）：

| Python 异常 | 触发场景 | Java 类比 |
|---|---|---|
| `NameError` | 未定义变量 | 编译错误（Java 无运行时等价） |
| `TypeError` | 类型操作不匹配 | `ClassCastException` 部分场景 |
| `ValueError` | 值不合法 | `NumberFormatException` |
| `KeyError` | 字典键不存在 | `null`（Java Map 返回 null 不抛异常） |
| `IndexError` | 列表索引越界 | `ArrayIndexOutOfBoundsException` |
| `FileNotFoundError` | 文件不存在 | `FileNotFoundException` |

## 3. try/except/else/finally 完整执行顺序

### 3.1 基础结构

```python
try:
    # 【尝试区】放可能出错的代码
    result = 10 / int(input("输入数字: "))
except ValueError:
    # 【捕获区】try 块中抛出 ValueError 时执行
    print("输入的不是数字！")
except ZeroDivisionError:
    # 【捕获区】try 块中抛出 ZeroDivisionError 时执行
    print("不能除以 0！")
else:
    # 【成功区】try 块没有发生任何异常时执行
    # 注意：else 里的代码如果出错了，不会被上面的 except 捕获！
    print(f"计算成功，结果是: {result}")
finally:
    # 【清理区】无论如何都执行，就算前面有 return 也会先执行 finally
    print("程序结束（无论是否出错都会看到这一行）")
```

**执行流程图解**：

```
进入 try 块
    |
    v
执行 try 中的代码
    |
    +--> 没出错? --> 跳过 except --> 执行 else 块 --> 执行 finally 块
    |
    +--> 出了 ValueError? --> 执行 except ValueError 块 --> 执行 finally 块
    |
    +--> 出了 ZeroDivisionError? --> 执行 except ZeroDivisionError 块 --> 执行 finally 块
    |
    +--> 出了未捕获的异常? --> 跳过所有 except --> 执行 finally 块 --> 异常继续向外抛出
```

### 3.2 finally 与 return 的交互（重要！）

这是一个经常被问到的面试题场景：

```python
def demo():
    try:
        print("第1步: try 开始")
        return "这是 try 的返回值"   # 注意这里有 return
    except Exception:
        print("第2步: except（不会执行）")
    else:
        print("第3步: else（不会执行，因为 try 中有 return）")
    finally:
        print("第4步: finally 无论如何都执行！")
    print("第5步: 这行永远不会执行")

result = demo()
print(f"最终返回值: {result}")
```

输出：
```
第1步: try 开始
第4步: finally 无论如何都执行！
最终返回值: 这是 try 的返回值
```

**关键理解**：虽然 try 中写了 `return`，但 Python 在执行 `return` 之前，**先执行了 finally 块**，然后再把值返回。finally 就像"临走前必须做的告别"，不管你是正常结束还是异常退出，都得先过 finally 这一关。

**Java 对比**：Java 的 try-catch-finally 行为几乎一样——finally 总在 return 之前执行。这是从 Java 学过来的同学可以直接迁移的知识。

## 4. 异常链：raise ... from ...

当你捕获一个异常后想抛出一个更高层的异常，可以用 `raise ... from ...` 保留"案发现场"：

```python
def load_config(filename):
    """加载配置文件"""
    try:
        with open(filename, "r", encoding="utf-8") as f:
            import json
            return json.load(f)
    except FileNotFoundError as e:
        # 把底层异常"包装"成业务异常，同时保留原始异常链
        raise RuntimeError(f"配置加载失败：找不到文件 {filename}") from e
    except json.JSONDecodeError as e:
        raise RuntimeError(f"配置加载失败：JSON 格式错误（第{e.lineno}行）") from e

# 调用方可以同时看到两个异常
try:
    config = load_config("config.json")
except RuntimeError as ex:
    print(f"业务异常: {ex}")
    print(f"根本原因: {ex.__cause__}")   # 通过 __cause__ 获取原始异常
```

**为什么要用异常链？** 就好比快递员送货失败，不是简单说"没送到"，而是告诉你是"客户不在家"或"地址写错了"。异常链让排查问题的人能追溯到最根本的原因。

**Java 对比**：Java 中 `throw new RuntimeException(msg, e)` 的第二个参数就是 cause，和 Python 的 `from e` 语义一致。

## 5. 自定义异常

当内置异常不足以表达业务含义时，创建自己的异常类：

```python
class OrderError(Exception):
    """订单相关的所有异常的基类"""

    def __init__(self, message, order_id=None):
        super().__init__(message)      # 调用父类 Exception 的构造
        self.order_id = order_id       # 附加业务信息

    def __str__(self):
        """控制 print(异常对象) 时的显示内容"""
        base = super().__str__()
        if self.order_id:
            return f"[订单 {self.order_id}] {base}"
        return base


# 创建更具体的子异常
class InsufficientStockError(OrderError):
    """库存不足"""
    pass

class PaymentFailedError(OrderError):
    """支付失败"""
    pass


# 使用
def place_order(order_id, item, quantity):
    stock = {"apple": 5, "banana": 3}
    if stock.get(item, 0) < quantity:
        raise InsufficientStockError(f"{item} 库存不足", order_id=order_id)
    print(f"订单 {order_id}: 购买 {quantity} 个 {item} 成功")

try:
    place_order("ORD-001", "apple", 100)
except OrderError as e:
    print(e)  # [订单 ORD-001] apple 库存不足
```

## 6. 上下文管理器（with 语句）

### 6.1 为什么需要上下文管理器？——从文件操作说起

不完善的代码：

```python
f = open("data.txt", "r", encoding="utf-8")
content = f.read()
# 如果上面某行出错了，f.close() 永远不会被调用
# 文件句柄泄漏！操作系统能同时打开的文件数有限
f.close()
```

使用 with 的完善代码：

```python
with open("data.txt", "r", encoding="utf-8") as f:
    content = f.read()
# 缩进结束，文件自动关闭！
# 就算 read() 中途出错，Python 也会保证文件被关闭
```

**`with open()` 等价于**：
```python
f = open("data.txt", "r", encoding="utf-8")
try:
    content = f.read()
finally:
    f.close()   # 无论是否出错都会关闭
```

`with open()` 不过是上面这种写法的"语法糖"，但省去了写 finally 的麻烦。

### 6.2 用 contextmanager 实现计时器

`@contextmanager` 让你用生成器函数来创建上下文管理器，比写类简单：

```python
from contextlib import contextmanager
import time

@contextmanager
def timer(name="代码块"):
    """一个计时的上下文管理器"""
    start = time.perf_counter()    # 进入时记录时间
    try:
        yield                       # 把控制权交给 with 块
    finally:
        elapsed = time.perf_counter() - start
        print(f"[{name}] 耗时: {elapsed:.3f} 秒")

# 使用
with timer("数据处理"):
    total = sum(range(10_000_000))  # 求和一千万个数
    print(f"结果: {total}")
# 输出:
# 结果: 49999995000000
# [数据处理] 耗时: 0.234 秒
```

**`yield` 是什么？** 在 `yield` 之前的代码相当于"准备工作"（进入 with 时执行），`yield` 之后的 finally 代码相当于"收尾工作"（退出 with 时执行，无论是否出错）。

### 6.3 写一个自定义上下文管理器类

用类的方法实现（实现 `__enter__` 和 `__exit__`）：

```python
class DatabaseConnection:
    """模拟数据库连接——一个上下文管理器"""

    def __init__(self, db_name):
        self.db_name = db_name
        self.connected = False

    def __enter__(self):
        # 进入 with 块时自动调用
        print(f"连接数据库: {self.db_name}")
        self.connected = True
        return self    # 返回值会绑定给 as 后的变量

    def __exit__(self, exc_type, exc_val, exc_tb):
        # 退出 with 块时自动调用（无论如何都会执行）
        print(f"断开数据库: {self.db_name}")
        self.connected = False
        # 返回 False = 不吞掉异常（正常行为）
        # 返回 True  = 吞掉异常（不推荐）
        return False

    def query(self, sql):
        if not self.connected:
            raise RuntimeError("未连接数据库")
        print(f"执行: {sql}")


# 使用
with DatabaseConnection("test_db") as db:
    db.query("SELECT * FROM users")
# 输出:
# 连接数据库: test_db
# 执行: SELECT * FROM users
# 断开数据库: test_db
```

**Java 对比**：Python 的 with 语句相当于 Java 的 try-with-resources（`try (Resource r = new Resource()) { ... }`），但 Python 的上下文管理器协议更灵活——任何对象实现 `__enter__`/`__exit__` 就能用于 with。

## 7. 基础练习

### 练习 1：安全的整数输入函数

**题目**：写一个函数 `safe_input_int(prompt)`，反复提示用户输入直到输入一个合法整数为止。如果用户输入非数字，提示错误并重新询问。

参考答案：

```python
def safe_input_int(prompt):
    """反复提示用户，直到输入一个合法整数为止"""
    while True:
        user_input = input(prompt)
        try:
            return int(user_input)    # 尝试转换
        except ValueError:
            # 转换失败时提示用户并重新循环
            print(f"'{user_input}' 不是一个整数，请重新输入。")


# 测试
age = safe_input_int("请输入年龄: ")
print(f"你的年龄是: {age}")
```

### 练习 2：安全地从字典取值

**题目**：写一个函数 `safe_get(data, key, default)`，从字典中安全地取出值。如果 key 不存在，返回 default；如果 data 本身不是字典，也返回 default 并提示错误。

参考答案：

```python
def safe_get(data, key, default=None):
    """安全地从字典取值"""
    try:
        return data[key]     # 尝试直接用 [] 取值
    except KeyError:
        return default       # key 不存在，返回默认值
    except TypeError:
        # data 根本就不是字典（可能是 None 或 list 等）
        print(f"警告: safe_get 期望一个字典，但收到了 {type(data).__name__}")
        return default


# 测试
d = {"name": "小明", "age": 20}
print(safe_get(d, "name", "未知"))     # 小明
print(safe_get(d, "score", 0))         # 0（key 不存在）
print(safe_get(None, "anything", 0))   # 0（并打印警告）
```

## 8. 进阶练习

### 练习 4：支持重试的文件读取

**题目**：实现函数 `read_file_with_retry(filepath, max_retries=3)`，读取指定文件内容。如果读取失败（文件不存在、权限不足等），等待递增的时间后重试（第 1 次等 0.5 秒，第 2 次等 1 秒，第 3 次等 2 秒）。全部失败后抛出带原始异常链的自定义异常。

参考答案：

```python
import time

class FileReadError(Exception):
    """文件读取失败的自定义异常"""
    def __init__(self, filepath, retries, message="文件读取失败"):
        super().__init__(message)
        self.filepath = filepath
        self.retries = retries

def read_file_with_retry(filepath, max_retries=3):
    """带重试机制的文件读取"""
    last_error = None
    for attempt in range(1, max_retries + 1):
        try:
            with open(filepath, "r", encoding="utf-8") as f:
                return f.read()
        except (FileNotFoundError, PermissionError, OSError) as e:
            last_error = e
            print(f"第 {attempt}/{max_retries} 次尝试失败: {e}")
            if attempt < max_retries:
                wait = 0.5 * (2 ** (attempt - 1))   # 指数退避: 0.5s, 1s, 2s
                time.sleep(wait)
    # 全部重试失败
    raise FileReadError(filepath, max_retries) from last_error


# 测试
try:
    content = read_file_with_retry("不存在的文件.txt", max_retries=3)
except FileReadError as e:
    print(f"最终失败: {e}，原始错误: {e.__cause__}")
```

### 练习 5：用 contextmanager 实现计时上下文

**题目**：用 `@contextmanager` 实现一个计时的上下文管理器，统计代码块的执行时间。

参考答案：

```python
from contextlib import contextmanager
import time

@contextmanager
def timed_block(name="操作"):
    """计时上下文管理器"""
    start = time.perf_counter()
    try:
        yield    # 把控制权交给 with 块中的代码
    finally:
        elapsed = time.perf_counter() - start
        print(f"[{name}] 耗时: {elapsed:.3f} 秒")


# 测试 1: 计时列表生成
with timed_block("生成 100 万个平方数"):
    squares = [x ** 2 for x in range(1_000_000)]

# 测试 2: 计时文件读写
with timed_block("写文件"):
    with open("test.txt", "w", encoding="utf-8") as f:
        for i in range(10000):
            f.write(f"第 {i} 行\n")

# 测试 3: 异常发生时也保证计时
try:
    with timed_block("出错的代码"):
        1 / 0
except ZeroDivisionError:
    print("捕获到了除零异常")
```

## 9. 常见错误

### 错误 1：bare except 吞掉所有异常

```python
# 错误：bare except 连 KeyboardInterrupt (Ctrl+C) 都吞掉
try:
    risky_operation()
except:                      # 等于 except BaseException
    pass                     # 什么也不做，异常无声无息消失了
```

正确写法：

```python
# 正确：至少指明 except Exception，并记录日志
try:
    risky_operation()
except Exception as e:
    print(f"操作失败: {e}")    # 至少让用户知道发生了什么
```

### 错误 2：把不会出错的代码放进 try 块

```python
# 不推荐：整个函数都放进 try，不知道哪些行可能出错
try:
    x = 1 + 1
    y = input("输入: ")
    z = int(y)
    print(x + z)
except ValueError:
    print("输入错误")
```

**问题**：`x = 1 + 1` 和第一个 `print` 不会出 ValueError，但也被放在 try 里了。如果这些代码出了其他类型的异常，会被漏掉。正确做法是只把可能出错的 `int(y)` 包起来，其余代码放在 else 或 try 外面。

### 错误 3：忘记 finally 中的 return 会覆盖 try 中的 return

```python
# 危险! finally 中的 return 会覆盖 try 中的返回值
def bad_func():
    try:
        return "try 的返回值"
    finally:
        return "finally 的返回值"   # 覆盖了上面的 return

print(bad_func())   # "finally 的返回值" —— try 的返回值丢失了！
```

正确做法：**永远不要在 finally 中使用 return / break / continue**。

### 错误 4：捕获异常后不处理直接 pass

```python
# 错误：静默忽略错误，排查问题时会非常痛苦
try:
    data = load_remote_data()
except ConnectionError:
    pass  # 连接失败了，但程序假装什么都没发生，继续往下跑
```

正确做法：至少记录日志，或者给变量设一个合理的默认值。

### 错误 5：异常类型写得太宽泛

```python
# 不推荐：用 Exception 捕获所有异常
try:
    value = int(user_input)
    result = 100 / value
except Exception as e:    # 太宽泛了，不知道具体是什么错误
    print(f"出错: {e}")
```

正确做法：根据实际情况捕获具体的异常类型：

```python
try:
    value = int(user_input)
    result = 100 / value
except ValueError:
    print("请输入一个数字！")
except ZeroDivisionError:
    print("不能为 0！")
```

### 错误 6：with 语句中忘记处理异常

```python
# 问题：with 只保证文件关闭，但如果 file.read() 出错了呢？
with open("data.txt", "r", encoding="utf-8") as f:
    data = f.read()
    numbers = [int(x) for x in data.split()]  # 如果文件内容不是纯数字呢？
```

正确做法：with 保证资源释放，但业务逻辑的异常仍需自己捕获：

```python
try:
    with open("data.txt", "r", encoding="utf-8") as f:
        data = f.read()
        numbers = [int(x) for x in data.split()]
except FileNotFoundError:
    print("文件不存在")
except ValueError:
    print("文件内容不是纯数字格式")
```

## 10. 本章小结

| 概念 | 核心要点 |
|---|---|
| 异常的作用 | 让程序在出错时可控，避免崩溃，提供友好提示 |
| 常见异常 | NameError / TypeError / ValueError / KeyError / IndexError / FileNotFoundError |
| try/except/else/finally | try 尝试, except 捕获, else 成功时执行, finally 必定执行 |
| finally 铁律 | 在 return 之前执行；不要在 finally 中用 return |
| 异常链 | `raise NewError(msg) from original_error` 保留根因 |
| 自定义异常 | 继承 Exception，添加业务属性，用 `super().__init__(msg)` |
| 上下文管理器 | `with` 语句; 实现 `__enter__`/`__exit__` 或用 `@contextmanager` + yield |
| 资源管理 | 文件、数据库连接、网络连接等用完必须释放，with 自动保证 |
