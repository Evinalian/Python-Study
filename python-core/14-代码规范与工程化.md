## 1. 从"能跑就行"到"还要给人看"

假设你写了一个数据处理脚本，它确实能跑通，功能也正确。但三个月后你回头维护它时，看到的却是：

```python
from math import *
from os import *
import sys, json, re

class data_processor:
    def __init__(s, raw):
        s.raw = raw
    def getResult(s):
        tmp = []
        for i in range(len(s.raw)):
            if s.raw[i] != "" and s.raw[i] != None:
                tmp.append(int(s.raw[i]))
        if len(tmp) == 0:
            return {}
        total = 0; max_val = -1; min_val = 999999
        for j in range(len(tmp)):
            total = total + tmp[j]
            if tmp[j] > max_val: max_val = tmp[j]
            if tmp[j] < min_val: min_val = tmp[j]
        d = {}
        d["total"] = total; d["max"] = max_val
        d["min"] = min_val; d["avg"] = total / len(tmp)
        return d
```

这段代码有三个层面的问题：

1. **格式混乱**：缩进缩在一行、分号滥用、import 乱七八糟
2. **不 Pythonic**：用 C 语言的索引循环来遍历 Python 列表
3. **命名糟糕**：`s`、`tmp`、`d`——三个月后你自己都不知道它们是什么

**本章目标**：把上面这样的代码，变成下面这样：

```python
"""A simple numeric data analyzer."""

from typing import Any


class DataProcessor:
    """Process raw string data into numeric statistics."""

    def __init__(self, raw_data: list[str]) -> None:
        self.raw_data = raw_data

    def get_result(self) -> dict[str, Any]:
        """Compute total, max, min, and average of valid numeric entries."""
        values = [int(s) for s in self.raw_data if s]

        if not values:
            return {}

        return {
            "total": sum(values),
            "max": max(values),
            "min": min(values),
            "avg": sum(values) / len(values),
        }
```

## 2. PEP 8 最实用规则

PEP 8 是 Python 官方的代码风格指南，全文很长。下面是你日常最需要遵守的 8 条规则。

### 2.1 命名约定

| 类型 | 命名方式 | 好例子 | 坏例子 |
|------|----------|--------|--------|
| 变量 | `snake_case` | `user_count`, `max_retry` | `userCount`, `uc` |
| 函数/方法 | `snake_case` | `calculate_total()`, `_internal()` | `CalculateTotal()`, `cal()` |
| 类 | `PascalCase` | `HttpClient`, `DataLoader` | `http_client`, `data_loader` |
| 常量 | `UPPER_CASE` | `MAX_CONNECTIONS`, `DEFAULT_PORT` | `maxConnections` |
| 模块/文件 | `snake_case` | `http_utils.py`, `data_loader.py` | `httpUtils.py` |
| 私有成员 | `_leading_underscore` | `_cache`, `_parse_raw()` | `__cache`（双下划线另有用处） |

**核心原则**：名字应该自解释。`a`、`tmp`、`data` 在超过 5 行的函数里就是阅读障碍。如果一段代码中有变量叫 `d`、`x1`、`x2`、`temp`，给它取个有意义的名字——你的同事和三个月后的你自己都会感谢你。

### 2.2 缩进用 4 个空格，禁止 Tab

```python
# 正确：4 空格缩进
def process(items):
    for item in items:           # 4 空格
        if item > 0:             # 8 空格
            print(item)          # 12 空格

# 错误：缩进不一致、混用 Tab
def process(items):
  for item in items:           # 2 空格——不符合 PEP 8
  	if item > 0:               # Tab——混用了 Tab 和空格
  	    print(item)            # 混乱
```

### 2.3 运算符两侧各一个空格，逗号后有空格

```python
# 正确
total = price * quantity + tax
names = ["Alice", "Bob", "Carl"]
def connect(host="localhost", port=8080):
    ...

# 错误
total=price*quantity+tax           # 挤在一起，难读
names=["Alice","Bob","Carl"]       # 逗号后没空格
```

**特别注意**：函数参数默认值的 `=` 两侧**不加空格**——这是让 IDE 区分"赋值语句"和"参数默认值"：

```python
# 赋值语句：= 两侧有空格
count = 10

# 参数默认值：= 两侧无空格
def connect(host="localhost", port=8080):
    ...
```

### 2.4 import 分行、分三组

```python
# 错误：一行导入多个模块，来源混在一起
import sys, json, os, numpy as np
from mypackage import utils
import pandas as pd

# 正确：每个 import 一行，按三组排列，组间空一行
# 1. 标准库
import os
import sys
from pathlib import Path

# 2. 第三方库
import numpy as np
import pandas as pd
from pydantic import BaseModel

# 3. 本地模块
from mypackage.config import settings
from .utils import parse_args
```

**绝对禁止** `from module import *`——它会把你不知道的名字全部倒进当前命名空间：

```python
from math import *      # 把 math 里所有东西都倒进来——包括你没用到的
from os import *        # os.system、os.path……几十个名字甩进全局

# 更糟糕的是：开发者根本分不清哪个名字来自哪个模块
# 静态分析工具（mypy、ruff）也很难追踪这些名字
```

正确做法：只导入你需要的。

### 2.5 行长度

- 推荐每行不超过 **79** 个字符（docstring/注释不超过 72）
- 现代项目通常放宽到 **88**（与 black 的默认值一致）

换行技巧——优先在括号内自然换行：

```python
# 括号内自然换行（最推荐）
result = some_long_function_name(
    first_argument, second_argument,
    third_argument, fourth_argument,
)

# 长字符串用隐式拼接
msg = (
    "This is a very long message that needs to be "
    "split across multiple lines for readability."
)

# 列表/字典也同理
config = {
    "database": "postgresql://localhost:5432/db",
    "api_key": "sk-xxxxxxxxxxxxx",
    "max_retries": 3,
}
```

### 2.6 注释：写"为什么"，不是"是什么"

```python
# 差：翻译代码——废话
# 循环遍历列表
for item in items:
    print(item)

# 差：解释语法
x = x + 1  # increment x by 1

# 好：解释"为什么这么做"——代码说不清的东西
# 使用 // 而非 /，因为索引必须是整数
mid = len(data) // 2

# 超时设为 10s 而非默认的 30s——该 API 的 P95 延迟只有 3s
timeout = 10
```

代码说"怎么做"（how），注释说"为什么这么做"（why）。好的代码名称本身就是最好的注释。

### 2.7 函数/类之间空 2 行，方法之间空 1 行

```python
# 模块级别：两个函数之间空 2 行
def first_function():
    pass


def second_function():  # 2 个空行
    pass


class MyClass:

    def method_one(self):  # 方法之间空 1 行
        pass

    def method_two(self):
        pass
```

### 2.8 文件末尾留一个空行

这是一个常见的细节——很多工具（如 Git）在没有末尾换行时会给出警告。确保每个 `.py` 文件最后一行是空行。

## 3. Pythonic 写法：摆脱 C/Java 思维

很多同学因为先学了 C 或 Java，写 Python 时会不自觉地沿用那些语言的习惯。下面是最常见的"翻译腔"及对应的地道写法。

### 3.1 直接遍历元素，不要用索引

```python
items = ["a", "b", "c"]

# 不推荐：C/Java 风格——通过索引间接访问
for i in range(len(items)):
    print(items[i])

# 推荐：Python 的 for 直接遍历可迭代对象中的元素
for item in items:
    print(item)
```

**为什么推荐后者**：Python 的 `for` 本质上是"逐个拿出可迭代对象里的东西"。它不像 C 的 `for (int i=0; i<n; i++)` 那样依赖索引。直接遍历更简洁，也避免了写错 `range(len())` 导致的 off-by-one 错误。

### 3.2 需要索引时用 enumerate

```python
tasks = ["复习 NumPy", "完成习题", "预习 Pandas"]

# 不推荐
for i in range(len(tasks)):
    print(f"{i + 1}. {tasks[i]}")

# 推荐：enumerate 同时给你索引和值
for i, task in enumerate(tasks, start=1):
    print(f"{i}. {task}")
```

### 3.3 并行遍历用 zip

```python
names = ["Alice", "Bob", "Carl"]
scores = [95, 87, 92]

# 不推荐
for i in range(len(names)):
    print(f"{names[i]}: {scores[i]}")

# 推荐
for name, score in zip(names, scores):
    print(f"{name}: {score}")
```

### 3.4 推导式代替 map/filter

```python
numbers = range(10)

# 不推荐：map/filter 嵌套 lambda，可读性差
evens_squared = list(map(lambda x: x ** 2, filter(lambda x: x % 2 == 0, numbers)))

# 推荐：推导式——从左读到右，逻辑清晰
evens_squared = [x ** 2 for x in numbers if x % 2 == 0]
```

推导式更接近自然语言："对 numbers 中的每个 x（如果 x 是偶数），计算 x 的平方"。

### 3.5 用 `in` 替代多重 `or`

```python
role = "moderator"

# 不推荐
if role == "admin" or role == "superadmin" or role == "moderator":
    ...

# 推荐
if role in ("admin", "superadmin", "moderator"):
    ...
```

### 3.6 利用"假值"判断空容器

```python
items = []

# 不推荐：像 Java 那样显式检查长度
if len(items) == 0:
    print("列表为空")

# 推荐：Python 的空容器在布尔上下文中视为 False
if not items:
    print("列表为空")
```

Python 中的"假值"（falsy）：`None`、`False`、`0`、`0.0`、`""`（空字符串）、`[]`（空列表）、`{}`（空字典）、`set()`（空集合）、`range(0)`。除此之外都是"真值"。

### 3.7 用 `with` 管理资源

```python
# 不推荐：手动 close，如果中间抛出异常，close 可能不执行
f = open("data.txt")
content = f.read()
f.close()

# 推荐：with 语句——无论是否发生异常，都会自动关闭文件
with open("data.txt") as f:
    content = f.read()
# 出了 with 块，文件自动关闭
```

不只文件，任何实现了上下文管理协议（`__enter__` / `__exit__`）的对象都支持 `with`：数据库连接、网络 socket、线程锁、临时目录等。

### 3.8 f-string 格式化

```python
name, age = "Alice", 25

# 不推荐：% 格式化（Python 2 残留）
msg = "%s is %d years old" % (name, age)

# 不推荐：.format()（Python 3 早期风格）
msg = "{} is {} years old".format(name, age)

# 推荐：f-string（Python 3.6+），变量直接嵌入
msg = f"{name} is {age} years old"

# f-string 支持表达式和格式控制
price = 12.345
print(f"Price: ${price:.2f}")              # Price: $12.35
print(f"{name.upper()} next year: {age + 1}")  # ALICE next year: 26
```

### 3.9 `join` 拼接字符串，不要用 `+` 循环

```python
words = ["Hello", "World", "Python"]

# 不推荐：每次 + 都会创建新的字符串对象，O(n^2) 复杂度
result = ""
for w in words:
    result += w + " "

# 推荐：join 一次性分配目标内存，O(n) 复杂度
result = " ".join(words)
print(result)  # "Hello World Python"
```

### 3.10 isinstance 而非 type() ==

```python
# 不推荐：type() == 不兼容子类
if type(x) == int:
    ...

# 推荐：isinstance 考虑继承关系——子类实例也算
if isinstance(x, int):
    ...

# 一次检查多种类型
if isinstance(x, (int, float)):
    ...
```

类比 Java：`type() ==` 相当于 Java 的 `getClass() ==`，`isinstance()` 相当于 Java 的 `instanceof`。后者总是更安全的选择。

## 4. 自动化工具：让机器管格式

手动遵守所有规则不现实。现代 Python 工程把格式检查交给机器。

### 4.1 black：无争议的代码格式化器

安装：

```bash
pip install black
```

使用：

```bash
# 格式化单个文件
black app.py

# 检查但不修改（CI 环境用）
black --check app.py

# 格式化整个目录
black src/
```

**black 的设计哲学**："所有可能引发争论的格式选项全部固定。"它几乎不提供配置项——行宽固定 88、双引号、末尾逗号等规则全部内置。团队成员不需要讨论格式，因为 black 说了算。

**实操演示**：假设你有一段格式混乱的代码 `messy.py`：

```python
def calc( a,b,c ):
    result=a+b*c
    if result>100:
       return "big"
    else:
        return "small"
print(calc( 10,20,3 ))
```

运行 `black messy.py` 后：

```python
def calc(a, b, c):
    result = a + b * c
    if result > 100:
        return "big"
    else:
        return "small"


print(calc(10, 20, 3))
```

缩进、空格、空行全部自动修正，你甚至不需要思考格式问题。

### 4.2 ruff：极速的 linter + 格式化器

ruff 用 Rust 编写，一个工具替代 flake8、isort、pyflakes 等传统选择，速度快 10-100 倍。

```bash
pip install ruff

# 检查代码质量问题
ruff check .

# 自动修复可修复的问题
ruff check --fix .

# 格式化（替代 black）
ruff format .
```

### 4.3 pyproject.toml 最简配置

现代 Python 项目把几乎所有工具的配置统一放在 `pyproject.toml` 中：

```toml
[project]
name = "my-project"
version = "0.1.0"
requires-python = ">=3.11"

[tool.ruff]
line-length = 88
target-version = "py311"

[tool.ruff.lint]
select = [
    "E",   # pycodestyle errors（基本错误）
    "W",   # pycodestyle warnings
    "F",   # pyflakes（未使用的 import 等）
    "I",   # isort（import 排序）
    "N",   # pep8-naming（命名规范）
    "UP",  # pyupgrade（建议使用新语法）
]

[tool.ruff.format]
quote-style = "double"
indent-style = "space"

[tool.mypy]
python_version = "3.11"
strict = true
```

### 4.4 pre-commit：在 git commit 前自动检查

pre-commit 是一个框架，在每次 `git commit` 时自动运行你指定的检查。如果检查不通过，提交就会被阻止——bug 在进入仓库之前就被拦下。

安装：

```bash
pip install pre-commit
```

创建 `.pre-commit-config.yaml`（最简版）：

```yaml
repos:
  # 基础检查：去尾部空格、文件末尾空行等
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
        args: ['--maxkb=500']

  # ruff：格式 + lint
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
```

安装 hooks（在项目根目录执行）：

```bash
pre-commit install
```

此后每次 `git commit` 都会自动运行 ruff。如果代码有格式问题，ruff 会阻止提交并告诉你需要修复什么。

## 5. 项目目录结构

对于中型项目，推荐的 `src-layout`：

```
my-project/
├── pyproject.toml              # 项目元数据和工具配置
├── .pre-commit-config.yaml     # pre-commit hooks
├── .gitignore                  # 忽略文件规则
├── .env.example                # 环境变量模板（不含真实密钥）
├── README.md                   # 项目说明
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── main.py
│       ├── config.py
│       ├── models/
│       ├── services/
│       └── utils/
└── tests/
    ├── __init__.py
    └── test_main.py
```

`src-layout` 的好处：强制你以"可安装包"的方式组织代码（`pip install -e .`），避免 Python 路径混乱导致的"能运行但不能导入"问题。

## 7. 常见错误

### 错误 1：混用 Tab 和空格缩进

```python
# 错误：有的行用 Tab，有的用空格
def foo():
	print("hello")     # 这行用 Tab
    print("world")     # 这行用 4 空格——混用了！

# 正确：统一用 4 个空格（现代编辑器可设"Tab 键转换为 4 空格"）
def foo():
    print("hello")
    print("world")
```

Python 3 直接禁止混用——会抛出 `TabError: inconsistent use of tabs and spaces in indentation`。

### 错误 2：from module import *

```python
# 错误：污染命名空间
from math import *     # 导入了 50+ 个名字，包括你没用到的

# 正确：只导入需要的
from math import sin, cos, pi
# 或导入模块本身
import math
print(math.sin(math.pi))  # 带命名空间前缀，清晰明了
```

### 错误 3：把 .env 提交到 git

一旦 `.env` 进入 git 历史，即便你马上删除，攻击者仍可从历史记录中找到密钥。发生后的补救措施：

```bash
git rm --cached .env
git commit -m "Remove .env from tracking"
# 然后立即去各服务平台轮换所有泄露的密钥！
```

### 错误 4：用 `type() ==` 而不是 `isinstance()`

```python
# 错误：子类实例无法通过检查
class MyInt(int):
    pass

x = MyInt(5)
if type(x) == int:        # False！虽然 x 确实是 int 的子类
    print("是整数")

# 正确
if isinstance(x, int):    # True——isinstance 认可子类
    print("是整数")
```

### 错误 5：注释翻译代码

```python
# 错误：废话注释——代码本身就是"是什么"的最好表达
# 初始化 counter 为 0
counter = 0

# 循环遍历列表
for item in items:
    # 打印每一项
    print(item)

# 正确：注释解释"为什么"
# 使用滑动窗口而非逐行扫描，因为日志文件按时间有序
for i in range(len(logs) - window_size + 1):
    ...
```

## 8. 练习题

### 基础练习

**练习 1：找出 PEP 8 违规**

下面这段代码违反了至少 6 条 PEP 8 规则，请逐一找出并说明：

```python
from math import *
import os, sys

class dataProcessor:
    def __init__(Self, name, Age):
        Self.name=name
        Self.age=Age

    def getInfo(Self):
        return f"{Self.name} is {Self.age} years old"

MAX_RETRY=3
```

<details>
<summary>参考答案（点击展开）</summary>

1. `from math import *`：禁止通配符导入，污染命名空间
2. `import os, sys`：每个 import 应独占一行
3. `class dataProcessor`：类名应为 PascalCase → `DataProcessor`
4. `Self`：self 应为小写（Python 约定）
5. `name, Age`：参数 `Age` 应是小写 `age`
6. `Self.name=name`：等号两侧缺少空格 → `Self.name = name`
7. `MAX_RETRY=3`：等号两侧缺少空格 → `MAX_RETRY = 3`
</details>

**练习 2：将不 Pythonic 的代码改成 Pythonic**

将下面的代码改写为 Pythonic 风格：

```python
def process_data(data_list):
    result = []
    for i in range(len(data_list)):
        if data_list[i] != None and data_list[i] > 0:
            result.append(data_list[i] * 2)
    return result
```

<details>
<summary>参考答案（点击展开）</summary>

```python
def process_data(data_list):
    """将列表中的正数乘以 2，忽略 None 和非正数"""
    return [x * 2 for x in data_list if x is not None and x > 0]
```

改动点：
- `range(len(data_list))` + `data_list[i]` → 直接遍历 `data_list`
- `!= None` → `is not None`（PEP 8 推荐用 `is` 比较 None）
- for 循环 + append → 列表推导式
</details>

## 9. 本章小结

| 层面 | 工具/做法 | 覆盖什么 |
|------|-----------|---------|
| 风格指南 | PEP 8 | 命名、缩进、空格、注释等规则 |
| 格式化 | black / ruff format | 自动修正缩进、引号、空格、换行 |
| Lint | ruff check | 发现未使用变量、错误 import、潜在 bug |
| 类型检查 | mypy | 提前发现类型不匹配（见第 13 章） |
| 提交前检查 | pre-commit | 阻止不符合规范的代码进入仓库 |
| 敏感信息 | .gitignore + .env.example | 密钥不进入 git 历史 |

**最终原则**：把格式、lint、类型检查全部自动化。把人解放出来思考架构和逻辑——这才是工程师应该花时间的地方。
