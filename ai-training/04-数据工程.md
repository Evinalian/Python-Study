## 1. 数据的决定性作用

### 1.1 数据 > 架构 > 超参

在深度学习社区中，有一个广泛认可的观点：对于大模型训练，**数据的地位高于模型架构，而模型架构又高于超参调优**。这不是说架构和超参不重要，而是在当前的 AI 工程实践中，数据质量的差距比架构改进带来的提升更显著。

举个例子：LLaMA 1（2023年）在性能上显著优于 GPT-3，尽管它的模型架构并没有根本性创新（主要是 RMSNorm、SwiGLU、RoPE 这些已公开的技术）。LLaMA 的关键秘密在于数据——它使用了更高质量、更精心策划的训练数据，特别是使用了大量高质量的学术和技术文本。

### 1.2 垃圾进垃圾出

Garbage In, Garbage Out（GIGO）是计算机科学的基本原则。在预训练中，这个原则体现得尤为明显：

- 如果训练数据中包含大量重复、低质量、错误的文本，模型会学到这些噪声
- 如果训练数据缺乏某些领域（如代码、数学），模型在这些领域上的能力会很弱
- 如果数据的语言分布不均衡，模型会偏向主导语言

OpenAI 的研究表明，在固定计算预算下，使用经过精心筛选的较小数据集训练的模型，其性能常常超过使用未经筛选的较大量数据训练的模型。换句话说，**数据质量比数据量更能决定模型的上限**。

### 1.3 数据工程的总体框架

一个完整的数据工程流水线通常包含以下阶段：

```
数据源收集 → 语言过滤 → 质量过滤 → 去重 → 隐私过滤 → 数据配比 → 数据打包 → 训练
```

每个阶段都会筛掉一部分数据。一个典型的经验法则：从原始数据到最终用于训练的数据，可能只保留了 10%-30%。这意味着数据工程中"丢弃"比"保留"更关键——知道什么不该放进训练集，往往比知道什么该放更重要。

## 2. 数据源收集

### 2.1 公开网络数据：CommonCrawl

CommonCrawl 是一个定期爬取整个互联网的非盈利组织，提供了 PB 级别的原始网页数据。自 2011 年以来，CommonCrawl 每个月释放一个爬取快照。这些数据是绝大多数开源大模型的训练数据基础。

**CommonCrawl 的优点**：
- 规模巨大：每个月释放的爬取数据约 200-300 TB（压缩后）
- 多样性高：覆盖数百种语言、各种主题
- 免费开放：不需要复杂的授权

**CommonCrawl 的挑战**：
- 数据极其脏乱：包含大量广告、导航栏、cookie 声明、版权声明等"网页家具"
- 重复内容多：不同 URL 可能有完全相同的内容（镜像站、转载）
- 需要大量预处理：从 WARC（Web ARChive）格式中提取纯文本，去除 HTML 标签

**处理 CommonCrawl 的常用流程**：
1. 从 WARC 文件中提取 HTML（使用 `warcio` 或 `trafilatura` 库）
2. 使用 boilerplate removal 工具提取正文（`trafilatura`、`jusText`、`readability`）
3. 过滤掉非自然语言的页面（菜单、产品列表、链接聚合页）
4. 语言检测和分类

### 2.2 高质量文本源

除了 CommonCrawl，以下来源通常被认为质量更高：

- **维基百科**：经过人工编辑和审核，质量高，多语言覆盖。但总数据量有限（英文维基约 20GB 纯文本）。
- **书籍**：Books3、Gutenberg 项目等。书籍的特点是语言优美、结构完整、长文本连续。对于训练模型的"文学素养"至关重要。
- **学术论文**：arXiv、PubMed、S2ORC 等。学术论文的特点是严谨、专业、包含公式和引用。对于训练模型的推理和专业知识至关重要。
- **代码**：GitHub、StackOverflow 等。代码数据让模型具备编程能力。需要注意的是代码需要过滤掉自动生成的低质量代码。
- **新闻**：新闻文章的特点是时效性强、事实准确（相对而言）、语言规范。
- **对话/论坛**：Reddit、StackExchange 等。这类数据让模型学会对话和问答的格式，但质量参差不齐。

### 2.3 数据源的伦理和法律考量

- **版权**：使用受版权保护的数据训练模型是否构成侵权，目前是一个法律灰区。不同地区有不同的规定。
- **个人信息**：训练数据中可能包含姓名、地址、电话等个人信息。需要做隐私过滤。
- **偏见和有害内容**：互联网数据中包含各种偏见和有害内容。模型可能学到并放大这些偏见。
- **许可协议**：使用 GitHub 代码需要考虑开源协议。GPL 等 copyleft 协议可能对模型输出有限制。

## 3. 质量过滤

### 3.1 基于规则的过滤

基于规则的方法是最直接的过滤手段，计算成本低，可以快速筛掉大量明显低质量的数据。常见的规则包括：

**文本长度过滤**：
- 太短（< 50 字符）：通常是导航、错误页面、评论
- 太长（> 单个文件上限）：可能是日志、自动生成的文本
- 典型保留范围：100-100,000 字符

**语言检测**：
- 使用 fastText 或 CLD2/CLD3 进行语言识别
- 对于多语言模型：保留目标语言的数据（置信度 > 0.7）
- 对于单语言模型（如 GPT-3 主要用于英文）：只保留该语言的数据

**词重复率（Repetition Ratio）**：
- 检测重复的 n-gram、行、段落
- 如果某个句子或段落在文档中多次出现，可能是模板生成的内容
- 典型阈值：最频繁的 2-gram/3-gram 占比不应超过某个比例

**特殊字符比例**：
- 非文字字符（标点、数字、特殊符号）占比不能过高
- 过高比例可能意味着乱码、base64 编码内容、或自动生成的数据
- 典型阈值：字母/汉字占比 > 60%

**HTML/代码残留**：
- 检测未清除的 HTML 标签、CSS、JavaScript
- 检测模板占位符（如 `{{variable}}`、`{% template %}`）

**启发式信号**：
- 是否包含"lorem ipsum"等占位文本
- 是否包含大段的数字、日期列表
- 是否以"Cookie policy"、"Terms of service"开头

### 3.2 基于模型的评分过滤

规则过滤只能处理明显的情况。对于更微妙的"质量"判断，需要训练分类器。

**KenLM 困惑度过滤**：
使用一个在高质量文本（如维基百科）上训练的 5-gram KenLM 语言模型，计算每个文档的困惑度。困惑度太高的文档（意味着字符序列在高质量文本中很少见），很可能质量较差。

```
如果 perplexity > 阈值（如 1000）→ 过滤
如果 perplexity < 阈值（如 5）→ 也可能是重复模板，考虑过滤
```

**Classifier-based 过滤**：
训练一个轻量级分类器（如 fastText 或小 BERT），使用标注过的"好/坏"数据进行训练。CCNet 和 CulturaX 等项目都使用了这种方法。

**LLM-as-a-judge**：
使用一个已有的 LLM 对数据质量进行评分。虽然成本高，但质量判断的准确度也最高。通常只对边界情况使用 LLM 评分。

### 3.3 质量过滤的实现示例

```python
"""
数据质量过滤工具
=================
常用的基于规则的文本质量过滤函数。

作者：Python+AI 学习教程
"""

import re
from collections import Counter


def compute_repetition_ratio(text: str, n: int = 3) -> float:
    """
    计算文本的重复率。

    统计最高频的 n-gram 占总 n-gram 的比例。
    比例过高说明文本中有大量重复内容（模板生成、机器翻译等）。

    参数:
        text: 输入文本
        n: n-gram 的 n 值（默认为 3，即 3-gram）
    返回:
        repetition_ratio: 取值范围 [0, 1]
    """
    text = text.lower()
    # 提取所有 n-gram
    ngrams = [text[i:i+n] for i in range(len(text) - n + 1)]
    if len(ngrams) == 0:
        return 0.0

    counter = Counter(ngrams)
    # 最高频 n-gram 出现次数 / 总 n-gram 数
    most_common_count = counter.most_common(1)[0][1]
    return most_common_count / len(ngrams)


def compute_alpha_ratio(text: str) -> float:
    """
    计算字母/汉字在文本中的占比。

    参数:
        text: 输入文本
    返回:
        alpha_ratio: 字母和汉字字符占所有字符的比例
    """
    if len(text) == 0:
        return 0.0

    # 统计字母和汉字字符
    alpha_chars = sum(1 for c in text if c.isalpha())
    return alpha_chars / len(text)


def compute_special_char_ratio(text: str) -> float:
    """
    计算特殊字符占比。

    特殊字符包括: 标点符号、特殊符号、非字母/数字/空格的字符。
    高比例可能表示乱码或非文本内容。

    参数:
        text: 输入文本
    返回:
        special_ratio: 取值范围 [0, 1]
    """
    if len(text) == 0:
        return 0.0

    special_count = sum(1 for c in text if not c.isalnum() and not c.isspace())
    return special_count / len(text)


def has_html_residue(text: str) -> bool:
    """
    检测文本中是否有 HTML 标签残留。

    返回:
        True: 发现 HTML 残留
        False: 未发现 HTML 残留
    """
    html_patterns = [
        r'<[a-zA-Z]+[^>]*>',  # 开放标签
        r'</[a-zA-Z]+>',      # 闭合标签
        r'&[a-zA-Z]+;',       # HTML 实体
        r'javascript:',       # JavaScript 引用
        r'<style\b',          # CSS 引用
    ]
    for pattern in html_patterns:
        if re.search(pattern, text):
            return True
    return False


def filter_document(text: str,
                    min_length: int = 100,
                    max_length: int = 100000,
                    min_alpha_ratio: float = 0.6,
                    max_special_ratio: float = 0.3,
                    max_repetition: float = 0.3,
                    check_html: bool = True) -> tuple:
    """
    综合质量过滤函数。

    对文档应用多项质量检查，返回是否通过以及具体的失败原因。

    参数:
        text: 输入文档文本
        min_length: 最小字符数
        max_length: 最大字符数
        min_alpha_ratio: 最低字母/汉字比例
        max_special_ratio: 最高特殊字符比例
        max_repetition: 最高重复率
        check_html: 是否检查 HTML 残留

    返回:
        (passed: bool, reason: str)
        如果 passed 为 False，reason 说明未通过的原因
    """
    # 1. 长度检查
    text_len = len(text)
    if text_len < min_length:
        return False, f"too_short ({text_len} < {min_length})"
    if text_len > max_length:
        return False, f"too_long ({text_len} > {max_length})"

    # 2. 字母比例检查
    alpha_ratio = compute_alpha_ratio(text)
    if alpha_ratio < min_alpha_ratio:
        return False, f"low_alpha_ratio ({alpha_ratio:.2f} < {min_alpha_ratio})"

    # 3. 特殊字符比例检查
    special_ratio = compute_special_char_ratio(text)
    if special_ratio > max_special_ratio:
        return False, f"high_special_chars ({special_ratio:.2f} > {max_special_ratio})"

    # 4. 重复率检查
    rep_ratio = compute_repetition_ratio(text, n=4)
    if rep_ratio > max_repetition:
        return False, f"high_repetition ({rep_ratio:.2f} > {max_repetition})"

    # 5. HTML 残留检查
    if check_html and has_html_residue(text):
        return False, "html_residue"

    return True, "passed"


if __name__ == "__main__":
    # 测试
    good_text = "深度学习是人工智能的一个重要分支。它使用多层神经网络来学习数据的层次化表示。"
    bad_text_short = "ab"  # 太短
    bad_text_repetitive = "hello hello hello hello hello hello hello "
    bad_text_html = "<div>Some text</div> with <script>alert('xss')</script>"

    for label, text in [
        ("好文本", good_text),
        ("太短", bad_text_short),
        ("重复", bad_text_repetitive),
        ("HTML残留", bad_text_html),
    ]:
        passed, reason = filter_document(text, min_length=10)
        print(f"{label}: passed={passed}, reason={reason}")
```

## 4. 去重

### 4.1 为什么去重如此重要

训练数据中的重复有多个层面：
- **同一个 URL 的不同爬取**：完全相同的页面被爬了多次
- **转载和镜像**：同一篇文章出现在多个网站上
- **模板化内容**：许多网站使用固定模板（如商品页、论坛帖），正文之外的部分高度重复
- **句子级重复**：高频短语或固定表达在不同文档中重复出现

数据重复的危害：
1. **浪费计算**：同样的内容被训练多次 = 浪费 GPU 时
2. **记忆而非泛化**：模型可能"背诵"重复的内容（特别是训练数据中的高频段落），而不是学习泛化的模式。这就是"训练数据泄露"导致 benchmark 失真。
3. **偏见放大**：重复的内容（如某个被反复转载的偏见观点）会在训练中被过度强化。

研究表明，CommonCrawl 中约 50% 的数据是重复的。这意味着去重可以将有效训练数据量扩大近一倍。

### 4.2 精确去重

**文档级精确去重**：对文档内容计算哈希（MD5 或 SHA-256），相同哈希的文档只保留一份。

```
hash = hashlib.md5(text.encode('utf-8')).hexdigest()
```

**增强方法**：在计算哈希前，对文本进行规范化：
- 统一空白字符（多个空格 → 单个空格）
- 小写化
- 移除标点符号
- Unicode 规范化（NFC）

这样，只有空白或大小写差异的文档也会被视为相同。

**关键数据结构**：布隆过滤器（Bloom Filter）。当数据量极大（数十亿文档）时，无法将所有哈希值放入内存。布隆过滤器使用多个哈希函数和一个位数组，以极小的内存占用实现"可能存在"的查询，适合做初步筛选。

### 4.3 近似去重：MinHash

精确去重只能找到完全相同的文档。近似去重能找到内容相似但不完全相同的文档（如略有修改的转载）。

**MinHash 的核心思想**：
给定两个文档 A 和 B，它们的 Jaccard 相似度定义为：
```
J(A, B) = |A ∩ B| / |A ∪ B|
```
其中 A 和 B 被视为 n-gram 的集合。

直接计算所有文档对的 Jaccard 相似度代价是 O(N²)（N 是文档数），对于数十亿文档完全不现实。

**MinHash 的巧妙之处**：
1. 将文档表示为其 n-gram 的集合
2. 对每个 n-gram 计算一个哈希值
3. 取最小的 k 个哈希值作为文档的"签名"（通常 k=128 或 256）
4. 两个文档的 MinHash 签名相同的比例 ≈ 它们的 Jaccard 相似度

这样，文档用固定长度的签名向量表示，相似度的计算从"集合交集"变成了"签名向量的匹配率"。

**MinHash 签名的大小**：k=128 个 64 位整数 = 1KB/文档。对于 10 亿文档，总签名约 1TB——可以存入磁盘分批处理。

**LSH（Locality-Sensitive Hashing）加速**：
将 k 个 MinHash 值分成 b 个 band，每个 band 包含 r 个值（k = b × r）。如果两个文档在一个 band 内所有 r 个值都相同，它们就是一个"候选对"。

通过调整 b 和 r，可以控制查全率和查准率的平衡：
- b 大（band 多）：召回率高（更多候选对被发现）
- r 大（band 内值多）：精确率高（候选对是真的相似的概率更高）

### 4.4 SimHash

SimHash 是另一种常用的近似去重算法，Google 用它来做网页去重。

**SimHash 的核心思想**：
1. 将文档表示为一个 f 维的向量（通常 f=64），初始化为 0
2. 对文档的每个特征（如每个 token 的哈希值），如果哈希值的第 i 位为 1，f 向量的第 i 维 +1；否则 -1
3. 最终的 f 维向量：正值的维度设为 1，负值的维度设为 0
4. 这样每个文档被压缩为一个 f 位的指纹

两个文档相似 ⇔ 它们的 SimHash 指纹的海明距离（不同位的数量）很小（如 < 3）。

**SimHash vs MinHash**：
- SimHash 计算更快（单次扫描文档）
- MinHash 的数学保证更强（签名匹配率 = Jaccard 相似度）
- 实际中，两者常结合使用：SimHash 初筛，MinHash 精筛

### 4.5 句子级去重

除了全文去重，还需要在句子级别去重。原因是：很多高频短语（如引用、谚语、广告语）在不同文档中反复出现，虽然不构成全文重复，但会在训练中过度强调。

在大规模处理中，句子级去重通常使用**后缀数组**（Suffix Array）高效地找到所有文档中重复出现的子串。具体做法：
1. 将所有训练数据拼接成一个巨大的文本
2. 构建后缀数组（所有后缀的排序数组）
3. 相邻后缀的最长公共前缀（LCP）如果超过阈值（如 50 个 token），则该子串为重复
4. 去除重复出现超过一定次数的子串

C4 数据集和 LLaMA 训练数据都采用了句子级去重。

### 4.6 文档级近似去重示例代码

```python
"""
MinHash 近似去重实现
=====================
使用 datasketch 库实现 MinHash + LSH 的近重复文档检测。

安装: pip install datasketch

作者：Python+AI 学习教程
"""

from datasketch import MinHash, MinHashLSH


def create_minhash(text: str, num_perm: int = 128, ngram_size: int = 5) -> MinHash:
    """
    为文档创建 MinHash 签名。

    参数:
        text: 文档文本
        num_perm: 排列数（签名长度），越大越精确但占更多内存
        ngram_size: n-gram 的大小
    返回:
        MinHash 对象
    """
    m = MinHash(num_perm=num_perm)
    # 提取 shingles（n-gram）
    text = text.lower()
    for i in range(len(text) - ngram_size + 1):
        shingle = text[i:i + ngram_size]
        m.update(shingle.encode('utf-8'))
    return m


def deduplicate_documents(documents: list,
                          threshold: float = 0.8,
                          num_perm: int = 128,
                          ngram_size: int = 5) -> list:
    """
    MinHash + LSH 去重。

    使用 LSH 高效发现相似文档对，移除相似文档。

    参数:
        documents: 文档文本列表
        threshold: Jaccard 相似度阈值（>= 此值视为重复）
        num_perm: MinHash 签名长度
        ngram_size: n-gram 大小
    返回:
        去重后的文档索引列表（保留第一个出现的文档）
    """
    # 创建 LSH 索引
    lsh = MinHashLSH(threshold=threshold, num_perm=num_perm)

    # 为每个文档创建 MinHash 签名并插入索引
    minhashes = []
    for i, doc in enumerate(documents):
        m = create_minhash(doc, num_perm, ngram_size)
        lsh.insert(i, m)
        minhashes.append(m)

    # 查找重复文档
    duplicate_indices = set()
    for i, m in enumerate(minhashes):
        if i in duplicate_indices:
            continue
        # 查询相似文档
        result = lsh.query(m)
        for j in result:
            if j > i:  # 只标记后面出现的重复文档
                duplicate_indices.add(j)

    # 返回非重复文档的索引
    kept_indices = [i for i in range(len(documents)) if i not in duplicate_indices]
    print(f"原始文档数: {len(documents)}, 去重后: {len(kept_indices)}, "
          f"移除重复: {len(duplicate_indices)}")

    return kept_indices


if __name__ == "__main__":
    # 测试
    test_docs = [
        "深度学习是人工智能的一个重要分支。它通过多层神经网络学习数据的层次化表示。",
        "深度学习是人工智能的一个重要分支。它通过多层神经网络学习数据的层次化表示。",  # 完全相同
        "深度学习是人工智能的一个重要分支，它使用神经网络来学习数据的表示。",  # 近似重复
        "今天天气真好，适合出去散步。",  # 完全不同的文档
        "深度学习已成为人工智能领域的核心技术之一。",  # 不同文档
    ]

    kept = deduplicate_documents(test_docs, threshold=0.5)
    print("保留的文档索引:", kept)
    for idx in kept:
        print(f"  [{idx}]: {test_docs[idx][:50]}...")
```

## 5. 数据配比

### 5.1 为什么需要配比

训练大模型的数据可以来自几十个甚至上百个不同的来源：CommonCrawl、维基百科、书籍、代码、论文、新闻、论坛...... 不同来源的数据在质量、风格、主题上差异巨大。如果简单地将所有数据混在一起训练，数据量大的来源（如 CommonCrawl 可能占 80%+）会主导训练，而高质量但数据量小的来源（如书籍、论文）的影响微乎其微。

**数据配比**就是决定每个来源在训练数据中各占多少比例。

### 5.2 配比设计的考量因素

**质量优先**：
高质量来源（维基百科、书籍、论文）应给予更高的权重。即使它们在数据总量中占比小，也应该在训练中被"多看几遍"（更高的采样概率）。

**多样性平衡**：
确保模型接触到不同领域、不同文体、不同语言的数据。如果全部是新闻数据，模型只会写新闻体；如果全部是代码，模型只懂编程。

**下游任务对齐**：
如果模型的主要用途是代码生成，代码数据的比例应提高；如果是对话，对话数据比例应提高。

**token 总量和经济性**：
不同来源的处理成本不同。CommonCrawl 虽然质量不如书籍，但获取成本极低，可以大量使用。配比设计需要在质量和成本间权衡。

### 5.3 Chinchilla 最优数据配比

DeepMind 的 Chinchilla 论文揭示了一个重要的发现：在给定计算预算下，模型大小和训练数据量应该等比放大——而不是像 GPT-3 那样专注于增大模型。这被称为 Chinchilla Optimal。

但 Chinchilla 还有一个常被忽视的贡献：它研究了不同数据来源的"价值"。通过消融实验，Chinchilla 发现：

- MassiveWeb（经过过滤的网页数据）占总数据约 61%
- 书籍占约 27%
- 维基百科占约 12%

这个配比并非"最优"的铁律，而是 Chinchilla 团队在实验中找到的一个良好平衡点。不同团队有不同的配比选择：

| 来源 | Chinchilla | LLaMA | Falcon | GPT-3 |
|------|-----------|-------|--------|-------|
| 网页 | 61% | 67% | 75%+ | 60% |
| 书籍 | 27% | 5% | <5% | 13% |
| 维基百科 | 12% | 5% | <5% | 3% |
| 代码 | - | 15% | <5% | 8% |
| 论文 | - | 3% | - | - |
| 其他 | - | 5% | 14% | 16% |

**关键观察**：
- Llama 增加了代码（15%）和学术论文（3%）的比例，这是 LLaMA 在代码和推理任务上表现优异的重要原因
- 网页数据始终是最大来源（60-80%），因其规模大、成本低
- 高质量来源即使占比小，也对模型质量有不成比例的巨大影响

### 5.4 配比的实现：采样策略

实际实现配比时，通常采用**采样概率**而非精确比例。训练中每次从混合数据集中采样一个 batch 时，以某个概率从来源 i 采样：

```
P(source_i) = weight_i / Σ weight_j
```

这样，来源的最终占比在期望上等于预设比例，而不需要物理上合并所有数据。

## 6. 数据打包

### 6.1 为什么需要打包

训练 Transformer 时，每个 batch 内的序列长度必须一致（因为 Attention 矩阵是 `(batch, seq_len, seq_len)` 的规则张量）。但自然文档长度差异巨大——一条推文可能只有 50 个 token，一篇论文可能有 10000 个 token。

最简单的处理方式是 truncation（截断）和 padding（填充）：
- 短文档 pad 到最长文档的长度
- 长文档截断到 max_seq_len

但这造成了严重浪费。如果 batch 中大多文档是 500 tokens 而有一个 4000 tokens 的长文档，那么大部分的计算都在处理 padding tokens——这些 padding 在 Attention 中被 mask 掉，不贡献任何学习信号，却消耗了计算和显存。

**数据打包**的解决方案是：将多个短文档拼接成一个"超长"文档（长度接近或等于 max_seq_len），然后在这个拼接序列上进行训练。不同文档之间用 `[EOS]` token 分隔。

### 6.2 打包的策略

**贪心打包（Greedy Packing）**：
1. 将文档按长度降序排列
2. 对每个长文档，尝试用短文档"填补"其后的剩余空间
3. 一个 pack 达到 max_seq_len（或填不满）关闭，开始新 pack

贪心打包简单高效，但在末尾可能有少量 token 浪费。

**最优打包（Optimal Packing）**：
将打包问题形式化为 bin-packing 问题（NP-hard），使用启发式算法求解。目标是最小化 packs 数量（即最小化总 padding）。Google 的 T5 训练使用了最优打包。

### 6.3 打包的注意事项

**文档边界**：
打包后，一个训练序列可能包含多个文档。Attention 计算时，不同文档之间不应该互相 attend。这意味着需要在 attention mask 中标记文档边界，避免信息跨文档"泄漏"。

**Loss 计算**：
Loss 应该只在文档内容上计算，不包括 `[EOS]` 分隔 token 和文档之间的 padding（如果有的话）。

**Shuffling 的影响**：
打包通常在 shuffling 之前进行。如果先 shuffle 再打包，会破坏文档边界，导致知识污染。

### 6.4 数据打包的简单实现

```python
"""
数据打包工具
=============
将长度不一的文档拼接成固定长度的训练序列。

作者：Python+AI 学习教程
"""

import random


def pack_documents(tokenized_docs: list,
                   max_seq_len: int = 2048,
                   eos_token_id: int = 2,
                   shuffle: bool = True) -> list:
    """
    将 tokenized 文档打包成固定长度的训练序列。

    使用贪心算法: 按长度排序，短文档填补长文档后的空隙。

    参数:
        tokenized_docs: 每个文档的 token ID 列表，如 [[101, 102, ...], [201, ...], ...]
        max_seq_len: 目标序列长度（包含 EOS）
        eos_token_id: EOS token 的 ID
        shuffle: 是否先打乱文档顺序
    返回:
        packs: 打包后的序列列表，每个序列长度为 max_seq_len
    """
    # 在每个文档末尾添加 EOS
    docs = [doc + [eos_token_id] for doc in tokenized_docs]

    # 过滤掉超过 max_seq_len 的文档（或截断）
    docs = [doc[:max_seq_len - 1] + [eos_token_id]
            if len(doc) >= max_seq_len else doc
            for doc in docs]

    # 按长度降序排列（贪心策略）
    docs.sort(key=len, reverse=True)

    if shuffle:
        # 对相似长度的文档组内打乱（保留长度排序的大结构）
        pass

    packs = []
    current_pack = []
    current_len = 0

    for doc in docs:
        doc_len = len(doc)

        if current_len + doc_len <= max_seq_len:
            # 可以放入当前 pack
            current_pack.extend(doc)
            current_len += doc_len
        else:
            # 当前 pack 已满，开新 pack
            if current_pack:
                # 填充到 max_seq_len（用 EOS 填充）
                padding_len = max_seq_len - current_len
                current_pack.extend([eos_token_id] * padding_len)
                packs.append(current_pack)

            # 开始新 pack
            current_pack = doc[:]
            current_len = doc_len

    # 处理最后一个 pack
    if current_pack:
        padding_len = max_seq_len - current_len
        current_pack.extend([eos_token_id] * padding_len)
        packs.append(current_pack)

    # 计算填充率
    total_tokens = sum(len(doc) for doc in docs)
    total_capacity = len(packs) * max_seq_len
    utilization = total_tokens / total_capacity * 100

    print(f"打包统计: {len(docs)} 个文档 → {len(packs)} 个序列")
    print(f"利用率: {utilization:.1f}% (总 token: {total_tokens}, 总容量: {total_capacity})")

    return packs


if __name__ == "__main__":
    # 模拟 tokenized 文档（不同长度）
    random.seed(42)
    tokenized_docs = [
        [random.randint(100, 50000) for _ in range(random.randint(50, 500))]
        for _ in range(100)
    ]

    packs = pack_documents(tokenized_docs, max_seq_len=1024)
    print(f"前3个序列的长度: {[len(p) for p in packs[:3]]}")
```

## 7. 数据版本管理

### 7.1 数据也需要版本控制

在大模型训练中，数据是不断演进的：
- 数据源更新（新的 CommonCrawl 快照、新的维基百科版本）
- 数据处理流程改进（更好的过滤规则、更精确的去重算法）
- 实验性数据混合（修改数据配比）

如果没有数据版本管理，会遇到严重问题：
- **不可复现**：无法确切知道"去年的那个模型"用了什么数据
- **难以调试**：性能变好或变差时，不确定是模型改动的效果还是数据改动的效果
- **协作困难**：团队成员对"当前数据"的理解不一致

### 7.2 DVC（Data Version Control）

DVC 是专为 ML 项目设计的数据版本控制工具。它的核心思想：**Git 管理代码和元数据，DVC 管理数据和模型文件**。

**DVC 的基本使用流程**：

```bash
# 初始化 DVC（在已有 Git 仓库中）
dvc init

# 添加数据文件到 DVC 跟踪
dvc add data/raw/corpus_v1.jsonl
# 这会生成 corpus_v1.jsonl.dvc 元数据文件（小而可以 git commit）
# 并将实际数据文件加入 .gitignore

# 将 .dvc 文件提交到 Git
git add data/raw/corpus_v1.jsonl.dvc .gitignore
git commit -m "Add raw corpus v1"

# 推送数据到远程存储（S3/GCS/SSH/本地）
dvc remote add myremote s3://my-bucket/data
dvc push

# 切换到之前的数据版本
git checkout <old-commit>  # 切换到代码的老版本
dvc checkout                # 切换数据到对应版本
```

**DVC 管道的威力**：

DVC 不仅管理数据，还管理整个数据处理流程：

```yaml
# dvc.yaml - 数据处理流程定义
stages:
  download:
    cmd: python scripts/download.py
    deps:
      - scripts/download.py
    outs:
      - data/raw/

  filter:
    cmd: python scripts/filter.py data/raw/ data/filtered/
    deps:
      - scripts/filter.py
      - data/raw/
    outs:
      - data/filtered/

  deduplicate:
    cmd: python scripts/dedup.py data/filtered/ data/deduped/
    deps:
      - scripts/dedup.py
      - data/filtered/
    outs:
      - data/deduped/
```

运行 `dvc repro` 会自动执行所有步骤，只重新计算变动过的环节。这确保了数据处理的可复现性。

### 7.3 数据集的元数据记录

推荐为每个数据集版本记录以下元数据：
- **版本号**：日期或语义版本
- **来源**：原始数据来源和获取时间
- **处理流程**：应用了哪些过滤/去重/混合步骤
- **统计信息**：文档数、token 数、语言分布、平均长度
- **数据配比**：各来源的精确占比
- **已知问题**：已知的偏差、噪声或遗漏

这些信息可以存储为 JSON/YAML，与 DVC 元数据一起管理。

## 10. 常见错误

### 错误 1：在训练集上做去重但在测试集上没做
**症状**：模型在 benchmark 上表现异常好，但在实际应用中表现差。或者模型输出中出现了 benchmark 测试用例的内容。
**原因**：训练数据中的测试集内容没有被清除。模型"作弊"了——它见过测试数据（可能是测试数据的内容以其他形式出现在训练数据中）。
**修复**：去重时要包括测试集在内。用 n-gram 重叠检测训练数据和测试数据之间的重复，并移除重叠的训练数据。

### 错误 2：数据打包时忘记文档边界的 attention mask
**症状**：模型训练正常，但在推理时跨句引用信息，产生幻觉。
**原因**：打包将多个文档拼成一个序列，但 Attention 计算时没有屏蔽文档之间的交互。模型学到了"跨文档的伪关系"。
**修复**：创建一个 block-diagonal attention mask，让 Attention 只在同一文档内进行。不同文档的 token 不能互相 attend。

### 错误 3：数据配比的"表面比例"和"有效比例"不一致
**症状**：训练后的模型在某些领域表现意外地差。
**原因**：不同来源的文档平均长度不同。如果按"文档数"计算配比（来源 A 取 1000 篇、来源 B 取 1000 篇），但 A 的文档平均 100 token、B 平均 10000 token，那么模型实际"看到"的 B 是 A 的 100 倍。
**修复**：按 token 数或字符数（而非文档数）计算配比。

### 错误 4：全部依赖自动过滤，放弃了人工审核
**症状**：数据看起来"干净"（各项指标正常），但模型输出中有奇怪的偏差或错误。
**原因**：自动过滤规则有盲区。某些看起来"正常"的文本实际上包含错误信息、偏见或低质量内容。
**修复**：定期进行人工审核——从各来源抽样几百到几千条数据，人工检查质量。这不能替代自动过滤，但能发现自动过滤的盲区。

### 错误 5：数据规范化损失了重要信息
**症状**：分词器解码后，原始文本的信息丢失（如特殊的空格布局、大小写）。
**原因**：过度激进的规范化（如全小写化、去除所有特殊字符）可能破坏对某些任务至关重要的信息。例如，代码数据中的缩进和大小写有语法含义。
**修复**：对不同来源使用不同的规范化策略。代码数据不做大小写转换；诗歌数据保留原始排版。

### 错误 6：去重时使用了太小的 n-gram
**症状**：去重率极高（如 90%），怀疑"正常"的内容也被移除了。
**原因**：使用 3-gram 或 4-gram 做去重时，常见的短语（如"the United States"）会在大量正常文档中出现，导致这些文档被误判为重复。
**修复**：使用足够大的 shingle size（如 5-gram 或 10-gram）来减少误匹配。对句子级去重使用较长的最小重复长度（如 50 token）。

### 错误 7：未考虑数据的时效性
**症状**：模型输出的知识过时（如"当前总统是..."）。
**原因**：训练数据可能覆盖了多年的内容，模型无法区分信息的新旧程度。
**修复**：在数据中保留时间戳信息（如果可用），并在数据配比中考虑时间权重。某些应用场景下，可能需要定期使用最新数据更新模型。

### 错误 8：忘记处理数据中的个人身份信息（PII）
**症状**：模型可能输出真实的邮箱、电话、人名等信息。
**原因**：训练数据中包含未被清除的个人信息。
**修复**：使用 PII 过滤工具（如 `presidio`）检测和遮蔽个人信息。对于姓名、邮箱、电话等常见 PII 类型，进行正则匹配和替换。

## 11. 本章小结

数据工程是模型训练中最耗时、最不被重视但最关键的一环：

1. **数据质量优先**：精心筛选的小数据集往往超过未筛选的大数据集。Garbage In, Garbage Out。
2. **多样化的数据源**：CommonCrawl 提供规模，维基百科/书籍/论文提供质量，代码赋予编程能力。
3. **多阶段过滤**：规则过滤（长度、语言、特殊字符）→ 模型评分过滤（困惑度、分类器）→ 去重（精确 + 近似 + 句子级）。
4. **去重是免费的午餐**：删除 50% 的重复数据意味着同样的计算预算下有效训练数据翻倍。
5. **数据配比是艺术**：没有"最优"配比，但有好的实践——保证多样性，给高质量来源更高的采样权重。
6. **打包减少浪费**：将小文档拼接成固定长度序列，padding 浪费从可能是 50% 降到 5% 以内。
7. **版本管理不可忽视**：DVC 提供了类似 Git 的体验来管理数据版本和数据处理流程。

在下一章，我们将学习分布式训练——当数据准备好后，如何用成百上千张 GPU 高效地训练模型。
