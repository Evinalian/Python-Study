## 5.1 模型保存与加载：不只是"保存文件"

### 5.1.1 核心问题：为什么不能像保存图片一样保存模型

当你在画图软件中做了一张图，保存为 PNG 文件，发给别人打开，所见即所得。你可能会期待训练好的模型也能这样"一键保存、一键加载"。但模型和图片有本质不同：PNG 文件不仅包含像素值，还包含了分辨率、色彩模式等解码信息；而 PyTorch 模型的"形状"是由 Python 代码定义的——层数、连接方式、激活函数类型——这些信息不是数据，而是指令。

这就引出了 PyTorch 模型持久化的核心设计选择：**参数（数据）和结构（代码）应该分开保存吗？**

PyTorch 的回答是：**分开。** 你保存参数值（`state_dict`），用代码定义结构。为什么这样设计？因为参数值是"快照"，可以在任意时刻保存和传输；而结构是"配方"，必须精确到代码层面才能执行。二者的生命周期完全不同——你可能会多次修改结构（加层、改激活函数），但参数值只在训练后产生。如果把它们绑在一起（保存整个模型对象），每次结构调整后旧的模型文件就作废了。

### 5.1.2 state_dict：模型参数的清单

在 PyTorch 中，每一个 `nn.Module` 子类都有一个 `state_dict()` 方法。它返回一个 **Python 有序字典（OrderedDict）**，键是层的名称，值是对应的参数张量（`nn.Parameter`）。你可以把它想象成一份"库存清单"——每个货架（层）上放了什么东西（参数张量），一清二楚。

```python
import torch
import torch.nn as nn

class SimpleModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(10, 5)   # 权重 5x10 + 偏置 5 = 55 个参数
        self.fc2 = nn.Linear(5, 2)    # 权重 2x5 + 偏置 2 = 12 个参数

    def forward(self, x):
        x = torch.relu(self.fc1(x))
        x = self.fc2(x)
        return x

model = SimpleModel()

# 查看 state_dict 的键和形状
for name, param in model.state_dict().items():
    print(f"{name:20s}  shape={str(param.shape):15s}  dtype={param.dtype}")
```

输出示例：

```
fc1.weight            shape=torch.Size([5, 10])  dtype=torch.float32
fc1.bias              shape=torch.Size([5])       dtype=torch.float32
fc2.weight            shape=torch.Size([2, 5])    dtype=torch.float32
fc2.bias              shape=torch.Size([2])       dtype=torch.float32
```

从这个输出中你能获得两个关键信息。第一，`state_dict` 只包含**可学习参数**（`nn.Parameter`），不包含层的结构信息（层数、激活函数类型、连接方式）。它不告诉你 `fc1` 后面接的是 ReLU 还是 Sigmoid，也不告诉你 `fc1` 和 `fc2` 之间有没有 Dropout。第二，参数命名遵循 `层名.参数类型` 的规则，这让你可以精确地访问或修改任意层的任意参数。

### 5.1.3 buffer 也在 state_dict 中

一个常见的误解是"buffer 不在 state_dict 中"，但事实恰恰相反。BatchNorm 的 `running_mean` 和 `running_var` 是 buffer（不可学习但在 forward 中更新），它们**会被 state_dict 包含**。这一点至关重要——如果你自己实现序列化而遗漏了 buffer，加载后 BatchNorm 的推理行为将完全错误。

```python
# 验证 state_dict 是否包含 buffer
bn = nn.BatchNorm1d(3)
for name, buf in bn.state_dict().items():
    print(f"{name:20s}  shape={str(buf.shape)}")
# 输出：
# weight              shape=torch.Size([3])
# bias                shape=torch.Size([3])
# running_mean        shape=torch.Size([3])
# running_var         shape=torch.Size([3])
# num_batches_tracked shape=torch.Size([])
```

可以看到 `running_mean`、`running_var`（以及 `num_batches_tracked`）都在其中。PyTorch 的设计原则是：`state_dict` 包含前向传播过程中需要持久化的**所有**状态——无论可学习还是不可学习。

### 5.1.4 方式一：保存 state_dict（推荐）

这是 PyTorch 官方推荐的方式。思想很简单：参数归参数，结构归结构。

**保存：**

```python
# 保存模型的 state_dict
torch.save(model.state_dict(), "model_weights.pth")
```

**加载：**

```python
# 1. 先重新实例化模型（必须！因为 state_dict 不含结构信息）
model = SimpleModel()  # 模型结构必须与保存时完全一致

# 2. 加载权重
model.load_state_dict(torch.load("model_weights.pth"))

# 3. 切换到评估模式（如果用于推理）
model.eval()
```

这段代码的每一步都有明确的动机：

| 步骤 | 为何必要 |
|------|---------|
| `torch.save(model.state_dict(), ...)` | 只保存参数值，文件小。`model.state_dict()` 返回 Python dict，`torch.save` 将其序列化 |
| `model = SimpleModel()` | 创建空模型——你必须重新定义结构，因为 `.pth` 中只有数值没有层定义 |
| `model.load_state_dict(...)` | 逐层拷贝参数值。键名必须完全匹配——如果代码中 `fc1` 被改名了，加载失败 |
| `model.eval()` | 切换为推理模式。如果你要接着训练，自然跳过这步 |

这种方式的灵活性在你做迁移学习时会得到充分体现：你可以创建一个与原始模型结构不同的新模型，然后只加载部分层的权重（通过 `load_state_dict` 的 `strict=False` 参数）。

### 5.1.5 方式二：保存整个模型对象（以及为什么你不该用它）

```python
# 保存整个模型
torch.save(model, "model_full.pth")

# 加载整个模型
model = torch.load("model_full.pth")
model.eval()
```

这种方式的诱惑是显而易见的：一行加载，不用重新定义模型类。但这个便利背后隐藏着严重的脆弱性。`torch.save(model, ...)` 底层使用 Python 的 `pickle` 模块序列化整个模型对象，pickle 保存的是对象的完整 Python 路径——包括它定义在哪个文件的哪个类。当你在另一台机器（或同一个项目的另一个目录）加载这个文件时，Python 需要找到**完全相同**的类定义。

```python
# 假设你在 my_project/models.py 中定义了 SimpleModel
# torch.save(model, "model_full.pth")

# 在另一个脚本中加载：
model = torch.load("model_full.pth")  # 可能报错！
# AttributeError: Can't get attribute 'SimpleModel' on <module '__main__'>
```

这个报错的原因是：pickle 记录的是 `my_project.models.SimpleModel`，但加载脚本中这个路径不存在。解决办法是确保加载时 Python 能定位到类定义：

```python
# 方案 A：在加载脚本中导入或定义相同的类
from my_project.models import SimpleModel  # 确保类在命名空间中
model = torch.load("model_full.pth")

# 方案 C（最稳妥）：使用 torch.serialization.add_safe_globals
torch.serialization.add_safe_globals([SimpleModel])
model = torch.load("model_full.pth", weights_only=False)
```

**安全性提醒**：PyTorch 2.x 默认 `torch.load` 的 `weights_only=True`，这意味着加载整个模型对象需要通过 `weights_only=False` 显式放松限制。这是因为 pickle 反序列化可以执行任意代码——如果你从不可信来源获取了模型文件，存在安全风险。

**对比总结：**

| 特性 | `state_dict` 方式 | 整个模型方式 |
|------|-------------------|-------------|
| 文件大小 | 小（仅参数） | 大（含结构） |
| 加载灵活性 | 高（可部分加载） | 低 |
| 跨版本兼容 | 好 | 差 |
| 使用便捷性 | 需重定义模型类 | 一行加载 |
| 安全性 | 高（`weights_only=True` 默认） | 低（pickle 风险） |
| 推荐程度 | **强烈推荐** | 不推荐 |

### 5.1.6 优化器的 state_dict：断点续训的另一半

如果模型的 state_dict 是"模型学会了什么"，优化器的 state_dict 就是"优化器记住了什么"。对于 SGD，优化器状态很少——基本只有学习率。但对于 Adam/AdamW，优化器持有大量的内部状态：每个参数的一阶矩 `exp_avg` 和二阶矩 `exp_avg_sq`。这些状态记录了梯度的历史信息，是优化器"知道该往哪走、该走大步还是小步"的基础。

如果断点续训时只恢复模型参数而不恢复优化器状态，Adam 的两组动量阵将重新从零开始累积，导致恢复后的训练行为与中断前完全不同——学习率自适应机制被重置了。

```python
import torch.optim as optim

model = SimpleModel()
optimizer = optim.Adam(model.parameters(), lr=0.001)

# 模拟几步训练
x = torch.randn(4, 10)
y = torch.randn(4, 2)
for _ in range(3):
    optimizer.zero_grad()
    loss = nn.MSELoss()(model(x), y)
    loss.backward()
    optimizer.step()

# 查看优化器的 state_dict 的键
print("Optimizer state_dict keys:")
for key in optimizer.state_dict().keys():
    print(f"  {key}")

# 保存模型 + 优化器
checkpoint = {
    "model_state_dict": model.state_dict(),
    "optimizer_state_dict": optimizer.state_dict(),
}
torch.save(checkpoint, "checkpoint.pth")
```

优化器的 `state_dict` 典型结构揭示了它维护了哪些信息：

```
{
    'state': {
        0: {  # 参数 ID（对应 model.parameters() 中的索引）
            'exp_avg': tensor(...),   # Adam 一阶矩（梯度均值）
            'exp_avg_sq': tensor(...), # Adam 二阶矩（梯度方差）
            'step': tensor(0)          # 步数
        },
        1: {...},
        ...
    },
    'param_groups': [{
        'lr': 0.001,
        'betas': (0.9, 0.999),
        'eps': 1e-08,
        'weight_decay': 0,
        'amsgrad': False,
        'params': [0, 1, 2, 3]  # 参数 ID 列表
    }]
}
```

## 5.2 训练中断与恢复：一个完整的场景故事

### 5.2.1 当你训练了 3 天，停电了

假设你在训练一个模型，预计需要 7 天。第 3 天晚上，服务器因为过热自动关机。早上你来到实验室，发现训练进程没了。

如果没有 checkpoint，你会面临两个选择：从头开始再训 7 天（浪费了前 3 天的计算和时间），或者放弃这个实验（浪费了 3 天的心血）。有了 checkpoint，你只需要从最后一个保存点恢复，继续训练剩下的 4 天——就像游戏里的"存档"。

这就是为什么生产级训练脚本必须包含 checkpoint 机制。它不是"锦上添花"的可选项，而是"保底"的生命线。

### 5.2.2 一个完整的 Checkpoint 应该包含什么

一个生产级别的 checkpoint 不只是"保存模型"——它要保存从当前时刻恢复到完全一致的训练状态所需的一切：

```python
checkpoint = {
    "epoch": epoch,                           # 当前 epoch 编号（知道训到哪了）
    "model_state_dict": model.state_dict(),   # 模型参数
    "optimizer_state_dict": optimizer.state_dict(),  # 优化器状态（Adam 动量等）
    "loss": loss.item(),                      # 最近的 loss 值（监控用）
    "scheduler_state_dict": scheduler.state_dict(),  # 学习率调度器状态
    "best_accuracy": best_accuracy,           # 历史最佳准确率
    "args": args_dict,                        # 训练参数记录（用于复现）
}
```

损失函数通常不需要保存——因为它不包含内部状态。但如果你使用了自定义的带状态的损失函数（极少见），那也要保存它的副本。

### 5.2.3 完整训练代码（含 Checkpoint 保存与恢复）

下面是一个完整的断点续训示例。请特别关注恢复逻辑中的初始化顺序——顺序错误会导致加载失败，而且报错信息通常不直观。

```python
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset
import os

# ============================================================
# 1. 定义模型
# ============================================================
class Classifier(nn.Module):
    def __init__(self, input_dim=20, hidden_dim=64, num_classes=3):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, num_classes),
        )

    def forward(self, x):
        return self.net(x)

# ============================================================
# 2. 超参数与配置
# ============================================================
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
LEARNING_RATE = 0.001
NUM_EPOCHS = 50
CHECKPOINT_DIR = "./checkpoints"
CHECKPOINT_PATH = os.path.join(CHECKPOINT_DIR, "checkpoint.pth")
BEST_MODEL_PATH = os.path.join(CHECKPOINT_DIR, "best_model.pth")
RESUME_TRAINING = True  # 是否从 checkpoint 恢复

os.makedirs(CHECKPOINT_DIR, exist_ok=True)

# ============================================================
# 3. 准备数据（模拟数据用于演示）
# ============================================================
torch.manual_seed(42)
X_train = torch.randn(1000, 20)
y_train = torch.randint(0, 3, (1000,))
X_val = torch.randn(200, 20)
y_val = torch.randint(0, 3, (200,))

train_loader = DataLoader(
    TensorDataset(X_train, y_train), batch_size=32, shuffle=True
)
val_loader = DataLoader(
    TensorDataset(X_val, y_val), batch_size=32, shuffle=False
)

# ============================================================
# 4. 初始化模型、损失函数、优化器、调度器
# ============================================================
model = Classifier(input_dim=20, hidden_dim=64, num_classes=3).to(device)
criterion = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=LEARNING_RATE)
scheduler = optim.lr_scheduler.StepLR(optimizer, step_size=10, gamma=0.5)

# ============================================================
# 5. 从 checkpoint 恢复训练（如果存在）
# ============================================================
start_epoch = 0
best_val_acc = 0.0

if RESUME_TRAINING and os.path.exists(CHECKPOINT_PATH):
    print(f"[INFO] 发现 checkpoint: {CHECKPOINT_PATH}，正在恢复...")
    checkpoint = torch.load(CHECKPOINT_PATH, map_location=device)

    model.load_state_dict(checkpoint["model_state_dict"])
    optimizer.load_state_dict(checkpoint["optimizer_state_dict"])
    scheduler.load_state_dict(checkpoint["scheduler_state_dict"])
    start_epoch = checkpoint["epoch"] + 1  # 从下一个 epoch 开始
    best_val_acc = checkpoint.get("best_accuracy", 0.0)

    print(f"[INFO] 恢复成功！从 Epoch {start_epoch} 继续训练")
    print(f"[INFO] 历史最佳验证准确率: {best_val_acc:.4f}")
else:
    print("[INFO] 未找到 checkpoint，从头开始训练")

# ============================================================
# 6. 验证函数
# ============================================================
@torch.no_grad()
def evaluate(model, dataloader, criterion, device):
    model.eval()
    total_loss = 0.0
    correct = 0
    total = 0
    for inputs, labels in dataloader:
        inputs, labels = inputs.to(device), labels.to(device)
        outputs = model(inputs)
        loss = criterion(outputs, labels)
        total_loss += loss.item() * inputs.size(0)
        _, predicted = torch.max(outputs, 1)
        correct += (predicted == labels).sum().item()
        total += labels.size(0)
    return total_loss / total, correct / total

# ============================================================
# 7. 训练循环（含 checkpoint 保存）
# ============================================================
for epoch in range(start_epoch, NUM_EPOCHS):
    # ---- 训练阶段 ----
    model.train()
    train_loss = 0.0
    for inputs, labels in train_loader:
        inputs, labels = inputs.to(device), labels.to(device)

        optimizer.zero_grad()
        outputs = model(inputs)
        loss = criterion(outputs, labels)
        loss.backward()
        optimizer.step()

        train_loss += loss.item() * inputs.size(0)

    train_loss /= len(train_loader.dataset)
    scheduler.step()  # 更新学习率

    # ---- 验证阶段 ----
    val_loss, val_acc = evaluate(model, val_loader, criterion, device)

    # ---- 打印进度 ----
    current_lr = optimizer.param_groups[0]["lr"]
    print(f"Epoch {epoch+1:3d}/{NUM_EPOCHS} | "
          f"Train Loss: {train_loss:.4f} | "
          f"Val Loss: {val_loss:.4f} | "
          f"Val Acc: {val_acc:.4f} | "
          f"LR: {current_lr:.6f}")

    # ---- 保存 checkpoint（每个 epoch） ----
    checkpoint = {
        "epoch": epoch,
        "model_state_dict": model.state_dict(),
        "optimizer_state_dict": optimizer.state_dict(),
        "scheduler_state_dict": scheduler.state_dict(),
        "loss": train_loss,
        "best_accuracy": best_val_acc,
    }
    torch.save(checkpoint, CHECKPOINT_PATH)
    # 每个 epoch 覆盖同一个文件。生产环境建议保留最近 N 个防止断电时损坏文件。

    # ---- 保存最佳模型 ----
    if val_acc > best_val_acc:
        best_val_acc = val_acc
        torch.save(model.state_dict(), BEST_MODEL_PATH)
        print(f"  >>> 新的最佳模型！Val Acc: {best_val_acc:.4f}，已保存至 {BEST_MODEL_PATH}")

print(f"\n训练完成！最佳验证准确率: {best_val_acc:.4f}")
```

### 5.2.4 恢复训练的关键细节

**细节 1：`map_location` 的不可或缺**

当你把训练好的 checkpoint 从 GPU 服务器拷到自己的 CPU 笔记本上做分析时，如果不在 `torch.load` 中指定 `map_location`，PyTorch 会尝试把张量加载到当初保存时的设备（GPU），而你的笔记本没有 CUDA——结果就是 RuntimeError。

```python
# 在 CPU 机器上加载 GPU 上保存的 checkpoint
checkpoint = torch.load("checkpoint.pth", map_location="cpu")

# 加载到指定 GPU
checkpoint = torch.load("checkpoint.pth", map_location="cuda:0")
```

**细节 2：`optimizer.load_state_dict` 的初始化顺序**

加载优化器状态时，必须保证两件事：(1) 模型已经存在并在正确的 device 上；(2) 优化器已经用模型参数初始化。如果你先加载优化器状态再创建模型，或者模型还没 `to(device)` 就加载优化器，会出现 device 不匹配的奇怪错误。

```python
model = MyModel().to(device)  # 先移到 device
optimizer = optim.Adam(model.parameters())  # 再创建优化器
checkpoint = torch.load("checkpoint.pth", map_location=device)
model.load_state_dict(checkpoint["model_state_dict"])
optimizer.load_state_dict(checkpoint["optimizer_state_dict"])  # 此时才能正确加载
```

**细节 3：Random Seed 的复现**

如果要在恢复训练后获得**完全一致**的结果（比如做科学实验时需要严格的确定性），仅保存模型和优化器状态是不够的——随机数生成器也有状态。对于绝大多数场景，你不需要这么严格的复现，但了解这个选项是好的：

```python
import random
import numpy as np

# 保存时：
checkpoint["python_seed"] = random.getstate()
checkpoint["numpy_seed"] = np.random.get_state()
checkpoint["torch_seed"] = torch.get_rng_state()

# 恢复时：
random.setstate(checkpoint["python_seed"])
np.random.set_state(checkpoint["numpy_seed"])
torch.set_rng_state(checkpoint["torch_seed"])
```

### 5.2.5 生产级别的 Checkpoint 策略

逐个 epoch 覆盖同一个文件的风险在于：如果恰好文件写入过程中断电（或磁盘满），checkpoint 文件会损坏。生产环境通常保留最近 N 个 checkpoint：

```python
import glob

def save_checkpoint(state, filename, keep_best_n=3):
    torch.save(state, filename)

    # 清理旧 checkpoint
    checkpoint_files = sorted(glob.glob(os.path.join(CHECKPOINT_DIR, "checkpoint_epoch_*.pth")))
    if len(checkpoint_files) > keep_best_n:
        for old_file in checkpoint_files[:-keep_best_n]:
            os.remove(old_file)
            print(f"[INFO] 删除旧 checkpoint: {old_file}")

# 使用：
save_checkpoint(
    checkpoint,
    os.path.join(CHECKPOINT_DIR, f"checkpoint_epoch_{epoch+1:03d}.pth"),
    keep_best_n=3,
)
```

## 5.3 模型评估模式：train 和 eval 的精确语义

### 5.3.1 两个方法影响哪些层

`model.train()` 和 `model.eval()` 不是"全局开关"——它们只影响特定类型的层。理解这个精确语义，你才能判断自己的模型在切换模式时到底会发生什么。

| 层类型 | `model.train()` 行为 | `model.eval()` 行为 |
|--------|---------------------|---------------------|
| `nn.Dropout` | 随机将输入元素置零（概率 p），保留元素乘以 1/(1-p) | 不置零，直接通过 |
| `nn.BatchNorm1d/2d/3d` | 使用当前 batch 的均值/方差，同时更新 running 统计量 | 使用历史累积的 running 统计量，不更新 |
| `nn.LayerNorm` | 通常不受影响 | 通常不受影响 |
| `nn.InstanceNorm` | 不受 train/eval 切换影响（设计如此） | 同上 |

### 5.3.2 演示 Dropout 的行为差异

```python
import torch
import torch.nn as nn

torch.manual_seed(42)
dropout = nn.Dropout(p=0.5)
x = torch.ones(1, 6)  # 全 1 的输入

# 训练模式：约一半的元素变为 0，其余变为 2（补偿因子 1/(1-p) = 2）
dropout.train()
print("Train mode (run 1):", dropout(x))
print("Train mode (run 2):", dropout(x))  # 每次结果不同！
print("Train mode (run 3):", dropout(x))

# 评估模式：所有元素保持不变
dropout.eval()
print("Eval  mode (run 1):", dropout(x))
print("Eval  mode (run 2):", dropout(x))
print("Eval  mode (run 3):", dropout(x))
```

注意 train 模式下保留的元素值变为 2 而不是 1——这是 PyTorch 的 inverted dropout 特性：训练时做缩放（乘以 1/(1-p)），使得 eval 时可以直接恒等映射而不需要额外的缩放操作。

### 5.3.3 演示 BatchNorm 的行为差异

```python
bn = nn.BatchNorm1d(3, track_running_stats=True)

torch.manual_seed(42)
x_train = torch.randn(8, 3)  # 训练数据
x_test = torch.randn(4, 3)   # 测试数据

# 训练模式：使用当前 batch 统计量，并更新 running stats
bn.train()
out_train = bn(x_train)
print(f"Train mode - running_mean: {bn.running_mean}")

# 评估模式：使用 running stats（固定不变）
bn.eval()
out_eval_1 = bn(x_test)
print(f"Eval  mode - running_mean: {bn.running_mean}")  # 不会因 x_test 而改变
out_eval_2 = bn(x_test)
print(f"Eval  mode (再次) - running_mean: {bn.running_mean}")
```

**关键规则**：推理时的标准写法必须是 `model.eval()` 后加 `torch.inference_mode()` 或 `torch.no_grad()`。只调用其中一个是不够的——`model.eval()` 控制 Dropout/BN 的行为模式，`no_grad()` 控制计算图构建。两者职责不同，缺一不可。

```python
# 正确的推理写法
model.eval()
with torch.no_grad():
    outputs = model(inputs)
```

### 5.3.4 `torch.no_grad()` vs `torch.inference_mode()`：不只是快一点

很多人认为这两个上下文管理器是"差不多的"。它们的共性确实很大——都禁用自动求导，不构建计算图。但 `inference_mode()`（PyTorch 1.9+）更彻底：它不仅禁用了梯度计算，还禁用了版本号追踪和 tensor/view 的 autograd 元数据记录。结果是 `inference_mode()` 比 `no_grad()` 快 10-15%，且更省显存。

代价是：在 `inference_mode()` 内部，你不能创建 `requires_grad=True` 的 tensor，也不能调用需要 autograd 元数据的操作。对推理场景而言，这些限制无所谓——推理本来就不需要梯度。因此结论很明确：**推理时用 `torch.inference_mode()`，只在有特殊需求时用 `torch.no_grad()`**。

```python
import torch
import time

x = torch.randn(1000, 1000)

# ---- torch.no_grad() ----
with torch.no_grad():
    y = x * 2 + 1
    print(f"no_grad: requires_grad={y.requires_grad}")

# ---- torch.inference_mode() (PyTorch 1.9+) ----
# 完整禁用 autograd，比 no_grad 更快
with torch.inference_mode():
    y = x * 2 + 1
    print(f"inference_mode: requires_grad={y.requires_grad}")

# 注意：inference_mode 内部无法创建 requires_grad=True 的 tensor
try:
    with torch.inference_mode():
        w = torch.randn(10, requires_grad=True)  # 报错！
except RuntimeError as e:
    print(f"\n错误（符合预期）: {e}")
```

**选择建议：**

| 场景 | 推荐 |
|------|------|
| 验证/测试循环 | `torch.inference_mode()` |
| 需要创建可求导 tensor 的上下文（极少见） | `torch.no_grad()` |
| 推理服务 | `torch.inference_mode()` |
| 最大兼容性（旧版 PyTorch） | `torch.no_grad()` |

## 5.4 模型导出：为什么部署不能用训练代码

### 5.4.1 根本问题：Python 和 PyTorch 是训练的利器，却是部署的负担

你在训练时享受的所有便利——动态计算图、Python 控制流、交互式调试——到了部署阶段都变成了障碍。生产环境需要的是：确定性（每次同样的输入必须得到同样的输出）、高性能（毫秒级延迟）、小体积（不需要 2GB 的 PyTorch 库）、跨平台（服务器上的 GPU、手机上的 NPU、浏览器里的 WebAssembly）。

你的训练脚本是用 Python 写的，里面满是 `if`/`for`/列表推导/第三方库调用。这种灵活的动态代码直接在生产环境中运行的代价太高了——每次推理都要 Python 解释器的开销、GIL 锁、无法做图优化。模型导出就是要把"灵活的 Python 程序"转化为"高效的、自包含的计算图"。

两条主流路径：TorchScript（PyTorch 自有格式，路径最短）和 ONNX（跨框架通用格式，生态最广）。

### 5.4.2 TorchScript：两种生成方式，两种思维模式

TorchScript 是 PyTorch 的中间表示（IR）。它把 Python 模型转化为一个独立于 Python 的图结构，可以在 C++ 运行时（LibTorch）中执行。TorchScript 的两种生成方式——trace 和 script——代表了两种截然不同的设计哲学。

#### 方式 A：`torch.jit.script`（分析源码）

script 直接读取你的 Python 源代码，把抽象语法树（AST）翻译成 TorchScript IR。它的思路是"我读你的代码，理解你的逻辑，用另一种语言实现它"。因此它能处理控制流——if 分支、for 循环、while 循环都能完整保留。

```python
import torch
import torch.nn as nn

class ScriptableModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc = nn.Linear(10, 5)

    def forward(self, x):
        # 支持控制流！
        if x.sum() > 0:
            x = torch.relu(self.fc(x))
        else:
            x = torch.sigmoid(self.fc(x))
        return x

model = ScriptableModel()

# 使用 torch.jit.script 转换
scripted_model = torch.jit.script(model)
print(scripted_model.code)  # 查看生成的 TorchScript 代码

# 保存
scripted_model.save("model_scripted.pt")

# 加载
loaded_model = torch.jit.load("model_scripted.pt")
```

script 的代价是对 Python 语法有严格限制——不是所有的 Python 语法都能翻译成 TorchScript IR。比如动态类型、`**kwargs`、某些生成器表达式都可能让 script 失败。如果你用到了大量 Python 特性，script 可能需要大量修改代码。

#### 方式 B：`torch.jit.trace`（追踪执行路径）

trace 的思路完全不同：它不去"读"代码，而是"看"代码跑一次——给模型一个示例输入，记录下哪些操作被执行了、按什么顺序、中间张量形状是什么。然后把这条执行轨迹固化成一个计算图。

```python
# 使用 trace
example_input = torch.randn(1, 10)
traced_model = torch.jit.trace(model, example_input)

# 保存
traced_model.save("model_traced.pt")

# 加载
loaded_model = torch.jit.load("model_traced.pt")
```

trace 的最大优点是简单——大部分前馈模型（CNN、MLP）可以无痛通过 trace 导出。最大缺点是**它只看一条路径**。如果模型中存在依赖于输入数据的 if 语句，trace 只会记录触发的那条分支，另一条分支永远不会出现在导出的计算图中。

#### script vs trace：一个决定性的对比

```python
class ControlFlowModel(nn.Module):
    def forward(self, x):
        # 当 x.sum() > 0 走分支 A，否则走分支 B
        if x.sum() > 0:
            return x * 2    # 分支 A
        else:
            return x * 3    # 分支 B

model = ControlFlowModel()

# ---- trace：只记录一条路径 ----
traced = torch.jit.trace(model, torch.tensor([1.0, 2.0]))  # sum > 0，走分支 A
print("Traced 输出 [1,2]:", traced(torch.tensor([1.0, 2.0])))   # [2, 4] --- 正确
print("Traced 输出 [-5,-6]:", traced(torch.tensor([-5.0, -6.0])))  # [-10, -12] --- 错误！

# ---- script：完整保留控制流 ----
scripted = torch.jit.script(model)
print("Script 输出 [1,2]:", scripted(torch.tensor([1.0, 2.0])))   # [2, 4]   --- 正确
print("Script 输出 [-5,-6]:", scripted(torch.tensor([-5.0, -6.0])))  # [-15, -18] --- 正确！
```

trace 在处理 `[-5, -6]` 时仍然返回 `x * 2`（分支 A 的路径），因为它的计算图中根本没有分支 B。这是一个"静默错误"——不会报错，但结果错了。

**选择决策：**

```
模型是否有 if/for 依赖于输入数据？
    ├── 是 → 使用 torch.jit.script
    └── 否（纯前馈） → 使用 torch.jit.trace（更简单）
```

### 5.4.3 ONNX 导出：跨框架通行证

ONNX（Open Neural Network Exchange）是一个行业联盟推动的跨框架模型交换格式。它的定位是"模型世界的中立语言"——PyTorch 训练、ONNX 导出、然后用 ONNX Runtime（CPU 推理）、TensorRT（NVIDIA GPU 优化）、OpenVINO（Intel 平台）或 CoreML（Apple 设备）来推理。

```python
import torch
import torch.nn as nn

class ONNXModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.conv = nn.Conv2d(3, 16, 3, padding=1)
        self.pool = nn.AdaptiveAvgPool2d(1)
        self.fc = nn.Linear(16, 10)

    def forward(self, x):
        x = torch.relu(self.conv(x))
        x = self.pool(x)
        x = x.view(x.size(0), -1)
        x = self.fc(x)
        return x

model = ONNXModel()
model.eval()

# 准备示例输入
dummy_input = torch.randn(1, 3, 32, 32)

# 导出为 ONNX
torch.onnx.export(
    model,                          # 模型
    dummy_input,                    # 示例输入
    "model.onnx",                   # 输出文件名
    export_params=True,             # 是否保存参数
    opset_version=14,               # ONNX opset 版本
    do_constant_folding=True,       # 是否做常量折叠优化
    input_names=["input"],          # 输入节点名
    output_names=["output"],        # 输出节点名
    dynamic_axes={                  # 动态轴配置
        "input": {0: "batch_size"},    # 第 0 维（batch）是动态的
        "output": {0: "batch_size"},
    },
)

print("ONNX 模型已导出至 model.onnx")

# ---- 验证 ONNX 模型 ----
import onnx  # pip install onnx

onnx_model = onnx.load("model.onnx")
onnx.checker.check_model(onnx_model)
print("ONNX 模型验证通过！")

# ---- 使用 ONNX Runtime 推理 ----
import onnxruntime  # pip install onnxruntime
import numpy as np

ort_session = onnxruntime.InferenceSession("model.onnx")

# 准备输入（numpy 格式）
ort_inputs = {"input": dummy_input.numpy()}
ort_outputs = ort_session.run(None, ort_inputs)
print(f"ONNX Runtime 输出 shape: {ort_outputs[0].shape}")

# 与 PyTorch 输出对比
with torch.no_grad():
    torch_output = model(dummy_input)
print(f"PyTorch 与 ONNX 输出最大误差: {(torch_output.numpy() - ort_outputs[0]).max():.8f}")
```

**`dynamic_axes` 的重要性**：如果不声明动态轴，导出的 ONNX 模型会固定 batch_size 为导出时的值。这意味着你训练时用的 batch_size=32，导出的模型就只能接受 batch=32 的输入。声明 `{0: "batch_size"}` 后，模型可以处理任意 batch_size。

**常见导出配置选项：**

| 参数 | 说明 | 建议值 |
|------|------|--------|
| `opset_version` | ONNX 算子集版本，越高支持的操作越多 | 14 或更高 |
| `do_constant_folding` | 预计算常量表达式，减少图节点 | `True` |
| `dynamic_axes` | 声明动态维度（如可变 batch size） | 按需设置 |
| `input_names` / `output_names` | 给输入/输出节点命名 | 有意义的名称 |

## 5.5 模型压缩简介

### 5.5.1 模型压缩要解决什么

一个 ResNet-50 大约有 25M 参数，以 FP32 存储约 100MB。在云端 GPU 上这不叫事，但在手机上（App 体积敏感）、IoT 设备上（内存以 KB/MB 计）、边缘服务器上（算力有限），100MB 的模型是不可接受的。模型压缩的三个方向——量化、剪枝、知识蒸馏——各从不同角度解决问题。

### 5.5.2 量化（Quantization）：降低数值精度

量化的核心思想是：既然 FP32（32 位浮点）中有大量精度对你的任务来说是冗余的，何不用 8 位整数来存？一张图片的像素值之间微小的 FP32 精度差异对分类结果几乎没有影响。将 FP32 权重映射到 INT8，模型体积立刻减为 1/4，而且整数乘加运算比浮点快得多（尤其在专用硬件上）。

**三种量化方式及其 tradeoff：**

**动态量化** 是最简单入门的：只量化权重（FP32 -> INT8），激活值在推理时动态量化。适合 LSTM/Transformer 这类权重比激活值占更多计算的模型。精度损失很小——通常不到 1% 的准确率下降。

```python
import torch.quantization

model = SimpleModel()
model.eval()

# 动态量化：对 nn.Linear 和 nn.LSTM 等层有效
quantized_model = torch.quantization.quantize_dynamic(
    model,
    {nn.Linear, nn.LSTM},
    dtype=torch.qint8,
)
```

**静态量化** 同时量化权重和激活值，效果更好但需要额外的校准步骤——用一批代表性数据跑一遍，确定激活值的量化范围。如果校准数据不能代表真实数据分布，精度损失会显著增大。

**量化感知训练（QAT）** 在训练过程中就模拟量化误差，让模型学会在低精度下工作。这是精度最高的量化方式，但需要修改训练流程。

| 方式 | 精度损失 | 速度提升 | 使用复杂度 | 适用场景 |
|------|---------|---------|-----------|---------|
| 动态量化 | 小 | 中（1.5-2x） | 极低 | LSTM / Transformer |
| 静态量化 | 中 | 高（2-4x） | 中 | CNN / 生产部署 |
| QAT | 最小 | 高（2-4x） | 高 | 对精度要求极高的场景 |

### 5.5.3 剪枝（Pruning）：移除冗余连接

剪枝的直觉来自生物学：人脑在发育过程中会"修剪"掉不常用的神经连接，保留关键通路。在神经网络中，大量权重值接近零，对整个网络的输出贡献微乎其微——它们占用存储和算力但不产生价值。剪枝将这些"不重要"的权重彻底置零，产生一个稀疏矩阵。

**非结构化剪枝**（移除单个权重）能在保持精度的前提下移除 50%-90% 的权重——但实际加速效果高度依赖硬件的稀疏矩阵支持。**结构化剪枝**（移除整个神经元/通道）直接缩小了矩阵维度，更容易被硬件加速，但精度损失更大。

```python
import torch.nn.utils.prune as prune

model = SimpleModel()

# 全局非结构化剪枝（L1 范数）：将 fc1 层中 L1 范数最小的 30% 权重置零
prune.l1_unstructured(model.fc1, name="weight", amount=0.3)

# 移除剪枝（使稀疏永久化）
prune.remove(model.fc1, "weight")
```

**剪枝方法汇总：**

| 方法 | 粒度 | 硬件加速 | 说明 |
|------|------|---------|------|
| `l1_unstructured` | 单个权重 | 困难 | 将不重要的单个权重置零 |
| `random_unstructured` | 单个权重 | 困难 | 随机置零 |
| `ln_structured` | 通道/神经元 | 容易 | 移除整行/整列，减小矩阵维度 |

### 5.5.4 知识蒸馏（Knowledge Distillation）：大模型教小模型

知识蒸馏的思路和前两种完全不同——它不修改原模型，而是用一个大模型（Teacher）训练一个小模型（Student）。关键洞见在于：Teacher 不仅告诉 Student"这张图是猫"，还告诉它"这张图有 70% 概率是猫，20% 概率是虎，8% 概率是狗"——这个概率分布包含了"猫和虎很像"的额外知识，比硬标签（100% 是猫，0% 是其他）信息量丰富得多。

**概念示意：**

```
输入图片（猫）
    │
    ├──> Teacher (ResNet-152) ──> soft labels: [猫:0.7, 虎:0.2, 狗:0.08, ...]
    │                                    └── 包含"猫和虎很像"的知识
    │
    └──> Student (MobileNet) ──> 学习目标：
         ├── 分类 loss（与真实标签）
         └── 蒸馏 loss（与 Teacher 的 soft labels，KL 散度）
```

**温度参数（Temperature）** 控制 soft labels 的"软度"。温度越高，概率分布越平滑（各类别差异越小），类间相似性信息越突出。温度设为 1 就是原始 softmax；温度趋于无穷，所有类别的概率相等。

```python
# 假设 teacher_model 已经训练好，student_model 是要训练的小模型
teacher_model.eval()
temperature = 4.0  # 温度参数，越高 soft labels 越"软"
alpha = 0.7        # 蒸馏 loss 权重

for inputs, labels in dataloader:
    # Teacher 预测（no_grad）
    with torch.no_grad():
        teacher_logits = teacher_model(inputs)

    # Student 预测
    student_logits = student_model(inputs)

    # 硬标签 loss（标准交叉熵）
    hard_loss = nn.CrossEntropyLoss()(student_logits, labels)

    # 软标签 loss（KL 散度，关注 Teacher 输出的分布）
    soft_loss = nn.KLDivLoss(reduction="batchmean")(
        torch.log_softmax(student_logits / temperature, dim=1),
        torch.softmax(teacher_logits / temperature, dim=1),
    ) * (temperature ** 2)  # 梯度缩放

    # 总 loss
    total_loss = alpha * soft_loss + (1 - alpha) * hard_loss

    total_loss.backward()
    optimizer.step()
```

**蒸馏的关键超参数：**

| 参数 | 作用 | 典型值 |
|------|------|--------|
| `temperature` | 控制 soft labels 的"软度"。越高各类别概率越接近 | 2 -- 20 |
| `alpha` | 蒸馏 loss 与硬标签 loss 的权重比 | 0.5 -- 0.9 |

## 5.8 常见错误

### 错误 1：忘记 model.eval() 导致推理结果不稳定

这个错误的内在逻辑：BatchNorm 在 train 模式下使用当前 batch 的统计量做归一化。推理时 batch 往往很小（甚至 batch=1），统计量高度不可靠。加上 Dropout 仍在随机丢弃神经元，导致每次推理结果不同且偏低。

```python
# ---- 正确的推理写法 ----
model = MyModelWithBN()
model.load_state_dict(torch.load("best.pth"))
model.eval()  # 必须调用！
with torch.inference_mode():
    predictions = model(test_data)
```

### 错误 2：在 inference_mode 下创建 requires_grad 张量

`inference_mode()` 彻底禁用了 autograd 系统，任何要求梯度的操作都会报错。这个设计是合理的——如果你在推理模式下需要梯度，那你就不是在推理。

```python
# ---- 错误 ----
with torch.inference_mode():
    x = torch.randn(10, requires_grad=True)  # RuntimeError!

# ---- 正确 ----
with torch.inference_mode():
    x = torch.randn(10)  # requires_grad=False
```

### 错误 3：保存和加载的模型结构不一致

`load_state_dict` 的键名匹配是基于字符串的——`fc1.weight` 必须对应 `fc1.weight`，连层的命名都不能差一个字符。如果加载失败报 size mismatch，先检查两边模型的参数形状是否一致。

```python
# 保存时
model = nn.Linear(10, 5)
torch.save(model.state_dict(), "weights.pth")

# 加载时——结构不同！
model = nn.Linear(10, 3)  # 输出维度 5 -> 3
# model.load_state_dict(torch.load("weights.pth"))  # RuntimeError: size mismatch
```

### 错误 4：torch.load 时未指定 map_location

这个错误几乎每个初学者都会遇到一次：在 Colab（有 GPU）上训练保存的 checkpoint，下载到自己的 CPU 笔记本上加载，报 CUDA 不可用的错误。

```python
# ---- 正确 ----
checkpoint = torch.load("gpu_checkpoint.pth", map_location="cpu")
```

### 错误 5：优化器 state_dict 加载顺序错误

优化器在创建时就与模型参数建立了绑定关系。如果你在加载 checkpoint 之前就创建了优化器但模型还没加载参数，或者模型和优化器不在同一设备上，就会出现 device 不匹配的问题。

```python
# ---- 正确顺序 ----
model = MyModel().to(device)             # 1. 创建模型并移至目标设备
optimizer = optim.Adam(model.parameters())  # 2. 用模型参数创建优化器
checkpoint = torch.load("checkpoint.pth", map_location=device)
model.load_state_dict(checkpoint["model_state_dict"])        # 3. 先加载模型
optimizer.load_state_dict(checkpoint["optimizer_state_dict"])  # 4. 再加载优化器
```

### 错误 6：混淆 script 和 trace

用 trace 导出含数据依赖控制流的模型会产生静默错误——不会报任何异常，但推理结果在某些输入上是错的。如果你不确定模型中是否有控制流（或者不知道预训练模型内部有没有），最保险的做法是用 script。

```python
# 含控制流的模型：必须用 script
class DynamicModel(nn.Module):
    def forward(self, x):
        if x.sum() > 0:
            return x * 2
        return x * 3

model = DynamicModel()
# traced = torch.jit.trace(model, torch.tensor([1.0]))  # 错误！会丢失分支
scripted = torch.jit.script(model)  # 正确
```

## 5.9 本章小结

**模型保存的设计哲学**：参数（state_dict）和结构（代码）应当分离。参数是数值快照，结构是可执行配方，二者的生命周期不同。`torch.save(model.state_dict())` 是唯一推荐的方式——文件小、安全、灵活、跨环境兼容。整个模型的 pickle 保存虽然"方便"，但跨环境脆弱且存在安全风险，不应用于生产环境。

**Checkpoint 不只是"保存模型"**：一个生产级 checkpoint 包含 epoch、模型参数、优化器状态、调度器状态。缺少任何一个组件，恢复后的训练行为都会与中断前不同。特别要注意：Adam/AdamW 的动量矩阵（exp_avg 和 exp_avg_sq）是优化器状态的核心——丢失了它们，自适应学习率的优势荡然无存。恢复训练的初始化顺序（模型 -> 优化器 -> 加载 state_dict）不容错乱。

**train 和 eval 的精确语义**：这两个方法不是"全局开关"，而只影响特定层的行为模式。Dropout 在 train 下随机丢弃、eval 下恒等映射；BatchNorm 在 train 下使用 batch 统计并更新 running stats、eval 下使用固定的全局统计。推理时必须 `model.eval() + torch.inference_mode()` 组合使用——前者控制层行为，后者禁用计算图。

**模型导出的本质**是将"灵活的 Python 程序"转化为"高效的自包含计算图"。TorchScript（PyTorch 自有格式）：`trace` 简单但不支持控制流，`script` 支持控制流但语法受限。ONNX（跨框架通用格式）：一次导出，多引擎运行（ONNX Runtime、TensorRT、OpenVINO 等）。`dynamic_axes` 让导出的模型支持可变 batch_size。

**模型压缩三大方向各有侧重**：量化降低精度（FP32 -> INT8），适合通用部署；剪枝移除冗余连接，适合稀疏计算的硬件；知识蒸馏用大模型教小模型，在精度和体积之间取得最佳平衡。
