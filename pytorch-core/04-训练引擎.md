## 1. 训练循环：理解每一个步骤

### 1.1 从一个"不学习的模型"说起

如果你只定义了一个模型却从不调用训练循环，模型中的参数将永远停留在随机初始化的状态。所谓"训练"，本质上是让模型参数沿着损失函数减小的方向逐步移动的过程。这个过程被分解为几个固定的步骤，每一步都有其不可替代的角色。

让我们从宏观视角理解整个训练循环。一次完整的训练包含两层循环：外层按 epoch 遍历数据集，内层按 batch 遍历每个小批量数据。每个 batch 的处理又遵循一个六步流水线：设置训练模式、清零梯度、前向传播、计算损失、反向传播、参数更新。这六步中任何一步的缺失或顺序错误都会导致训练失败，而且不同步骤的失败表现各异——有的会直接报错，有的则悄无声息地让模型学不到东西（这正是最危险的）。

下面是训练循环的伪代码，建立了整体框架之后我们再逐行深挖每一步的"为什么"：

```
for epoch in range(num_epochs):
    # === 训练阶段 ===
    model.train()
    for batch in train_loader:
        optimizer.zero_grad()       # 清零上一步的梯度
        output = model(input)       # 前向传播
        loss = criterion(output, target)  # 计算损失
        loss.backward()             # 反向传播，计算梯度
        optimizer.step()            # 根据梯度更新参数
        record_loss(loss)

    # === 验证阶段 ===
    model.eval()
    with torch.no_grad():
        for batch in val_loader:
            output = model(input)
            val_loss = criterion(output, target)
            record_val_loss(val_loss)
```

### 1.2 完整代码实现

在深入每一步的原理之前，先把可运行的代码立起来，这样后面的解释有了具体的锚点。以下代码将训练和验证分别封装为独立函数——这种封装不是形式上的"好看"，而是因为训练和验证的行为模式截然不同：训练需要梯度计算和参数更新，验证则需要彻底关闭这些机制。把它们分开，代码的意图就一目了然。

```python
import torch
import torch.nn as nn
from torch.utils.data import DataLoader

def train_one_epoch(model, loader, criterion, optimizer, device):
    """执行一个 epoch 的训练，返回平均 loss"""
    model.train()
    total_loss = 0.0
    n_batches = 0

    for batch_x, batch_y in loader:
        batch_x = batch_x.to(device)
        batch_y = batch_y.to(device)

        optimizer.zero_grad()
        outputs = model(batch_x)
        loss = criterion(outputs, batch_y)
        loss.backward()
        optimizer.step()

        total_loss += loss.item()
        n_batches += 1

    return total_loss / n_batches

@torch.no_grad()
def evaluate(model, loader, criterion, device):
    """在验证/测试集上评估模型，返回平均 loss 和准确率（分类任务）"""
    model.eval()
    total_loss = 0.0
    correct = 0
    total = 0
    n_batches = 0

    for batch_x, batch_y in loader:
        batch_x = batch_x.to(device)
        batch_y = batch_y.to(device)

        outputs = model(batch_x)
        loss = criterion(outputs, batch_y)

        total_loss += loss.item()
        n_batches += 1

        # 计算准确率（仅分类任务）
        _, predicted = outputs.max(1)     # 取 logit 最大值的索引
        total += batch_y.size(0)
        correct += predicted.eq(batch_y).sum().item()

    avg_loss = total_loss / n_batches
    accuracy = 100.0 * correct / total
    return avg_loss, accuracy
```

**代码解读**：注意 `train_one_epoch` 中的 `model.train()` 和 `evaluate` 中的 `model.eval()` 分别位于函数的第一行——这绝非偶然。`train_one_epoch` 没有使用 `@torch.no_grad()` 装饰器，因为它必须构建计算图才能做反向传播；而 `evaluate` 则用 `@torch.no_grad()` 彻底关闭了自动求导。另外，`evaluate` 中 `correct += predicted.eq(batch_y).sum().item()` 这行的 `.item()` 调用值得特别注意：它将 GPU 上的单元素张量转为 Python 标量，不这么做的话，`correct` 变量会持有整个计算图的引用，导致显存泄漏。

### 1.3 每一步的详细解释

现在让我们逐行拆解训练循环的每一步，回答三个核心问题：这一步做了什么、为什么必须这么做、如果不这么做会发生什么。

| 步骤 | 代码 | 为什么这样做 | 不这样做会发生什么 |
|------|------|-------------|-------------------|
| 1. 训练模式 | `model.train()` | 启用 Dropout（随机置零神经元）、启用 BatchNorm 的 batch 统计量更新 | Dropout 不工作（无正则化效果），BatchNorm 不更新 running stats（推理不稳定） |
| 2. 梯度清零 | `optimizer.zero_grad()` | `backward()` 的梯度是**累加**的（设计特性）。如果不清零，本次的梯度会叠加在上次的梯度上 | 梯度不断累加，相当于使用了错误的梯度方向，参数往随机方向更新 |
| 3. 前向传播 | `output = model(input)` | 数据流经网络各层，构建计算图。`output` 中的每个元素都记录了如何从参数计算得到（通过 `grad_fn`） | 没有前向传播就无法计算 loss |
| 4. 计算损失 | `loss = criterion(output, target)` | 将预测与真实标签比较，得到一个标量值。`backward()` 只能从标量开始反向传播 | 如果 loss 不是标量，backward 会报错 |
| 5. 反向传播 | `loss.backward()` | 沿计算图从 loss 反向计算所有 `requires_grad=True` 的参数的梯度，结果存到 `.grad` 属性 | 参数的 .grad 为 None，optimizer.step() 无法更新参数 |
| 6. 参数更新 | `optimizer.step()` | 对每个参数执行 `param = param - lr * param.grad`（SGD 的情况）。Adam 等优化器的更新公式更复杂，但概念相同 | 参数永远不会改变——模型不会学习 |
| 7. 记录 loss | `total_loss += loss.item()` | `.item()` 将单元素 Tensor 转为 Python float，释放计算图引用，避免内存累积 | 计算图持续占用内存，GPU 显存可能溢出 |

**关于梯度累加的深层原因**：PyTorch 设计 `backward()` 的梯度累加行为不是 bug，而是一个精心设计的特性。试想，如果每次 `backward()` 都自动覆盖梯度，那么当你需要从多个 loss 来源累积梯度时（比如多任务学习、GAN 的判别器/生成器交替训练），反而需要手动管理梯度的保存和累加。梯度累加的设计给了开发者最大的灵活性——你可以在一个 batch 内多次调用 `backward()` 来累积来自不同 loss 的梯度，然后再一次性 `step()`。代价是你必须在每次新 batch 开始时手动清零。这个 tradeoff（灵活性 vs 易出错）在 PyTorch 的很多设计决策中反复出现。

### 1.4 设备迁移的时机

GPU 训练中有一个常被忽视的细节：模型和数据分别在什么时候搬到 GPU 上？直觉上你可能会想到"让数据提前待在 GPU 上不是更快吗"，但这个直觉是错误的。模型在训练开始前一次性搬到 GPU，之后不再移动——因为模型参数在整个训练过程中需要持久存在。但数据完全不同：数据集通常远大于 GPU 显存，你不可能把所有数据都塞进显存里。

正确的做法是模型初始化后立即 `to(device)`，而数据在每个 batch 加载时才从 CPU 搬到 GPU：

```python
# 推荐做法：将模型在训练开始前一次性移到 GPU
model = MyModel().to(device)

# 数据在每批处理时移（Dataset 中保持 CPU Tensor）
for batch_x, batch_y in train_loader:
    batch_x = batch_x.to(device)
    batch_y = batch_y.to(device)
```

**为什么不把 Dataset 中的数据预先放到 GPU？** 原因有三层。第一层最直接：GPU 显存通常远小于系统内存（比如 24GB vs 256GB），装不下整个数据集。第二层是技术约束：DataLoader 的 `num_workers` 多进程预加载机制工作在 CPU 内存空间，子进程无法直接写入 GPU 显存（GPU 显存是进程独占的资源）。第三层是性价比：每批数据传输的 PCIe 开销相对于计算时间来说很小，尤其是配合 `pin_memory=True` 和 `non_blocking=True` 之后，传输可以和计算重叠。

### 1.5 梯度累积：用小显存模拟大 batch

当你读到一篇论文说"我们用 batch_size=1024 训练"，而你的 GPU 只能容纳 batch_size=32 时，梯度累积就是你的救命稻草。它的原理非常简单：PyTorch 的 `backward()` 会累加梯度到 `.grad` 属性而不是覆盖它。利用这个特性，你可以连续处理 32 个小 batch（`1024 / 32 = 32`），每个小 batch 都调用 `backward()` 但不调用 `step()` 和 `zero_grad()`，等梯度累积够了 32 次之后，再一次性 `step()` 更新参数并清零。

关键细节在于 loss 的缩放：因为每个小 batch 的 loss 是在 32 个样本上平均出来的，而你又累积了 32 个这样的 loss，所以总梯度相当于在 1024 个样本上的平均——前提是**每个小 batch 的 loss 除以累积步数**。如果不除以累积步数，梯度会被放大，等价于使用了比目标大得多的学习率。

```python
# 目标 batch_size = 128，但显存只够 32
# 解决方案：每 4 个小 batch 才做一次 optimizer.step()

accumulation_steps = 4  # 128 / 32 = 4

model.train()
optimizer.zero_grad()

for i, (batch_x, batch_y) in enumerate(train_loader):
    batch_x, batch_y = batch_x.to(device), batch_y.to(device)
    outputs = model(batch_x)
    loss = criterion(outputs, batch_y)

    # 除以累积步数，使得总梯度的量级与真实 batch_size 一致
    loss = loss / accumulation_steps
    loss.backward()  # 梯度累加（不清零）

    if (i + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()

# 处理最后不足 accumulation_steps 的部分
if (i + 1) % accumulation_steps != 0:
    optimizer.step()
    optimizer.zero_grad()
```

**梯度累积的一个重要局限**需要你心中有数：BatchNorm 层在每次 forward 时使用当前 mini-batch 的均值和方差做归一化。当你用梯度累积时，每个小 batch 的 BatchNorm 统计量只基于 32 个样本（而非 128 个），因此梯度累积**不等价于**真正的大 batch 下的 BatchNorm 行为。如果你的模型严重依赖 BatchNorm（这几乎是所有现代 CNN 的共性），而 batch_size 差别又很大时，梯度累积的结果和真正大 batch_size 的结果会有偏差。对此，一种改进方案是使用 GroupNorm 或 LayerNorm 替代 BatchNorm（它们在 batch 维度上的行为与 batch_size 无关），但这不是总能替换的。

## 2. 验证循环：`model.eval()` 与 `torch.no_grad()`

### 2.1 故事场景：为什么验证和训练不能混用

想象你在准备一场考试（训练），你需要在大量的练习题上反复练习，每次练习后你都会根据答案（label）检查自己的错误并修正（梯度更新）。但考试当天（验证/测试），你不能再翻看答案——否则成绩就不真实了。训练的"翻答案"行为对应模型中的 Dropout（随机丢弃神经元，强迫网络学习冗余表示）和 BatchNorm（用当前批次的统计量归一化）。验证时必须关闭这些训练行为，否则每次评估的结果都不一样，你无法判断模型到底有没有进步。

`model.eval()` 和 `model.train()` 改变的是某些特定层的行为，主要体现在两个组件上：

**Dropout** 在训练模式下随机将一部分神经元输出置为零（比例为 p），并将保留的神经元输出放大 1/(1-p) 倍以保持期望值不变。换言之，训练时每个神经元有 p 的概率"被暂时切除"，网络必须学会在任何神经元组合下都能工作——这是一种强制冗余学习。进入 eval 模式后，Dropout 彻底停止工作，输入直接通过，之前那个 1/(1-p) 的缩放因子也不再需要，因为 PyTorch 在训练时已经做了缩放。

**BatchNorm** 在训练模式下不仅用当前 batch 的均值和方差做归一化，还会用指数移动平均更新全局统计量 `running_mean` 和 `running_var`——这两个量记录了整个训练集上的统计信息。进入 eval 模式后，BatchNorm 不再计算当前 batch 的统计量（因为验证时 batch 可能很小，统计不稳定），而是直接使用训练期间累积的 `running_mean` 和 `running_var`。

如果在验证时忘记调用 `model.eval()`，后果是双重的：Dropout 仍然在随机丢弃神经元，导致每次验证结果不同且偏低（因为一半神经元被关掉了）；BatchNorm 使用验证 batch 自身的小样本统计量做归一化，因为验证时 batch 通常较小，统计不稳定，结果波动剧烈。

**验证 Dropout 和 BatchNorm 在 train/eval 下的行为差异**：

```python
import torch
import torch.nn as nn

model = nn.Sequential(
    nn.Linear(10, 20),
    nn.BatchNorm1d(20),
    nn.ReLU(),
    nn.Dropout(0.5),
    nn.Linear(20, 5),
)

x = torch.randn(8, 10)
# 重复运行多次 compare
model.train()
out_train_1 = model(x)
out_train_2 = model(x)

model.eval()
out_eval_1 = model(x)
out_eval_2 = model(x)

# train 模式：每次输出都不同（因为 Dropout 随机）
print(f"Train mode outputs equal: {(out_train_1 == out_train_2).all().item()}")  # False

# eval 模式：每次输出相同（Dropout 关闭，BN 使用 running stats）
print(f"Eval mode outputs equal: {(out_eval_1 == out_eval_2).all().item()}")   # True
```

**关键认识**：这个实验揭示了训练和推理之间一个根本性的不对称——训练是随机的（Stochastic），推理是确定的（Deterministic）。训练需要随机性来探索参数空间和防止过拟合；推理需要确定性来保证可复现的结果。`model.train()` 和 `model.eval()` 就是这两种模式的开关。

### 2.2 `torch.no_grad()`：不仅仅是"不需要梯度"

很多人把 `torch.no_grad()` 简单理解为"验证时不需要梯度所以关掉"，但它有两个更深层的收益，任一单拿出来都足以成为你使用它的理由。

第一个收益是**显存**。在正常的前向传播中，PyTorch 会默默构建一张计算图——记录每个中间激活值是从哪个输入、经过哪个运算产生的。这张图是反向传播的路线图，但它占用的显存巨大（有时比模型参数本身还大）。在验证时，你不需要这张图，因为你不做 `backward()`。`torch.no_grad()` 告诉 PyTorch："别建图了"，于是那些中间激活值用完就能被释放，显存节省通常有 30%-50%。

第二个收益是**速度**。构建计算图本身需要 CPU/GPU 时间。省掉这一步，验证速度通常能提升 10%-20%。在大型数据集上跑验证时，这个提升不是小数目。

第三个收益是**防御性编程**。如果你的验证代码中不小心调用了 `backward()`（可能是一个复制粘贴的错误），`no_grad` 会让它直接报错，而不是悄无声息地修改模型参数。

```python
@torch.no_grad()   # 装饰器或上下文管理器均可
def evaluate(model, loader, criterion, device):
    model.eval()
    for batch_x, batch_y in loader:
        outputs = model(batch_x)
        # ...
```

**有 `no_grad()` vs 无 `no_grad()` 的性能对比**：

```python
import time

model = SimpleCNN().cuda()
x = torch.randn(128, 1, 28, 28).cuda()

# 有 no_grad
model.eval()
start = time.time()
with torch.no_grad():
    for _ in range(1000):
        _ = model(x)
torch.cuda.synchronize()
time_with = time.time() - start
print(f"With no_grad:    {time_with:.3f}s")

# 无 no_grad
model.eval()
start = time.time()
for _ in range(1000):
    _ = model(x)
torch.cuda.synchronize()
time_without = time.time() - start
print(f"Without no_grad: {time_without:.3f}s")
print(f"Speedup: {time_without / time_with:.1f}x")
```

**为什么 eval 模式下的 Dropout 不需要缩放？** 这是一个常见困惑。训练时 Dropout 随机关闭比例为 p 的神经元，并将剩余神经元的输出乘以 1/(1-p)。这个缩放操作（称为 inverted dropout）让输出期望值保持不变。因此进入 eval 模式时，所有神经元都在工作，不需要任何额外缩放——PyTorch 已经把缩放内化在训练阶段了。这是和原始 Dropout 论文（需要 eval 时乘 1-p）的重要区别。

## 3. 优化器：从 SGD 到 AdamW 的演进史

优化器的选择不是一场"谁最新谁最好"的选美。每种优化器都是为了解决前一代优化器的某个具体痛点而生的。理解了这个演进逻辑，你才能在具体场景中做出有理有据的选择。

### 3.1 起点：SGD -- 朴素但诚实

最原始的 SGD 更新规则只有一个动作：沿着梯度反方向走一步，步长由 `lr` 决定。

```
param = param - lr * grad
```

这条规则的优势是简单——没有隐藏状态，没有魔法，每一个参数更新都直接反映当前梯度。代价也很明显：梯度是局部的、带噪声的。在一批数据上算出来的梯度未必是全局最优方向，SGD 会在损失曲面上走 Z 字形，收敛缓慢。

### 3.2 Momentum：给球加上惯性

Momentum 的灵感来自物理学：一个球从山坡上滚下来，不仅受当前位置重力的影响，还带着之前累积的速度。将这个直觉翻译成数学：

```
v_t = momentum * v_{t-1} + grad_t
param = param - lr * v_t
```

`v_t` 是"速度"，它不仅包含当前梯度 `grad_t`，还包含历史梯度的指数衰减和 `v_{t-1}`。当多个连续 batch 的梯度方向一致时，Momentum 加速前进；当梯度方向剧烈震荡时，历史速度起到平滑作用，减少震荡幅度。`momentum` 参数（通常设为 0.9）控制"惯性"的大小。设为 0 就退化为普通 SGD；设为 1 则完全不衰减，那显然不行。

```python
# 基础 SGD
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)

# SGD + Momentum（推荐）
optimizer = torch.optim.SGD(model.parameters(), lr=0.01, momentum=0.9)

# SGD + Momentum + Nesterov（稍好于标准 Momentum）
optimizer = torch.optim.SGD(model.parameters(), lr=0.01,
                             momentum=0.9, nesterov=True)
```

Nesterov Momentum 是对标准 Momentum 的一个小改进：标准 Momentum 是先算当前梯度再加历史速度，Nesterov 则是先按历史速度走一步，在那一步的位置上再算梯度。这个"向前看一步"的策略让更新方向更精准，通常能带来略微更好的收敛。

**SGD 的调参经验**：

- 学习率是最重要的超参数。CV 任务通常从 0.1 开始，NLP 从 0.01 开始
- Momentum 通常设为 0.9（几乎不需要调）
- 如果 loss 震荡剧烈，降低 lr；如果 loss 下降太慢，提高 lr
- SGD 通常需要配合学习率调度器（StepLR 或 CosineAnnealing）
- 使用 `weight_decay=1e-4` 到 `5e-4` 进行正则化

### 3.3 Adam：为每个参数定制学习率

SGD + Momentum 解决了方向平滑问题，但还存在一个更深层的问题：同一个全局学习率对所有参数一视同仁。但实际中，不同的参数需要不同的更新步长——频繁出现的特征对应的参数应该用小步长（已经学得差不多了），稀有特征对应的参数应该用大步长（还需要多学）。更微妙的是，在训练的不同阶段，同一个参数的"合适步长"也在变化。

Adam 的核心创新是维护两个指数移动平均：一阶矩 `m_t`（梯度均值，类似 Momentum 的速度）和二阶矩 `v_t`（梯度平方的均值，衡量梯度的不确定性）。在更新时，`m_t` 除以 `sqrt(v_t)` 的效果是：如果某个参数的梯度一直很稳定（v_t 小），它的有效学习率就大；如果梯度震荡剧烈（v_t 大），有效学习率就小。因此 Adam 对学习率的初始设置远不如 SGD 敏感——你用 lr=1e-3 和 lr=1e-4 可能都能训练出来。

```python
# Adam: 自适应学习率 + 动量
optimizer = torch.optim.Adam(model.parameters(), lr=0.001, betas=(0.9, 0.999))
```

Adam 的更新公式（简化版）：

```
m_t = beta1 * m_{t-1} + (1 - beta1) * grad_t        # 一阶矩
v_t = beta2 * v_{t-1} + (1 - beta2) * grad_t^2       # 二阶矩
param = param - lr * m_t / (sqrt(v_t) + eps)          # 参数更新
```

**为什么 Adam 有时泛化不如 SGD？** 这个问题的答案至今仍在学术界讨论。一个广为接受的观点是：Adam 的自适应学习率在训练后期"太聪明了"——它能把训练 loss 压得非常低，但泛化性能反而比 SGD 差。SGD 的均匀步长带来了隐式的正则化效果（类似噪声），而 Adam 的精细调控反而削弱了这种正则化。这也是为什么很多 CV 论文报告的最好结果仍然是在 ImageNet 上用 SGD + Momentum 训练出来的。

### 3.4 AdamW：解耦权重衰减的最后一公里

Adam 有一个隐蔽的问题：weight_decay 的实现方式。在 Adam 中，weight_decay 是通过在梯度上加上 `weight_decay * param` 来实现的（等价于 L2 正则化）。这看起来无害，但注意梯度会被 Adam 的自适应学习率缩放——也就是说，`weight_decay * param` 这一项也进入了 `m_t` 和 `v_t` 的计算。结果就是：不同参数的 weight_decay 效果不同，大梯度的参数受到的正则化比小梯度的参数弱。这破坏了正则化的均匀性。

AdamW 的解决方案直击要害：把 weight_decay 从自适应机制中完全解耦出来——直接在参数上做衰减，不经过 `m_t` 和 `v_t`。

```python
# AdamW: Adam + 解耦权重衰减（推荐替代 Adam）
optimizer = torch.optim.AdamW(model.parameters(), lr=0.001, weight_decay=0.01)
```

**Adam vs AdamW 的数学本质**：

```
# Adam (旧式 L2 正则化)
grad = grad + weight_decay * param      # weight_decay 进入了自适应机制
# 问题：weight_decay 和自适应学习率耦合，正则化效果不均匀

# AdamW (解耦权重衰减)
param = param - lr * weight_decay * param   # 直接在参数上衰减
# weight_decay 完全独立于梯度方向
```

AdamW 的解耦设计在实践中带来了显著的泛化提升，以至于它已经取代 Adam 成为默认推荐。

**Adam/AdamW 的调参经验**：

- lr: CV 任务通常 1e-4 到 1e-3；Transformer 通常 1e-4
- weight_decay: 通常 0.01 到 0.1（AdamW 用大值，Adam 用小值或 0）
- betas: 几乎总是用默认值 (0.9, 0.999)
- eps: 默认 1e-8，通常不需要改

### 3.5 优化器选择指南

这里的推荐不是教条，而是基于大量实践经验形成的共识：

| 场景 | 推荐 | 原因 |
|------|------|------|
| CV（CNN 训练） | SGD + Momentum (lr=0.1) | 经典配置，调参经验丰富，泛化性能好 |
| Transformer/BERT | AdamW (lr=1e-4) | 标准配置，warmup + decay |
| GAN / 强化学习 | Adam (lr=1e-4) | 对学习率鲁棒 |
| 快速实验/原型 | AdamW (lr=1e-3) | 默认参数通常 work，不需要精细调参 |
| 图像生成（扩散） | AdamW (lr=1e-4) | 稳定收敛 |
| 推荐系统 | Adam (lr=1e-3) | 稀疏梯度场景 |

**优化器收敛曲线的概念性对比**：

```
Loss
 |  SGD (无 momentum):  缓慢下降，震荡大
 |  SGD + Momentum:     较快下降，前期震荡
 |  Adam:               下降最快，前期收敛极快
 |  AdamW:              与 Adam 类似，但泛化更好
 |
 |  SGD 在大 batch 下 + 长训练，最终可能超越 Adam
 |  Adam 在前期最快，但最终精度可能不如精调 SGD
 +------------------------------------------------> Epochs
```

这条曲线揭示了一个反直觉的结论：**在训练的不同阶段，最好的优化器可能不同**。Adam 前期快但后期可能被 SGD 超越。一些高阶技术（如 SWATS）甚至会在训练中途从 Adam 切换到 SGD，各取所长。

### 3.6 不同优化器的完整代码对比

```python
import torch.optim as optim

def create_optimizer(optimizer_name, model, lr, **kwargs):
    """根据名称创建优化器"""
    if optimizer_name == "sgd":
        return optim.SGD(model.parameters(), lr=lr,
                         momentum=kwargs.get("momentum", 0.9),
                         weight_decay=kwargs.get("weight_decay", 1e-4),
                         nesterov=kwargs.get("nesterov", True))
    elif optimizer_name == "adam":
        return optim.Adam(model.parameters(), lr=lr,
                          betas=kwargs.get("betas", (0.9, 0.999)),
                          weight_decay=kwargs.get("weight_decay", 0))
    elif optimizer_name == "adamw":
        return optim.AdamW(model.parameters(), lr=lr,
                           betas=kwargs.get("betas", (0.9, 0.999)),
                           weight_decay=kwargs.get("weight_decay", 0.01))
    elif optimizer_name == "rmsprop":
        return optim.RMSprop(model.parameters(), lr=lr,
                             weight_decay=kwargs.get("weight_decay", 0))
    else:
        raise ValueError(f"Unknown optimizer: {optimizer_name}")

# 使用
optimizer = create_optimizer("adamw", model, lr=1e-3, weight_decay=0.01)
```

## 4. 学习率调度器：训练的"变速杆"

### 4.1 为什么不能用一个固定学习率走到头

想象你在一个完全陌生的黑暗空间中寻找最低点。一开始你对地形一无所知，应该迈大步（大学习率），快速探索。当你接近谷底时（训练后期），你需要迈小步（小学习率），否则会在最低点附近来回震荡而无法精确落入。

这就是学习率调度的核心直觉：**训练的早中晚期对学习率的需求不同**。早期需要大步快速靠近最优区域；中期需要中等步长在区域内搜索；晚期需要小步长精细定位最优解。

如果你用固定的大学习率，早期收敛快但后期无法收敛到精确解（震荡）。如果用固定的小学习率，后期精细但早期收敛极慢，可能在有限的训练时长内走不到最优区域。学习率调度器就是解决这个矛盾的工具——它让学习率随着训练进展自动调整。

### 4.2 常用调度器及适用场景

PyTorch 提供了丰富的调度器，它们之间的核心区别在于**学习率随时间变化的曲线形状**。选择合适的调度器，就是选择合适的下降曲线。

```python
import torch.optim.lr_scheduler as lr_scheduler

optimizer = torch.optim.AdamW(model.parameters(), lr=0.001)

# StepLR: 每 step_size 个 epoch，lr 乘以 gamma
scheduler = lr_scheduler.StepLR(optimizer, step_size=30, gamma=0.1)
# epoch 0-29: lr=0.001, epoch 30-59: lr=0.0001, epoch 60-89: lr=0.00001

# MultiStepLR: 在指定的 epoch 处衰减
scheduler = lr_scheduler.MultiStepLR(optimizer, milestones=[30, 60, 90], gamma=0.1)

# ExponentialLR: 每 epoch 衰减 gamma
scheduler = lr_scheduler.ExponentialLR(optimizer, gamma=0.95)

# CosineAnnealingLR: 余弦曲线衰减到 eta_min
scheduler = lr_scheduler.CosineAnnealingLR(optimizer, T_max=100, eta_min=1e-6)
# lr 沿余弦曲线从 0.001 平滑衰减到 1e-6

# CosineAnnealingWarmRestarts: 带重启的余弦衰减
scheduler = lr_scheduler.CosineAnnealingWarmRestarts(optimizer, T_0=10, T_mult=2)
# 每 10 个 epoch 重启一次，每次重启周期乘 2

# ReduceLROnPlateau: 当指标停滞时降低 lr
scheduler = lr_scheduler.ReduceLROnPlateau(
    optimizer, mode='min', factor=0.5, patience=10
)
# 当 val_loss 连续 10 个 epoch 不下降时，lr *= 0.5

# OneCycleLR: 一个周期内先升后降
scheduler = lr_scheduler.OneCycleLR(
    optimizer, max_lr=0.01, steps_per_epoch=len(train_loader), epochs=100
)
```

**StepLR 和 MultiStepLR** 是最经典的阶梯式衰减。在预设的 epoch 处将学习率骤降为原来的 gamma 倍。优点是规则清晰、可预期；缺点是衰减时机是人为设定的，可能与实际训练动态不匹配。

**CosineAnnealingLR** 是当前最流行的选择之一。它沿余弦曲线平滑衰减学习率——开始时下降缓慢（在高学习率区域多停留一会），中间加速下降，末尾又放缓（在低学习率区域精细搜索）。这个"两头慢中间快"的节奏被大量实验证明优于阶梯式衰减。它的变体 CosineAnnealingWarmRestarts 更进一步：每当学习率降到最低点时"重启"到初始值，每个周期产生一个新的局部最优解，你可以取这些解的集成（Snapshot Ensembling）。

**ReduceLROnPlateau** 是唯一一个"数据驱动"的调度器——它不预设衰减时间，而是监控验证 loss，每当 loss 停滞（plateau）时就降低学习率。这在你不确定需要训练多久的场景中特别有用，但它要求你有一个可靠的验证集。

调度器在训练循环中的调用位置：

```python
for epoch in range(num_epochs):
    train_loss = train_one_epoch(model, train_loader, criterion, optimizer, device)
    val_loss, val_acc = evaluate(model, val_loader, criterion, device)

    # StepLR / CosineAnnealingLR（每个 epoch 调用一次）
    scheduler.step()

    # ReduceLROnPlateau（需要传入监控指标）
    # scheduler.step(val_loss)
```

**关键细节**：`scheduler.step()` 必须在每个 epoch 结束时调用一次，而不是每个 batch ——如果你在每个 batch 后都调用它，学习率会在一个 epoch 内频繁衰减，训练会失控。唯一的例外是 OneCycleLR，它明确设计为每个 batch 更新一次。

### 4.3 Cosine Annealing 的数学原理

CosineAnnealingLR 之所以叫"余弦"，是因为学习率沿半段余弦曲线下降：

```
lr = eta_min + 0.5 * (eta_max - eta_min) * (1 + cos(pi * t / T_max))

其中:
    eta_max: 初始学习率
    eta_min: 最小学习率
    t: 当前 epoch (0 到 T_max-1)
    T_max: 半个余弦周期
```

这个公式在 `t=0` 时 `cos(0)=1`，`lr = eta_min + (eta_max - eta_min) = eta_max`；在 `t=T_max` 时 `cos(pi)=-1`，`lr = eta_min`。中间的变化是平滑的。与阶梯式衰减相比，余弦衰减的平滑性避免了学习率突变带来的训练扰动。同时，cos 函数在 0 和 pi 附近变化慢、在 pi/2 附近变化快的特性，天然实现了"两头慢中间快"的节奏。

### 4.4 Warmup：训练起步的谨慎阶段

有个问题我们一直没讨论：训练刚一开始的时候，模型参数是随机初始化的，梯度方向高度不稳定。如果这时候就用目标学习率（比如 0.1 或 1e-3），巨大的初始梯度会让参数一步跨出很远，可能导致训练初期震荡甚至发散。这在 Transformer 训练中尤其严重——没有 warmup 的 Transformer 几乎训不动。

Warmup 的策略简单而有效：让学习率在最初 N 个 epoch 从接近零线性增长到目标值，给优化器时间建立稳定的梯度统计量（Adam 的 `m_t` 和 `v_t`）之后，再进入正常的调度。

**手动实现线性 Warmup**：

```python
class WarmupScheduler:
    """线性 warmup：lr 从 warmup_start 线性增加到 optimizer 的初始 lr"""
    def __init__(self, optimizer, warmup_epochs, warmup_start=1e-6):
        self.optimizer = optimizer
        self.warmup_epochs = warmup_epochs
        self.warmup_start = warmup_start
        self.base_lrs = [group['lr'] for group in optimizer.param_groups]
        self.current_epoch = 0

    def step(self):
        self.current_epoch += 1
        if self.current_epoch <= self.warmup_epochs:
            # 线性增长
            progress = self.current_epoch / self.warmup_epochs
            for i, param_group in enumerate(self.optimizer.param_groups):
                param_group['lr'] = self.warmup_start + \
                                    (self.base_lrs[i] - self.warmup_start) * progress

# 使用方式：先用 WarmupScheduler.step()，再接主调度器
warmup = WarmupScheduler(optimizer, warmup_epochs=5)
main_scheduler = lr_scheduler.CosineAnnealingLR(optimizer, T_max=95)

for epoch in range(100):
    train_one_epoch(...)
    if epoch < 5:
        warmup.step()         # warmup 阶段
    else:
        main_scheduler.step()  # 主调度器
```

**使用 PyTorch 内置方式（更简洁）**：

```python
from torch.optim.lr_scheduler import LinearLR, CosineAnnealingLR, SequentialLR

optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)

warmup_scheduler = LinearLR(
    optimizer, start_factor=1e-3, total_iters=5
)
main_scheduler = CosineAnnealingLR(optimizer, T_max=95)
scheduler = SequentialLR(
    optimizer,
    schedulers=[warmup_scheduler, main_scheduler],
    milestones=[5]
)

for epoch in range(100):
    train_one_epoch(...)
    scheduler.step()
```

**使用 LambdaLR 自定义任意 schedule**：

```python
# 自定义学习率函数: warmup 5 epoch + cosine decay 95 epoch
def lr_lambda(epoch):
    if epoch < 5:
        return (epoch + 1) / 5  # warmup: 0.2 -> 1.0
    else:
        # cosine decay from 1.0 to 0
        progress = (epoch - 5) / 95
        return 0.5 * (1 + torch.cos(torch.tensor(torch.pi * progress)).item())

scheduler = lr_scheduler.LambdaLR(optimizer, lr_lambda)
```

`LambdaLR` 是最灵活的调度器：你提供一个函数，输入 epoch 编号，输出学习率乘因子。这让你可以实现任意形状的学习率曲线，不受预定义模板的限制。

**Warmup 的推荐设置**：Transformer 通常 warmup 4000 步或 5-10 个 epoch。CNN 训练如果不使用非常大的学习率，warmup 通常不是必需的，但加上也不会有什么坏处。一般来说，你的学习率越大，warmup 就越有必要。

## 5. 梯度裁剪：防止训练崩溃的兜底

梯度爆炸是一个真实且常见的问题，尤其在 RNN/LSTM 的训练中。当一个 batch 的数据恰好触发了某种极端模式时，反向传播的梯度可以呈指数级增长。如果 optimizer 在这样巨大的梯度上做 `step()`，参数会一步跨出很远——远到损失函数的值变成 NaN（Not a Number），训练就崩溃了。

梯度裁剪的思路直截了当：如果梯度的 L2 范数超过了预设的上限 `max_norm`，就按比例缩小使得范数恰好等于 `max_norm`。如果没超过，什么都不做。

```python
# 将整个模型参数的梯度范数限制在 max_norm 以内
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 在训练循环中的位置：loss.backward() 之后、optimizer.step() 之前
loss.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)  # 梯度裁剪
optimizer.step()
```

梯度裁剪在训练循环中的位置是锁死的：必须在 `backward()` 之后（梯度已产生）、`step()` 之前（参数尚未更新）。这个位置保证了裁剪操作直接作用于即将被用来更新参数的梯度。

梯度裁剪的等价实现清晰地揭示了它的工作原理：

```python
# 等价于：
total_norm = torch.sqrt(sum(p.grad.norm(2)**2 for p in model.parameters()))
if total_norm > max_norm:
    scale = max_norm / total_norm
    for p in model.parameters():
        p.grad *= scale
```

你不需要自己写这段——`clip_grad_norm_` 已经做了并且做了优化——但理解这个等价逻辑有助于你判断什么时候需要调大或调小 `max_norm`。

**按值裁剪（clip by value）** 是另一种形式的梯度裁剪，但它更粗暴：直接将每个梯度元素限制在 [-clip_value, clip_value] 之间。

```python
torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=1.0)
```

`clip_grad_norm_` 更常用，因为它保留了梯度的方向信息（只是整体缩放），而 `clip_grad_value_` 会扭曲方向。常用 `max_norm` 值：RNN 训练一般用 1.0 或 5.0；Transformer 训练通常用 1.0。

## 6. 过拟合：理解 bias-variance tradeoff

### 6.1 什么是 bias-variance tradeoff

过拟合不是一个 bug，它是机器学习中**偏差-方差权衡（bias-variance tradeoff）**的自然体现。理解这个概念比记住任何正则化技巧都重要，因为它解释了"为什么"这些技巧有效。

**偏差（Bias）**衡量模型预测的平均值偏离真实值的程度。高偏差意味着模型太简单，无法捕捉数据中的模式——欠拟合。**方差（Variance）**衡量模型对不同训练集的敏感程度。高方差意味着模型过度捕捉了训练数据中的噪声，换一批训练数据预测结果就完全不同——过拟合。

这两个东西是打架的：降低偏差通常会增加方差，降低方差通常会增加偏差。你的目标是找到两者之间的平衡点。当你拥有大量高质量数据时，你可以降低方差而不显著增加偏差（数据增强了模型的泛化基础）；当数据有限时，你必须通过各种正则化手段来压低方差——这就是本章要讲的过拟合应对策略。

### 6.2 Dropout：让神经元学会"独立"

Dropout 的核心思想非常优雅：每次前向传播时随机"关闭"一部分神经元（将输出置零），迫使网络不能依赖任何单个神经元——它必须学习对任何神经元子集都能工作的冗余表示。这等价于在每个训练 step 中，你都在训练一个不同的、随机减薄的子网络；测试时，所有这些子网络以某种方式集成了起来。

```python
model = nn.Sequential(
    nn.Linear(784, 256),
    nn.ReLU(),
    nn.Dropout(p=0.5),    # 50% dropout 率
    nn.Linear(256, 10),
)
```

关于 Dropout 的 `p` 参数，有一个常见的认知偏差：p 不是"保留概率"而是"丢弃概率"。Dropout(p=0.5) 意味着每个神经元有 50% 的概率被丢弃。CNN 中在卷积层后通常用较小的 p（0.1-0.3），因为卷积层的空间共享本身就有正则化效果；全连接层后通常用较大的 p（0.2-0.5）。

**为什么 eval 模式下不需要 Dropout？** 因为 Dropout 的"集成"效果是在训练过程中通过随机丢弃实现的。在测试时，你需要整个网络的完整能力来做预测——你不是在训练，不需要正则化。而且，如果在测试时还随机丢弃神经元，预测结果就会不稳定（每次都不一样），这显然不可接受。

### 6.3 早停 (Early Stopping)：适可而止的智慧

从过拟合的视角看，训练过程可以分为两个阶段。第一阶段：模型同时学习训练集和验证集上的模式，两个 loss 都在下降。第二阶段：模型开始记忆训练集中的噪声和特例（这些对泛化无帮助），训练 loss 继续下降但验证 loss 开始回升。早停就是在第二阶段开始前停止训练——在验证 loss 不再改善时收手。

早停不仅是防过拟合的手段，还是节省计算资源的实用策略：与其固定跑 200 个 epoch，不如让模型在合适的时候自动停下来。

```python
class EarlyStopping:
    def __init__(self, patience=10, min_delta=0.0):
        """
        patience: 连续多少个 epoch 无改善后停止
        min_delta: 多小的改善算"有效"
        """
        self.patience = patience
        self.min_delta = min_delta
        self.counter = 0
        self.best_loss = float('inf')
        self.early_stop = False

    def __call__(self, val_loss):
        if val_loss < self.best_loss - self.min_delta:
            self.best_loss = val_loss
            self.counter = 0                  # 有改善，重置计数器
        else:
            self.counter += 1                 # 无改善，计数+1
            if self.counter >= self.patience:
                self.early_stop = True
        return self.early_stop

# 使用
early_stopping = EarlyStopping(patience=10)
for epoch in range(num_epochs):
    train_loss = train_one_epoch(...)
    val_loss, val_acc = evaluate(...)
    if early_stopping(val_loss):
        print(f"Early stopping at epoch {epoch}")
        break
```

**patience 的艺术**：设得太小（比如 2），模型可能因为一次偶然的小波动就提前终止；设得太大（比如 50），早停就失去了意义。通常 5-15 是合理的范围。`min_delta` 用于过滤微小波动——验证 loss 从 0.351 降到 0.350 这种细微变化不算"有效改善"。

**带模型保存的早停（推荐）**：

```python
class EarlyStoppingWithCheckpoint:
    """带最佳模型保存的早停"""
    def __init__(self, patience=10, min_delta=0.0, save_path="best_model.pth"):
        self.patience = patience
        self.min_delta = min_delta
        self.save_path = save_path
        self.counter = 0
        self.best_loss = float('inf')
        self.early_stop = False

    def __call__(self, val_loss, model):
        if val_loss < self.best_loss - self.min_delta:
            self.best_loss = val_loss
            self.counter = 0
            # 保存最佳模型
            torch.save(model.state_dict(), self.save_path)
            return False
        else:
            self.counter += 1
            if self.counter >= self.patience:
                self.early_stop = True
            return self.early_stop
```

### 6.4 权重衰减 (Weight Decay)：让参数"安分"一些

权重衰减从损失函数的角度看等价于 L2 正则化——在损失函数中加入参数范数的惩罚项，促使所有参数向零收缩。直观理解：如果不加任何正则化，优化器会给某些特征分配极端大的权重（因为训练集中这些特征恰好非常"管用"）；但在验证集中，这些特征的分布可能不同，大权重反而导致错误。权重衰减对大参数施加持续的"拉力"，让模型不轻易赋予任何单个特征过大的影响力。

```python
# AdamW 中通过 weight_decay 参数实现（推荐）
optimizer = torch.optim.AdamW(model.parameters(), lr=0.001, weight_decay=0.01)

# SGD 中同样支持
optimizer = torch.optim.SGD(model.parameters(), lr=0.01, weight_decay=5e-4)
```

常用值：`1e-4 ~ 1e-2`。过大会导致欠拟合——参数被强行拉向零，模型表达力不足。

**Weight Decay vs L2 Regularization 的区别**：

```
L2 Regularization:
    loss_total = loss_orig + (lambda/2) * ||w||^2
    grad = grad_orig + lambda * w
    在 Adam 中，lambda * w 项也被自适应学习率缩放

Weight Decay (AdamW):
    loss_total = loss_orig  (不加 L2 项)
    param = param - lr * grad_orig  (Adam 步骤)
    param = param - lr * weight_decay * param  (直接衰减)
    权重衰减独立于梯度方向
```

### 6.5 数据增强：最有效的正则化是更多样化的数据

数据增强不修改模型也不修改损失函数——它通过扩充训练数据的多样性来直接打击过拟合的根源（训练数据有限）。每一次随机裁剪、翻转、颜色抖动，本质上都是对模型说："你看到的这张图片只是这个物体的一个视角，不要过度拟合这个特定视角。"

```python
train_transform = transforms.Compose([
    transforms.RandomHorizontalFlip(),
    transforms.RandomCrop(32, padding=4),
    transforms.ColorJitter(brightness=0.2),
    transforms.ToTensor(),
])
```

数据增强在应对策略中的独特地位：它几乎总是有效，几乎没有代价（只增加一些数据预处理计算），且与其他策略正交（可以叠加使用）。

### 6.6 标签平滑 (Label Smoothing)

标签平滑针对的是过拟合的一种微妙形式：模型对训练标签过于自信。在标准训练中，你要求模型对正确类别输出概率 1.0，对其他类别输出 0.0。标签平滑将 1.0 稍微"匀"一点给其他类别——比如正确类别变为 0.9，其他 9 个类别各分得 0.01。这样模型不会被鼓励对任何单个类别产生极端确信，从而为类间相似性留出了建模空间。

```python
# 使用 CrossEntropyLoss 的 label_smoothing 参数（PyTorch 1.10+）
criterion = nn.CrossEntropyLoss(label_smoothing=0.1)
```

标签平滑在 ImageNet 等大型分类任务上通常能带来 0.2-0.5% 的 top-1 精度提升——不是革命性的，但几乎没有代价。在知识蒸馏中，标签平滑还有一个额外的好处（或者说坏处，取决于视角）：它会让 teacher 模型的输出更"平滑"，影响蒸馏效果。

### 6.7 应对策略的优先级

过拟合的应对策略按"效果/代价比"排列：

| 优先级 | 方法 | 适用情况 | 效果 |
|--------|------|----------|------|
| 1 | 数据增强 | 始终推荐，尤其小数据集 | 非常有效 |
| 2 | Dropout | 全连接层过拟合 | 有效，几乎没有代价 |
| 3 | Weight Decay | 配合 AdamW，设 0.01~0.1 | 有效 |
| 4 | 早停 | 防止训练过久 | 有效，节省时间 |
| 5 | 标签平滑 | 分类任务 | 轻微提升泛化 |
| 6 | 减小模型 | 如果以上所有都无效（模型太大） | 根本性解决 |
| 7 | 增加数据 | 如果可以获取更多数据 | 最有效但成本高 |

这个优先级表格遵循一个原则：**先用低代价/高收益的策略，把高代价的策略留作最后手段**。数据增强和 Dropout 在绝大多数场景下已经能解决大部分过拟合问题。

## 7. Metrics 计算：准确率、精确率、召回率、F1

分类任务中，loss 是训练的信号（告诉优化器该往哪走），但 loss 本身不直观——loss=0.23 到底意味着模型好还是不好？你需要人类可解释的指标来评估模型的实际表现。准确率是最直观的，但它有一个致命缺陷：在类别不平衡的数据集上，模型可以靠"全猜多数类"获得高准确率而不学到任何东西。

精确率、召回率、F1 从不同角度弥补了准确率的盲区。精确率回答："模型说它是猫的图片中，有多少真的是猫？"（宁缺毋滥）。召回率回答："所有猫的图片中，模型找出了多少？"（宁滥毋缺）。F1 是精确率和召回率的调和平均——当两者都很重要时使用。

### 7.1 基础准确率

```python
def compute_accuracy(outputs, labels):
    """计算分类准确率"""
    _, preds = outputs.max(1)
    correct = preds.eq(labels).sum().item()
    return correct / labels.size(0)
```

### 7.2 使用 sklearn 计算多指标

```python
from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score
import numpy as np

def compute_metrics(outputs, labels, average="macro"):
    """计算多个分类指标"""
    _, preds = outputs.max(1)

    preds_np = preds.cpu().numpy()
    labels_np = labels.cpu().numpy()

    metrics = {
        "accuracy": accuracy_score(labels_np, preds_np),
        "precision": precision_score(labels_np, preds_np, average=average, zero_division=0),
        "recall": recall_score(labels_np, preds_np, average=average, zero_division=0),
        "f1": f1_score(labels_np, preds_np, average=average, zero_division=0),
    }
    return metrics
```

`average="macro"` 的含义是：先计算每个类别的指标，然后取算术平均（每个类别权重相同，不受样本数影响）。`average="micro"` 则是将所有类别的预测结果汇总后一次性计算（受样本数影响，大类别的贡献更大）。`average="weighted"` 是按各类别样本数加权平均。在类别不平衡的场景中，macro 更能暴露小类别的分类能力。

### 7.3 纯 PyTorch 实现（不依赖 sklearn）

在推理部署环境或竞赛代码中，你可能不想引入 sklearn 依赖。以下是纯 PyTorch 的实现：

```python
def precision_recall_f1_pytorch(outputs, labels, num_classes, average="macro"):
    """纯 PyTorch 实现多分类指标"""
    _, preds = outputs.max(1)

    # 对每个类别计算 TP, FP, FN
    tp = torch.zeros(num_classes, dtype=torch.int64)
    fp = torch.zeros(num_classes, dtype=torch.int64)
    fn = torch.zeros(num_classes, dtype=torch.int64)

    for c in range(num_classes):
        tp[c] = ((preds == c) & (labels == c)).sum()
        fp[c] = ((preds == c) & (labels != c)).sum()
        fn[c] = ((preds != c) & (labels == c)).sum()

    # 每个类别的 precision, recall, f1
    precision_per_class = tp / (tp + fp).clamp(min=1)
    recall_per_class = tp / (tp + fn).clamp(min=1)
    f1_per_class = 2 * precision_per_class * recall_per_class / \
                   (precision_per_class + recall_per_class).clamp(min=1e-8)

    if average == "macro":
        return {
            "precision": precision_per_class.mean().item(),
            "recall": recall_per_class.mean().item(),
            "f1": f1_per_class.mean().item(),
        }
    elif average == "micro":
        total_tp = tp.sum()
        total_fp = fp.sum()
        total_fn = fn.sum()
        micro_precision = total_tp / (total_tp + total_fp).clamp(min=1).item()
        micro_recall = total_tp / (total_tp + total_fn).clamp(min=1).item()
        micro_f1 = 2 * micro_precision * micro_recall / (micro_precision + micro_recall + 1e-8)
        return {"precision": micro_precision, "recall": micro_recall, "f1": micro_f1}
```

```python
log_dir = f"./runs/{datetime.now().strftime('%Y%m%d_%H%M%S')}"
writer = SummaryWriter(log_dir)

        loss.backward()
        optimizer.step()
        train_loss += loss.item()

        if batch_idx % 100 == 0:
            writer.add_scalar("Loss/train_batch", loss.item(), global_step)

    avg_train_loss = train_loss / len(train_loader)

    # 验证
    val_loss, val_acc = evaluate(model, val_loader, criterion, device)

    # 记录 epoch 级别指标
    writer.add_scalar("Loss/train", avg_train_loss, epoch)
    writer.add_scalar("Loss/val", val_loss, epoch)
    writer.add_scalar("Accuracy/val", val_acc, epoch)
    writer.add_scalar("LR", optimizer.param_groups[0]['lr'], epoch)

    # 记录梯度直方图（每 5 个 epoch 记录一次，减少日志体积）
    if epoch % 5 == 0:
        for name, param in model.named_parameters():
            if param.grad is not None:
                writer.add_histogram(f"gradients/{name}", param.grad, epoch)

    scheduler.step()

writer.close()
print(f"TensorBoard log saved to: {log_dir}")
```

**为什么梯度直方图要每 5 个 epoch 才记录一次？** 因为每个参数的梯度值通常有数千到数百万个元素，写入频繁会急剧膨胀 TensorBoard 日志文件的大小。5 个 epoch 一次是"能看到趋势"和"日志文件不要太大"之间的平衡。

## 9. 混合精度训练 (AMP) 简要引入

混合精度训练是当前 GPU 训练的一个标准配置——你几乎不需要额外成本就能获得约 40% 的显存节省和 50-80% 的速度提升。它的原理是利用 FP16（半精度浮点）做大部分运算（快且省显存），但保留一份 FP32 的权重副本用于参数更新（保证数值稳定性）。第 6 章将深入讲解精度格式和底层原理，这里先给出即用的代码模板：

```python
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()  # 梯度缩放器，防止 float16 下溢

for batch_x, batch_y in train_loader:
    batch_x, batch_y = batch_x.to(device), batch_y.to(device)
    optimizer.zero_grad()

    # 前向传播使用 autocast: 自动选择 float16 或 float32
    with autocast():
        outputs = model(batch_x)
        loss = criterion(outputs, batch_y)

    # 反向传播使用 scaler: 缩放 loss 防止梯度下溢
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)          # 在裁剪前恢复梯度尺度

    # 梯度裁剪（在 unscale 之后）
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

    scaler.step(optimizer)              # 使用缩放的梯度更新参数
    scaler.update()                     # 更新缩放因子
```

**GradScaler 的必要性**：FP16 能表示的最小正数约 6e-5。训练中很多梯度值远小于这个数，直接在 FP16 下会变成 0（下溢），导致对应参数永远无法更新。GradScaler 的做法是把 loss 放大（乘以一个很大的 scaling factor，初始为 65536），这样反向传播产生的梯度也被放大，脱离 FP16 的下溢区。在 `step()` 更新参数前再把梯度缩回原始尺度。整个过程是透明的——参数更新发生在 FP32 精度下，不受影响。

**AMP 的关键组件**：

| 组件 | 作用 |
|------|------|
| `autocast()` | 自动决定哪些操作用 float16、哪些用 float32 |
| `GradScaler` | 动态缩放 loss 防止 float16 梯度过小导致下溢 |
| `scaler.scale(loss)` | 将 loss 乘以缩放因子 |
| `scaler.unscale_(optimizer)` | 在 step 前恢复梯度原始尺度 |
| `scaler.step(optimizer)` | 安全地更新参数 |
| `scaler.update()` | 动态调整缩放因子 |

## 10. 检查点保存与恢复

### 10.1 state_dict vs 整个模型

PyTorch 提供两种保存方式：只保存参数（`state_dict`）和保存整个模型对象。它们的区别不仅仅是"一个文件大一个文件小"——而是涉及灵活性、兼容性和安全性。

**只保存参数（推荐）**：文件只含参数字典，轻量且灵活。加载时需要重新实例化模型结构。这种方式允许你修改模型代码后仍然加载旧权重（只要参数名匹配），也可以只加载预训练模型的前几层做迁移学习。

```python
# 保存
torch.save(model.state_dict(), "model_weights.pth")

# 加载
model = MyModel()
model.load_state_dict(torch.load("model_weights.pth"))
model = model.to(device)
```

**保存整个模型（不推荐）**：底层用 pickle 序列化整个 Python 对象，包含模型结构。问题在于 pickle 记录了类的完整模块路径——如果你的类定义位置变了，或者类名改了，加载就会失败。此外，pickle 反序列化存在安全风险。

```python
# 保存
torch.save(model, "model_full.pth")

# 加载
model = torch.load("model_full.pth")
```

**保存训练状态（用于断点续训）**：

```python
checkpoint = {
    "epoch": epoch,
    "model_state_dict": model.state_dict(),
    "optimizer_state_dict": optimizer.state_dict(),
    "scheduler_state_dict": scheduler.state_dict(),
    "loss": train_loss,
    "best_val_acc": best_val_acc,
}
torch.save(checkpoint, "checkpoint_epoch_10.pth")

# 恢复训练
checkpoint = torch.load("checkpoint_epoch_10.pth")
model.load_state_dict(checkpoint["model_state_dict"])
optimizer.load_state_dict(checkpoint["optimizer_state_dict"])
scheduler.load_state_dict(checkpoint["scheduler_state_dict"])
start_epoch = checkpoint["epoch"] + 1
best_val_acc = checkpoint["best_val_acc"]
```

### 10.2 自动保存最佳模型

```python
class ModelCheckpoint:
    """自动保存最佳模型"""
    def __init__(self, save_path="best_model.pth", mode="min"):
        self.save_path = save_path
        self.mode = mode
        self.best_metric = float("inf") if mode == "min" else float("-inf")

    def step(self, metric, model, optimizer=None, epoch=None):
        """如果 metric 更好，保存 checkpoint"""
        is_better = (self.mode == "min" and metric < self.best_metric) or \
                    (self.mode == "max" and metric > self.best_metric)

        if is_better:
            self.best_metric = metric
            checkpoint = {
                "model_state_dict": model.state_dict(),
                "metric": metric,
            }
            if optimizer is not None:
                checkpoint["optimizer_state_dict"] = optimizer.state_dict()
            if epoch is not None:
                checkpoint["epoch"] = epoch

            torch.save(checkpoint, self.save_path)
            print(f"Best model saved! metric={metric:.4f}")
            return True
        return False

# 使用
checkpoint = ModelCheckpoint("best_model.pth", mode="max")  # 准确率越高越好
for epoch in range(num_epochs):
    ...
    val_acc = evaluate(...)
    checkpoint.step(val_acc, model, optimizer, epoch)
```

## 11. 训练可视化：tqdm 进度条

在终端里盯着成百上千行 loss 数字刷屏不仅枯燥，而且无法快速感知训练速度。tqdm 在 DataLoader 上包装一层进度条，实时显示当前 batch 的 loss、处理速度（samples/sec）和预计剩余时间，让你对训练节奏有直观把握。

```python
from tqdm import tqdm

def train_one_epoch(model, loader, criterion, optimizer, device):
    model.train()
    total_loss = 0.0

    # 用 tqdm 包装 DataLoader，显示进度条
    pbar = tqdm(loader, desc="Training", leave=False)
    for batch_x, batch_y in pbar:
        batch_x, batch_y = batch_x.to(device), batch_y.to(device)

        optimizer.zero_grad()
        outputs = model(batch_x)
        loss = criterion(outputs, batch_y)
        loss.backward()
        optimizer.step()

        total_loss += loss.item()
        # 实时更新进度条显示当前 loss
        pbar.set_postfix({'loss': f'{loss.item():.4f}'})

    return total_loss / len(loader)
```

### 完整训练主循环（带 tqdm + 日志）

```python
import torch
import torch.nn as nn
from torch.utils.data import DataLoader
from tqdm import tqdm

def train(model, train_loader, val_loader, criterion, optimizer,
          scheduler, num_epochs, device, early_stopping=None):
    """完整的训练主循环"""
    history = {'train_loss': [], 'val_loss': [], 'val_acc': []}

    epoch_pbar = tqdm(range(num_epochs), desc="Epochs")
    for epoch in epoch_pbar:
        # ---- 训练 ----
        train_loss = train_one_epoch(model, train_loader, criterion, optimizer, device)

        # ---- 验证 ----
        val_loss, val_acc = evaluate(model, val_loader, criterion, device)

        # ---- 记录 ----
        history['train_loss'].append(train_loss)
        history['val_loss'].append(val_loss)
        history['val_acc'].append(val_acc)

        # ---- 调度器 ----
        scheduler.step()

        # ---- 显示 ----
        current_lr = optimizer.param_groups[0]['lr']
        epoch_pbar.set_postfix({
            'train_loss': f'{train_loss:.4f}',
            'val_loss': f'{val_loss:.4f}',
            'val_acc': f'{val_acc:.2f}%',
            'lr': f'{current_lr:.2e}',
        })

        # ---- 早停 ----
        if early_stopping and early_stopping(val_loss):
            print(f"Early stopping at epoch {epoch+1}")
            break

    return history
```

## 12. 训练中常见问题的排查 Checklist

当训练不按预期进行时，不要盲目调参。遵循系统的排查流程。

### 12.1 Loss 不下降

- [ ] 学习率是否太小？（尝试增大 10 倍）
- [ ] 学习率是否太大？（尝试减小 10 倍）
- [ ] 优化器是否正确初始化？
- [ ] `optimizer.zero_grad()` 是否在每次迭代前调用？
- [ ] `optimizer.step()` 是否在 `loss.backward()` 之后调用？
- [ ] 数据加载是否正确？（打印一批数据检查 shape 和数值范围）
- [ ] 标签格式是否匹配损失函数要求？

### 12.2 Loss 变为 NaN

NaN 的根本原因是某次运算的数值超出了浮点数的表示范围。排查的优先级链条是：学习率过大（在反向传播中产生巨大梯度）> 数据中有 NaN > 没有梯度裁剪 > 损失函数输入非法值。

- [ ] 学习率是否太大？
- [ ] 是否存在除零操作？
- [ ] 数据中是否有 NaN？检查：`print(torch.isnan(data).any())`
- [ ] 是否使用了梯度裁剪？增加 `clip_grad_norm_`
- [ ] 是否正确初始化了参数？
- [ ] 损失函数是否可能输入了非法值（如 BCELoss 的 log(0)）？

### 12.3 训练集 loss 远低于验证集 loss（过拟合）

- [ ] 增加数据增强
- [ ] 增加 Dropout 率
- [ ] 增大 weight_decay
- [ ] 使用早停
- [ ] 减少模型参数量
- [ ] 收集更多训练数据

### 12.4 训练集 loss 和验证集 loss 都很高（欠拟合）

这意味着模型还没有从数据中学到任何有用的模式——要么是模型太简单（无法表达数据中的复杂模式），要么是学习率太小（参数几乎没动），要么是数据本身有问题。

- [ ] 学习率是否太小？
- [ ] 模型容量是否不够？（增加层数或隐藏单元数）
- [ ] 数据集是否过于复杂？

### 12.5 训练不稳定（loss 剧烈震荡）

根源在于梯度的方差太大——每个 batch 算出来的梯度方向差异巨大，导致参数更新像喝醉的人走路。增大 batch_size 能降低梯度的方差（样本多了，估计就更稳定）；换 AdamW 能让学习率自适应调整；Warmup 能让训练初期平滑过渡。

- [ ] batch_size 是否太小？（增大 batch_size）
- [ ] 学习率是否太大？
- [ ] 是否使用了 SGD？（考虑换 AdamW）
- [ ] 是否使用了 Warmup？

### 12.6 GPU 显存不足 (Out of Memory)

- [ ] 减小 batch_size
- [ ] 使用梯度累积
- [ ] 使用混合精度训练（AMP）
- [ ] 减小模型大小（减少层数/隐藏单元）
- [ ] 使用 `torch.cuda.empty_cache()` 清理缓存

### 12.7 训练速度慢

- [ ] `num_workers` 是否设置足够？（0 -> 4/8）
- [ ] `pin_memory` 是否设为 True？
- [ ] 是否在 `no_grad()` 下做验证？
- [ ] 是否使用了不必要的 `.item()` 调用？
- [ ] batch_size 是否填满了 GPU？

## 13. 代码实战：MNIST 分类完整训练流程

我们将本章所有组件串联成一条完整的 MNIST 训练流水线。这个脚本虽短，但包含了训练循环、验证、学习率调度和 tqdm 进度条这四个核心要素：

```python
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader
from torchvision import datasets, transforms
from tqdm import tqdm

# ---- 1. 数据准备 ----
transform = transforms.Compose([
    transforms.ToTensor(),
    transforms.Normalize((0.1307,), (0.3081,))  # MNIST 的统计量
])

train_dataset = datasets.MNIST('./data', train=True, download=True, transform=transform)
test_dataset  = datasets.MNIST('./data', train=False, transform=transform)

train_loader = DataLoader(train_dataset, batch_size=128, shuffle=True, pin_memory=True)
test_loader  = DataLoader(test_dataset, batch_size=128, shuffle=False, pin_memory=True)

# ---- 2. 模型定义 ----
class MNISTCNN(nn.Module):
    def __init__(self):
        super().__init__()
        self.conv1 = nn.Conv2d(1, 32, 3, padding=1)
        self.conv2 = nn.Conv2d(32, 64, 3, padding=1)
        self.pool = nn.MaxPool2d(2, 2)
        self.fc1 = nn.Linear(64 * 7 * 7, 128)
        self.fc2 = nn.Linear(128, 10)
        self.dropout = nn.Dropout(0.25)
        self.relu = nn.ReLU()

    def forward(self, x):
        x = self.pool(self.relu(self.conv1(x)))
        x = self.pool(self.relu(self.conv2(x)))
        x = x.view(x.size(0), -1)
        x = self.dropout(self.relu(self.fc1(x)))
        x = self.fc2(x)
        return x

# ---- 3. 训练配置 ----
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = MNISTCNN().to(device)
criterion = nn.CrossEntropyLoss()
optimizer = optim.AdamW(model.parameters(), lr=0.001, weight_decay=0.01)
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=10)

# ---- 4. 训练 ----
num_epochs = 10
for epoch in range(num_epochs):
    # 训练
    model.train()
    train_loss = 0.0
    pbar = tqdm(train_loader, desc=f"Epoch {epoch+1}/{num_epochs}")
    for data, target in pbar:
        data, target = data.to(device), target.to(device)
        optimizer.zero_grad()
        output = model(data)
        loss = criterion(output, target)
        loss.backward()
        optimizer.step()
        train_loss += loss.item()
        pbar.set_postfix({'loss': f'{loss.item():.4f}'})

    # 验证
    model.eval()
    correct = 0
    total = 0
    with torch.no_grad():
        for data, target in test_loader:
            data, target = data.to(device), target.to(device)
            output = model(data)
            _, predicted = output.max(1)
            total += target.size(0)
            correct += predicted.eq(target).sum().item()

    scheduler.step()
    avg_train_loss = train_loss / len(train_loader)
    acc = 100.0 * correct / total
    print(f"Epoch {epoch+1}: Train Loss={avg_train_loss:.4f}, Test Acc={acc:.2f}%")
```

## 14. 自定义 Trainer 类（封装训练逻辑）

将训练循环封装为 `Trainer` 类，不仅仅是减少重复代码（Don't Repeat Yourself），更是将"训练"抽象为一个可配置、可扩展的模块。当你需要尝试不同的模型、数据集或训练策略时，你只改配置，不改训练逻辑。

```python
class Trainer:
    """通用训练器"""

    def __init__(self, model, train_loader, val_loader, criterion,
                 optimizer, scheduler=None, device="cpu",
                 grad_clip=None, early_stopping=None,
                 checkpoint_dir="./checkpoints",
                 use_amp=False):
        self.model = model
        self.train_loader = train_loader
        self.val_loader = val_loader
        self.criterion = criterion
        self.optimizer = optimizer
        self.scheduler = scheduler
        self.device = device
        self.grad_clip = grad_clip
        self.early_stopping = early_stopping
        self.checkpoint_dir = checkpoint_dir
        self.use_amp = use_amp and (device.type == "cuda")

        if self.use_amp:
            self.scaler = torch.cuda.amp.GradScaler()

        self.history = {"train_loss": [], "val_loss": [], "val_acc": []}
        self.best_val_acc = 0.0

    def train_epoch(self):
        self.model.train()
        total_loss = 0.0

        pbar = tqdm(self.train_loader, desc="Training", leave=False)
        for batch_x, batch_y in pbar:
            batch_x, batch_y = batch_x.to(self.device), batch_y.to(self.device)
            self.optimizer.zero_grad()

            if self.use_amp:
                with torch.cuda.amp.autocast():
                    outputs = self.model(batch_x)
                    loss = self.criterion(outputs, batch_y)
                self.scaler.scale(loss).backward()
                if self.grad_clip:
                    self.scaler.unscale_(self.optimizer)
                    torch.nn.utils.clip_grad_norm_(self.model.parameters(), self.grad_clip)
                self.scaler.step(self.optimizer)
                self.scaler.update()
            else:
                outputs = self.model(batch_x)
                loss = self.criterion(outputs, batch_y)
                loss.backward()
                if self.grad_clip:
                    torch.nn.utils.clip_grad_norm_(self.model.parameters(), self.grad_clip)
                self.optimizer.step()

            total_loss += loss.item()
            pbar.set_postfix({"loss": f"{loss.item():.4f}"})

        return total_loss / len(self.train_loader)

    @torch.no_grad()
    def validate(self):
        self.model.eval()
        total_loss = 0.0
        correct = 0
        total = 0

        for batch_x, batch_y in self.val_loader:
            batch_x, batch_y = batch_x.to(self.device), batch_y.to(self.device)
            outputs = self.model(batch_x)
            loss = self.criterion(outputs, batch_y)

            total_loss += loss.item()
            _, preds = outputs.max(1)
            total += batch_y.size(0)
            correct += preds.eq(batch_y).sum().item()

        avg_loss = total_loss / len(self.val_loader)
        acc = 100.0 * correct / total
        return avg_loss, acc

    def fit(self, num_epochs):
        for epoch in range(num_epochs):
            train_loss = self.train_epoch()
            val_loss, val_acc = self.validate()

            self.history["train_loss"].append(train_loss)
            self.history["val_loss"].append(val_loss)
            self.history["val_acc"].append(val_acc)

            if self.scheduler:
                if isinstance(self.scheduler, torch.optim.lr_scheduler.ReduceLROnPlateau):
                    self.scheduler.step(val_loss)
                else:
                    self.scheduler.step()

            current_lr = self.optimizer.param_groups[0]['lr']
            print(f"Epoch {epoch+1:3d} | "
                  f"Train Loss: {train_loss:.4f} | "
                  f"Val Loss: {val_loss:.4f} | "
                  f"Val Acc: {val_acc:.2f}% | "
                  f"LR: {current_lr:.2e}")

            # 保存最佳模型
            if val_acc > self.best_val_acc:
                self.best_val_acc = val_acc
                torch.save(self.model.state_dict(),
                          f"{self.checkpoint_dir}/best_model.pth")

            # 早停
            if self.early_stopping and self.early_stopping(val_loss):
                print(f"Early stopping at epoch {epoch+1}")
                break

        return self.history


# 使用 Trainer
trainer = Trainer(
    model=model,
    train_loader=train_loader,
    val_loader=val_loader,
    criterion=nn.CrossEntropyLoss(),
    optimizer=optim.AdamW(model.parameters(), lr=1e-3),
    scheduler=optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=10),
    device=device,
    grad_clip=1.0,
    use_amp=torch.cuda.is_available(),
    checkpoint_dir="./checkpoints",
)

history = trainer.fit(num_epochs=20)
print(f"Best val accuracy: {trainer.best_val_acc:.2f}%")
```

## 15. 分布式训练的简要预告

当单个 GPU 的显存或算力不足以支持训练时（尤其是在大模型时代这几乎是常态），需要用到分布式训练。DataParallel 是最简单但效率最低的入门方案——它用多线程把 batch 拆分到多张卡上计算，但主 GPU 承担了过多的 scatter/gather 工作，其他卡经常空闲等待。更专业的选择是 DistributedDataParallel (DDP)，每个 GPU 运行一个独立进程，通过 NCCL 后端高效通信。第 6 章将深入讲解 DDP 的完整配置和使用。

```python
# 单机多卡训练的最简方式 —— DataParallel（适合快速实验）
model = MyModel()
if torch.cuda.device_count() > 1:
    model = nn.DataParallel(model)
model = model.to(device)
```

## 常见错误

### 错误 1: 忘记 `optimizer.zero_grad()`

这是新手最常犯的错误。`backward()` 累加梯度到 `.grad` 属性而不覆盖。如果每个 batch 不清零，梯度会叠加——你可能在处理第 10 个 batch 时，`.grad` 里装的是前 9 个 batch 的梯度累积。模型参数会朝一个随机、不可预测的方向更新。

```python
# 错误
for data, target in loader:
    output = model(data)
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()   # 梯度不断累积，相当于用了错误梯度方向！
    # 缺少 optimizer.zero_grad()

# 正确
for data, target in loader:
    optimizer.zero_grad()   # 每次先清零
    output = model(data)
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()
```

### 错误 2: 验证时忘记 `model.eval()` 和 `no_grad()`

这个错误的隐蔽之处在于代码不会报错——模型照样输出结果，但 Dropout 仍在随机丢弃神经元，BatchNorm 在使用验证 batch 自己的统计量而非全局统计量。你得到的准确率偏低且每次运行结果不同。

```python
# 错误
def validate(model, loader):
    model.train()  # 错误！Dropout 仍然激活
    for data, target in loader:
        output = model(data)  # 构建了计算图，浪费显存

# 正确
@torch.no_grad()
def validate(model, loader):
    model.eval()
    for data, target in loader:
        output = model(data)
```

### 错误 3: `loss.backward()` 后没有 `optimizer.step()`

这是一个"代码运行不报错但模型永远学不会"的错误。梯度算出来了，但参数从未更新。

```python
# 正确
loss.backward()
optimizer.step()
```

### 错误 4: `scheduler.step()` 的位置错误

调度器要求每个 epoch 更新一次而非每个 batch。在每个 batch 后都调用 `scheduler.step()` 会导致学习率在一个 epoch 内急剧衰减，训练完全失控。

```python
# 正确
for epoch in range(num_epochs):
    for batch in loader:
        ...
        optimizer.step()
    scheduler.step()  # 每个 epoch 调用一次
```

### 错误 5: 在 `no_grad()` 中调用 `loss.backward()`

`no_grad()` 禁用了计算图构建，而 `backward()` 需要计算图来做反向传播——两者直接冲突。

```python
# 错误
with torch.no_grad():
    output = model(data)
    loss = criterion(output, target)
    loss.backward()  # RuntimeError: 计算图已被禁用！

# 正确
output = model(data)          # 在 no_grad 外
loss = criterion(output, target)
loss.backward()
```

### 错误 6: 在验证循环中忘记 `with torch.no_grad()` 导致显存溢出

验证循环中如果不加 `no_grad()`，每个 batch 的计算图都会被保留，显存持续增长。在小数据集上可能察觉不到，但在大数据集或长时间验证时，显存会耗尽。

```python
# 正确
model.eval()
with torch.no_grad():           # 必须加
    for data, target in val_loader:
        output = model(data)    # 不构建计算图
        loss = criterion(output, target)
```

## 本章小结

训练循环看似简单——六行代码一个 for 循环——但每一步背后都有精确的设计意图。理解这些意图，你才能在训练出问题时知道该检查哪里，而不是盲目调参。

**六步流水线**：`model.train()` -> `zero_grad()` -> `forward` -> `loss` -> `backward()` -> `step()`。这六步的顺序不容错乱。`zero_grad()` 必须在 `forward` 之前（否则上次的梯度还没清），`backward()` 必须在 `step()` 之前（否则无需可用的梯度），`model.train()` 必须在整个循环之外但在 forward 之前。

**模式切换**：`model.train()` 和 `model.eval()` 本质上控制的是 Dropout 和 BatchNorm 的两种行为模式——训练需要随机性和批统计，推理需要确定性和全局统计。忘记切换是最隐蔽的 bug。

**优化器的演进**：SGD（简单但需要精调）-> SGD + Momentum（加速和稳定）-> Adam（自适应学习率，上手容易）-> AdamW（修正 weight_decay 的实现，泛化更好）。这不是"新的替代旧的"的简单升级，每种优化器都有其最佳适用场景。SGD + Momentum 在 CV 领域仍然地位稳固，AdamW 在 NLP/Transformer 领域是标准配置。

**学习率调度**：训练的早中晚期对步长需求不同。早期大步探索，晚期小步精调。CosineAnnealingLR 以平滑的余弦曲线优于传统的阶梯式衰减。Warmup 是训练起步的缓冲器，在 Transformer 训练中不可或缺。

**过拟合的本质**是 bias-variance tradeoff。应对策略不是越多越好，而是按优先级逐层防御：数据增强（给模型看更多变体）> Dropout（强制冗余学习）> Weight Decay（惩罚大权重）> 早停（在 overfit 之前收手）。数据永远是最好的正则化器。

**验证的规则**永远记牢：`model.eval()` + `torch.no_grad()`。两者缺一不可。

**混合精度训练**只用几行代码就能换来 40% 显存节省和 50%+ 速度提升。`GradScaler` 解决了 FP16 梯度下溢的问题，让混合精度训练变得可靠。

**实用工具**：TensorBoard 让你能看到训练的全貌（loss 曲线、梯度分布、权重直方图），而不仅仅是几个滚动数字。tqdm 让终端里的训练进度一目了然。尽早将这些工具集成到你的训练流程中。

当训练出问题时，不要跳进"试试这个学习率能不能 work"的随机尝试。用 Checklist 系统排查：loss 不降（检查学习率和梯度清零）、NaN（检查学习率太大/数据异常/没有梯度裁剪）、过拟合（按优先级加正则化）、欠拟合（增大模型或减小正则化）。
