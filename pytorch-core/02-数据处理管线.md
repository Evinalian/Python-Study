## 1. 为什么需要 DataLoader

在进入具体的 API 之前，我们先站在一个更高的视角来理解：DataLoader 到底解决了什么问题？

想象你正在写一个训练循环。在没有 DataLoader 的情况下，你需要手工完成以下所有事情：

- **批量组装**：将 N 个独立样本的 Tensor 堆叠为 shape `(N, ...)` 的批次。这不仅仅是调用 `torch.stack`——你需要确保所有样本具有相同的 dtype 和 shape，否则堆叠会失败。
- **随机打乱**：每个 epoch 需要重新 shuffle 样本顺序。如果不打乱，模型会学到样本的顺序模式而非内容本身，导致泛化能力差。而手写一个 shuffle 算法需要考虑内存效率（不能把整个数据集拷贝一份来 shuffle）。
- **并行加载**：如果你从磁盘读取大图片（如 ImageNet 的 224x224 图像），I/O 可能成为整个训练管线的瓶颈。单进程加载意味着 GPU 在大部分时间都在等 CPU 搬运数据，利用率极低。
- **内存管理**：可以使用"预取"（prefetching）技术——在当前 batch 在 GPU 上运算时，CPU 后台加载下一个 batch。这需要处理多线程同步和队列管理。

PyTorch 的 `DataLoader` 将这些问题统一封装在一套 API 中。其核心设计是一个两层抽象：

```
Dataset        — 定义"如何读取第 i 个样本"
DataLoader     — 定义"如何将样本组装成批次并高效供给训练循环"
```

这种**关注点分离**（Separation of Concerns）的设计是 DataLoader 最精妙的地方。Dataset 只关心数据本身——文件从哪里来、如何解析、如何预处理单个样本。DataLoader 只关心批次组装的编排——取多少样本、用什么顺序、用几个进程、组装成什么格式。你可以独立修改其中的任意部分而不影响另一端。比如你可以把 Dataset 从 CSV 文件读取改为数据库读取，而训练循环和 DataLoader 配置不需要任何改动。

### 1.1 DataLoader 的内部工作流程

理解 DataLoader 内部的步骤顺序有助于排查性能瓶颈——当训练速度不理想时，你可以定位到是哪个环节出了问题：是 Dataset 的 `__getitem__` 太慢（I/O 瓶颈）？还是 collate 太慢（CPU 计算瓶颈）？还是 batch_size 太小（GPU 利用率低）？

```
1. DataLoader 根据 Sampler 生成一批索引 (indices)
2. 对这批索引，多进程并行调用 Dataset.__getitem__(idx)
3. 将返回的样本列表传给 collate_fn 组装成一个批次
4. 如果 pin_memory=True，将批次数据拷贝到锁页内存
5. 将批次 yield 给训练循环
```

每一步都可能成为瓶颈：第 2 步受磁盘 I/O 和解码速度影响（用 `num_workers` 并行化来缓解），第 4 步受 CPU-GPU 传输带宽影响（用 `pin_memory` 来加速）。后续我们会逐一展开这些参数。

## 2. Dataset 基类

Dataset 是整个数据管线的起点。PyTorch 设计了一个极简的抽象接口——只需实现两个方法即可接入整个生态：

- `__len__(self)`：返回数据集总样本数
- `__getitem__(self, idx)`：返回第 idx 个样本

这个设计的精妙之处在于，它把**随机访问的复杂度**完全封装在了 Dataset 内部。DataLoader 不需要知道数据在磁盘上怎么存的、是什么格式、是否需要解压——它只需要通过这两个方法来获取样本。这使得 Dataset 可以表示任何形式的数据：磁盘上的图像文件、内存中的 Tensor、远程数据库中的记录、甚至实时产生的数据流（通过 IterableDataset）。

### 2.1 抽象接口

```python
from torch.utils.data import Dataset

class MyDataset(Dataset):
    def __init__(self, ...):
        # 在此处加载或记录数据路径
        pass

    def __len__(self):
        # 返回数据集总样本数
        return 0

    def __getitem__(self, idx):
        # 返回第 idx 个样本，通常是一个 (data, label) 元组
        return None
```

`DataLoader` 通过 `__len__` 知道有多少样本（用于计算 epoch 结束、进度条百分比、以及 Sampler 的总采样数），通过 `__getitem__(idx)` 按索引取样本。这两个方法看起来简单，但有一个重要的性能含义：`__getitem__` 在每次取样本时被调用，因此如果预处理很重（如读大图片、做复杂变换），应该尽量在 `__init__` 中预加载或使用内存缓存，避免 I/O 成为瓶颈。

### 2.2 TensorDataset：最简单的情况

当数据已经是 Tensor 时，不需要自定义 Dataset。`TensorDataset` 是 PyTorch 提供的最简单封装——它将传入的所有 Tensor 沿第 0 维对齐，`__getitem__(i)` 返回每个 Tensor 的第 i 个切片组成的元组。

这是一个在实际项目中使用频率极高的 Dataset 子类。任何已经从文件预处理为 Tensor 格式的数据（如量化后的特征、预计算的词向量）都可以直接丢给 `TensorDataset`。

```python
from torch.utils.data import TensorDataset, DataLoader

# 假设有 1000 个样本，每个 10 维特征，标签是整数
X = torch.randn(1000, 10)
y = torch.randint(0, 3, (1000,))

dataset = TensorDataset(X, y)
dataloader = DataLoader(dataset, batch_size=32, shuffle=True)

# 使用
for batch_X, batch_y in dataloader:
    print(batch_X.shape)  # torch.Size([32, 10])
    print(batch_y.shape)  # torch.Size([32])
    break
```

输出确认了 DataLoader 正确地将 1000 个样本按 batch_size=32 分批，每个 batch 的 shape 为 `(32, 10)` 和 `(32,)`。

**TensorDataset 支持多个输入**：

```python
# 多个特征 Tensor + 标签
X1 = torch.randn(1000, 10)   # 特征集 1
X2 = torch.randn(1000, 5)    # 特征集 2
y = torch.randint(0, 3, (1000,))

dataset = TensorDataset(X1, X2, y)
loader = DataLoader(dataset, batch_size=32, shuffle=True)

for batch_x1, batch_x2, batch_y in loader:
    print(batch_x1.shape, batch_x2.shape, batch_y.shape)
    # torch.Size([32, 10]) torch.Size([32, 5]) torch.Size([32])
    break
```

多输入场景在实践中很常见：比如一个推荐系统可能有用户特征、物品特征和上下文特征三组输入，用 `TensorDataset` 可以便捷地将它们对齐包装。

### 2.3 自定义 Dataset：从 CSV 文件加载

当数据以文件形式存储（CSV、JSON、图像文件等），你需要自定义 Dataset。下面是一个从 CSV 文件读取表格数据的标准实现。

这个 Dataset 的关键设计决策是：**在 `__init__` 中完成 I/O 重操作（读取 CSV 文件），在 `__getitem__` 中只做轻量的 Tensor 转换**。如果把 `pd.read_csv` 放在 `__getitem__` 中，每个样本的获取都会触发完整的文件读取，这将是灾难性的性能。

```python
import pandas as pd
from torch.utils.data import Dataset

class CSVDataset(Dataset):
    """从 CSV 文件读取特征和标签的 Dataset"""

    def __init__(self, csv_path, label_col="label"):
        """
        csv_path: CSV 文件路径
        label_col: 标签列的名称
        """
        df = pd.read_csv(csv_path)                 # 读取整个 CSV
        self.labels = df[label_col].values         # 提取标签列
        self.features = df.drop(columns=[label_col]).values.astype(np.float32)

    def __len__(self):
        return len(self.labels)                    # 返回样本数

    def __getitem__(self, idx):
        # 返回 (特征 Tensor, 标签 Tensor)
        x = torch.tensor(self.features[idx])       # shape (n_features,)
        y = torch.tensor(self.labels[idx])         # 标量
        return x, y
```

> 注意：`__getitem__` 在每次取样本时被调用，因此如果预处理很重（如读图片、做变换），应尽量在 `__init__` 中预加载或使用缓存。

**进阶版 CSVDataset —— 支持预处理**：

真实的表格数据往往需要标准化。以下进阶版在 `__init__` 中计算了训练集的均值和标准差，并在 `__getitem__` 返回前自动标准化。同时提供了 `inverse_transform` 方法用于将标准化后的预测值"反标准化"回原始尺度——这在回归任务中非常实用。

```python
class CSVDatasetV2(Dataset):
    """支持特征预处理的 CSV Dataset"""

    def __init__(self, csv_path, label_col="label",
                 feature_cols=None, normalize=True):
        df = pd.read_csv(csv_path)

        # 指定特征列（如果为 None，只用数值列）
        if feature_cols is None:
            feature_cols = df.select_dtypes(include=[np.number]).columns.tolist()
            feature_cols = [c for c in feature_cols if c != label_col]

        self.labels = df[label_col].values.astype(np.int64)
        features = df[feature_cols].values.astype(np.float32)

        # 可选：标准化
        if normalize:
            self.mean = features.mean(axis=0, keepdims=True)
            self.std = features.std(axis=0, keepdims=True) + 1e-8
            features = (features - self.mean) / self.std
        else:
            self.mean = None
            self.std = None

        self.features = features
        self.feature_cols = feature_cols

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        x = torch.tensor(self.features[idx], dtype=torch.float32)
        y = torch.tensor(self.labels[idx], dtype=torch.long)
        return x, y

    def inverse_transform(self, x_normalized):
        """将标准化后的特征反标准化"""
        if self.mean is None:
            return x_normalized
        return x_normalized * self.std + self.mean
```

`1e-8` 这个微小的常数加在标准差上是为了防止除以零——当某个特征的所有值都相同（方差为 0）时，标准化变成除以零的错误。这是数据处理中的经典防御性编程技巧。

### 2.4 从文件目录加载图像的自定义 Dataset

图像数据是深度学习中最常见的数据形式之一。PyTorch 的 `torchvision.datasets.ImageFolder` 已经实现了按目录结构加载图像的功能，但理解其内部原理有助于你在遇到非标准目录结构时自定义 Dataset。

实现图像 Dataset 的核心挑战是：**图像解码（从 JPEG/PNG 到像素数组）是一个 I/O 重操作，必须在 `__getitem__` 中完成**（因为把所有图像预加载到内存通常不现实）。这导致 `__getitem__` 成为性能瓶颈——解决方法是用多进程的 `DataLoader`（`num_workers > 0`）来并行化图像解码。

```python
import os
from PIL import Image
from torch.utils.data import Dataset

class ImageFolderDataset(Dataset):
    """从目录结构加载图像：root/class_A/xxx.jpg, root/class_B/xxx.jpg"""

    def __init__(self, root_dir, transform=None):
        """
        root_dir: 根目录，其子目录名为类别名
        transform: 图像预处理变换（torchvision.transforms.Compose）
        """
        self.samples = []          # 存储 (path, label) 对
        self.classes = sorted(os.listdir(root_dir))  # 类别名列表

        for label, class_name in enumerate(self.classes):
            class_dir = os.path.join(root_dir, class_name)
            if not os.path.isdir(class_dir):
                continue
            for fname in os.listdir(class_dir):
                if fname.lower().endswith(('.png', '.jpg', '.jpeg')):
                    self.samples.append((os.path.join(class_dir, fname), label))

        self.transform = transform

    def __len__(self):
        return len(self.samples)

    def __getitem__(self, idx):
        img_path, label = self.samples[idx]
        image = Image.open(img_path).convert("RGB")  # 用 PIL 读取

        if self.transform:
            image = self.transform(image)             # 应用变换

        return image, label
```

`Image.open(img_path).convert("RGB")` 中的 `.convert("RGB")` 是一个容易被忽略但很重要的步骤。许多图像可能是灰度图（单通道）、RGBA（四通道）等格式，`.convert("RGB")` 统一将它们转为三通道 RGB，确保输入到模型的 Tensor 的通道数始终为 3。

**进阶版 —— 支持缓存和错误处理**：

在生产环境中，图像数据集中经常混杂着损坏的文件、不支持的格式、甚至 0 字节的文件。以下实现展示了如何优雅地处理这些异常情况，避免训练过程中因为某个坏文件而崩溃。

```python
class RobustImageDataset(Dataset):
    """更健壮的图像 Dataset：支持缓存、路径列表、错误处理"""

    def __init__(self, image_paths, labels, transform=None,
                 cache_images=False):
        """
        image_paths: 图像路径列表
        labels: 标签列表（与 image_paths 一一对应）
        """
        self.image_paths = image_paths
        self.labels = labels
        self.transform = transform
        self.cache_images = cache_images

        # 可选：预加载所有图像到内存
        if cache_images:
            self.cached = []
            for path in image_paths:
                try:
                    img = Image.open(path).convert("RGB")
                    self.cached.append(img)
                except Exception as e:
                    print(f"警告：无法加载 {path}: {e}")
                    self.cached.append(None)
        else:
            self.cached = None

    def __len__(self):
        return len(self.image_paths)

    def __getitem__(self, idx):
        if self.cached is not None:
            image = self.cached[idx]
            if image is None:
                # 损坏的图像，返回全黑图像
                image = Image.new("RGB", (224, 224), (0, 0, 0))
        else:
            try:
                image = Image.open(self.image_paths[idx]).convert("RGB")
            except Exception:
                image = Image.new("RGB", (224, 224), (0, 0, 0))

        label = self.labels[idx]

        if self.transform:
            image = self.transform(image)

        return image, label
```

缓存策略的选择取决于数据集大小和可用内存：对于小数据集（几千张图片），`cache_images=True` 可以将所有图像预加载到内存，消除 I/O 瓶颈；对于 ImageNet 级别的数据集，缓存是不现实的，必须依赖多进程并行加载。

### 2.5 文本数据 Dataset

文本数据与图像和表格数据有本质差别：**文本是变长的**。不同的句子有不同的单词数量，而神经网络（特别是嵌入层之后的线性层）要求固定长度的输入。这引入了两个新需求：词表映射（word -> token ID）和序列填充/截断（padding/truncation）。

下面是一个文本分类 Dataset 的实现，包含了词表映射和截断/填充的逻辑。

```python
from torch.utils.data import Dataset
import torch

class TextClassificationDataset(Dataset):
    """文本分类 Dataset：支持词表映射和截断/padding"""

    def __init__(self, texts, labels, vocab, max_length=128):
        """
        texts: 文本列表
        labels: 标签列表
        vocab: dict，word -> id
        max_length: 最大序列长度
        """
        self.texts = texts
        self.labels = labels
        self.vocab = vocab
        self.max_length = max_length
        self.unk_id = vocab.get("<UNK>", 0)
        self.pad_id = vocab.get("<PAD>", 0)

    def __len__(self):
        return len(self.texts)

    def __getitem__(self, idx):
        text = self.texts[idx]
        label = self.labels[idx]

        # 文本 -> token IDs
        tokens = []
        for word in text.split():
            tokens.append(self.vocab.get(word, self.unk_id))

        # 截断或填充到 max_length
        if len(tokens) > self.max_length:
            tokens = tokens[:self.max_length]
        else:
            tokens = tokens + [self.pad_id] * (self.max_length - len(tokens))

        return torch.tensor(tokens, dtype=torch.long), torch.tensor(label, dtype=torch.long)


# 使用示例
texts = ["I love deep learning", "PyTorch is amazing", "Machine learning rocks"]
labels = [1, 1, 1]

# 构建简单词表
vocab = {"<PAD>": 0, "<UNK>": 1, "I": 2, "love": 3, "deep": 4,
         "learning": 5, "PyTorch": 6, "is": 7, "amazing": 8,
         "Machine": 9, "rocks": 10}
dataset = TextClassificationDataset(texts, labels, vocab, max_length=10)

for i in range(len(dataset)):
    tokens, label = dataset[i]
    print(f"样本 {i}: tokens={tokens}, label={label}")
```

注意这里的一个简化：padding 是在 Dataset 的 `__getitem__` 中做，这意味着**每个样本都被 pad 到了固定的最大长度**。对于短句子多的数据集，这会浪费大量计算（大量 pad token 参与运算但实际不需要）。更好的做法是将 padding 延迟到 `collate_fn` 中——在组装 batch 时只 pad 到该 batch 的最长长度。我们将在 collate_fn 部分详细对比这两种方案。

### 2.6 多模态 Dataset（图像+文本）

很多现代深度学习应用需要同时处理多种类型的数据——比如图文匹配、视觉问答、视频理解等。多模态 Dataset 需要从不同来源获取不同类型的数据，在同一索引处将它们对齐。

```python
class MultiModalDataset(Dataset):
    """同时包含图像和文本的数据集"""

    def __init__(self, image_paths, texts, labels,
                 image_transform=None, tokenizer_fn=None):
        self.image_paths = image_paths
        self.texts = texts
        self.labels = labels
        self.image_transform = image_transform
        self.tokenizer_fn = tokenizer_fn  # 函数：text -> tensor

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        # 图像
        image = Image.open(self.image_paths[idx]).convert("RGB")
        if self.image_transform:
            image = self.image_transform(image)

        # 文本
        text = self.texts[idx]
        if self.tokenizer_fn:
            text_tokens = self.tokenizer_fn(text)
        else:
            text_tokens = text  # 返回原始文本

        label = torch.tensor(self.labels[idx])
        return (image, text_tokens), label
```

这个 Dataset 返回的每个样本是一个 `((image, text_tokens), label)` 的嵌套结构。在 batch 组装时，你需要自定义 `collate_fn` 来处理这种非标准的样本结构——图像的 batch 可以用 `torch.stack`，文本序列可能需要 padding。多模态的 `collate_fn` 将在后续内容中展示。

### 2.7 常见的 Dataset 子类一览

PyTorch 生态提供了多个 Dataset 子类来覆盖常见场景。了解它们可以避免重复造轮子。

| Dataset 类 | 来源 | 用途 |
|------------|------|------|
| `TensorDataset` | `torch.utils.data` | 纯 Tensor 数据 |
| `ConcatDataset` | `torch.utils.data` | 拼接多个 Dataset |
| `Subset` | `torch.utils.data` | 从 Dataset 中取子集 |
| `ImageFolder` | `torchvision.datasets` | 按文件夹组织的图像分类 |
| `DatasetFolder` | `torchvision.datasets` | 通用的按文件夹组织的文件数据 |
| `IterableDataset` | `torch.utils.data` | 适合流式数据或无法随机访问的数据 |

`ConcatDataset` 和 `Subset` 的组合特别实用——你可以把多个来源的数据集拼在一起，然后用 `Subset` 按索引划分训练/验证集，不需要物理复制文件。

```python
# ConcatDataset 使用示例
from torch.utils.data import ConcatDataset, Subset

dataset1 = TensorDataset(torch.randn(500, 10), torch.randint(0, 3, (500,)))
dataset2 = TensorDataset(torch.randn(300, 10), torch.randint(0, 3, (300,)))
combined = ConcatDataset([dataset1, dataset2])
print(f"ConcatDataset 大小: {len(combined)}")  # 800

# Subset 使用示例：取前 100 个样本
subset = Subset(combined, indices=range(100))
print(f"Subset 大小: {len(subset)}")  # 100
```

### 2.8 IterableDataset（流式数据）

当数据量远超内存容量，或数据是实时产生的（如传感器流、用户行为日志、网络爬虫结果），标准的基于索引的 Dataset 不够用——因为你无法 `__len__` 返回一个有意义的总数，也无法保证能随机访问任意位置的数据。

这就是 `IterableDataset` 的用武之地。它改用 `__iter__` 方法（而非 `__getitem__`）来产生样本。DataLoader 会调用 `__iter__` 来获取一个迭代器，然后从中逐个取出样本并组装批次。

```python
from torch.utils.data import IterableDataset, DataLoader

class RandomDataStream(IterableDataset):
    """生成无限的随机数据流"""

    def __init__(self, num_features=10):
        self.num_features = num_features

    def __iter__(self):
        # 不断 yield 样本
        while True:
            x = torch.randn(self.num_features)
            y = torch.randint(0, 5, (1,)).item()
            yield x, y

# IterableDataset 不支持 shuffle（除非用特殊的 buffer shuffle）
stream = RandomDataStream(num_features=10)
loader = DataLoader(stream, batch_size=32)

for batch_x, batch_y in loader:
    print(batch_x.shape, batch_y.shape)
    break  # 只取第一批，否则无限循环
```

> `IterableDataset` 的多 worker 模式需要注意数据分片：每个 worker 会独立调用 `__iter__`，可能导致数据重复。需要通过 worker 信息 (`torch.utils.data.get_worker_info()`) 做分片处理。

多 worker 分片的标准写法是在 `__iter__` 中根据 `worker_info` 把数据集切分成多个不重叠的片段，让每个 worker 只遍历自己负责的那一段。

## 3. DataLoader 核心参数

DataLoader 的 API 只有一行，但每个参数的选择都会影响训练效率和模型行为。理解每个参数背后的含义比记忆默认值更重要。

```python
from torch.utils.data import DataLoader

loader = DataLoader(
    dataset,                # Dataset 实例
    batch_size=64,          # 每批样本数
    shuffle=True,           # 每个 epoch 是否随机打乱
    num_workers=4,          # 加载数据的子进程数（0 = 主进程加载）
    pin_memory=True,        # 将数据锁页到 CPU 内存，加速 CPU -> GPU 传输
    drop_last=True,         # 丢弃最后不足 batch_size 的批次
    collate_fn=None,        # 自定义批次组装函数
)
```

下面详细解释每个参数。

### 3.1 `batch_size`

batch_size 是影响训练效率和收敛质量的最关键超参数之一。它控制每次更新参数时使用多少个样本的梯度平均值。

- **太小**（如 1-4）：GPU 利用率低，因为 GPU 的大规模并行能力没有被充分利用。同时梯度噪声大，可能导致训练不稳定。但小 batch 有时能帮助模型跳出局部最优（因为梯度方向更随机）。
- **太大**（如 512-1024）：可能超出 GPU 显存（Out of Memory）。同时过大的 batch 可能导致泛化性能下降（被称为"泛化差距"现象）。实践中，当 batch_size 增大时，通常需要相应调整学习率（线性缩放法则：batch_size 翻倍，学习率也翻倍）。
- **实践经验**：CV 任务 32-256，NLP 任务 16-128，取决于显存。建议从 32 开始，逐步增大直到显存接近满载。

**batch_size 对训练的影响 —— 对比实验**：

以下代码通过控制变量实验展示了不同 batch_size 对训练速度的影响。注意在 GPU 上，你看到的加速源于更高的并行度——GPU 的 SM（流多处理器）在处理大矩阵时利用率更高。

```python
import time
import torch
import torch.nn as nn
from torch.utils.data import TensorDataset, DataLoader

# 模拟数据
X = torch.randn(10000, 100)
y = torch.randint(0, 10, (10000,))
dataset = TensorDataset(X, y)

model = nn.Linear(100, 10).cuda()
criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)

for batch_size in [1, 16, 64, 256]:
    loader = DataLoader(dataset, batch_size=batch_size, shuffle=True)

    model.zero_grad()
    start = time.time()
    for batch_x, batch_y in loader:
        batch_x, batch_y = batch_x.cuda(), batch_y.cuda()
        output = model(batch_x)
        loss = criterion(output, batch_y)
        loss.backward()
        optimizer.step()
        model.zero_grad()

    elapsed = time.time() - start
    print(f"batch_size={batch_size:3d}: {elapsed:.3f}s")
```

典型输出（GPU）：
```
batch_size=  1: 2.345s  (GPU 利用率极低)
batch_size= 16: 0.421s
batch_size= 64: 0.187s  (GPU 利用率高)
batch_size=256: 0.156s  (继续提升但边际效益递减)
```

从结果可以看出，batch_size 从 1 增加到 64 带来的加速是超线性的（GPU 终于"吃饱"了），但从 64 到 256 的改进是边际递减的（GPU 已经接近满载）。在实践中，通常选择 batch_size 使得 GPU 显存利用率在 80-90% 左右。

### 3.2 `shuffle`

shuffle 参数控制每个 epoch 是否重新随机排列样本顺序。

- `True`：每个 epoch 开始时重新随机排列样本顺序，**训练时必须设为 True**。如果不打乱，模型会在每个 epoch 以完全相同的顺序看到样本，可能导致模型记住样本顺序而非学习内容本身。
- `False`：验证集/测试集保持原始顺序即可——评估只需要前向传播，不涉及梯度更新，因此样本顺序不影响结果。

**为什么不 shuffle 会导致问题**：

深层原因在于 SGD 的收敛假设：随机梯度下降的收敛性依赖于"每个 batch 是数据分布的一个无偏估计"。如果数据按类别排序，那么每个 batch 看到的都是单一类别的样本，梯度方向会严重偏离真实梯度方向。

```
# 假设数据按类别排序：[A类样本, A类样本, ..., B类样本, B类样本, ...]
# 不 shuffle 时，每个 batch 只包含一个类别的样本：
#   batch 1: 全是 A 类
#   batch 2: 全是 A 类
#   ...
#   batch N: 全是 B 类
# 梯度在每个 batch 只学到单一类别的模式，导致训练不稳定
```

### 3.3 `num_workers`

`num_workers` 控制用于数据加载的并行子进程数量。这是解决 I/O 瓶颈的主要手段。

```python
# num_workers=0: 主进程加载（慢，但易调试）
# num_workers=4: 4 个子进程并行加载（快，推荐）

# 常见设置
import os
num_workers = min(os.cpu_count(), 8)  # 一般不超过 CPU 核心数
```

加载图像/大文件时，`num_workers` 能显著加速；纯内存数据（TensorDataset）设置 `num_workers=0` 即可——多进程的启动和通信开销反而会拖慢速度。

**num_workers 对加载速度的影响 —— 对比实验**：

```python
import time
from torch.utils.data import Dataset, DataLoader

# 模拟 I/O 密集型 Dataset
class SlowDataset(Dataset):
    def __init__(self, n=10000):
        self.n = n
    def __len__(self):
        return self.n
    def __getitem__(self, idx):
        time.sleep(0.001)  # 模拟磁盘读取延迟
        return torch.randn(3, 224, 224), 0

dataset = SlowDataset(500)
for nw in [0, 2, 4, 8]:
    loader = DataLoader(dataset, batch_size=32, num_workers=nw)
    start = time.time()
    for batch_x, batch_y in loader:
        pass
    print(f"num_workers={nw}: {time.time()-start:.2f}s")
```

典型输出：
```
num_workers=0: 5.21s (串行)
num_workers=2: 2.68s
num_workers=4: 1.45s
num_workers=8: 0.89s
```

多 worker 加速的原理是：当 worker 1 在等待磁盘读取图片时，worker 2 的 CPU 空闲，可以处理别的图片——I/O 等待期被 CPU 计算期所掩盖。但 worker 数量并不是越多越好：超过 CPU 核心数后，上下文切换的开销会超过并行带来的收益。

> Windows 下使用 `num_workers > 0` 可能因 multiprocessing 问题卡死。解决方案：将 DataLoader 创建语句放在 `if __name__ == "__main__":` 保护中，或设置 `num_workers=0`。

### 3.4 `pin_memory`

`pin_memory` 涉及一个底层的计算机体系结构概念：锁页内存（pinned memory）vs 可分页内存（pageable memory）。

- **普通内存（pageable memory）**：操作系统可以随时将数据交换到磁盘（swap），GPU 无法直接通过 DMA（直接内存访问）从这种内存拷贝数据。
- **锁页内存（pinned memory）**：操作系统保证数据始终驻留在物理 RAM 中，GPU 可以直接通过 DMA 高速传输，无需 CPU 介入拷贝。

`pin_memory=True` 将 batch 数据拷贝到锁页内存区域，使得后续的 CPU-to-GPU 传输可以异步进行（配合 `non_blocking=True`），加速约 2-3 倍。代价是锁页内存不能被 swap 出去，因此占用更多可用的物理 RAM。

```python
# 典型搭配
loader = DataLoader(dataset, batch_size=64, num_workers=4, pin_memory=True)
for batch_x, batch_y in loader:
    batch_x = batch_x.to("cuda", non_blocking=True)  # non_blocking 配合 pin_memory
    batch_y = batch_y.to("cuda", non_blocking=True)
    # non_blocking=True 让 CPU 在传输数据的同时继续做其他事情
```

`non_blocking=True` 的作用是：启动 GPU 传输后立即返回（不等待传输完成），这样 CPU 可以继续加载下一个 batch 的数据。如果下一个操作需要使用 GPU 上的数据，PyTorch 会自动同步（隐式等待传输完成），所以你通常不需要手动同步。

### 3.5 `drop_last`

- `True`：丢弃最后一批不足 `batch_size` 的样本
- `False`：保留最后一批（可能小于 `batch_size`）

如果模型中有 `BatchNorm`（依赖批次统计量计算均值和方差），强烈建议 `drop_last=True`。原因在于：BatchNorm 的效果严重依赖批次统计量的稳定性。如果最后一批只有 2 个样本（而正常 batch 有 64 个），这个 mini-batch 统计量会极不稳定，可能引入噪声甚至导致梯度异常。

```python
# drop_last 对 epoch 批次数量的影响
dataset = TensorDataset(torch.randn(100, 10), torch.randint(0, 5, (100,)))

loader_drop = DataLoader(dataset, batch_size=32, drop_last=True)
loader_keep = DataLoader(dataset, batch_size=32, drop_last=False)

print(f"drop_last=True: {len(loader_drop)} batches")   # 3 (32+32+32=96, 丢弃4)
print(f"drop_last=False: {len(loader_keep)} batches")  # 4 (32+32+32+4)
```

### 3.6 `prefetch_factor` 和 `persistent_workers`

这两个参数用于进一步优化加载效率，特别是在大规模训练时。

```python
loader = DataLoader(
    dataset,
    batch_size=64,
    num_workers=4,
    prefetch_factor=4,       # 每个 worker 预取 4 批数据
    persistent_workers=True,  # worker 进程在 epoch 之间保持存活
)
```

- `prefetch_factor`：每个 worker 提前加载的批次数，默认 2。增大可减少 GPU 等待时间——当 GPU 在处理当前 batch 时，worker 已经开始加载后面的多个 batch。但增大也意味着更多内存被预取数据占用。
- `persistent_workers=True`：worker 进程在 epoch 之间不会被销毁重建。每个 epoch 结束时销毁和重建 worker 进程有不可忽略的启动开销（fork 子进程 + 重新初始化 Dataset），对于 epoch 较短（数据集较小）的场景，这个开销可能占到总训练时间的 10-20%。

### 3.7 `sampler` 参数

如果指定了 `sampler`，则 `shuffle` 必须为 `False`（sampler 已经决定了采样顺序）。这是为了防止冲突——两者同时在控制采样顺序，结果不可预测。

```python
from torch.utils.data import RandomSampler, SequentialSampler

# RandomSampler: 随机采样（等价于 shuffle=True）
sampler = RandomSampler(dataset)
loader = DataLoader(dataset, batch_size=32, sampler=sampler)

# SequentialSampler: 顺序采样（等价于 shuffle=False）
sampler = SequentialSampler(dataset)
loader = DataLoader(dataset, batch_size=32, sampler=sampler)
```

## 4. `collate_fn`：自定义批次组装

现在我们来深度讲解 DataLoader 中最灵活但最容易让人困惑的参数：`collate_fn`。

### 4.0 为什么需要 collate_fn —— 一个完整的故事

假设你正在做一个 NLP 文本分类项目。你的 Dataset 的 `__getitem__` 返回一个 token ID 序列和一个标签，看起来一切正常。

但当你把 DataLoader 的 `batch_size` 设为 32，开始迭代时，程序崩溃了：

```
RuntimeError: stack expects each tensor to be equal size, but got [5] at entry 0 and [12] at entry 1
```

原因很简单：第一个句子有 5 个 token，第二个有 12 个 token。`torch.stack` 要求所有 Tensor 形状完全一致，而不同长度的序列无法直接堆叠。

这就是 `collate_fn` 存在的根本理由：**DataLoader 不知道如何组装你的样本**。默认的 collate_fn 只能处理所有样本同形的情况。当你遇到变长序列、异质结构（图像+文本）、或需要返回非 Tensor 数据（如文件名、元数据），就必须自定义 collate_fn。

DataLoader 内部获取 `batch_size` 个样本后，调用 `collate_fn` 将它们组装成一个批次 Tensor。默认的 `collate_fn` 会将列表中的样本沿第 0 维堆叠：

```python
# 默认行为等价于
def default_collate(batch):
    # batch 是 list of (x, y) tuples
    xs, ys = zip(*batch)
    return torch.stack(xs, 0), torch.stack(ys, 0)
```

### 4.1 处理变长序列

当样本长度不一致时（如 NLP 中不同长度的句子），默认 `torch.stack` 会失败。

```python
# 变长数据：3 个样本，长度分别为 2, 4, 3
data = [
    (torch.tensor([1, 2]), 0),
    (torch.tensor([3, 4, 5, 6]), 1),
    (torch.tensor([7, 8, 9]), 0),
]

# 默认 collate_fn 会报错：stack expects each tensor to be equal size
```

**方案一：padding 到该 batch 的最大长度**。这是最高效的方案——只在每个 batch 内部做 padding，不需要每个样本都 pad 到全局最大长度。

```python
from torch.nn.utils.rnn import pad_sequence

def collate_variable_length(batch):
    """自定义 collate：将变长序列 padding 到批次内最大长度"""
    sequences, labels = zip(*batch)

    # pad_sequence: 将 list of tensors 填充到相同长度（补 0）
    # batch_first=True: 输出 shape (batch_size, max_len)
    padded = pad_sequence(sequences, batch_first=True, padding_value=0)

    # 记录每个样本的真实长度（给 RNN pack_padded_sequence 用）
    lengths = torch.tensor([len(seq) for seq in sequences])

    labels = torch.tensor(labels)
    return padded, lengths, labels

# 使用
loader = DataLoader(dataset, batch_size=16, collate_fn=collate_variable_length)
```

`pad_sequence` 的行为需要仔细理解：它会找出序列列表中最长的那个长度 `max_len`，然后把其他序列用 `padding_value`（通常是 0）填充到 `max_len`。这意味着不同 batch 的序列长度可能不同——第一个 batch 的最长序列是 15，第二个 batch 的最长是 8。这比全局 padding 节省了大量计算。

**处理变长序列的完整方案 —— 配合 RNN 的 pack_padded_sequence**：

仅仅 padding 还不够——RNN 在计算时会把 padding 的位置也纳入计算，产生无意义的隐藏状态。`pack_padded_sequence` 告诉 RNN "这些位置是 padding，跳过它们"。这不仅能提高计算效率，还能保证输出的隐藏状态不受 padding 值的污染。

```python
import torch.nn as nn
from torch.nn.utils.rnn import pack_padded_sequence, pad_packed_sequence

# collate_fn（同上）
def collate_fn(batch):
    sequences, labels = zip(*batch)
    lengths = torch.tensor([len(s) for s in sequences])
    padded = pad_sequence(sequences, batch_first=True)
    labels = torch.tensor(labels)
    return padded, lengths, labels

# 在模型中使用
class LSTMModel(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim, num_classes):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.lstm = nn.LSTM(embed_dim, hidden_dim, batch_first=True)
        self.fc = nn.Linear(hidden_dim, num_classes)

    def forward(self, padded, lengths):
        embedded = self.embedding(padded)              # (batch, seq, embed)

        # pack: 告诉 LSTM 忽略 padding 部分
        packed = pack_padded_sequence(embedded, lengths.cpu(),
                                      batch_first=True, enforce_sorted=False)
        packed_output, (h_n, c_n) = self.lstm(packed)

        # 使用最后一个有效时间步的隐藏状态做分类
        # 或者将 packed_output 再 unpack 取每个序列的真实长度处
        output, _ = pad_packed_sequence(packed_output, batch_first=True)
        # 取每个序列的最后一个非 padding token
        last_outputs = output[torch.arange(len(lengths)), lengths - 1]
        logits = self.fc(last_outputs)
        return logits
```

`enforce_sorted=False` 允许序列不必按长度降序排列——这是一个重要的便利性选项。早期的 PyTorch 版本要求序列必须先从长到短排序，否则会因 CUDA 内核限制而失败。新版本放宽了这个限制。

### 4.2 处理复杂样本结构

有时样本包含不应被 stack 的元数据（如文件名、原始文本）。collate_fn 可以灵活处理这种情况——对 Tensor 做 stack，对字符串保持原样。

```python
def collate_with_metadata(batch):
    """每个样本包含 (image, label, filename)，前两个正常堆叠，filename 保持为列表"""
    images, labels, filenames = zip(*batch)
    return torch.stack(images), torch.stack(labels), list(filenames)
```

这种模式在实际项目中很常见：你需要保留文件名以便后续将模型预测与原图关联（如目标检测的可视化输出）。

### 4.3 自定义 batch 结构 —— 字典格式

当模型有多个输入时（如 HuggingFace Transformers 的 `input_ids` + `attention_mask`），将 batch 组装为字典格式可以提高代码的可读性和可维护性。

```python
def collate_as_dict(batch):
    """将批次组装为字典格式，便于管理多输入"""
    # 假设每个样本是 dict: {"input_ids": ..., "attention_mask": ..., "labels": ...}
    batch_dict = {}
    for key in batch[0].keys():
        values = [item[key] for item in batch]
        if isinstance(values[0], torch.Tensor):
            batch_dict[key] = torch.stack(values)
        else:
            batch_dict[key] = values  # 非 Tensor 保持列表
    return batch_dict

# 使用
loader = DataLoader(dataset, batch_size=16, collate_fn=collate_as_dict)
for batch in loader:
    # batch["input_ids"], batch["attention_mask"], batch["labels"]
    pass
```

字典格式的 batch 相比于元组有几个优势：通过 key 访问而非按位置索引减少了出错概率；新增或删除一个字段不需要修改所有下游代码中的索引位置。

### 4.4 collate_fn 同时处理图像和文本（多模态）

多模态是 collate_fn 最复杂的应用场景——同一个 batch 中既有可堆叠的图像 Tensor（固定 shape），又有变长的文本序列（需要 padding），还有标量标签。

```python
def collate_multimodal(batch):
    """处理 (image, text_tokens, label) 三元组"""
    images, texts, labels = zip(*batch)

    # 图像可以直接 stack
    images = torch.stack(images)  # (batch, C, H, W)

    # 文本可能变长，需要 padding
    texts_padded = pad_sequence(texts, batch_first=True, padding_value=0)
    text_lengths = torch.tensor([len(t) for t in texts])

    labels = torch.tensor(labels)

    return {"image": images,
            "text": texts_padded,
            "text_lengths": text_lengths,
            "label": labels}
```

这个 collate_fn 体现了对异构数据的处理策略：对同质的数据统一堆叠，对异质的数据各自特殊处理。返回字典格式使得下游代码可以按名称访问不同类型的特征。

## 5. transforms：数据预处理与增强

在图像深度学习任务中，原始图片很少能直接输入模型。你需要调整尺寸（使所有输入一致）、转换为 Tensor 格式、以及做数据增强（提高模型泛化能力）。`torchvision.transforms` 提供了一套标准化的工具来完成这些操作。

transforms 的核心设计模式是**组合式管线**（Compose）：每个 transform 是一个简单的可调用对象，`Compose` 将它们串联成一个管线，数据依次流过每个 transform。这种设计让你可以像搭积木一样组合变换，而不需要写复杂的预处理函数。

```python
from torchvision import transforms

# 定义变换管线
transform = transforms.Compose([
    transforms.Resize((224, 224)),           # 调整大小
    transforms.RandomHorizontalFlip(p=0.5),  # 随机水平翻转（数据增强）
    transforms.ToTensor(),                   # PIL Image -> Tensor [0, 1]
    transforms.Normalize(                    # 标准化
        mean=[0.485, 0.456, 0.406],         # ImageNet 均值
        std=[0.229, 0.224, 0.225]           # ImageNet 标准差
    ),
])
```

一个常见的陷阱是 **transform 的顺序**。`ToTensor()` 将 PIL Image 转为 Tensor（值域 [0, 1]），`Normalize` 需要 Tensor 作为输入。如果你把 `Normalize` 放在 `ToTensor` 前面，会收到类型错误——因为 Normalize 期望 Tensor 而非 PIL Image。此外，任何随机变换（RandomCrop、RandomFlip、ColorJitter）都应该在 `ToTensor()` 之前调用，因为它们的设计目标是操作 PIL Image。

### 5.1 常用变换详解

下面的清单涵盖了最常用的几何变换和像素变换。理解每个变换的物理意义有助于你为特定任务选择合适的增强策略。例如，对医学图像做随机水平翻转是合理的（左右镜像），但做随机垂直翻转可能不合理（上下颠倒的 X 光片没有医学意义）。

```python
# ----- 几何变换 -----
transforms.Resize((256, 256))       # 缩放到 256x256
transforms.Resize(256)              # 保持长宽比，短边缩放到 256
transforms.CenterCrop(224)          # 中心裁剪 224x224
transforms.RandomCrop(224)          # 随机位置裁剪 224x224
transforms.RandomResizedCrop(224,   # 随机区域缩放裁剪到 224x224
                              scale=(0.08, 1.0),  # 裁剪面积占比
                              ratio=(3/4, 4/3))   # 长宽比范围
transforms.RandomHorizontalFlip(p=0.5)   # 50% 概率水平翻转
transforms.RandomVerticalFlip(p=0.5)     # 50% 概率垂直翻转
transforms.RandomRotation(15)       # +/- 15 度内随机旋转
transforms.RandomAffine(degrees=0, translate=(0.1, 0.1))  # 随机平移
transforms.RandomPerspective(p=0.5) # 随机透视变换

# ----- 像素变换 -----
transforms.ColorJitter(
    brightness=0.2, contrast=0.2,  # 亮度/对比度在 0.8~1.2 间随机
    saturation=0.2, hue=0.1        # 饱和度/色相微调
)
transforms.Grayscale(num_output_channels=1)  # 转灰度
transforms.RandomGrayscale(p=0.1)  # 10% 概率转灰度
transforms.GaussianBlur(kernel_size=5)  # 高斯模糊

# ----- 格式转换 -----
transforms.ToTensor()               # PIL 或 numpy [H, W, C] 0-255 -> Tensor [C, H, W] 0-1
transforms.ToPILImage()             # Tensor -> PIL Image

# ----- 标准化 -----
transforms.Normalize(mean, std)     # (x - mean) / std，逐通道

# ----- 自动增强（AutoAugment）-----
transforms.AutoAugment(policy=transforms.AutoAugmentPolicy.IMAGENET)
# 基于搜索的自动数据增强策略

# ----- RandAugment -----
transforms.RandAugment(num_ops=2, magnitude=9)
# 随机选择 N 种增强操作，统一强度
```

`RandomResizedCrop` 是 ImageNet 训练的标准操作，也是最强的数据增强之一。它的 `scale=(0.08, 1.0)` 意味着裁剪区域可以小到原始图像的 8%，大到 100%。这种极端的缩放变化迫使模型学会在多尺度下识别物体。

AutoAugment 和 RandAugment 是自动数据增强的两个代表性方法。AutoAugment 通过强化学习搜索最优的增强策略组合；RandAugment 则简化为随机选择 N 种操作并统一强度——二者都比手工设计的增强管线有更好的效果，但计算开销也更大。

### 5.2 `ToTensor()` 的细节

`ToTensor()` 看似简单，但有几个细节需要注意：
1. 它将像素值从 `[0, 255]` 缩放到 `[0.0, 1.0]`（隐式除以 255）
2. 它将维度从 HWC（PIL 格式）转换为 CHW（PyTorch 格式）
3. 如果输入是 numpy 数组且 dtype 是 uint8，也是除以 255

```python
# PIL Image: (H, W, C), pixel in [0, 255]
# ToTensor: (C, H, W), pixel in [0.0, 1.0]
# 等价于：先除以 255，再调整维度顺序

import torch
from PIL import Image
import numpy as np

# 手动验证 ToTensor 的行为
img = Image.new("RGB", (10, 20), color=(128, 64, 32))
tensor = transforms.ToTensor()(img)
print(tensor.shape)   # torch.Size([3, 20, 10]) — 注意维度变化
print(tensor[:, 0, 0])  # tensor([0.5020, 0.2510, 0.1255]) — 原值/255
```

如果你处理的图像的像素值范围已经不对（如已经是 [0, 1] 的浮点数），再经过 `ToTensor()` 除以 255 会导致数值几乎全为 0。因此，在使用 `ToTensor()` 之前要确认输入的数据格式。

### 5.3 `Normalize` 的必要性

神经网络对输入尺度非常敏感。如果不同特征的数据范围差距巨大（如特征 1 在 [0, 1]，特征 2 在 [0, 1000]），相对于特征 2 的参数更新会远大于特征 1——因为梯度的大小与输入值成正比。这导致优化器需要为不同参数使用不同的学习率。

标准化（Normalization）通过将每个通道的分布变为均值 0、标准差 1，让所有特征在大致相同的尺度上变化，从而保证各参数的梯度更新步长相对一致。

ImageNet 的 `mean=[0.485, 0.456, 0.406]` 和 `std=[0.229, 0.224, 0.225]` 是 ImageNet 数据集上数百万张图片的统计值。**关键规则**：如果你用 ImageNet 预训练权重做微调，必须使用这些值（因为预训练模型是在这样的标准化分布上训练的）；如果你从头训练，应该用你自己的数据集统计量来计算 mean 和 std——使用 ImageNet 的统计量对你自己的数据集没有物理意义。

**计算自定义数据集的 mean/std**：

```python
def compute_mean_std(dataset):
    """计算数据集的逐通道均值和标准差"""
    loader = DataLoader(dataset, batch_size=64, shuffle=False, num_workers=4)

    mean = torch.zeros(3)
    std = torch.zeros(3)
    total_pixels = 0

    for images, _ in loader:
        # images: (batch, C, H, W)
        batch_pixels = images.size(0) * images.size(2) * images.size(3)
        total_pixels += batch_pixels
        mean += images.mean(dim=[0, 2, 3]) * images.size(0)
        std += images.std(dim=[0, 2, 3]) * images.size(0)

    mean /= len(dataset)
    std /= len(dataset)
    return mean.tolist(), std.tolist()

# 使用
# mean, std = compute_mean_std(train_dataset)
# transform = transforms.Compose([
#     transforms.ToTensor(),
#     transforms.Normalize(mean=mean, std=std),
# ])
```

这个计算逻辑需要注意：我们是对每个 batch 计算其 mean 和 std，然后按 batch 大小加权平均。如果直接对整个数据集的像素数组算一次性的 mean/std，内存可能不够。这个滑动加权的方式避免了内存爆炸。

### 5.5 自定义 Transform 类

当内置的 transforms 不能满足需求时，可以自定义 Transform。自定义 Transform 只需实现 `__call__` 方法，接收一个输入（通常是 PIL Image 或 Tensor），返回变换后的结果。

以下展示了两个自定义 Transform：添加高斯噪声（模拟传感器噪声）和随机擦除（模拟遮挡，提高模型对部分遮挡的鲁棒性）。

```python
class AddGaussianNoise:
    """自定义 Transform：添加高斯噪声"""

    def __init__(self, mean=0.0, std=0.1):
        self.mean = mean
        self.std = std

    def __call__(self, tensor):
        # tensor 是 (C, H, W) 的 Tensor (ToTensor 之后)
        noise = torch.randn(tensor.size()) * self.std + self.mean
        return tensor + noise

    def __repr__(self):
        return f"{self.__class__.__name__}(mean={self.mean}, std={self.std})"


class RandomErasingCustom:
    """自定义随机擦除"""

    def __init__(self, p=0.5, scale=(0.02, 0.1)):
        self.p = p
        self.scale = scale

    def __call__(self, tensor):
        if torch.rand(1).item() > self.p:
            return tensor

        _, h, w = tensor.shape
        area = h * w
        target_area = np.random.uniform(*self.scale) * area
        aspect_ratio = np.random.uniform(0.3, 3.3)

        erase_h = int(np.sqrt(target_area * aspect_ratio))
        erase_w = int(np.sqrt(target_area / aspect_ratio))

        if erase_h < h and erase_w < w:
            top = np.random.randint(0, h - erase_h)
            left = np.random.randint(0, w - erase_w)
            tensor[:, top:top+erase_h, left:left+erase_w] = 0.0

        return tensor


# 使用自定义 Transform
custom_transform = transforms.Compose([
    transforms.Resize((224, 224)),
    transforms.ToTensor(),
    AddGaussianNoise(mean=0.0, std=0.05),  # 自定义变换
    transforms.Normalize(mean=[0.5]*3, std=[0.5]*3),
])
```

需要注意的是，自定义 Transform 在 Compose 管线中的位置很重要。`AddGaussianNoise` 操作在 Tensor 上（需要 `ToTensor` 之后），且应该在 `Normalize` 之前——因为噪声应该在标准化的数据上添加才有意义（噪声范围和标准化后的数据在相同尺度上）。

## 6. 数据增强策略

数据增强通过在训练时对原始样本施加随机变换，生成更多"变体"。从信息论的角度看，数据增强不会引入新的信息——它只是在利用"人类对图像内容的先验知识"来生成有效的训练样本。例如，水平翻转一张猫的图片，它仍然是一只猫——这种"不变性"是数据增强有价值的根本原因。

### 6.1 图像分类的标准增强管线

以下是一个被广泛验证过的增强管线，适用于大多数自然图像分类任务。

```python
# 训练时的增强管线
train_transform = transforms.Compose([
    transforms.RandomResizedCrop(224, scale=(0.8, 1.0)),  # 随机裁剪+缩放
    transforms.RandomHorizontalFlip(p=0.5),                # 随机水平翻转
    transforms.ColorJitter(brightness=0.2, contrast=0.2),  # 颜色抖动
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.485, 0.456, 0.406],
                         std=[0.229, 0.224, 0.225]),
])

# 验证时的管线（不做增强，只需 resize + normalize）
val_transform = transforms.Compose([
    transforms.Resize(256),
    transforms.CenterCrop(224),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.485, 0.456, 0.406],
                         std=[0.229, 0.224, 0.225]),
])
```

### 6.2 为什么要区分训练/验证的 transform

这是数据增强最基本的原则：**训练用随机增强，验证只用确定性变换**。

训练时的随机变换（RandomCrop、Flip、ColorJitter）让模型在每次 epoch 看到略有不同的输入——同一个样本在不同 epoch 中的版本都不一样。这等效于把训练集"放大"了多倍（理论上无限倍），因为每次随机采样的结果都不同。

但验证和测试需要**确定性**：对同一张图片，每次前向传播必须得到完全相同的输出，否则你就无法公平地比较不同模型的性能——不同的预测结果到底是模型差异还是数据差异导致的？因此验证管线只做 resize + center crop + normalize，没有任何随机性。

> 核心原则：训练用随机变换增强，验证只用确定性的 resize + normalize。

### 6.3 不同视觉任务的数据增强策略

不同任务需要的增强策略差异很大——强增强对分类任务有帮助（迫使模型学到不变性），但对语义分割可能有害（改变了像素级的标签对应关系）。

**分类任务**：

```python
train_transform = transforms.Compose([
    transforms.RandomResizedCrop(224),
    transforms.RandomHorizontalFlip(),
    transforms.ColorJitter(0.2, 0.2, 0.2, 0.1),
    transforms.ToTensor(),
    transforms.Normalize(mean, std),
])
```

**语义分割（需要同步变换图像和 mask）**：

这是一个关键的区别：分割任务中图像和像素级标签（mask）必须经过**完全相同**的几何变换。如果你对图像做了随机水平翻转但 mask 没有翻转，"猫在左边"的标注就和"猫在右边"的图像对不上了。

```python
import random

class SegmentationTransform:
    """对图像和 mask 同步做相同变换"""

    def __init__(self, size=256):
        self.size = size

    def __call__(self, image, mask):
        # 同步 Resize
        image = transforms.Resize((self.size, self.size))(image)
        mask = transforms.Resize((self.size, self.size), interpolation=transforms.InterpolationMode.NEAREST)(mask)

        # 同步随机水平翻转
        if random.random() > 0.5:
            image = transforms.RandomHorizontalFlip(p=1)(image)  # p=1 强制翻转
            mask = transforms.RandomHorizontalFlip(p=1)(mask)    # p=1 强制翻转

        # ToTensor
        image = transforms.ToTensor()(image)
        mask = transforms.ToTensor()(mask)
        # 或 mask = torch.from_numpy(np.array(mask)).long()

        image = transforms.Normalize(mean, std)(image)

        return image, mask
```

Mask 的 resize 使用 `InterpolationMode.NEAREST`（最近邻插值）而非默认的双线性插值——因为 mask 的值是离散的类别标签（0, 1, 2, ...），用双线性插值会产生"类别 1.5"这样的无意义值。

**目标检测（使用 torchvision 内置）**：

```python

# v2 API 无需特殊处理即可同步变换 image + bbox + mask
```

## 7. 数据采样器 (Sampler)

Sampler 决定了 DataLoader 按什么顺序从 Dataset 中取样本。它是 shuffle 参数的泛化版本——当简单的随机或顺序采样不够用（如类别不平衡、分布式训练）时，需要用自定义 Sampler 来控制采样逻辑。

### 7.1 内置采样器

PyTorch 提供了五种常用采样器，覆盖了大多数使用场景。

```python
from torch.utils.data import (
    RandomSampler, SequentialSampler,
    SubsetRandomSampler, WeightedRandomSampler,
    BatchSampler
)

# 1. SequentialSampler: 按顺序采样 (0, 1, 2, ..., n-1)
sampler = SequentialSampler(dataset)

# 2. RandomSampler: 随机采样（有放回或无放回）
sampler = RandomSampler(dataset, replacement=False, num_samples=None)

# 3. SubsetRandomSampler: 从指定索引中随机采样
# 常用于将训练集的一部分作为验证集
indices = list(range(100))  # 使用前 100 个样本
sampler = SubsetRandomSampler(indices)

# 4. WeightedRandomSampler: 按权重采样
# 适用于类别不平衡问题
class_counts = [500, 300, 200]  # 3 个类别的样本数
class_weights = 1.0 / torch.tensor(class_counts, dtype=torch.float)
sample_weights = class_weights[labels]  # 每个样本的权重
sampler = WeightedRandomSampler(
    weights=sample_weights,
    num_samples=len(sample_weights),
    replacement=True,
)

# 5. BatchSampler: 将另一个 sampler 的输出组装成 batch
batch_sampler = BatchSampler(
    RandomSampler(dataset),
    batch_size=32,
    drop_last=False
)
loader = DataLoader(dataset, batch_sampler=batch_sampler)
```

### 7.2 WeightedRandomSampler 处理类别不平衡

类别不平衡是现实世界数据中最常见的问题之一——比如欺诈检测中，99.9% 的样本都是正常的，只有 0.1% 是欺诈的。如果不加处理，模型只需始终预测"正常"就能达到 99.9% 的准确率，但这样的模型毫无价值。

WeightedRandomSampler 通过用倒数权重（`1/类别样本数`）来重采样，使得少数类被采样的概率与多数类相等。`replacement=True` 意味着允许重复采样，这对于少数类样本数量不够的情况是必需的。

```python
import numpy as np
from collections import Counter

def create_balanced_sampler(labels):
    """为不平衡数据集创建加权采样器"""

    # 统计每个类别的样本数
    class_counts = Counter(labels)
    n_samples = len(labels)

    # 计算每个样本的权重 = 1 / 该样本所属类别的样本数
    class_weights = {cls: n_samples / count for cls, count in class_counts.items()}
    sample_weights = [class_weights[label] for label in labels]

    sampler = WeightedRandomSampler(
        weights=sample_weights,
        num_samples=n_samples,
        replacement=True
    )
    return sampler

# 使用
labels = [0, 0, 0, 1, 2]  # 极不平衡：0 类占多数
sampler = create_balanced_sampler(labels)
# 现在 DataLoader 采样时，各类别会被平衡采样
```

### 7.3 分布式采样器（多 GPU）

在多 GPU 的分布式训练（DDP）中，每个 GPU 只负责处理数据集的一部分。`DistributedSampler` 自动将数据集按 GPU 数量切分为互不重叠的片段。

```python
from torch.utils.data.distributed import DistributedSampler

# DDP 训练时，每个 GPU 只处理数据集的一部分
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,  # GPU 数量
    rank=rank,                # 当前 GPU 的编号
    shuffle=True,
)

loader = DataLoader(dataset, batch_size=32, sampler=sampler)

# 每个 epoch 开始时需要调用 set_epoch 以确保 shuffle 不同
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)
    for batch in loader:
        ...
```

`sampler.set_epoch(epoch)` 是一个容易被忽略但至关重要的调用。如果不调用它，每个 epoch 的 shuffle 结果是完全相同的——因为 DistributedSampler 用 epoch 作为随机种子的一部分。`set_epoch` 确保了不同 epoch 有不同的 shuffle 顺序。

## 8. 代码实战：完整的图像分类 DataLoader

下面是一个从零开始构建的完整图像分类数据管线示例。它串联了 Dataset、transform 和 DataLoader 的所有知识点。

```python
import os
import torch
from torch.utils.data import Dataset, DataLoader
from torchvision import transforms
from PIL import Image

# ----- 模拟数据：在当前目录创建 fake_img/ 目录和一些 dummy 图像 -----

class CatDogDataset(Dataset):
    """猫狗分类的自定义 Dataset"""
    def __init__(self, root_dir, transform=None):
        self.samples = []
        # 假设目录结构: root_dir/cat/xxx.jpg, root_dir/dog/xxx.jpg
        for label, cls in enumerate(["cat", "dog"]):
            cls_dir = os.path.join(root_dir, cls)
            if os.path.isdir(cls_dir):
                for fname in os.listdir(cls_dir):
                    if fname.endswith(('.jpg', '.png')):
                        self.samples.append((os.path.join(cls_dir, fname), label))
        self.transform = transform

    def __len__(self):
        return len(self.samples)

    def __getitem__(self, idx):
        img_path, label = self.samples[idx]
        image = Image.open(img_path).convert("RGB")
        if self.transform:
            image = self.transform(image)
        return image, label

# 定义 transform
train_transform = transforms.Compose([
    transforms.Resize((128, 128)),
    transforms.RandomHorizontalFlip(),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.5, 0.5, 0.5], std=[0.5, 0.5, 0.5]),
])

# 构建 DataLoader
train_dataset = CatDogDataset("./data/cats_dogs", transform=train_transform)
train_loader = DataLoader(
    train_dataset,
    batch_size=32,
    shuffle=True,
    num_workers=2,
    pin_memory=True,
)

# 遍历一个批次
for images, labels in train_loader:
    print(f"Batch images shape: {images.shape}")  # [32, 3, 128, 128]
    print(f"Batch labels shape: {labels.shape}")  # [32]
    break
```

输出验证了 DataLoader 正确地将图像 resize 到 128x128、堆叠成 32 的 batch，并应用了标准化。

## 9. 性能优化建议

数据加载往往是一个深度学习项目中最大的隐性瓶颈——你很难在代码中直接看到它慢，但 GPU 利用率一直上不去就是最明显的信号。下表汇总了各优化点及其适用场景。

| 优化点 | 做法 | 适用场景 |
|--------|------|----------|
| `num_workers` | 设为 CPU 核心数（通常 4-8） | 从磁盘加载图片/文件 |
| `pin_memory` | 设为 True | 使用 GPU 训练 |
| `prefetch_factor` | 设为 2-4 | num_workers > 0 时预取更多批次 |
| `persistent_workers` | 设为 True | 避免每个 epoch 重启 worker 进程 |
| 预加载到内存 | 在 `__init__` 中把所有数据读入内存 | 小数据集（< 几 GB） |
| 使用 .to(non_blocking=True) | 配合 pin_memory，异步传输 | GPU 训练 |
| 序列化存储 | 使用 LMDB / HDF5 存储预处理的 Tensor | 大图像数据集 |
| DALI | NVIDIA DALI 库，GPU 端数据预处理 | 极大规模训练 |

```python
loader = DataLoader(
    dataset,
    batch_size=128,
    shuffle=True,
    num_workers=4,
    pin_memory=True,
    prefetch_factor=2,
    persistent_workers=True,
)
```

**完整的最佳实践配置**：

```python
def create_dataloader(dataset, batch_size, is_train=True, num_workers=None):
    """创建配置良好的 DataLoader"""

    if num_workers is None:
        import os
        num_workers = min(os.cpu_count() or 1, 8)

    loader = DataLoader(
        dataset,
        batch_size=batch_size,
        shuffle=is_train,
        num_workers=num_workers,
        pin_memory=torch.cuda.is_available(),
        drop_last=is_train,                    # 训练时丢弃最后不完整 batch
        persistent_workers=num_workers > 0,    # 有 worker 时才设置
        prefetch_factor=2 if num_workers > 0 else None,
    )
    return loader

# 使用
train_loader = create_dataloader(train_dataset, batch_size=128, is_train=True)
val_loader = create_dataloader(val_dataset, batch_size=128, is_train=False)
```

这个 `create_dataloader` 函数封装了所有最佳实践配置，可以作为项目中 DataLoader 创建的标准模板。它的关键设计是：根据 `is_train` 参数自动选择 shuffle、drop_last 等设置，根据 GPU 可用性自动设置 pin_memory。

## 10. 大规模数据集的加载策略

当数据集的大小超过物理内存时，需要特殊的加载策略。

### 10.1 延迟加载（Lazy Loading）

核心思想是：Dataset 的 `__init__` 只记录文件路径的清单（一个极小的文本文件），`__getitem__` 被调用时才读取和解码图像。这样内存中只保存路径字符串，不保存图像数据。

```python
class LargeImageDataset(Dataset):
    """只存储路径，在 __getitem__ 时动态读取"""

    def __init__(self, manifest_file, transform=None):
        self.samples = []
        with open(manifest_file, 'r') as f:
            for line in f:
                path, label = line.strip().split(',')
                self.samples.append((path, int(label)))
        self.transform = transform

    def __len__(self):
        return len(self.samples)

    def __getitem__(self, idx):
        path, label = self.samples[idx]
        image = Image.open(path).convert("RGB")  # 动态加载
        if self.transform:
            image = self.transform(image)
        return image, label
```

### 10.2 LMDB 存储

对于海量小文件（如 ImageNet 120 万张图片），文件系统的随机读取性能会成为瓶颈——每张图片都是一次独立的文件系统调用（open + read + close），开销极大。LMDB（Lightning Memory-Mapped Database）将所有小文件打包为一个高效的键值存储，大幅减少文件系统的 IOPS。

```python
import lmdb
import pickle

class LMDBDataset(Dataset):
    """从 LMDB 数据库读取数据"""

    def __init__(self, lmdb_path, transform=None):
        self.lmdb_path = lmdb_path
        self.transform = transform

        # 打开 LMDB 环境
        self.env = lmdb.open(lmdb_path, readonly=True, lock=False,
                             readahead=False, meminit=False)
        with self.env.begin() as txn:
            self.length = txn.stat()['entries']

    def __len__(self):
        return self.length

    def __getitem__(self, idx):
        with self.env.begin() as txn:
            data = txn.get(str(idx).encode())
            image, label = pickle.loads(data)

        if self.transform:
            image = self.transform(image)

        return image, label
```

LMDB 特别适合**随机读取**场景（DataLoader 的 shuffle 模式就是随机读取）。因为它使用内存映射，读取操作直接从操作系统页缓存中获取数据，避免了传统文件系统的 open/seek/read/close 四步开销。

## 常见错误

数据管线的错误通常不会在代码语法层面暴露，而是在运行时以 shape 不匹配、类型错误、性能低下的形式出现。理解每个错误的根本原因能帮助你建立正确的数据管线心智模型。

### 错误 1: `__len__` 返回 0

**错误原因**：`DataLoader` 通过 `__len__` 判断数据集是否有样本。如果 `__len__` 返回 0，DataLoader 认为数据集为空，迭代时不会 yield 任何东西。这通常是因为 `__init__` 中的数据加载逻辑有 bug（路径不对、文件不存在等），导致没有添加任何样本。

```python
# 错误
class BadDataset(Dataset):
    def __init__(self):
        self.data = torch.randn(100, 10)

    def __getitem__(self, idx):
        return self.data[idx]

    # 忘记定义 __len__!

# DataLoader 报错: TypeError: object of type 'BadDataset' has no len()

# 正确
class GoodDataset(Dataset):
    def __init__(self):
        self.data = torch.randn(100, 10)

    def __len__(self):
        return len(self.data)          # 必须实现

    def __getitem__(self, idx):
        return self.data[idx]
```

### 错误 2: `__getitem__` 返回不一致类型

**错误原因**：DataLoader 的默认 collate_fn 假设所有样本具有相同的结构。如果有些样本返回 `(tensor, label)` 而另一些只返回 `tensor`，collate_fn 在尝试 `zip(*batch)` 和 `torch.stack` 时会遇到结构不匹配的错误。

```python
# 错误
class InconsistentDataset(Dataset):
    def __getitem__(self, idx):
        if idx % 2 == 0:
            return torch.randn(10), 0      # 奇数返回 tuple
        else:
            return torch.randn(10)          # 偶数只返回 tensor

# DataLoader 的默认 collate_fn 试图 stack 不同结构时会出错

# 正确：始终返回相同结构
class ConsistentDataset(Dataset):
    def __getitem__(self, idx):
        return torch.randn(10), 0           # 始终返回 (tensor, label)
```

### 错误 3: 训练时忘记设 `shuffle=True`

**错误原因**：不 shuffle 意味着模型在每个 epoch 以完全相同的顺序看到样本。这违背了 SGD 的基本假设——每个 batch 应该是数据分布的无偏样本。具体表现为训练 loss 可能很低但验证 loss 很高（模型记住了训练集的顺序而非内容）。

```python
# 错误：每个 epoch 样本顺序完全相同
train_loader = DataLoader(train_dataset, batch_size=64, shuffle=False)
# 模型可能学会样本顺序而非内容本身，泛化能力差

# 正确
train_loader = DataLoader(train_dataset, batch_size=64, shuffle=True)
```

### 错误 4: `ToTensor()` 在 `Normalize` 之后调用

**错误原因**：`Normalize` 的输入必须是 Tensor（或 numpy 数组），而 `ToTensor` 负责将 PIL Image 转为 Tensor。顺序颠倒是逻辑错误——Normalize 无法处理 PIL Image 类型。

```python
# 错误：先 normalize 后 ToTensor，顺序颠倒！
transform = transforms.Compose([
    transforms.Normalize(mean=[0.5], std=[0.5]),  # 此时还是 PIL Image，报错！
    transforms.ToTensor(),
])
# TypeError: pic should be PIL Image or ndarray. Got <class 'torch.Tensor'>

# 正确
transform = transforms.Compose([
    transforms.ToTensor(),                          # 先转为 Tensor [0, 1]
    transforms.Normalize(mean=[0.5], std=[0.5]),    # 再标准化
])
```

### 错误 5: GPU Tensor 传给 DataLoader 做 shuffle

**错误原因**：DataLoader 内部操作（如 shuffle 时的随机索引生成、collate_fn 中的 stack）是在 CPU 上执行的。如果 Dataset 存储的是 GPU Tensor，shuffle 和 collate 过程会涉及隐式的 GPU->CPU 传输，既慢又容易出错。

```python
# 错误
X_gpu = torch.randn(1000, 10).cuda()
y_gpu = torch.randint(0, 5, (1000,)).cuda()
dataset = TensorDataset(X_gpu, y_gpu)
# DataLoader 的默认 collate_fn 内部操作（stack）可能需要 CPU Tensor

# 正确：Dataset 中存 CPU Tensor，在训练循环中再移到 GPU
dataset = TensorDataset(X, y)   # X, y 在 CPU
loader = DataLoader(dataset, batch_size=64)
for bx, by in loader:
    bx, by = bx.to("cuda"), by.to("cuda")  # 训练时再移
```

### 错误 6: `num_workers > 0` 时 Dataset 使用了不可 pickle 的对象

**错误原因**：多 worker 模式使用 Python 的 `multiprocessing`，通过序列化（pickle）将 Dataset 对象传递给子进程。如果 Dataset 的 `__init__` 中保存了不可序列化的对象（如文件句柄、数据库连接、lambda 函数等），pickle 过程会失败并报错。

```python
# 错误：在 __init__ 中打开文件句柄
class FileDataset(Dataset):
    def __init__(self, path):
        self.f = open(path, "r")      # 文件句柄不能被 pickle

# 正确：在 __getitem__ 中打开或使用预加载
class FileDataset(Dataset):
    def __init__(self, path):
        with open(path, "r") as f:
            self.data = f.readlines()  # 预加载到内存

    def __getitem__(self, idx):
        return self.data[idx]
```

### 错误 7: Windows 下 num_workers > 0 导致卡死

**错误原因**：Windows 的 multiprocessing 使用 `spawn` 模式（而非 Linux 的 `fork` 模式）。`spawn` 会在子进程中重新导入主模块，如果主模块顶层有不受保护的代码（如在 `if __name__ == "__main__"` 之外的 DataLoader 创建），子进程会递归创建新的子进程，导致死循环。

```python
# 问题：Windows 的 multiprocessing 使用 spawn 模式，
# 子进程会重新导入主模块，如果不在 if __name__ == "__main__" 保护下会死循环

# 正确
if __name__ == "__main__":
    # 所有 DataLoader 创建和训练代码放在这里
    dataset = MyDataset(...)
    loader = DataLoader(dataset, num_workers=4, ...)
    for batch in loader:
        ...

# 或者在脚本顶层设置 num_workers=0
```

### 错误 8: 忘记对验证集的 transform 去除数据增强

**错误原因**：如果在验证时使用了随机变换，每次前向传播会因为随机采样的不同而变化，导致同一个模型在同一批验证数据上得到完全不同的预测。这样就无法可靠地评估模型性能——你无法区分性能差异是来自超参数调整还是随机增强的运气。

```python
# 错误：验证集也用了数据增强
val_transform = transforms.Compose([
    transforms.RandomResizedCrop(224),   # 错误！验证不需要随机裁剪
    transforms.RandomHorizontalFlip(),   # 错误！验证不需要随机翻转
    transforms.ToTensor(),
    transforms.Normalize(mean, std),
])
# 后果：每次验证结果都不同，无法可靠评估模型性能

# 正确
val_transform = transforms.Compose([
    transforms.Resize(256),
    transforms.CenterCrop(224),
    transforms.ToTensor(),
    transforms.Normalize(mean, std),
])
```

## 本章小结

数据管线是整个深度学习系统的入口。管线设计的好坏直接决定了训练的效率和模型的最终质量。通过本章的学习，你应当掌握以下核心认知：

**关于 Dataset 和 DataLoader 的分离设计**：这是 PyTorch 数据加载体系最精妙的地方。Dataset 封装"数据是什么"（来源、格式、单样本的读取逻辑），DataLoader 封装"数据如何供给"（批次大小、采样顺序、并行度、组装方式）。这种分离意味着你可以在不修改 Dataset 代码的情况下，通过调整 DataLoader 的参数来优化训练效率，也可以在保持 DataLoader 配置的情况下切换数据来源。

**关于 collate_fn 的本质**：它不是一种特殊的 API，而是一个普通的 Python 函数。DataLoader 把 `batch_size` 个样本的列表传给这个函数，函数返回"组装好的批次"。这给了你完全的自由——你可以返回元组、字典、namedtuple、或者任何适合模型输入的数据结构。变长序列的 padding 只是 collate_fn 最经典的应用场景之一。

**关于数据增强的哲学**：训练时用随机增强来扩充数据的"有效尺寸"，验证时用确定性变换来确保评估的可靠性。这个原则简单但至关重要——违反它会导致你无法可靠地评估和比较模型。

**关于性能优化**：DataLoader 的性能优化是递进的——`num_workers` 解决 I/O 瓶颈，`pin_memory` 解决传输瓶颈，`persistent_workers` 解决进程启动开销。理解各优化的作用位置，才能在遇到性能问题时精准定位。

要点速查：

- **DataLoader** 封装了批量加载、shuffle、并行 I/O、内存预取的完整管线
- **Dataset** 只需实现 `__len__` 和 `__getitem__`，即可接入 DataLoader
- `TensorDataset` 适用于纯 Tensor 数据；自定义 Dataset 适用于文件读取、预处理等场景
- 可以为图像、文本、表格、多模态等各类数据构建 Dataset
- `collate_fn` 是解决变长数据、异质批次的关键扩展点，支持 padding、字典组装等
- `transforms.Compose` 组合图像预处理操作，`ToTensor` 必须在 `Normalize` 之前
- 训练使用随机增强，验证仅使用确定性变换 —— 这是标准做法
- 可以编写自定义 Transform 类，只需实现 `__call__` 方法
- Sampler 决定了数据采样顺序：RandomSampler、WeightedRandomSampler 等用于不同场景
- `num_workers` + `pin_memory` 是 GPU 训练的性能基石
- `persistent_workers` + `prefetch_factor` 进一步优化加载效率
- 大规模数据集使用延迟加载或 LMDB 等序列化存储
