## 1. 为什么要微调

### 1.1 Prompt 工程的天花板

Prompt Engineering 无疑是使用大语言模型（LLM）最便捷的方式——你不需要任何 GPU，不需要训练代码，只需要写几行文字就能让模型完成各种任务。但在很多实际场景中，仅靠 Prompt 是不够的。这并非因为你的 Prompt 写得不够好，而是因为 Prompt 工程本身存在天然的上限。

让我们用一个具体的案例来理解这个上限。

假设你正在为一家三甲医院开发一个病历摘要生成系统。医生需要将长篇的入院记录（通常 2000-5000 字）压缩为 200 字的结构化摘要，包含主诉、现病史、既往史、体格检查、辅助检查、初步诊断六个部分。你可能会设计这样一个 Prompt：

```
你是一位经验丰富的临床医师，请根据以下入院记录生成病历摘要。摘要必须包含以下六个部分：
1. 主诉
2. 现病史
3. 既往史
4. 体格检查
5. 辅助检查
6. 初步诊断
```

把这个 Prompt 发给 GPT-4，它能生成看起来还不错的摘要。但当你实际部署后，医生们会反馈以下问题：

- **医学专业术语使用不一致**："血压升高"有时被简化为"高血压"，有时写成"BP增高"，但在正式病历中应该写"血压升高（140/90mmHg）"并标注具体数值。
- **阴性体征遗漏**：入院记录中写到"无头痛、恶心、呕吐"，这些阴性体征对鉴别诊断很关键，但模型有时会漏掉。
- **时间描述不规范**：病历中时间应该用"入院前X天"而不是"前天"、"上周二"这样的相对表达。
- **格式不稳定**：有时摘要输出为纯文本段落，有时用 Markdown 列表，有时用编号——而医院系统要求固定的段落格式。
- **幻觉问题**：模型有时会"编造"不存在的检查结果或诊断，这在医疗场景中是不可接受的。

你尝试优化 Prompt——加入更详细的指令、提供示例（Few-shot）、使用输出格式约束（JSON Schema）。效果确实有所改善，但格式不稳定的问题仍然存在，幻觉率只能降低到 5%，而医疗场景需要的是几乎为 0% 的幻觉率。

这就是 Prompt 工程的天花板：**你只能改变模型的"输入上下文"，无法改变模型的"内部知识分布"**。模型的所有行为本质上都是预训练阶段学到的统计模式的采样结果。当你的需求与预训练数据中的主流模式不一致时——比如你需要一个特定的、规范的医学写作风格——仅靠 Prompt 无法稳定地扭转这种分布偏移。

### 1.2 微调解决什么问题

微调（Fine-tuning）之所以强大，是因为它**直接修改模型的权重**。这意味着你不仅改变了模型的输入，还改变了模型的"大脑"。具体来说，微调解决以下四个层次的问题：

**第一层：风格适应（Style Adaptation）**

让模型学会一种特定的表达风格或格式规范。这是最浅层的微调，但也是最常见的。比如上面的病历摘要案例——通过微调，你可以让模型的输出格式变得像呼吸一样自然，而不是每次都需要在 Prompt 里反复强调格式要求。微调后的模型"学会"了这种风格，即使在 Prompt 很简短的情况下也能稳定输出符合规范的文本。

**第二层：领域知识注入（Domain Knowledge Injection）**

LLM 的预训练数据覆盖了互联网上的大部分公开文本，但对于某些专业领域（法律、医学、金融、芯片设计等），公开数据远远不够。许多专业知识存在于企业内部文档、付费数据库中，从未出现在预训练语料里。通过微调，你可以把这些私有领域知识"注入"到模型中，让模型学会这些专业的术语、逻辑和规范。

**第三层：任务格式化（Task Formatting）**

预训练模型本质上是"文本补全"模型——它只会续写文本，不知道什么是"回答问题"、什么是"翻译"、什么是"摘要"。指令微调（Instruction Tuning）通过大量格式化的问答对训练，教会模型理解"指令-回复"的交互模式。如果你有自己的特殊任务格式（比如内部的 SQL 查询模板、自定义的 JSON 交互协议），微调可以让模型原生地理解这些格式。

**第四层：偏好对齐（Preference Alignment）**

即使模型能完成任务，它的回答可能不符合人类的价值观和偏好。比如模型可能给出技术上正确但粗鲁的回答，或者在应该拒绝回答的时候给出不安全的内容。通过偏好对齐（RLHF/DPO），你可以让模型的输出更符合特定用户群体的期望。

### 1.3 什么时候该微调——决策框架

下面是一个实用的决策框架，帮助你在 Prompt 工程和微调之间做出选择：

| 维度 | 适合 Prompt 工程 | 适合微调 |
|------|-----------------|---------|
| 任务稳定性 | 格式可以灵活调整 | 要求严格的格式和风格一致性 |
| 数据量 | 零样本或少样本（0-20个示例） | 有成百上千条高质量示例 |
| 领域特殊性 | 通用领域、常识性任务 | 专业领域、私有知识 |
| 延迟要求 | 可以接受较长的 Prompt | 需要压缩 Prompt 以降低延迟和成本 |
| 幻觉控制 | 可以接受偶发幻觉 | 需要严格控制幻觉率 |
| 迭代速度 | 需要快速实验和调整 | 有明确的业务指标和评估体系 |
| 成本预算 | 按量付费（API 调用） | 有 GPU 或愿意投资训练成本 |

一个简化的判断标准：**如果你发现自己在 Prompt 里写了超过 2000 字的指令、仍然需要多次 API 调用重试才能得到稳定输出，那么微调就是更好的选择。**

## 2. 全量微调 vs 高效微调

### 2.1 全量微调（Full Fine-tuning）

全量微调是最直接的微调方式：加载预训练模型，在所有训练样本上更新模型的全部参数（通常几十亿到几千亿个）。

**工作原理**：与预训练完全相同——前向传播计算 loss，反向传播计算梯度，优化器更新每一个权重矩阵。区别只是数据量更少（几万条 vs 几万亿 token）、学习率更小（通常是预训练的 1/10 到 1/100）。

**优点**：
- 效果上限最高。所有参数都可以适配新任务，理论上能学到最多的领域知识。
- 不需要额外的工程复杂度。就是标准的训练循环。

**缺点**：
- **显存需求极大**。以 LLaMA-7B 为例，全量微调需要约 60-80GB 显存（模型参数 14GB（FP16）、梯度 14GB、优化器状态 28-42GB、激活值若干 GB）。这意味着一张 A100（80GB）勉强能跑，但 batch size 会非常受限。
- **训练成本高**。7B 模型全量微调通常需要 4-8 张 A100 跑数小时到数天。
- **灾难性遗忘（Catastrophic Forgetting）**：模型在学习新任务时可能会"忘记"预训练阶段学到的通用知识。尤其在数据量少、学习率偏高时尤为明显。
- **部署和存储成本高**：每个微调任务产出一个完整的模型副本（几十 GB），如果为不同客户/任务都要微调，存储成本线性增长。

### 2.2 高效微调（Parameter-Efficient Fine-Tuning, PEFT）

高效微调的核心思想是：**不修改（或只极少修改）原始预训练权重，而是通过添加少量新参数或选择性地更新部分参数来适配下游任务**。

这背后的直觉是：预训练模型已经学到了丰富的语言知识（语法、常识、推理能力）。下游任务通常只需要"重组"这些知识，而不是从头学习。就像一个已经精通 10 种语言的翻译，你只需要告诉他"把这个领域的术语翻译得正式一些"，不需要重新教他语法。

PEFT 的主要方法包括：

**Adapter**：在 Transformer 层之间插入小型神经网络模块。只训练这些 Adapter（约占原模型参数的 1-5%），冻结原始权重。这是最早的 PEFT 方法之一。

**Prefix Tuning / Prompt Tuning**：在输入层添加可学习的虚拟 token（软提示）。这些 token 的 embedding 是可训练参数，而模型本身完全冻结。可训练参数极少（<0.1%），但效果上限也较低。

**LoRA（Low-Rank Adaptation）**：在不改变原有权重矩阵的前提下，通过低秩分解的方式添加可训练的"增量矩阵"。LoRA 是目前最主流的 PEFT 方法，兼具参数量少（常用配置下 <1%）和效果好两个优点。下一章会专门深入讲解 LoRA。

### 2.3 选择策略

```
你的 GPU 资源是多少？
├── 单卡 < 24GB → QLoRA（4-bit 量化 + LoRA）
├── 单卡 24-48GB → LoRA（7B 模型）
├── 单卡 48-80GB → LoRA（13B-30B 模型）
├── 多卡 8×A100 → 可以考虑全量微调（7B-13B）
└── 多卡 32×A100+ → 全量微调（70B+）

你的任务复杂度如何？
├── 简单的风格/格式适配 → LoRA 足够
├── 大量领域知识注入 → LoRA（r 设大一些，如 64-128）
├── 需要改变模型基础能力 → 全量微调或继续预训练
└── 多任务微调 → LoRA（每个任务一个 adapter，可插拔）
```

对于大多数读者（CS 本科生，单张消费级 GPU），LoRA/QLoRA 几乎是唯一可行的选择。本书后续章节也以 LoRA/QLoRA 为主线。

## 3. 基座模型选择

### 3.1 为什么基座模型的选择很重要

微调不是"从零开始训练"，而是在已有的基座模型之上做适配。基座模型决定了你的微调效果的上限——如果基座模型本身的中文能力很弱，你在它上面做中文医疗微调，效果也不会好到哪里去。

选择合适的基座模型需要考虑以下维度：
- 预训练数据的质量和覆盖范围（中文够不够多？有没有你需要的领域数据？）
- 模型架构和规模（参数量、上下文长度、是否有特殊设计如 MoE）
- 开源协议（是否允许商用？是否允许微调和再分发？）
- 社区生态（HuggingFace 上有没有成熟的工具链？有没有人做过类似微调？）
- 推理性能（推理速度、显存占用、是否支持量化）

### 3.2 主流开源基座模型对比

#### LLaMA 系列（Meta）

LLaMA 是 Meta 发布的开源大语言模型系列，是目前开源生态中最具影响力的基座模型之一。

**特点**：
- **英语能力一流**：预训练数据以英语为主，英文理解和生成能力非常出色
- **社区生态最繁荣**：HuggingFace 上绝大多数微调教程、工具和变体都是以 LLaMA 为基础。你遇到的任何问题几乎都有人遇到过
- **中文能力一般**：LLaMA 1/2 的中文预训练数据很少，中文效果明显弱于英语。LLaMA 3 显著改善了多语言能力，但中文仍不如专门的国产模型
- **开源协议**：LLaMA 2/3 基本允许商用（有限制条款），LLaMA 1 不允许商用

**适合场景**：英文为主的微调任务，或需要利用最成熟的社区生态进行快速实验。

#### Qwen 系列（阿里通义千问）

Qwen 是阿里巴巴发布的开源模型系列，近年来在国内开发者中非常受欢迎。

**特点**：
- **中文能力顶尖**：预训练数据中中文占比很高，中文理解和生成能力在开源模型中数一数二
- **多尺寸全覆盖**：从 0.5B 到 72B（甚至 110B），覆盖从移动端部署到大规模推理的所有场景
- **优秀的 Chat Template**：Qwen 的 Chat Template 设计规范，指令遵循能力强
- **工具调用能力强**：Qwen 系列原生支持 Function Calling，非常适合 Agent 场景
- **长上下文支持好**：Qwen2.5 支持 128K token 的上下文窗口
- **开源协议宽松**：Apache 2.0 协议，允许商用

**适合场景**：中文任务微调的首选基座模型，尤其是需要强指令遵循、工具调用或 Agent 能力的场景。

#### DeepSeek 系列（深度求索）

DeepSeek 以极高的性价比和出色的性能在开源社区引起广泛关注。

**特点**：
- **MoE 架构**：DeepSeek-V2/V3 采用 Mixture of Experts（混合专家）架构，推理时只激活部分参数，推理成本极低
- **性价比极高**：在多个 benchmark 上达到 GPT-4 水平，但推理成本只有 GPT-4 的 1/10 甚至更低
- **中文能力优秀**：在中文理解和生成任务上表现出色
- **强推理能力**：DeepSeek-R1 系列尤其擅长数学推理和代码生成
- **开源协议友好**：MIT 协议，完全开放

**适合场景**：需要高性价比推理的场景，尤其是数学、编程等推理密集型任务的中文微调。

**注意**：MoE 架构的模型微调与 Dense 模型有一些不同，需要特别注意分布式训练配置和显存管理。

#### ChatGLM 系列（智谱 AI）

ChatGLM 是国内最早开源的大语言模型之一，有深厚的社区基础。

**特点**：
- **双向注意力 + 自回归**：使用 GLM（General Language Model）架构，既有双向注意力也有自回归能力
- **中文理解能力好**：对中文语境和本土化表达的理解很好
- **模型尺寸适中**：主要是 6B 和 9B，适合消费级 GPU 微调
- **历史沉淀**：从 GLM-130B 到 ChatGLM-4，积累了丰富的实践经验
- **开源协议**：部分版本允许商用，需查看具体协议

**适合场景**：中文对话场景，尤其是资源受限（单卡 24GB 以下）时的微调。

### 3.3 选型决策指南

```
你的任务场景是什么？
├── 中文任务 + 指令微调 + Agent
│   → 首选 Qwen2.5（中文强 + 指令遵循好 + 工具调用原生支持）
│
├── 中文任务 + 推理密集型（数学/代码）
│   → DeepSeek（推理能力极强 + 性价比高）
│
├── 中文任务 + 低资源（6B 级别、单卡 <16GB）
│   → ChatGLM-6B 或 Qwen2.5-7B
│
├── 英文任务 + 利用社区生态
│   → LLaMA 3（社区最成熟 + 英文顶级）
│
├── 多语言任务
│   → Qwen2.5 或 LLaMA 3（多语言覆盖好）
│
└── 需要完全可控的商用许可
    → Qwen2.5（Apache 2.0）或 DeepSeek（MIT）
```

## 4. 指令数据工程

指令数据工程是微调过程中最耗时、最关键也最容易被低估的环节。有一句流传很广的话："高质量的数据比更好的算法更重要"。在微调中尤其如此——如果你用劣质数据训练，无论 LoRA 的 rank 调得多高、学习率设置得多精妙，最终的模型效果都不会好。

我们把这部分拆分为五个步骤来详细讲解：数据收集 → 数据清洗 → 数据格式化 → 数据配比 → 数据质检。

### 4.1 数据收集

#### 4.1.1 数据来源分类

指令微调数据的来源可以分为三类：人工标注、模型生成、开源数据集。这三者有各自的优劣势和适用场景。

**人工标注**：
这是最传统也是质量最高的数据获取方式。由领域专家或经过培训的标注员编写"指令-回复"对。

优点：
- 质量可控：你可以定义详细的标注规范，确保每条数据都符合标准
- 领域专业知识：标注员可以注入 LLM 不具备的专业知识（如法律条款的最新解读、企业内部业务规则）
- 多样性保证：你可以主动设计覆盖各种场景和边界情况的指令

缺点：
- 成本高：专业领域的标注员时薪通常在 100-500 元，一条高质量数据的标注成本在 5-50 元
- 速度慢：一个标注员一天只能产出几十到上百条高质量数据
- 规模受限：对于需要万条以上数据的微调场景，纯人工标注不现实

在实践中，最经济的做法是"人机协作"——先用 GPT-4 等强模型生成初稿，再由人工审核和修改。

**模型生成（Self-Instruct / Evol-Instruct）**：
利用强大的 LLM（通常是 GPT-4 或 Claude）自动生成训练数据，然后用这些数据来微调一个较弱或较小的模型。

Self-Instruct 的工作流程：
1. 人工编写少量的"种子指令"（seed instructions，约 100-200 条）
2. 将种子指令发给 GPT-4，让它生成新的、多样化的指令
3. GPT-4 同时为每条指令生成回复
4. 过滤掉低质量的生成结果
5. 用过滤后的数据微调目标模型

Evol-Instruct 是 Self-Instruct 的升级版——它不只是一次性地生成指令，而是通过"进化"策略让指令逐渐变得复杂：
- In-Depth Evolving：让指令变得更深入（加约束条件、增加推理步数）
- In-Breadth Evolving：让指令覆盖更多样化的主题

优点：
- 成本低：通过 API 调用生成数万条高质量数据的成本通常在几百到几千元
- 速度快：并行调用可以在几小时内生成大量数据
- 多样性好：通过设计好的 Prompt，可以引导 GPT-4 生成各种类型的数据

缺点：
- 有"蒸馏"的嫌疑：本质上是用一个强模型的能力来训练弱模型，弱模型的上限受限于强模型
- 可能带入强模型的偏见和错误
- 生成的数据可能存在模式化（template-like）问题，缺乏真实用户输入的多样性

**开源数据集**：
社区中有大量公开的指令微调数据集。以下是一些常用数据集：

| 数据集 | 规模 | 主要语言 | 内容特点 |
|--------|------|---------|---------|
| Alpaca | 52K | 英文 | 最早的开源指令数据集，由 GPT-3 生成 |
| ShareGPT | ~90K | 多语言 | 真实的 ChatGPT 对话记录，质量参差不齐 |
| OpenOrca | ~1M | 英文 | 从 FLAN 等学术数据集演化而来，质量较高 |
| BELLE | 1.5M+ | 中文 | 中文指令数据集，覆盖面广 |
| Firefly | 1.1M+ | 中文 | 包含多种 NLP 任务类型的指令数据 |
| UltraChat | 1.4M+ | 英文 | 多轮对话数据，对话质量较高 |

使用开源数据集的注意事项：
- **必须二次清洗**：开源数据集通常质量参差不齐，直接使用会导致模型效果不稳定
- **注意许可证**：有些数据集不允许商用（如 ShareGPT 的使用条款存在争议）
- **去重和去偏**：开源数据集可能包含大量重复或高度相似的数据，也可能在某些主题上有偏差
- **与自有数据混合**：纯开源数据集微调出的模型缺乏差异性，建议与自有的高质量数据混合使用

### 4.2 数据清洗

数据收集完成后，不能直接用于训练。数据清洗是决定微调效果的关键环节。洗得不够，垃圾数据会污染模型；洗得过度，训练数据会变得过于同质化。

#### 4.2.1 规则过滤

规则过滤是第一道防线，通过预定义的规则快速筛除明显有问题的数据。

常见的过滤规则包括：

**长度过滤**：
- 指令太短（<5 个字符）：如"你好"、"谢谢"这种几乎没有学习价值的数据
- 指令太长（>2000 个字符）：可能是复制粘贴的文档，不是自然指令
- 回复太短（<10 个字符）：如只有一个"好"字的回复
- 回复太长（>8000 个字符）：可能导致显存溢出或训练不稳定

**语言过滤**：
- 使用语言检测工具（如 fasttext 或 lingua）检测文本语言
- 去掉非目标语言的数据（中文微调任务就只保留中文数据）
- 去掉中英混合过于严重且无意义的数据（如随机掺杂的英文单词）

**内容过滤**：
- 包含敏感词或不当内容的数据
- 包含非打印字符的乱码数据
- 包含明显的模板残留（如"[请在此处填写]"这样的占位符）
- 回复中包含"作为AI语言模型"这类 GPT 模型自带的拒绝话术（在微调数据中应替换为正面回答）

**格式过滤**：
- 检查 JSON 格式是否正确（对于要求结构化输出的数据）
- 检查指令和回复是否成对存在、不缺少字段

#### 4.2.2 质量评分

仅靠规则过滤不够。很多数据虽然"看起来"正常，但实际质量很低——指令表述模糊、回复答非所问、回复信息量不足等。我们需要对每条数据打分，只保留高分数据。

**基于语言模型的质量评分**：
用 GPT-4 或类似模型对每条数据评分。评分标准通常包括：
- 指令清晰度：用户意图是否明确
- 回复准确性：回复是否正确回答了问题
- 回复完整性：是否包含足够的细节和解释
- 回复有用性：对用户是否有实际帮助
- 格式规范性：是否符合预期的输出格式

**基于规则的质量评分**：
- 回复的词汇丰富度（unique token 比例）：如果回复只用了一些高频词，可能是敷衍的回复
- 回复的信息密度：包含多少事实性陈述（可以用 NER 提取的实体数量）
- 回复的一致性：对同一问题的多次回答是否保持一致

#### 4.2.3 去重

重复数据会浪费训练算力，还有可能导致模型过拟合到某些高频模式上。

**精确去重**：
- 对每条指令（或指令+回复的组合）计算哈希值
- 完全相同的哈希值代表完全相同的文本，直接去重

**近似去重（语义去重）**：
很多数据不是完全重复，但语义高度相似。比如"帮我写一封辞职信"和"能不能帮我起草一份离职信？"——这两条指令不同但语义等价，需要的回复也相同。

常用的近似去重方法：
- **MinHash**：通过 hash 函数集合来估算两个集合的 Jaccard 相似度。适合文本级别的近似去重
- **SimHash**：通过降维后的指纹来判断相似度。适合检测高度相似的文本
- **语义嵌入去重**：使用 sentence embedding（如 text2vec 或 OpenAI Embedding）将文本转为向量，计算余弦相似度，相似度高于阈值（如 0.9）的视为近似重复

**注意**：去重阈值不能设得太激进。有些任务就是需要相似但略有不同的训练数据（比如风格微调中需要大量同风格但不同内容的样本）。过度去重会导致模型缺乏足够的训练数据。

### 4.3 数据格式化：Chat Template

数据清洗完成后，需要将数据统一格式化为模型能理解的 Chat Template 格式。Chat Template 是微调中最容易出错也最重要的细节之一。

#### 4.3.1 Chat Template 的结构

现代对话模型（LLaMA、Qwen 等）的 Chat Template 通常包含三个角色：

```
<|im_start|>system
你是一个有帮助的AI助手。
<|im_end|>
<|im_start|>user
请帮我翻译下面的句子为英文：今天天气真好。
<|im_end|>
<|im_start|>assistant
Please translate the following sentence to English: The weather is really nice today.
<|im_end|>
```

这里面最关键的认知是：**Chat Template 不是简单地拼接文本，而是使用特殊 token 来标记角色转换**。这些特殊 token 在预训练或指令微调阶段已经教会了模型"看到这个 token 意味着角色切换"，所以微调时必须使用与基座模型一致的模板。

不同的模型使用不同的特殊 token：
- LLaMA 系列：`<s>[INST] ... [/INST] ... </s>`
- Qwen 系列：`<|im_start|>system\n...<|im_end|>\n<|im_start|>user\n...<|im_end|>\n<|im_start|>assistant\n...<|im_end|>`
- ChatGLM 系列：`[gMASK]<sop> ... <eop>`
- DeepSeek 系列：与 LLaMA 类似或自定义格式

**为什么 Chat Template 如此重要？**

如果你用错了模板，模型在推理时"看不懂"你的格式，输出会非常混乱——可能把用户输入也当作自己该重复的内容，或者完全忽略 system prompt。更隐蔽的问题是：训练时用了一种模板、推理时用了另一种，模型虽然在训练时 loss 在下降，但推理时效果很差（这被称为 train-inference mismatch）。

幸运的是，HuggingFace 的 tokenizer 内置了 `apply_chat_template` 方法，会自动使用正确的模型模板：

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")

messages = [
    {"role": "system", "content": "你是一个有帮助的AI助手。"},
    {"role": "user", "content": "请帮我翻译：今天天气真好。"},
    {"role": "assistant", "content": "The weather is really nice today."},
]

# 自动应用正确的 Chat Template
formatted = tokenizer.apply_chat_template(
    messages,
    tokenize=False,  # 返回字符串而不是 token id
)
print(formatted)
```

#### 4.3.2 数据格式化的标准流程

无论原始数据是什么格式（JSON、CSV、Parquet），最终都要转换为统一的格式：

```python
# 标准的指令微调数据格式
{
    "messages": [
        {"role": "system", "content": "系统指令（可选）"},
        {"role": "user", "content": "用户输入"},
        {"role": "assistant", "content": "期望的回复"}
    ]
}
```

对于多轮对话：

```python
{
    "messages": [
        {"role": "system", "content": "你是一个编程助手。"},
        {"role": "user", "content": "Python 中如何读取文件？"},
        {"role": "assistant", "content": "可以使用 open() 函数..."},
        {"role": "user", "content": "那写文件呢？"},
        {"role": "assistant", "content": "同样使用 open()，指定模式为 'w'..."}
    ]
}
```

**多轮对话数据的价值**：单轮数据只能让模型学会"一次性回答"，而多轮数据能让模型学会"在上下文中对话"——理解指代（"那"指什么）、在之前的回答基础上补充、纠正之前回答中的错误等。建议训练数据中至少有 20-30% 的多轮对话数据。

### 4.4 数据配比

数据配比指的是不同类型训练数据在总数中的比例分配。这是一个需要反复实验的环节。

#### 4.4.1 为什么数据配比重要

不同的训练数据会给模型带来不同的能力偏重。如果某种类型的数据占比过高，模型就会偏向于那种类型的输出——即使输入明显是另一种类型的问题。

举例来说，如果你的训练数据中 80% 是代码生成任务，20% 是通用对话，微调后的模型可能会把用户的日常闲聊也当成编程请求，总是试图输出代码。

#### 4.4.2 常见的数据配比策略

```
通用指令微调的推荐配比：
├── 通用问答/闲聊          30-40%
├── 知识问答/事实性查询     20-25%
├── 文本生成/改写/翻译      15-20%
├── 代码/推理              10-15%
├── 安全/拒绝样本           5-10%
└── 多轮对话                5-10%
```

如果你的微调有专门的目标（比如只做医疗问答），那么目标类型的数据占比可以提高到 50-70%，但仍需保留一些通用能力数据以防止灾难性遗忘。

对于偏好对齐（DPO），数据配比需要专门设计：

```
DPO 训练数据的推荐配比：
├── 有帮助性对比对          40%
├── 安全性对比对            30%
├── 真实性对比对            20%
├── 格式/表达质量对比对      10%
```

### 4.5 数据质检

数据准备完成后的最后一道关卡是质量检查。质检可以防止明显的错误数据进入训练流程。

#### 4.5.1 人工抽查

随机抽取 100-200 条数据，由人工逐一检查。检查清单：

- [ ] 指令是否清晰易懂？
- [ ] 回复是否正确回答了指令？
- [ ] 回复是否完整、有帮助？
- [ ] 格式是否符合 Chat Template 规范？
- [ ] 有没有事实性错误或幻觉？
- [ ] 有没有不安全或不适当的内容？
- [ ] System prompt 是否适合目标使用场景？

如果人工抽查的通过率低于 90%，说明数据准备流程有问题，需要回溯到数据清洗阶段重新处理。

#### 4.5.2 LLM-as-Judge 自动质检

对于大规模数据集（万条以上），人工抽查无法覆盖足够样本。这时可以使用 LLM-as-Judge 对全量数据进行自动评分。

基本流程：
1. 设计评分标准（准确性、完整性、有用性等维度）
2. 用 GPT-4 或同等水平的模型作为评判官
3. 对每条数据打分（1-5 分或 A/B/C/D 等级）
4. 设定阈值，过滤掉低分数据
5. 对评分结果进行人工校准（抽查评分是否合理）

LLM-as-Judge 的局限性：
- GPT-4 也有自己的偏好和偏见，评分可能不客观
- GPT-4 对某些专业领域的问题可能无法准确判断对错
- 成本不低：对十万条数据用 GPT-4 评分，费用可能在几百到几千美元

因此，推荐的做法是：**LLM-as-Judge 初筛 + 人工校准**。先让 GPT-4 批量评分过滤掉明显的低质数据，然后人工抽样验证评分质量。

## 5. 完整的数据准备 Pipeline 代码实现

下面是一个完整的数据准备 Pipeline 实现，涵盖了从原始数据到训练就绪数据的全流程。

```python
"""
完整的指令微调数据准备 Pipeline。

覆盖以下环节：
1. 数据加载（支持 JSON/JSONL/CSV 多种格式）
2. 规则过滤（长度、语言、内容过滤）
3. 质量评分（基于规则的质量指标）
4. 近似去重（基于 MinHash LSH）
5. 格式化为 Chat Template
6. 数据配比采样
7. 最终质检报告

依赖：pip install datasets pandarallel datasketch tqdm
"""
import os
import re
import json
import hashlib
import random
from pathlib import Path
from typing import Optional
from collections import Counter
from dataclasses import dataclass, field

from tqdm import tqdm
from datasets import Dataset, load_dataset, concatenate_datasets
from datasketch import MinHash, MinHashLSH

# 尝试导入语言检测库（可选）
try:
    import lingua
    LANGUAGE_DETECTOR = lingua.LanguageDetectorBuilder.from_all_spoken_languages().build()
except ImportError:
    LANGUAGE_DETECTOR = None
    print("提示: 安装 lingua-language-detector 以启用语言过滤功能")


# ============================================================
# 1. 数据加载
# ============================================================

def load_data_from_jsonl(file_path: str) -> Dataset:
    """从 JSONL 文件加载数据，每行一个 JSON 对象。"""
    data = []
    with open(file_path, "r", encoding="utf-8") as f:
        for line in f:
            line = line.strip()
            if line:
                data.append(json.loads(line))
    return Dataset.from_list(data)


def load_data_from_json(file_path: str) -> Dataset:
    """从单个 JSON 文件加载数据（JSON 数组）。"""
    with open(file_path, "r", encoding="utf-8") as f:
        data = json.load(f)
    if not isinstance(data, list):
        data = [data]
    return Dataset.from_list(data)


def load_data(source: str, source_type: str = "auto") -> Dataset:
    """
    统一的数据加载接口，支持多种格式。

    参数:
        source: 文件路径、目录路径或 HuggingFace 数据集名称
        source_type: "auto"(自动检测), "jsonl", "json", "csv", "parquet", "hf"

    返回:
        HuggingFace Dataset 对象
    """
    if source_type == "hf":
        return load_dataset(source, split="train")

    path = Path(source)

    if source_type == "auto":
        if path.is_dir():
            # 加载目录中的所有 JSONL 文件
            datasets_list = []
            for f in sorted(path.glob("*.jsonl")):
                datasets_list.append(load_data_from_jsonl(str(f)))
            for f in sorted(path.glob("*.json")):
                datasets_list.append(load_data_from_json(str(f)))
            if datasets_list:
                return concatenate_datasets(datasets_list)
            else:
                raise FileNotFoundError(f"目录 {source} 中未找到 JSONL/JSON 文件")
        elif path.suffix == ".jsonl":
            return load_data_from_jsonl(str(path))
        elif path.suffix == ".json":
            return load_data_from_json(str(path))
        else:
            raise ValueError(f"不支持的文件格式: {path.suffix}")

    elif source_type == "jsonl":
        return load_data_from_jsonl(source)
    elif source_type == "json":
        return load_data_from_json(source)
    else:
        raise ValueError(f"不支持的数据源类型: {source_type}")


# ============================================================
# 2. 规则过滤
# ============================================================

@dataclass
class FilterConfig:
    """过滤规则配置。"""
    min_instruction_chars: int = 10        # 指令最少字符数
    max_instruction_chars: int = 4000      # 指令最多字符数
    min_response_chars: int = 20           # 回复最少字符数
    max_response_chars: int = 8000         # 回复最多字符数
    target_language: Optional[str] = None  # 目标语言（如 "CHINESE"），None 表示不限制
    filter_urls: bool = True               # 是否过滤含 URL 的回复（不稳定的内容）
    filter_code_blocks: bool = False       # 是否过滤含代码块的文本（如果不需要代码能力）
    filter_repetitive: bool = True         # 是否过滤高度重复的回复
    min_response_unique_ratio: float = 0.3 # 回复的独特 token 比例低于此值则过滤
    blacklist_patterns: list = field(default_factory=list)  # 黑名单正则模式


def char_length_filter(example: dict, config: FilterConfig) -> bool:
    """长度过滤。"""
    instruction = example.get("instruction", example.get("user", ""))
    response = example.get("output", example.get("response", example.get("assistant", "")))

    if len(instruction.strip()) < config.min_instruction_chars:
        return False
    if len(instruction.strip()) > config.max_instruction_chars:
        return False
    if len(response.strip()) < config.min_response_chars:
        return False
    if len(response.strip()) > config.max_response_chars:
        return False
    return True


def language_filter(example: dict, config: FilterConfig) -> bool:
    """语言过滤。检测指令的语言是否符合目标语言。"""
    if config.target_language is None or LANGUAGE_DETECTOR is None:
        return True

    instruction = example.get("instruction", example.get("user", ""))
    if len(instruction.strip()) < 20:
        return True  # 太短的文本语言检测不可靠，不过滤

    result = LANGUAGE_DETECTOR.compute_language_confidence(instruction.strip())
    if result is None:
        return True  # 无法检测，保留

    language, confidence = result
    return language.name == config.target_language and confidence > 0.7


def url_filter(example: dict, config: FilterConfig) -> bool:
    """URL 过滤。过滤回复中包含 URL 的数据。"""
    if not config.filter_urls:
        return True
    response = example.get("output", example.get("response", example.get("assistant", "")))
    # 匹配 http/https URL
    url_pattern = re.compile(r'https?://\S+')
    return not bool(url_pattern.search(response))


def repetitive_filter(example: dict, config: FilterConfig) -> bool:
    """重复内容过滤。检测回复中是否有过度的重复模式。"""
    if not config.filter_repetitive:
        return True
    response = example.get("output", example.get("response", example.get("assistant", "")))

    # 将回复分词（简易的中文分词：按字符切分）
    tokens = list(response)
    if len(tokens) == 0:
        return False

    unique_ratio = len(set(tokens)) / len(tokens)
    return unique_ratio >= config.min_response_unique_ratio


def blacklist_filter(example: dict, config: FilterConfig) -> bool:
    """黑名单模式过滤。"""
    if not config.blacklist_patterns:
        return True
    instruction = example.get("instruction", example.get("user", ""))
    response = example.get("output", example.get("response", example.get("assistant", "")))
    combined = instruction + " " + response
    for pattern in config.blacklist_patterns:
        if re.search(pattern, combined):
            return False
    return True


def apply_filters(dataset: Dataset, config: FilterConfig) -> Dataset:
    """
    依次应用所有过滤规则。

    返回:
        过滤后的 Dataset 和一个统计摘要
    """
    initial_count = len(dataset)

    dataset = dataset.filter(lambda x: char_length_filter(x, config),
                             desc="长度过滤")
    after_length = len(dataset)

    dataset = dataset.filter(lambda x: language_filter(x, config),
                             desc="语言过滤")
    after_lang = len(dataset)

    dataset = dataset.filter(lambda x: url_filter(x, config),
                             desc="URL过滤")
    after_url = len(dataset)

    dataset = dataset.filter(lambda x: repetitive_filter(x, config),
                             desc="重复内容过滤")
    after_repetitive = len(dataset)

    dataset = dataset.filter(lambda x: blacklist_filter(x, config),
                             desc="黑名单过滤")
    after_blacklist = len(dataset)

    print(f"\n===== 过滤统计 =====")
    print(f"初始数据量:                {initial_count:>8}")
    print(f"长度过滤后:                {after_length:>8}  (-{initial_count - after_length})")
    print(f"语言过滤后:                {after_lang:>8}  (-{after_length - after_lang})")
    print(f"URL过滤后:                 {after_url:>8}  (-{after_lang - after_url})")
    print(f"重复内容过滤后:            {after_repetitive:>8}  (-{after_url - after_repetitive})")
    print(f"黑名单过滤后:              {after_blacklist:>8}  (-{after_repetitive - after_blacklist})")
    print(f"最终保留率:                {after_blacklist/initial_count*100:.1f}%\n")

    return dataset

# 3. 质量评分
# ============================================================

def compute_quality_score(example: dict) -> dict:
    """
    基于规则计算质量评分（0-100）。

    评分维度：
    - 指令信息量：指令的词数/复杂度
    - 回复信息量：回复的长度和实体密度
    - 回复多样性：独特字符占比
    - 结构完整性：是否有明显的格式问题
    """
    instruction = example.get("instruction", example.get("user", ""))
    response = example.get("output", example.get("response", example.get("assistant", "")))

    score = 0.0

    # 维度1：指令信息量（20分）
    inst_len = len(instruction.strip())
    if inst_len >= 50:
        score += 20
    elif inst_len >= 20:
        score += 15
    elif inst_len >= 10:
        score += 10
    else:
        score += 5

    # 维度2：回复信息量（30分）
    resp_len = len(response.strip())
    if resp_len >= 500:
        score += 30
    elif resp_len >= 200:
        score += 25
    elif resp_len >= 100:
        score += 20
    elif resp_len >= 50:
        score += 15
    else:
        score += 5

    # 维度3：回复多样性（20分）
    if len(response.strip()) > 0:
        unique_ratio = len(set(list(response))) / len(list(response))
        if unique_ratio > 0.6:
            score += 20
        elif unique_ratio > 0.4:
            score += 15
        elif unique_ratio > 0.2:
            score += 10
        else:
            score += 5

    # 维度4：结构完整性（30分）
    has_punctuation = bool(re.search(r'[。！？，、；：""''（）]', response))
    has_paragraphs = '\n' in response
    is_proper_length = 50 <= resp_len <= 3000

    score += 10 if has_punctuation else 0
    score += 10 if has_paragraphs else 0
    score += 10 if is_proper_length else 0

    return {"quality_score": min(score, 100)}


def filter_by_quality(dataset: Dataset, min_score: float = 40.0) -> Dataset:
    """过滤掉质量分数低于阈值的样本。"""
    dataset = dataset.map(compute_quality_score, desc="计算质量分数")
    before = len(dataset)
    dataset = dataset.filter(lambda x: x["quality_score"] >= min_score,
                             desc="质量过滤")
    after = len(dataset)
    print(f"质量过滤: {before} -> {after} (移除 {before - after} 条低质量数据)")
    return dataset


# ============================================================
# 4. 近似去重（基于 MinHash LSH）
# ============================================================

def create_minhash_signature(text: str, num_perm: int = 128) -> MinHash:
    """
    为文本创建 MinHash 签名。

    MinHash 是一种用于快速估计两个集合之间 Jaccard 相似度的算法。
    它通过多个哈希函数将集合"压缩"为固定大小的签名，
    两个签名的相似程度可以近似衡量原始集合的 Jaccard 相似度。
    """
    m = MinHash(num_perm=num_perm)
    # 使用字符级 n-gram（中文用 1-3 gram）
    for n in range(1, 4):
        for i in range(len(text) - n + 1):
            m.update(text[i:i+n].encode('utf-8'))
    return m


def deduplicate_with_minhash(
    dataset: Dataset,
    threshold: float = 0.85,
    num_perm: int = 128,
) -> Dataset:
    """
    使用 MinHash + LSH 进行近似去重。

    参数:
        dataset: 输入数据集
        threshold: Jaccard 相似度阈值，超过此值视为重复
        num_perm: MinHash 的排列数，越大越精确但越慢

    返回:
        去重后的数据集
    """
    print(f"开始近似去重（threshold={threshold}）...")

    # 创建 LSH 索引
    lsh = MinHashLSH(threshold=threshold, num_perm=num_perm)

    kept_indices = []
    duplicate_count = 0

    for i, example in enumerate(tqdm(dataset, desc="MinHash去重")):
        instruction = example.get("instruction", example.get("user", ""))
        response = example.get("output", example.get("response", example.get("assistant", "")))
        combined = instruction + " " + response

        m = create_minhash_signature(combined, num_perm)

        # 查询是否有相似的文档
        result = lsh.query(m)
        if len(result) == 0:
            # 没有重复
            lsh.insert(i, m)
            kept_indices.append(i)
        else:
            duplicate_count += 1

    print(f"去重完成: 移除 {duplicate_count} 条近似重复数据，"
          f"保留 {len(kept_indices)} 条")

    return dataset.select(kept_indices)

# 5. 格式化为 Chat Template
# ============================================================

def format_to_chat_template(example: dict, system_prompt: str = "") -> dict:
    """
    将原始数据转换为标准的 messages 格式。

    输入格式（兼容多种字段名）：
    - instruction / user: 用户指令
    - output / response / assistant: 期望回复
    - system: 系统指令（可选）

    输出格式：
    {"messages": [{"role": "system", "content": "..."},
                  {"role": "user", "content": "..."},
                  {"role": "assistant", "content": "..."}]}
    """
    instruction = example.get("instruction", example.get("user", ""))
    response = example.get("output", example.get("response", example.get("assistant", ""))
    sys_prompt = example.get("system", system_prompt)

    messages = []

    if sys_prompt:
        messages.append({"role": "system", "content": sys_prompt})

    messages.append({"role": "user", "content": instruction.strip()})
    messages.append({"role": "assistant", "content": response.strip()})

    return {"messages": messages}

def sample_by_category(
    dataset: Dataset,
    category_field: str,
    target_ratios: dict[str, float],
    total_samples: Optional[int] = None,
    seed: int = 42,
) -> Dataset:
    """
    按类别进行配比采样。

    参数:
        dataset: 包含类别字段的数据集
        category_field: 类别字段名（如 "task_type"）
        target_ratios: 目标配比（如 {"qa": 0.4, "code": 0.3, "chat": 0.3}）
        total_samples: 目标总样本数，None 表示按最少类别能提供的数量
        seed: 随机种子

    返回:
        配比后的数据集
    """
    random.seed(seed)

    # 按类别分组
    categories = {}
    for example in dataset:
        cat = example.get(category_field, "other")
        if cat not in categories:
            categories[cat] = []
        categories[cat].append(example)

    print(f"\n===== 数据配比 =====")
    for cat, samples in categories.items():
        print(f"  {cat}: {len(samples)} 条")

    # 计算每个类别的目标数量
    if total_samples is None:
        # 根据配比和最小类别数量自动计算
        min_available = min(len(v) for v in categories.values())
        total_samples = int(min_available / min(target_ratios.values()))

    sampled_data = []
    for cat, ratio in target_ratios.items():
        n_needed = int(total_samples * ratio)
        available = categories.get(cat, [])
        n_actual = min(n_needed, len(available))
        sampled = random.sample(available, n_actual)
        sampled_data.extend(sampled)
        print(f"  {cat}: 目标 {n_needed} -> 实际采样 {n_actual}")

    random.shuffle(sampled_data)
    return Dataset.from_list(sampled_data)

# 7. 最终质检报告
# ============================================================

def generate_quality_report(dataset: Dataset) -> dict:
    """
    生成数据集的质量报告。

    包含：
    - 总数据量
    - 指令长度分布
    - 回复长度分布
    - System prompt 覆盖率
    - 多轮对话占比
    - 字符编码检查
    """
    total = len(dataset)
    if total == 0:
        return {"error": "空数据集"}

    inst_lengths = []
    resp_lengths = []
    has_system = 0
    multi_turn = 0
    encoding_issues = 0

    for example in dataset:
        # 兼容多种字段名
        messages = example.get("messages", [example])
        if isinstance(messages, list):
            # messages 格式
            user_content = ""
            assistant_content = ""
            has_sys = False
            for msg in messages:
                if msg["role"] == "system":
                    has_sys = True
                elif msg["role"] == "user":
                    user_content += msg["content"]
                elif msg["role"] == "assistant":
                    assistant_content += msg["content"]
            if has_sys:
                has_system += 1
            # 检查是否多轮（超过3条消息）
            if len([m for m in messages if m["role"] in ("user", "assistant")]) > 2:
                multi_turn += 1
        else:
            user_content = example.get("instruction", example.get("user", ""))
            assistant_content = example.get("output", example.get("response", example.get("assistant", ""))

        inst_lengths.append(len(user_content))
        resp_lengths.append(len(assistant_content))

        # 检查编码问题
        try:
            user_content.encode('utf-8')
            assistant_content.encode('utf-8')
        except UnicodeEncodeError:
            encoding_issues += 1

    # 计算长度分布
    def percentile(data, p):
        sorted_data = sorted(data)
        idx = int(len(sorted_data) * p / 100)
        return sorted_data[min(idx, len(sorted_data) - 1)]

    report = {
        "总数据量": total,
        "指令长度": {
            "平均": sum(inst_lengths) / total,
            "中位数": percentile(inst_lengths, 50),
            "P10": percentile(inst_lengths, 10),
            "P90": percentile(inst_lengths, 90),
            "最小": min(inst_lengths),
            "最大": max(inst_lengths),
        },
        "回复长度": {
            "平均": sum(resp_lengths) / total,
            "中位数": percentile(resp_lengths, 50),
            "P10": percentile(resp_lengths, 10),
            "P90": percentile(resp_lengths, 90),
            "最小": min(resp_lengths),
            "最大": max(resp_lengths),
        },
        "System Prompt 覆盖率": f"{has_system/total*100:.1f}%",
        "多轮对话占比": f"{multi_turn/total*100:.1f}%",
        "编码问题数量": encoding_issues,
    }

    return report

def run_data_pipeline(

    """
    运行完整的数据准备 Pipeline。

    参数:
        data_source: 数据源路径或 HuggingFace 数据集名称
        output_path: 输出文件路径（JSONL）
        system_prompt: 全局 system prompt（如果原始数据没有指定）
        config: 过滤规则配置
        min_quality_score: 最低质量分数
        dedup_threshold: 去重相似度阈值

    返回:
        处理后的 Dataset
    """
    if config is None:
        config = FilterConfig()

    print("=" * 50)
    print("  指令微调数据准备 Pipeline")
    print("=" * 50)

    # Step 1: 加载数据
    print("\n[1/6] 加载数据...")
    dataset = load_data(data_source)

    # Step 2: 规则过滤
    print(f"\n[2/6] 规则过滤...")
    dataset = apply_filters(dataset, config)

    # Step 3: 质量评分
    print(f"\n[3/6] 质量评分...")
    dataset = filter_by_quality(dataset, min_quality_score)

    dataset = deduplicate_with_minhash(dataset, dedup_threshold)

    report = generate_quality_report(dataset)

    output_dir.mkdir(parents=True, exist_ok=True)
    dataset.to_json(output_path, orient="records", lines=True, force_ascii=False)
    print(f"\n处理完成！数据已保存至 {output_path}")

    return dataset
```

## 常见错误

1. **不检查原始数据质量就直接微调**：很多初学者拿到开源数据集就直接训练，结果模型效果很差。切记：数据质量比数据数量更重要。1000 条高质量数据的效果往往优于 10000 条低质量数据。

2. **Chat Template 用错**：训练时用了 Qwen 的模板，推理时却用了 LLaMA 的模板，导致模型输出混乱。解决方案：始终使用 `tokenizer.apply_chat_template()` 来自动应用正确的模板。

3. **System prompt 不一致**：训练数据中有的有 system prompt、有的没有、有的内容不同。训练时模型会学到"system prompt 不重要，可以忽略"，导致推理时无法通过 system prompt 控制模型行为。解决方案：训练时统一 system prompt，或在推理时使用与训练时一致的 system prompt。

4. **去重阈值设得太高**：把所有"稍微相似"的数据都去掉了，导致训练数据缺乏多样性，模型过拟合到少数模式上。一般来说，去重阈值 0.85-0.9 比较合适。

5. **回复中保留了"作为AI语言模型"等拒绝话术**：微调数据中不应该包含模型拒绝回答的样本，除非你专门希望模型学会拒绝。应该将拒绝话术替换为正面回答。

6. **数据中混入了非目标语言**：中英文混合数据如果没有经过语言过滤，会导致模型在中文任务中时不时输出英文，或者中英混杂。

7. **训练和测试数据有重叠**：去重时没有检查训练集和测试集之间的重叠，导致测试集评估结果不可靠，"高分"只是过拟合的表现。

8. **只使用单轮对话数据**：模型无法学会在多轮上下文中正确理解和响应，导致多轮对话效果差。

9. **数据配比与实际使用场景不匹配**：训练数据 80% 是代码，但实际部署时 90% 的请求是通用问答——模型会"说代码"。配比应该反映预期使用场景的分布。

10. **忽略了特殊 token 的正确使用**：Chat Template 中的 `<|im_start|>`、`<|im_end|>` 等特殊 token 需要在 tokenizer 中正确注册。如果在 `tokenizer_config.json` 中这些 token 没有被正确定义，微调后的模型会行为异常。
