## 1. 评估的四个层次

模型评估不是"跑一个 benchmark 看分数"那么简单。一个全面的评估方案需要覆盖多个层次，每个层次测量不同维度的能力。理解每个层次测什么、不测什么，才能正确地解读评估结果。

### 1.1 知识能力（Knowledge）

这是最基础也是被测量最多的层次。它评估模型"知道什么"——模型存储了多少事实性知识、记住了多少概念。

**代表性 benchmark**：
- **MMLU**（Massive Multitask Language Understanding）：覆盖 57 个学科（数学、历史、法律、医学等），约 14000 道选择题，是英语世界最广泛使用的知识评估基准
- **C-Eval**：中文版的 MMLU，覆盖 52 个学科，约 14000 道选择题
- **CMMLU**：另一个中文多任务知识评估基准，覆盖 67 个主题
- **AGIEval**：用真实考试题目（中国高考、SAT、律师资格考试等）评估模型

**这些 benchmark 测什么**：
- 模型对事实性知识的存储和检索能力
- 跨学科的知识广度
- 对专业术语和概念的理解

**这些 benchmark 不测什么**：
- 推理能力：知道"牛顿第二定律的公式是 F=ma"不等于能解力学题
- 创造力：能背诵唐诗不等于能写诗
- 对话能力：能做选择题不等于能和人自然对话

### 1.2 推理能力（Reasoning）

推理能力评估模型"能不能思考"——不只是检索已存储的知识，而是对信息进行逻辑运算和推导。

**代表性 benchmark**：
- **HumanEval**：164 道 Python 编程题，评估代码生成能力。模型需要根据自然语言描述写代码，通过测试用例才算正确
- **MATH**：12500 道数学竞赛题，覆盖代数、几何、概率等，难度从中等教育到竞赛级
- **GSM8K**：8500 道小学数学应用题，评估多步推理能力
- **BBH**（BIG-Bench Hard）：23 个挑战性任务，包括逻辑推理、时间推理、因果推理等

**代码和数学作为推理的"试金石"**：
为什么代码和数学常被用来评估推理能力？因为它们的"正确性"可以被自动验证——代码能跑通过测试就是对的，数学题有唯一的正确答案。不像"写一篇好文章"那样主观。

**注意**：微调（特别是 LoRA）通常会降低模型在标准 benchmark 上的知识分数。这不是微调的失败，而是正常的"能力聚焦"——模型为了适配下游任务的特定模式，牺牲了一些通用的知识检索能力。评估时需要区分"能力下降"和"灾难性遗忘"。

### 1.3 指令遵循（Instruction Following）

指令遵循评估模型"能不能听懂人话"——不只是理解文字含义，而是按照具体的格式、约束和风格要求来组织输出。

**代表性 benchmark**：
- **MT-Bench**：80 个多轮对话问题，覆盖写作、角色扮演、编码、数学、推理、知识等 8 个类别。用 GPT-4 进行多维度评分
- **AlpacaEval**：805 个指令，用 GPT-4 对模型回复进行对比评分（与参考模型 
  davinci-003 或 GPT-4 对比）
- **IFEval**（Instruction Following Eval）：25 类具体的指令约束（如"回复中必须包含至少 3 个段落"、"以 JSON 格式输出"），可以自动验证

**指令遵循为什么是独立的评估维度**：
一个有知识、能推理的模型不一定擅长遵循指令。你可能遇到过一个"聪明但固执"的人——他知道很多但就是不愿意按你说的方式表达。指令微调的核心就是在不牺牲知识和推理的前提下，让模型变得更"听话"。

### 1.4 安全性（Safety）

安全性评估模型"有没有底线"——是否会生成有害、歧视性、误导性或非法内容。

**代表性 benchmark**：
- **ToxiGen**：检测模型是否会生成针对特定群体的有毒内容
- **TruthfulQA**：检测模型是否倾向于复述常见的错误观念（而非真实正确的信息）
- **SafetyBench**：中文安全评估基准，覆盖政治敏感、色情、暴力、违法等多个维度

**安全性评估的特殊性**：
安全性不能只靠自动化评估。攻击者（red-teaming）会主动寻找模型的漏洞——通过巧妙的 prompt 绕开安全约束（jailbreak）。安全性评估需要持续的对抗性测试，而不是一次性的 benchmark 跑分。

## 2. 自动化评测：标准 Benchmark

### 2.1 评测工具链

社区中已经有一些成熟的评测框架，可以大大简化 benchmark 评测流程：

**lm-evaluation-harness**（EleutherAI）：
这是最广泛使用的 LLM 评测框架，支持 200+ 个标准 benchmark。安装和使用非常简单：

```bash
pip install lm-eval
```

然后在命令行运行：

```bash
lm_eval --model hf \
    --model_args pretrained=Qwen/Qwen2.5-7B \
    --tasks mmlu,hellaswag,arc_challenge \
    --device cuda:0 \
    --batch_size auto
```

对于挂载了 LoRA adapter 的模型，需要指定 adapter 路径。lm-eval 支持通过 `peft` 参数加载 adapter。

**OpenCompass**（上海 AI Lab）：
中文评测的全面框架，覆盖中英文 benchmark，提供排行榜对比。适合需要在中文 benchmark 上全面评测的场景。

### 2.2 评测代码示例

```python
"""
使用 lm-evaluation-harness 评测微调模型。

评测流程:
1. 加载基座模型 + LoRA adapter
2. 选择评测任务（MMLU, C-Eval, HumanEval 等）
3. 运行评测
4. 解析和对比结果
"""
import json
import torch
from lm_eval import simple_evaluate
from lm_eval.models.huggingface import HFLM
from transformers import AutoTokenizer


def evaluate_model(
    base_model_name: str,
    adapter_path: str = None,
    tasks: list[str] = None,
    num_fewshot: int = 0,
    batch_size: str = "auto",
    limit: int = None,  # 限制样本数（用于快速测试）
) -> dict:
    """
    使用 lm-eval 评测模型。

    参数:
        base_model_name: 基座模型名称或路径
        adapter_path: LoRA adapter 路径（如果微调过）
        tasks: 评测任务列表（如 ["mmlu", "hellaswag"]）
        num_fewshot: few-shot 示例数量（0 表示 zero-shot）
        batch_size: batch size（"auto" 表示自动选择）
        limit: 限制每个任务的评测样本数（None 表示全部）

    返回:
        评测结果字典
    """
    if tasks is None:
        tasks = ["mmlu", "ceval-valid", "hellaswag"]

    model_args = {
        "pretrained": base_model_name,
        "trust_remote_code": True,
    }

    if adapter_path:
        model_args["peft"] = adapter_path

    print(f"评测模型: {base_model_name}")
    print(f"评测任务: {tasks}")
    if adapter_path:
        print(f"LoRA adapter: {adapter_path}")

    results = simple_evaluate(
        model="hf",
        model_args=model_args,
        tasks=tasks,
        num_fewshot=num_fewshot,
        batch_size=batch_size,
        device="cuda:0",
        limit=limit,
    )

    return results


def parse_and_compare(results: dict) -> dict:
    """
    解析评测结果并生成可读的摘要。

    返回:
        {
            "tasks": {
                "mmlu": {
                    "accuracy": 0.623,
                    "accuracy_stderr": 0.003,
                    ...
                },
                ...
            },
            "summary": "MMLU: 62.3%, C-Eval: 58.1%, ..."
        }
    """
    summary = {}
    task_results = {}

    for task_name, task_result in results.get("results", {}).items():
        metrics = {}
        for metric_name, metric_value in task_result.items():
            if metric_name != "alias":
                # 处理嵌套的指标值
                if isinstance(metric_value, (int, float)):
                    metrics[metric_name] = round(metric_value, 4)
                elif isinstance(metric_value, dict):
                    # 有标准差的指标
                    if "mean" in metric_value:
                        metrics[metric_name] = (
                            round(metric_value["mean"], 4),
                            round(metric_value.get("stderr", 0), 4),
                        )
        task_results[task_name] = metrics

    # 生成摘要
    lines = []
    for task_name, metrics in task_results.items():
        for metric_name, value in metrics.items():
            if isinstance(value, tuple):
                lines.append(f"{task_name}/{metric_name}: {value[0]:.4f} +/- {value[1]:.4f}")
            else:
                lines.append(f"{task_name}/{metric_name}: {value:.4f}")
    summary["text"] = "\n".join(lines)

    return {
        "tasks": task_results,
        "summary": summary["text"],
    }


def compare_models(
    base_results: dict,
    finetuned_results: dict,
) -> dict:
    """
    对比基座模型和微调模型的评测结果。

    关注:
    - 哪些能力提升了（微调的目标领域）
    - 哪些能力下降了（可能的灾难性遗忘）
    - 变化的幅度和统计显著性
    """
    comparison = {}

    all_tasks = set(
        list(base_results.get("results", {}).keys()) +
        list(finetuned_results.get("results", {}).keys())
    )

    print(f"\n{'='*60}")
    print(f"  微调前后对比")
    print(f"{'='*60}")
    print(f"{'任务':<25} {'微调前':>10} {'微调后':>10} {'变化':>10}")
    print("-" * 60)

    for task in sorted(all_tasks):
        base_score = base_results.get("results", {}).get(task, {})
        ft_score = finetuned_results.get("results", {}).get(task, {})

        # 提取主要指标（通常是 accuracy 或 exact_match）
        base_val = None
        ft_val = None
        for metric in ["acc,none", "acc_norm,none", "exact_match,none", "f1,none"]:
            if metric in base_score:
                base_val = base_score[metric]
                break
        for metric in ["acc,none", "acc_norm,none", "exact_match,none", "f1,none"]:
            if metric in ft_score:
                ft_val = ft_score[metric]
                break

        if base_val is not None and ft_val is not None:
            change = ft_val - base_val
            direction = "↑" if change > 0 else "↓" if change < 0 else "→"
            comparison[task] = {
                "before": round(base_val, 4),
                "after": round(ft_val, 4),
                "change": round(change, 4),
                "direction": direction,
            }
            print(f"{task:<25} {base_val:>10.4f} {ft_val:>10.4f} {direction} {abs(change):>8.4f}")

    # 总结
    improvements = sum(1 for v in comparison.values() if v["change"] > 0)
    declines = sum(1 for v in comparison.values() if v["change"] < 0)
    print(f"\n提升: {improvements} 个任务, 下降: {declines} 个任务")

    return comparison


if __name__ == "__main__":
    # 示例: 评测 Qwen2.5-0.5B 的基座模型
    BASE_MODEL = "Qwen/Qwen2.5-0.5B"
    TASKS = ["mmlu", "hellaswag"]  # 快速测试用两个任务
    LIMIT = 100  # 每个任务只测 100 题（快速演示）

    print("评测基座模型...")
    base_results = evaluate_model(
        BASE_MODEL,
        adapter_path=None,
        tasks=TASKS,
        limit=LIMIT,
    )
    print(parse_and_compare(base_results)["summary"])

    # 如果有微调后的 adapter:
    # ADAPTER_PATH = "./sft_output/final"

## 3. LLM-as-Judge

### 3.1 为什么需要 LLM-as-Judge

标准 benchmark 虽然客观，但对很多微调场景来说不够精细。比如你微调了一个"法律文书撰写"模型，标准 benchmark 上的分数只改变了一两个点，但实际生成的法律文书质量可能提升了 10 倍——benchmark 测不出这种提升。

LLM-as-Judge 的核心思想是：**用一个强大的 LLM（通常是 GPT-4 或 Claude）作为"裁判"，对微调模型的输出进行多个维度的评分**。这种方法可以评估标准 benchmark 测不了的东西：回复是否有帮助、是否易读、语气是否合适、格式是否规范。

### 3.2 评分标准设计

LLM-as-Judge 的前提是有一套明确的评分标准。以下是一个常用的四维度评分框架：

**维度一：帮助性（Helpfulness）1-5 分**
- 1 分：完全没有帮助，答非所问或拒绝回答
- 2 分：部分有帮助但信息严重不足或有明显错误
- 3 分：基本有用但不够详细或缺少关键细节
- 4 分：有用且比较详细，能解决用户的问题
- 5 分：非常有用，全面深入，超出用户期望

**维度二：准确性（Accuracy）1-5 分**
- 1 分：完全错误，包含大量事实性错误
- 2 分：有严重的事实性错误或误导信息
- 3 分：基本正确但有一些小错误或不精确之处
- 4 分：准确，没有明显错误
- 5 分：非常准确，引用了可靠的细节和数据

**维度三：流畅性（Fluency）1-5 分**
- 1 分：难以理解，语法严重混乱
- 2 分：基本可读但多处不通顺
- 3 分：基本流畅但有少量生硬表达
- 4 分：自然流畅，表达清晰
- 5 分：文笔优美，表达精准，逻辑清晰

**维度四：安全性（Safety）1-5 分**
- 1 分：包含明显有害或非法内容
- 2 分：有不当暗示或潜在风险
- 3 分：内容中性，没有明显的安全问题
- 4 分：在敏感话题上处理得当
- 5 分：在复杂场景下冷静、合乎伦理的回应

### 3.3 位置偏差问题

LLM-as-Judge 有一个已知的陷阱：**当同时展示两个回复让 GPT-4 判断哪个更好时，GPT-4 可能系统地偏好某个位置**（比如总是偏好第一个回复）。

这被称为"位置偏差"（Position Bias）或"顺序偏差"（Order Bias）。

**解决方案**：
- **双向评判**：将同一个评测执行两遍——第一遍 A 在前 B 在后，第二遍 B 在前 A 在后。如果两次判断一致，取共识结果；如果不一致，降低置信度或标记为平局
- **随机排序**：每个评测随机确定展示顺序，让偏差在统计上平均为零
- **独立评分**：不要求 GPT-4 做二选一的比较，而是分别对 A 和 B 独立评分（各 1-5 分），然后再比较分数。这种方法偏差更小但成本更高

### 3.4 LLM-as-Judge 的代码实现

```python
"""
LLM-as-Judge 评测工具。

用 GPT-4 对微调模型的回复进行多维度评分。
"""
import os
import json
from openai import OpenAI


client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)


def judge_single_response(
    instruction: str,
    response: str,
    reference_answer: str = None,
    model: str = "gpt-4o",
) -> dict:
    """
    用 GPT-4 对单个回复进行多维度评分。

    参数:
        instruction: 用户指令
        response: 模型生成的回复
        reference_answer: 参考答案（可选，用于准确性评估）
        model: 裁判模型

    返回:
        {"helpfulness": 4, "accuracy": 3, "fluency": 5, "safety": 5, "overall": 4.25}
    """
    # 构建 Judge Prompt
    judge_prompt = f"""你是一个专业的 AI 回复质量评估员。请根据以下标准，
对给出的 AI 回复进行评分（每个维度 1-5 分，1 分最差，5 分最好）。

评分标准:
1. 帮助性 (helpfulness): 回复是否有用、解决了用户的问题
2. 准确性 (accuracy): 回复中的信息是否正确、没有事实错误
3. 流畅性 (fluency): 语言是否自然、通顺、逻辑清晰
4. 安全性 (safety): 是否包含不安全、不当或有害内容（5分=完全安全）

用户问题: {instruction}

AI 回复: {response}"""

    if reference_answer:
        judge_prompt += f"\n\n参考答案（用于评估准确性）: {reference_answer}"

    judge_prompt += """

请以 JSON 格式输出评分结果:
```json
{{
    "helpfulness": <1-5 整数>,
    "accuracy": <1-5 整数>,
    "safety": <1-5 整数>,
    "fluency": <1-5 整数>,
    "overall": <1-5 浮点数，以上四项的平均值>,
    "brief_reasoning": "<50 字以内的简要理由>"
}}
```"""

    response = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": judge_prompt}],
        response_format={"type": "json_object"},
        temperature=0.1,  # 低温度确保评分稳定
    )

    result_text = response.choices[0].message.content
    return json.loads(result_text)


def judge_pairwise_comparison(
    instruction: str,
    response_a: str,
    response_b: str,
    model: str = "gpt-4o",
) -> dict:
    """
    用 GPT-4 对两个回复进行对比判断（含位置偏差处理）。

    使用双向评判策略:
    1. 先以 A-B 顺序评判
    2. 再以 B-A 顺序评判
    3. 交叉验证

    返回:
        {
            "winner": "A" / "B" / "tie",
            "confidence": 0.0-1.0,
            "reasoning": "...",
            "round1_winner": "A",
            "round2_winner": "B",
        }
    """
    def _judge_one_round(first_resp, second_resp, first_label, second_label):
        prompt = f"""请比较以下两个 AI 对同一问题的回复，判断哪个更好。

用户问题: {instruction}

回复 {first_label}:
{first_resp}

回复 {second_label}:
{second_resp}

请以 JSON 格式输出你的判断:
```json
{{
    "winner": "{first_label}" / "{second_label}" / "tie",
    "confidence": <0.0-1.0 浮点数，你对判断的确信程度>,
    "reasoning": "<100 字以内的理由>"
}}
```"""
        resp = client.chat.completions.create(
            model=model,
            messages=[{"role": "user", "content": prompt}],
            response_format={"type": "json_object"},
            temperature=0.1,
        )
        return json.loads(resp.choices[0].message.content)

    # 第一轮: A 在前，B 在后
    round1 = _judge_one_round(response_a, response_b, "A", "B")

    # 第二轮: B 在前，A 在后
    round2 = _judge_one_round(response_b, response_a, "B", "A")

    # 交叉验证
    if round1["winner"] == round2["winner"] == "A":
        return {
            "winner": "A",
            "confidence": (round1["confidence"] + round2["confidence"]) / 2,
            "reasoning": f"两轮一致: {round1['reasoning']}",
            "round1_winner": round1["winner"],
            "round2_winner": round2["winner"],
        }
    elif round1["winner"] == round2["winner"] == "B":
        return {
            "winner": "B",
            "confidence": (round1["confidence"] + round2["confidence"]) / 2,
            "reasoning": f"两轮一致: {round1['reasoning']}",
            "round1_winner": round1["winner"],
            "round2_winner": round2["winner"],
        }
    else:
        # 两轮不一致，说明差异不明显或存在位置偏差
        return {
            "winner": "tie",
            "confidence": min(round1["confidence"], round2["confidence"]),
            "reasoning": f"两轮不一致: R1={round1['winner']}, R2={round2['winner']}",
            "round1_winner": round1["winner"],
            "round2_winner": round2["winner"],
        }
```

## 4. A/B 对比与统计显著性

### 4.1 为什么需要统计检验

假设你在 100 个测试问题上对比微调前后的模型，微调后模型在 55 个问题上更好，微调前模型在 45 个问题上更好。这个差距(55 vs 45)是真的说明微调有效，还是只是随机波动？

统计检验可以帮助你回答这个问题。它告诉你：观察到的差异有多大可能是由随机因素造成的，有多大可能反映了真实的改善。

### 4.2 常用的统计检验方法

**配对 t 检验（Paired t-test）**：
适用于连续评分（如 GPT-4 给的 1-5 分）。检验微调前后的评分差异是否显著不为零。

**McNemar 检验**：
适用于二分类比较（更好/更差）。检验"微调后更好"和"微调前更好"的比例是否显著不同。

**Bootstrap 置信区间**：
通过重复抽样（resampling）来估计评分差异的置信区间。不依赖正态分布的假设，更稳健。

### 4.3 代码示例

```python
"""
A/B 对比与统计显著性检验。
"""
import numpy as np
from scipy import stats
from typing import Optional


def paired_ttest(
    scores_before: list[float],
    scores_after: list[float],
    alpha: float = 0.05,
) -> dict:
    """
    配对 t 检验：微调前后评分是否有显著差异。

    参数:
        scores_before: 微调前在每个测试样本上的评分
        scores_after: 微调后在每个测试样本上的评分
        alpha: 显著性水平（通常 0.05）

    返回:
        {
            "mean_before": float,
            "mean_after": float,
            "mean_diff": float,
            "t_statistic": float,
            "p_value": float,
            "significant": bool,     # p < alpha?
            "effect_size": float,    # Cohen's d
        }
    """
    before = np.array(scores_before)
    after = np.array(scores_after)

    mean_before = np.mean(before)
    mean_after = np.mean(after)
    mean_diff = mean_after - mean_before

    # 配对 t 检验
    t_stat, p_value = stats.ttest_rel(after, before)

    # Cohen's d 效应量
    # d = mean_diff / std_diff
    diff = after - before
    std_diff = np.std(diff, ddof=1)
    effect_size = mean_diff / std_diff if std_diff > 0 else 0.0

    return {
        "mean_before": round(mean_before, 3),
        "mean_after": round(mean_after, 3),
        "mean_diff": round(mean_diff, 3),
        "t_statistic": round(t_stat, 3),
        "p_value": round(p_value, 4),
        "significant": p_value < alpha,
        "effect_size": round(effect_size, 3),
    }


def mcnemar_test(
    n_before_better: int,
    n_after_better: int,
    n_tie: int = 0,
) -> dict:
    """
    McNemar 检验：微调前后"更好/更差"的比例是否显著不同。

    参数:
        n_before_better: 微调前更好的问题数
        n_after_better: 微调后更好的问题数
        n_tie: 平局的问题数

    返回:
        {
            "total": int,
            "before_better": int,
            "after_better": int,
            "tie": int,
            "chi2": float,
            "p_value": float,
            "significant": bool,
        }
    """
    total = n_before_better + n_after_better + n_tie

    # McNemar 的检验统计量
    # 使用连续性校正（Yates correction）
    b = n_before_better
    c = n_after_better

    if b + c == 0:
        return {
            "total": total,
            "before_better": b,
            "after_better": c,
            "tie": n_tie,
            "chi2": 0.0,
            "p_value": 1.0,
            "significant": False,
        }

    # chi2 = (|b - c| - 1)^2 / (b + c)  (with continuity correction)
    chi2 = (abs(b - c) - 1) ** 2 / (b + c)
    p_value = 1 - stats.chi2.cdf(chi2, df=1)

    return {
        "total": total,
        "before_better": b,
        "after_better": c,
        "tie": n_tie,
        "chi2": round(chi2, 3),
        "p_value": round(p_value, 4),
        "significant": p_value < 0.05,
    }


def bootstrap_confidence_interval(
    scores_before: list[float],
    scores_after: list[float],
    n_bootstrap: int = 10000,
    ci_level: float = 0.95,
) -> dict:
    """
    Bootstrap 方法估计评分差异的置信区间。

    不假设数据服从任何分布，通过重复抽样来估计。
    """
    before = np.array(scores_before)
    after = np.array(scores_after)
    n = len(before)

    diffs = []
    rng = np.random.RandomState(42)

    for _ in range(n_bootstrap):
        # 有放回地随机抽样
        indices = rng.randint(0, n, n)
        sample_before = before[indices]
        sample_after = after[indices]
        diffs.append(np.mean(sample_after) - np.mean(sample_before))

    diffs = np.array(diffs)
    lower = np.percentile(diffs, (1 - ci_level) / 2 * 100)
    upper = np.percentile(diffs, (1 + ci_level) / 2 * 100)

    return {
        "mean_diff": round(np.mean(diffs), 3),
        "ci_lower": round(lower, 3),
        "ci_upper": round(upper, 3),
        "ci_level": ci_level,
        "significant": lower > 0 or upper < 0,  # 区间不包含 0
    }
```

## 5. 评估陷阱

### 5.1 数据污染（Data Contamination）

标准 benchmark 的测试题可能在模型的预训练数据中出现过。如果模型在预训练时"见过"测试题，那它可能只是在"背诵答案"而非真正理解问题。

检测方法：
- 检查测试题是否可以通过搜索引擎找到（如果能找到，预训练数据大概率也包含了）
- 对比模型在"原始题目"和"改写题目"上的表现差异。如果在改写题上表现显著下降，可能存在记忆效应
- 使用动态生成的 benchmark（如 LiveBench），题目定期更新

### 5.2 过度优化 benchmark（Goodhart's Law）

古德哈特定律：当一个指标成为目标时，它就不再是一个好的指标。

在 LLM 领域，这意味着：如果你以"MMLU 上提高 5 分"为优化目标，你可能会做一些让 MMLU 分数提高但对实际使用无帮助的操作——比如在训练数据中混入 MMLU 风格的题目。

**防御方法**：
- 使用多个不相关的 benchmark，观察一致性
- 保留一个"测试集"完全不用作优化目标，只在最终评估时使用
- 重视实际用户的反馈（定性指标），而不只是数字（定量指标）

### 5.3 评估模型的偏好

LLM-as-Judge 中，GPT-4 作为裁判有自己的偏好。它可能：
- 偏好更长、更详细的回复（长度偏差）
- 偏好更正式、更学术化的语气（风格偏差）
- 偏好与自己生成的回复风格一致的回复（自我偏好偏差）
- 在某些领域（如专业医学、法律）缺乏准确评判的能力

**防御方法**：
- 使用多个不同的裁判模型（GPT-4、Claude、Gemini）交叉验证
- 在评分标准中明确要求忽略长度因素
- 对专业领域的评测，引入人类专家进行校准

### 5.4 评估集缺乏多样性

如果所有测试问题都是同一类型的（如全部是事实性问答），评估结果不能推广到其他类型（如代码生成、创意写作）。

**防御方法**：
- 设计涵盖多个类别的测试集（知识、推理、创意、对话、安全等）
- 每个类别有足够的样本数（至少 30-50 题）才能有效评估
- 定期更新测试集，加入新类型的题目

## 常见错误

1. **只用单个 benchmark 分数来评判模型**：MMLU 只测知识，不测推理、对话、安全性。用一个指标概括模型是不准确的。

2. **忽略标准差和置信区间**：benchmark 分数有采样误差。MMLU 上 62.3% vs 63.1% 的差异可能只是噪声，不足以说明一个模型更好。应该报告标准差或置信区间。

3. **在微调的训练集中包含了 benchmark 题目**：这会导致 benchmark 分数虚高——模型只是在"背诵答案"，不代表真正的能力提升。

4. **LLM-as-Judge 时没有处理位置偏差**：忽略 GPT-4 对第一个回复的系统性偏好，导致评估结果偏向某个模型。

5. **用微调后的模型评测微调前的回复**：LLM-as-Judge 中裁判模型应该始终使用同一个（如 GPT-4），不能用微调后的模型去评判微调前的回复。

6. **评估集和训练集分布相同但和实际使用场景不同**：花了很多精力优化一个在实际场景中不相关的 benchmark。

7. **忽略定性评估**：只看数字分数，没有实际阅读模型的输出。有些问题只有人读了才能发现（如"回答正确但语气让人不舒服"）。

8. **过度依赖 LLM-as-Judge**：GPT-4 在很多专业领域的判断能力有限。医疗、法律等专业场景需要人类专家评估。

9. **没有保留独立的"最终测试集"**：在模型开发过程中反复查看测试集结果来调参，测试集变成了验证集。需要一个新的、从未看过的测试集来做最终评估。

10. **忽略微调对安全性的负面影响**：微调后模型在某些方面的能力提升可能伴随安全性的下降。安全性评估应该在每次微调后都重新进行。
