## 1. LoRA 权重合并

### 1.1 为什么要合并

LoRA 微调后，你实际上拥有两个独立的部分：
1. **基座模型**（Base Model）：原始的预训练权重（不动）
2. **LoRA Adapter**：微调过程中学到的小矩阵（几 MB 到几百 MB）

在推理时，模型的计算过程是：`y = W*x + B*A*x`。这意味着每次前向传播都要做两次矩阵乘法（W*x 和 B*A*x），比直接用合并后的权重（W_merged*x）多了一步。

虽然这个额外开销不大（通常 <5%），但在以下场景中，合并权重是必要的或有益的：

- **部署到 Ollama**：Ollama 需要完整的模型权重文件（GGUF 格式），不支持在推理时动态加载 LoRA adapter
- **部署到 vLLM**：虽然 vLLM 支持 LoRA adapter 的热加载，但合并后可以减少显存碎片并略微提高吞吐量
- **分发模型**：给别人分享一个合并后的完整模型比分享"基座模型 + adapter"更方便
- **后续量化**：GGUF、GPTQ 等量化工具需要完整的模型权重，不能直接量化 LoRA adapter

### 1.2 合并方法

HuggingFace PEFT 库提供了非常简单的合并方法：

```python
"""
将 LoRA adapter 合并回基座模型的完整代码。
"""
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel


def merge_lora_weights(
    base_model_name: str,
    adapter_path: str,
    output_path: str,
) -> None:
    """
    将 LoRA adapter 合并到基座模型并保存完整权重。

    参数:
        base_model_name: 基座模型名称或路径
        adapter_path: LoRA adapter 路径
        output_path: 合并后模型保存路径

    流程:
    1. 加载基座模型
    2. 加载并挂载 LoRA adapter
    3. 调用 merge_and_unload() 合并权重
    4. 保存合并后的完整模型
    """
    print(f"加载基座模型: {base_model_name}")

    # 加载基座模型（以推理精度加载）
    model = AutoModelForCausalLM.from_pretrained(
        base_model_name,
        torch_dtype=torch.bfloat16,
        device_map="auto",
        trust_remote_code=True,
    )

    # 加载 tokenizer
    tokenizer = AutoTokenizer.from_pretrained(
        base_model_name,
        trust_remote_code=True,
    )

    print(f"加载 LoRA adapter: {adapter_path}")

    # 挂载 LoRA adapter
    model = PeftModel.from_pretrained(model, adapter_path)

    print("合并 LoRA 权重到基座模型...")

    # merge_and_unload() 做了什么：
    # 1. 对每个有 LoRA adapter 的线性层，计算 W_merged = W + (alpha/r) * B * A
    # 2. 将 W_merged 替换原始的 W
    # 3. 移除 LoRA adapter 结构（A 和 B 矩阵被删除）
    # 4. 返回一个普通的（非 PeftModel）模型
    model = model.merge_and_unload()

    print(f"保存合并后的模型到: {output_path}")

    # 保存完整模型（不只是 adapter）
    model.save_pretrained(output_path, safe_serialization=True)
    tokenizer.save_pretrained(output_path)

    print("合并完成！")
    print(f"模型大小约: {sum(p.numel() * 2 for p in model.parameters()) / 1e9:.2f} GB (BF16)")


if __name__ == "__main__":
    merge_lora_weights(
        base_model_name="Qwen/Qwen2.5-7B",
        adapter_path="./sft_output/final",
        output_path="./merged_model",
    )
```

### 1.3 合并的注意事项

**显存需求**：合并过程需要同时加载基座模型和 LoRA adapter。对于 7B 模型，这大约是 14GB（BF16）。合并本身是一个快速的矩阵加法操作。

**精度保持**：`merge_and_unload()` 在 BF16/FP16 精度下合并。LoRA 的 A 和 B 矩阵也是 BF16/FP16 精度。合并后的精度损失通常可以忽略不计。

**多 adapter 合并**：如果你对同一基座模型训练了多个 LoRA adapter（如一个用于医疗、一个用于法律），每个都可以独立合并。但要注意：两个 adapter 不能同时合并到同一个模型——它们是互斥的增量。

**不可逆操作**：合并后 LoRA adapter 的 A 和 B 矩阵被删除。如果你想保留单独 adapter 的灵活性，在合并前先备份 adapter 权重。

## 2. 模型量化

### 2.1 为什么需要量化

微调后的模型即使经过 LoRA 合并，仍然占据大量存储和显存。以 7B 模型为例：
- BF16/FP16：约 14GB
- FP32：约 28GB

对于本地部署（单张消费级 GPU）或边缘设备部署，14GB 仍然太大了。量化可以将模型压缩到 1/4 甚至更小的体积，同时尽可能保持模型质量。

### 2.2 GGUF 格式：Ollama 的量化方案

**GGUF**（GGML Universal Format）是 llama.cpp 项目使用的量化格式，也是 Ollama 背后的核心技术。

**GGUF 的特点**：
- 将模型权重和所有元数据（tokenizer、配置文件、Chat Template）打包到一个文件中
- 支持多种量化级别：Q2_K、Q3_K_S、Q4_K_M、Q5_K_M、Q6_K、Q8_0 等
- 可以在 CPU 上高效推理（通过 llama.cpp 的优化）
- 支持 GPU 加速（通过 CUDA/Metal/Vulkan 后端）
- 文件大小和质量的权衡可以通过量化级别灵活控制

**常见量化级别及其效果**：

| 量化级别 | 7B 模型大小 | 质量损失 | 推荐场景 |
|---------|-----------|---------|---------|
| Q2_K | ~2.8 GB | 较大 | 极致压缩，质量要求低 |
| Q3_K_M | ~3.3 GB | 中等 | 存储极度受限 |
| Q4_K_M | ~4.1 GB | 很小 | **最常用**，性价比最优 |
| Q5_K_M | ~4.8 GB | 极小 | 追求高质量 |
| Q6_K | ~5.3 GB | 几乎无 | 接近 FP16 质量 |
| Q8_0 | ~7.0 GB | 几乎无 | 高精度部署 |

Q4_K_M 是社区公认的"甜点"级别——在质量和体积之间取得了最佳平衡。对于大多数微调模型，从 FP16 到 Q4_K_M 的质量损失通常 <1-2%。

**将 HuggingFace 模型转换为 GGUF**：

转换过程需要两个步骤：
1. 将 HuggingFace 格式转为 FP16 GGUF（使用 `convert_hf_to_gguf.py` 脚本）
2. 对 GGUF 文件进行量化（使用 `llama-quantize` 工具）

```bash
# 步骤 1: 转换为 FP16 GGUF
python convert_hf_to_gguf.py ./merged_model --outtype f16 --outfile model-f16.gguf

# 步骤 2: 量化为 Q4_K_M
llama-quantize model-f16.gguf model-Q4_K_M.gguf Q4_K_M
```

转换脚本来自 llama.cpp 仓库（https://github.com/ggerganov/llama.cpp）。

### 2.3 GPTQ/AWQ：GPU 推理专用量化

GGUF 的优势在于灵活（CPU + GPU 混合）、格式统一（一个文件包含一切）。但在纯 GPU 推理场景中，GPTQ 和 AWQ 通常能提供更高的推理吞吐量。

**GPTQ（Post-Training Quantization）**：
- 基于逐层量化的方法，在校准数据集上最小化量化误差
- 通常使用 INT4 或 INT8 精度
- 需要校准数据集（几百到几千个样本）来完成量化过程
- 推理时需要特定的 kernel（如 AutoGPTQ 库）

**AWQ（Activation-Aware Weight Quantization）**：
- 在 GPTQ 的基础上，进一步考虑了激活值的分布
- 核心洞察：不是所有权重对模型输出同等重要。那些与"显著激活值"（salient activations）相乘的权重对精度影响更大
- 通过保护"重要权重通道"的精度，在同等量化级别下质量优于 GPTQ
- 需要校准数据集

**GGUF vs GPTQ vs AWQ 对比**：

| 特性 | GGUF | GPTQ | AWQ |
|------|------|------|-----|
| 推理框架 | llama.cpp, Ollama | AutoGPTQ, vLLM | AutoAWQ, vLLM |
| CPU 推理 | 优秀 | 不支持 | 不支持 |
| GPU 推理 | 良好 | 优秀 | 优秀 |
| 量化速度 | 快（无需校准） | 较慢（需要校准） | 较慢（需要校准） |
| 文件体积 | 较小 | 较小 | 较小 |
| 质量（同级别） | 良好 | 好 | 更好 |
| 推荐场景 | 本地部署、Ollama | vLLM 高吞吐 | vLLM 高质量 |

## 3. 本地部署方案

### 3.1 Ollama 部署

Ollama 是目前最流行的本地 LLM 部署工具。它将模型下载、量化、推理、API 服务封装在一条命令中，极大降低了部署门槛。

**安装 Ollama**：

```bash
# Linux/Mac
curl -fsSL https://ollama.com/install.sh | sh

# Windows: 从 https://ollama.com/download 下载安装包
```

**导入自定义 GGUF 模型**：

1. 创建一个 Modelfile：

```
FROM ./model-Q4_K_M.gguf

TEMPLATE """{{ if .System }}<|im_start|>system
{{ .System }}<|im_end|>
{{ end }}{{ if .Prompt }}<|im_start|>user
{{ .Prompt }}<|im_end|>
{{ end }}<|im_start|>assistant
"""

PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER stop "<|im_end|>"
```

2. 创建 Ollama 模型：

```bash
ollama create my-finetuned-model -f Modelfile
```

3. 运行模型：

```bash
# 命令行交互
ollama run my-finetuned-model

# 作为 API 服务（默认端口 11434）
ollama serve
```

4. 通过 API 调用：

```python
import requests
import json

response = requests.post(
    "http://localhost:11434/api/chat",
    json={
        "model": "my-finetuned-model",
        "messages": [
            {"role": "user", "content": "解释一下什么是机器学习"}
        ],
        "stream": False,
    },
)

print(response.json()["message"]["content"])
```

### 3.2 vLLM 高性能推理

vLLM 是目前最高性能的开源 LLM 推理框架，广泛用于生产环境。它的核心优势包括：

- **PagedAttention**：将 KV cache 像操作系统的虚拟内存一样分页管理，几乎零浪费的显存利用率
- **连续批处理（Continuous Batching）**：动态地将新请求加入正在处理的 batch，显著提升吞吐量
- **高吞吐量**：相比 HuggingFace Transformers 原生推理，vLLM 的吞吐量通常提升 10-30 倍
- **支持 LoRA adapter 热加载**：可以在一个基座模型上同时服务多个 LoRA adapter

**安装 vLLM**：

```bash
pip install vllm
```

**部署合并后的模型**：

```python
"""
使用 vLLM 部署微调后的模型。
"""
from vllm import LLM, SamplingParams


def deploy_with_vllm(
    model_path: str,
    port: int = 8000,
) -> None:
    """
    使用 vLLM 部署模型为 OpenAI 兼容的 API 服务。

    vLLM 的 API 服务与 OpenAI API 格式兼容，
    客户端可以直接使用 OpenAI SDK 调用。
    """
    # vLLM 提供了 OpenAI-compatible 的 API server
    # 在命令行中运行:
    # python -m vllm.entrypoints.openai.api_server \
    #     --model ./merged_model \
    #     --served-model-name my-model \
    #     --port 8000 \
    #     --max-model-len 4096 \
    #     --gpu-memory-utilization 0.9

    print(f"请用以下命令启动 vLLM 服务:")
    print(f"python -m vllm.entrypoints.openai.api_server \\")
    print(f"    --model {model_path} \\")
    print(f"    --served-model-name my-model \\")
    print(f"    --port {port} \\")
    print(f"    --max-model-len 4096 \\")
    print(f"    --gpu-memory-utilization 0.9")


def test_vllm_local(model_path: str, test_prompts: list[str]):
    """
    不启动 API server，直接在本地用 vLLM 引擎推理。
    """
    print(f"加载 vLLM 模型: {model_path}")
    llm = LLM(
        model=model_path,
        trust_remote_code=True,
        max_model_len=4096,
        gpu_memory_utilization=0.9,
    )

    sampling_params = SamplingParams(
        temperature=0.7,
        top_p=0.9,
        max_tokens=256,
    )

    outputs = llm.generate(test_prompts, sampling_params)

    for prompt, output in zip(test_prompts, outputs):
        print(f"\nPrompt: {prompt}")
        print(f"Response: {output.outputs[0].text}")
        print("-" * 50)


if __name__ == "__main__":
    test_prompts = [
        "解释一下什么是机器学习",
        "Python 中如何处理异常？",
    ]
    test_vllm_local("./merged_model", test_prompts)
```

**vLLM 的关键参数**：

- `--gpu-memory-utilization 0.9`：允许 vLLM 使用 90% 的 GPU 显存。设置过低会浪费显存，设置过高可能导致 OOM
- `--max-model-len 4096`：最大上下文长度。vLLM 会为 KV cache 预留显存。设置过大会浪费显存，设置过小会截断长输入
- `--tensor-parallel-size N`：使用 N 张 GPU 做张量并行。对于 70B 模型，通常设置为 2 或 4
- `--enable-lora`：启用 LoRA adapter 热加载。可以同时服务多个 adapter

## 4. FastAPI 封装微调模型为 API

### 4.1 为什么自建 API

虽然 Ollama 和 vLLM 已经提供了 API，但有时你需要更多的控制：
- 在推理前后添加业务逻辑（如敏感词过滤、输入预处理、输出后处理）
- 实现自定义的认证和限流
- 集成到现有的微服务体系中
- 支持非标准的请求格式

### 4.2 FastAPI 服务实现

```python
"""
用 FastAPI 将微调模型封装为 API 服务。

提供 OpenAI 兼容的 /v1/chat/completions 接口，
以及自定义的 /health 和 /models 接口。

依赖: pip install fastapi uvicorn pydantic
"""
import time
from typing import Optional
from contextlib import asynccontextmanager

import torch
from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel, Field
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel


# ============================================================
# 数据模型
# ============================================================

class ChatMessage(BaseModel):
    role: str = Field(..., description="角色: system, user, assistant")
    content: str = Field(..., description="消息内容")


class ChatCompletionRequest(BaseModel):
    model: str = Field(..., description="模型名称")
    messages: list[ChatMessage] = Field(..., description="对话消息列表")
    temperature: float = Field(0.7, ge=0.0, le=2.0, description="采样温度")
    max_tokens: int = Field(512, ge=1, le=4096, description="最大生成 token 数")
    top_p: float = Field(0.9, ge=0.0, le=1.0, description="nucleus sampling 参数")
    stream: bool = Field(False, description="是否流式输出")


class ChatCompletionChoice(BaseModel):
    index: int = 0
    message: ChatMessage
    finish_reason: str = "stop"


class UsageInfo(BaseModel):
    prompt_tokens: int = 0
    completion_tokens: int = 0
    total_tokens: int = 0


class ChatCompletionResponse(BaseModel):
    id: str = "chatcmpl-001"
    object: str = "chat.completion"
    created: int = Field(default_factory=lambda: int(time.time()))
    model: str
    choices: list[ChatCompletionChoice]
    usage: UsageInfo


# ============================================================
# 模型管理
# ============================================================

class ModelManager:
    """
    管理模型的加载和推理。

    使用懒加载和单例模式，只在第一次请求时加载模型。
    """

    def __init__(self):
        self.model = None
        self.tokenizer = None
        self._device = "cuda" if torch.cuda.is_available() else "cpu"

    def load(self, model_path: str, adapter_path: Optional[str] = None):
        """加载模型。"""
        print(f"加载 tokenizer 从: {model_path}")
        self.tokenizer = AutoTokenizer.from_pretrained(
            model_path, trust_remote_code=True
        )
        if self.tokenizer.pad_token is None:
            self.tokenizer.pad_token = self.tokenizer.eos_token

        print(f"加载模型从: {model_path}")
        self.model = AutoModelForCausalLM.from_pretrained(
            model_path,
            torch_dtype=torch.bfloat16 if self._device == "cuda" else torch.float32,
            device_map="auto" if self._device == "cuda" else None,
            trust_remote_code=True,
        )

        # 如果有 LoRA adapter，挂载它
        if adapter_path:
            print(f"挂载 LoRA adapter 从: {adapter_path}")
            self.model = PeftModel.from_pretrained(self.model, adapter_path)

        self.model.eval()
        print(f"模型就绪。设备: {self._device}")
        print(f"参数量: {sum(p.numel() for p in self.model.parameters()):,}")

    def generate(self, request: ChatCompletionRequest) -> ChatCompletionResponse:
        """根据请求生成回复。"""
        if self.model is None or self.tokenizer is None:
            raise RuntimeError("模型尚未加载")

        # 构建 messages（与训练时一致的 Chat Template）
        messages = [
            {"role": msg.role, "content": msg.content}
            for msg in request.messages
        ]

        # 应用 Chat Template
        input_text = self.tokenizer.apply_chat_template(
            messages,
            tokenize=False,
            add_generation_prompt=True,
        )

        # Tokenize
        inputs = self.tokenizer(input_text, return_tensors="pt")
        if self._device == "cuda":
            inputs = {k: v.cuda() for k, v in inputs.items()}

        prompt_tokens = inputs["input_ids"].shape[1]

        # 生成
        with torch.no_grad():
            outputs = self.model.generate(
                **inputs,
                max_new_tokens=request.max_tokens,
                temperature=request.temperature,
                top_p=request.top_p,
                do_sample=True,
                pad_token_id=self.tokenizer.pad_token_id,
                eos_token_id=self.tokenizer.eos_token_id,
            )

        # 解码（只取新生成的 token）
        generated_ids = outputs[0][prompt_tokens:]
        response_text = self.tokenizer.decode(
            generated_ids, skip_special_tokens=True
        )

        completion_tokens = len(generated_ids)

        return ChatCompletionResponse(
            model=request.model,
            choices=[
                ChatCompletionChoice(
                    message=ChatMessage(role="assistant", content=response_text),
                    finish_reason="stop",
                )
            ],
            usage=UsageInfo(
                prompt_tokens=prompt_tokens,
                completion_tokens=completion_tokens,
                total_tokens=prompt_tokens + completion_tokens,
            ),
        )


# ============================================================
# FastAPI 应用
# ============================================================

# 全局模型管理器
model_manager = ModelManager()

# 应用启动和关闭事件
@asynccontextmanager
async def lifespan(app: FastAPI):
    """应用生命周期管理。"""
    # 启动时加载模型
    import os
    model_path = os.getenv("MODEL_PATH", "./merged_model")
    adapter_path = os.getenv("ADAPTER_PATH", None)
    model_manager.load(model_path, adapter_path)
    yield
    # 关闭时可以做清理工作


app = FastAPI(
    title="Fine-tuned LLM API",
    description="微调模型 API 服务",
    version="1.0.0",
    lifespan=lifespan,
)

# 允许跨域请求
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)


@app.get("/health")
async def health_check():
    """健康检查接口。"""
    return {
        "status": "healthy",
        "model_loaded": model_manager.model is not None,
    }


@app.get("/v1/models")
async def list_models():
    """列出可用模型（OpenAI 兼容）。"""
    return {
        "object": "list",
        "data": [
            {
                "id": "finetuned-model",
                "object": "model",
                "created": int(time.time()),
                "owned_by": "user",
            }
        ],
    }


@app.post("/v1/chat/completions", response_model=ChatCompletionResponse)
async def chat_completions(request: ChatCompletionRequest):
    """
    Chat Completions API（OpenAI 兼容）。

    请求格式:
    {
        "model": "finetuned-model",
        "messages": [
            {"role": "system", "content": "你是一个有帮助的AI助手。"},
            {"role": "user", "content": "解释一下机器学习"}
        ],
        "temperature": 0.7,
        "max_tokens": 512
    }
    """
    try:
        response = model_manager.generate(request)
        return response
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

## 5. 成本分析：自部署 vs API 调用

### 5.1 自部署的成本

**硬件成本**：
- 入门（7B 模型）：RTX 4090 24GB（约 15000 元）
- 标准（13B-30B 模型）：RTX A6000 48GB（约 40000 元）
- 高端（70B+ 模型）：A100 80GB 或 H100（约 80000-200000 元）

**运维成本**：
- 电费：高端 GPU 约 300W/h，24 小时运行一个月约 150-300 元电费
- 维护人力：配置环境、解决故障、更新模型（时间成本）
- 带宽：如果需要公网访问，需要服务器带宽费用

**隐性成本**：
- 模型优化时间（量化、合并、配置 vLLM）
- 故障排查时间（OOM、推理速度优化、并发性能调优）

### 5.2 API 调用的成本

以 GPT-4 API 为例：
- GPT-4o: 输入 $2.50/1M tokens, 输出 $10.00/1M tokens
- GPT-4o-mini: 输入 $0.15/1M tokens, 输出 $0.60/1M tokens
- 国内 API（如 Qwen API、DeepSeek API）：通常更便宜

**每月 API 调用的典型成本**（假设平均每天 10000 次调用，每次输入 500 token + 输出 300 token）：
- GPT-4o-mini: 约 $150/月（约 1050 元/月）
- Qwen-Max API: 约 100-500 元/月

### 5.3 如何选择

```
你的月 API 调用量是多少？
├── < 1000 次/天  → 用 API（省心、零运维）
├── 1000-10000 次/天 → 考虑自部署（如果对延迟敏感或需要定制）
├── > 10000 次/天 → 自部署（长期成本更低）

你的延迟要求是什么？
├── 可以接受 1-3 秒 → API 足够
├── 需要 < 500ms → 自部署 + vLLM 高吞吐
└── 需要 < 100ms → 自部署 + 量化 + vLLM + 多卡

你对模型控制程度的要求？
├── 只需要标准能力 → API
├── 需要自定义模型行为 → 自部署微调模型
└── 需要频繁更新模型 → 自部署（API 更新周期不可控）

数据安全要求？
├── 非敏感数据 → API
├── 敏感数据（医疗、金融、内部数据）→ 自部署（数据不出境/不出公司）
```

一个混合策略在不少场景下很实用：日常开发测试用 API（灵活、零运维），正式生产环境用自部署（可控、低成本、低延迟）。

## 常见错误

1. **合并后忘记保存 tokenizer**：`merge_and_unload()` 只处理模型权重，tokenizer 需要单独保存。如果只保存了模型权重没保存 tokenizer，加载时会报错或使用错误的 tokenizer。

2. **GGUF 转换时 Chat Template 丢失**：转换为 GGUF 时，Chat Template 信息可能不会自动保留。需要在 Modelfile 中手动指定 TEMPLATE，确保与训练时一致。

3. **vLLM 部署时 GPU 显存分配不合理**：`gpu-memory-utilization` 设置太低浪费显存和吞吐量，设置太高可能导致 OOM。建议从 0.85 开始，根据实际监控调整。

4. **Ollama 导入后模型行为异常**：通常是因为 Chat Template 不对。训练时用的 Qwen 模板，但 Ollama 的 Modelfile 中没有正确配置 TEMPLATE。务必检查并测试。

5. **FastAPI 服务没有做并发控制**：默认 uvicorn 是单线程的。如果多个请求同时到达，会排队等待。生产环境应使用多个 worker 或引入队列机制。

6. **量化级别选择不当**：为了省空间选了 Q2_K，结果模型质量严重下降。量化是"质量换体积"的权衡，Q4_K_M 是大多数场景的最佳起点。

7. **忽略推理时的内存泄漏**：长时间运行的推理服务可能出现显存缓慢增长（内存泄漏）。确保每次推理后释放中间 tensor，定期重启服务。

8. **模型加载了完整的 FP32 权重**：7B 模型 FP32 占用 28GB，很多 GPU（24GB）放不下。部署时务必使用 BF16/FP16 或量化格式。

9. **API 服务没有设置超时**：如果模型生成时间过长（因为输入太长或 max_tokens 太大），客户端会一直等待。应该在服务端设置生成超时。

10. **自部署后忽略了安全更新**：开源模型和推理框架（llama.cpp、vLLM、transformers）会持续有安全更新和性能优化。定期更新依赖是必要的。
